# pit-017: forEach 中使用 async/await 无效

## 环境
- 所有 JavaScript 环境

## 症状
在 `Array.forEach` 的回调中使用了 `async/await`，但代码没有按预期等待异步操作完成。日志显示所有异步操作被"同时触发"，然后外层代码直接继续执行。

## 根因
`forEach` 的回调是同步执行的。当回调是 `async` 函数时，`forEach` 会依次调用每个 `async` 函数（返回 Promise），但不会等待这些 Promise 完成。`await` 只在 `async` 函数内部暂停，不影响 `forEach` 的迭代循环。

## 修复

```diff
// ❌ 问题：forEach + async，不会等待
async function saveAll(items) {
-  items.forEach(async (item) => {
-    await saveToDatabase(item);
-  });
-  console.log('全部保存完成'); // 立即执行，数据可能还没保存完
}

// ✅ 修复方案一：for...of 循环（串行）
async function saveAll(items) {
+  for (const item of items) {
+    await saveToDatabase(item);
+  }
+  console.log('全部保存完成');
}

// ✅ 修复方案二：Promise.all 并行执行
async function saveAll(items) {
+  await Promise.all(
+    items.map(item => saveToDatabase(item))
+  );
+  console.log('全部保存完成');
}

// ✅ 修复方案三：Promise.allSettled（部分失败不中断）
async function saveAll(items) {
+  const results = await Promise.allSettled(
+    items.map(item => saveToDatabase(item))
+  );
+  const failed = results.filter(r => r.status === 'rejected');
+  if (failed.length > 0) {
+    console.error(`${failed.length} 项保存失败`);
+  }
+  console.log('全部保存完成');
}
```

## 原理
`Array.prototype.forEach` 的设计是同步迭代。它会依次对每个元素调用回调函数，但不会等待回调返回的 Promise。具体过程：

```javascript
items.forEach(callback);
// 等价于:
for (let i = 0; i < items.length; i++) {
  callback(items[i], i, items); // 不等待返回值
}
```

当 callback 是 `async` 函数时，它返回一个 Promise，但 forEach 忽略这个返回值。每个 async callback 中的 `await` 只暂停那个 callback 的执行，不影响 forEach 的迭代。

## 架构视角 (Architecture Perspective)

### 架构层面根因
forEach + async 问题的架构根因是**同步/异步迭代语义未在架构层面区分**：
- **迭代模式与并发策略耦合在业务代码中**：for...of（串行）、Promise.all（并行）、Promise.allSettled（容错并行）、p-limit（受控并发）是不同的并发策略，但开发者在业务代码中直接选择，没有通过架构层抽象。正确的做法是提供一个"批量执行引擎"，声明"这批操作需要什么并发策略"。
- **forEach 被过度使用**：`forEach` 在 JavaScript 中是最通用的迭代方法，但它也是最弱的——不支持 async、不支持 break/continue、不支持返回值。架构应该限制 `forEach` 的使用场景，强制在需要异步操作时使用更合适的迭代模式。
- **缺少异步操作的"事务"概念**：批量保存操作"全成功或全回滚"的语义没有在代码中表达。多个独立的 `async` 调用之间没有"事务边界"——部分成功部分失败时，成功的数据已经写入但无法回滚。架构应提供"异步事务"抽象。

**架构防护**：
- **批量操作引擎 (Batch Operation Engine)**：
```typescript
// @app/utils/batch.ts
export enum ConcurrencyStrategy {
  Serial,        // 串行: for...of
  Parallel,      // 并行: Promise.all
  FailSafe,      // 容错: Promise.allSettled
  RateLimited,   // 限流: p-limit(5)
}

export async function batchProcess<T, R>(
  items: T[],
  fn: (item: T) => Promise<R>,
  strategy: ConcurrencyStrategy = ConcurrencyStrategy.Parallel
): Promise<R[]> {
  switch (strategy) {
    case ConcurrencyStrategy.Serial:
      return serialProcess(items, fn);
    case ConcurrencyStrategy.Parallel:
      return Promise.all(items.map(fn));
    case ConcurrencyStrategy.FailSafe:
      return failSafeProcess(items, fn);
    case ConcurrencyStrategy.RateLimited:
      return rateLimitedProcess(items, fn, 5);
  }
}

// 业务代码只声明策略
const results = await batchProcess(users, saveUser, ConcurrencyStrategy.FailSafe);
```

### 跨层影响链

```
上游影响 (消费者):
  - 批量操作的调用方期望"全部完成"但实际只获得"全部启动"
  - 后续代码依赖批量操作的结果，但数据可能尚未写入

下游影响 (依赖):
  - 数据库连接池被瞬间打满（无控制的并发写入）
  - 后端 API 被 N 个并行请求冲垮（无 rate limiting）
  - 前端状态管理中被部分更新的数据导致 UI 不一致

部署影响:
  - 本地开发时数据量小（3 条）→ 表现正常
  - 生产环境数据量大（1000 条）→ 爆发式失败
  - SSR/SSG 构建时的数据获取中 forEach 问题导致构建产物数据不完整

监控影响:
  - 批量操作的完成时间分布（P95/P99）
  - 部分失败率（Promise.allSettled 的必要性数据）
  - API 限流错误率（频繁触发 429 Too Many Requests）
```

### 相关架构模式
- **数据流模式 — 乐观更新** (`patterns.md` 第四节) — 批量保存操作中的乐观更新需要等待所有请求完成后才能确认成功或回滚，forEach+async 无法满足此需求。
- **CQRS 前端化** (`patterns.md` 第三节) — 批量操作可以作为单一 Command 提交给后端，由后端处理并发，前端只关心最终结果。这从根本上消除了前端的并发管理需求。
- **Event Bus（反模式）** (`patterns.md` 第六节) — forEach+async 与 Event Bus 有相同的问题：事件（操作）被发出后无法追踪完成状态。集中处理（for...of/Promise.all）是更好的选择。

## 预防

1. **永远不要在 `forEach` 回调中使用 `async/await`**。
2. **ESLint**：`@typescript-eslint/no-misused-promises` 可以检测此问题。
3. **选择正确的迭代模式**：
   - 需要串行 → `for...of` + `await`
   - 需要并行 → `Promise.all(map(...))`
   - 需要部分容忍 → `Promise.allSettled(map(...))`
4. **Code Review 关键词**：搜索 `forEach.*async` 正则模式。

## 回退

```bash
# 如果 Promise.all 并发导致后端压力过大
# 方案A：限制并发数
# 使用 p-limit 库
import pLimit from 'p-limit';
const limit = pLimit(5); // 最多 5 个并发
await Promise.all(items.map(item => limit(() => saveItem(item))));

# 方案B：如果 for...of 串行太慢
# 改为分块并行:
for (let i = 0; i < items.length; i += 5) {
  await Promise.all(items.slice(i, i + 5).map(saveItem));
}

```
