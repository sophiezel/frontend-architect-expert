# pit-014: FlatList 缺少 keyExtractor 导致渲染错乱

## 环境
- React Native `FlatList` 组件
- 所有 RN 版本

## 症状
列表滚动时项目闪烁、图片错位、点击事件绑定错误。删除/插入项目后列表项目重复或消失。数据与 UI 不对应。

## 根因
`FlatList` 是虚拟列表，通过复用少量"窗口"内的 item 组件实现高性能。它使用 key 来追踪每个 item。如果不提供 `keyExtractor`，FlatList 使用 `item.key` 或 `item.id` 降级到数组索引。当数据更新（增删改）时索引变化，导致错误的组件被复用——React Native 不会像 React DOM 那样发出 key warning。

## 修复

```diff
// ❌ 问题：缺少 keyExtractor
<FlatList
  data={users}
  renderItem={({ item }) => <UserCard user={item} />}
- // 没有 keyExtractor → 使用索引 → 数据更新时错乱
/>

// ✅ 修复：提供稳定的 keyExtractor
<FlatList
  data={users}
  renderItem={({ item }) => <UserCard user={item} />}
+ keyExtractor={(item) => String(item.id)}
/>

// ✅ 更好的做法：确保数据源有唯一 id
<FlatList
  data={users}
  renderItem={({ item }) => <UserCard user={item} />}
+ keyExtractor={(item, index) => {
+   return item.id ? String(item.id) : `${index}-${item.name}`;
+ }}
/>
```

```diff
// ❌ 问题：使用 index 作为 keyExtractor（与默认行为相同）
<FlatList
  data={items}
- keyExtractor={(item, index) => String(index)}
/>

// ✅ 修复
<FlatList
  data={items}
+ keyExtractor={(item) => item._id || item.uuid || String(item.id)}
/>
```

## 原理
FlatList 是 React Native 的虚拟列表实现，类似于 Web 中的 react-window 或 react-virtualized。它只渲染可视区域内的 item（加上少量缓冲区），滚动时复用离开视口的 item 组件来渲染新进入视口的 item。

Key 的作用：FlatList 通过 key 来决定"哪个 item 组件代表哪个数据项"。如果 key 基于索引，当数据顺序变化时，同一个位置的 key 相同但数据不同，组件被错误复用。

## 架构视角 (Architecture Perspective)

### 架构层面根因
FlatList keyExtractor 缺失的架构根因与 pit-003（React 列表 key）同源但增加了**跨平台差异**维度：
- **React Native 的静默失败模式**：React DOM 对缺少 key 会发出 Warning，但 React Native 的 LogBox 对 FlatList 缺少 keyExtractor 不总是警告。架构缺少"跨平台诊断能力一致性"的保障——同样的错误在不同平台应有同等级别的可见性。
- **虚拟列表的身份管理契约被忽略**：FlatList 作为虚拟列表，其核心契约是"item 的身份标识必须在 data 变化时保持稳定"。这个契约应该在组件接口类型层面强制——通过 TypeScript 泛型约束 `data` 中每项必须有 `id: string`。
- **数据层到视图层的身份传输链路在 Native 端断裂**：在 React Native 中，数据从原生模块回调 → JS 线程 → FlatList 的路径中，身份信息如果在 Bridge 序列化过程中变成不稳定的引用（每次回调创建新对象），即使用了 `keyExtractor`，如果提取的值不稳定也无济于事。架构需要在数据进入 JS 线程后立即进行"身份归一化"。

### 系统级预防策略

**静态防护**：
```tsx
// TypeScript: 强制 data 中每项有 id
type Identifiable = { id: string | number };

function TypedFlatList<T extends Identifiable>(props: FlatListProps<T>) {
  return <FlatList {...props} keyExtractor={(item) => String(item.id)} />;
}

// 禁止直接使用 FlatList，强制使用 TypedFlatList
// ESLint: no-restricted-imports
rules: {
  'no-restricted-imports': ['error', {
    paths: [{
      name: 'react-native',
      importNames: ['FlatList'],
      message: '请使用 TypedFlatList，它强制要求 id 字段和 keyExtractor'
    }]
  }]
}
```

**CI 防护**：
```bash
# CI lint: 检测 FlatList 缺少 keyExtractor
grep -rl "<FlatList" src/ | xargs grep -L "keyExtractor" &&
  echo "ERROR: FlatList 必须提供 keyExtractor" && exit 1

# React Native 测试: 列表增删后验证 item 身份
test('FlatList 删除项后剩余项身份不变', () => {
  const items = [{ id: 1, text: 'A' }, { id: 2, text: 'B' }, { id: 3, text: 'C' }];
  const { getByTestId, rerender } = render(
    <FlatList data={items} renderItem={...} keyExtractor={i => i.id} />
  );
  // 删除中间项
  const newItems = items.filter(i => i.id !== 2);
  rerender(<FlatList data={newItems} renderItem={...} keyExtractor={i => i.id} />);
  // 验证剩余项的状态未串位（通过 testID 定位）
  expect(getByTestId('item-3')).toBeTruthy();
});
```

**架构防护**：
- **封装平台原生列表组件**：与 pit-003 类似，封装 `<AppFlatList>` 强制 keyExtractor。但需额外处理：
  - React Native 的 `extraData` 属性（列表项依赖外部状态时的更新触发）
  - `getItemLayout` 优化（固定高度列表的性能提升）
  - `maintainVisibleContentPosition` (iOS) 和 `onScrollToIndexFailed` 错误处理
- **数据归一化中间层**：所有从网络/原生模块进入的数据，在进入 FlatList 之前必须经过归一化（`normalize(data) → { ids, entities }`），确保每项有稳定的 `id` 字段。

**流程防护**：
- Code Review 检查项："FlatList 是否有 keyExtractor？keyExtractor 是否使用数据稳定 ID（非 index）？extraData 是否需要设置？"

### 跨层影响链

```
上游影响 (消费者):
  - 父组件通过 props 传入的 data 数组中对象引用变化但 ID 相同 → 正常
  - 但如果 keyExtractor 使用引用（如 item => item）→ 每次 data 更新全量重建

下游影响 (依赖):
  - FlatList 内嵌的可交互子组件（Switch、TextInput、Video）状态串位
  - 列表项内 Image 组件加载错误图片（缓存的图片 URL 未清空）
  - 列表项的 PanResponder/手势事件绑定错误

部署影响:
  - React Native OTA 更新（CodePush）中，列表逻辑修复需要用户重启 App
  - App Store 审核中列表闪烁会导致功能性拒绝

监控影响:
  - Flipper 插件检测 item 复用异常
  - 自定义事件: FlatList 视图复用率（visible items / rendered views）
  - 崩溃率: 列表项数据访问错误的 JS Exception
```

### 相关架构模式
- **pit-003: 列表渲染缺少 key** — 同源问题，架构解决方案共享。React Native 的不同在于静默失败。
- **虚拟列表模式** (`patterns.md` 第四节) — FlatList 是虚拟列表在 React Native 的实现，key 的稳定性直接决定视图复用的正确性。
- **状态管理模式** (`patterns.md` 第一节) — Zustand/Jotai 在 React Native 中同样适用，选择器精确订阅可以减少 `extraData` 的使用。

## 预防

1. **始终提供 `keyExtractor`** — 作为 Code Review 必检项。
2. **数据模型中包含稳定唯一 ID**：从后端返回的每个列表项都应有 `id` 字段。
3. **自定义 ESLint 规则**：对 `<FlatList` 检测是否有 `keyExtractor` 属性。
4. **`extraData` 属性**：如果列表项依赖外部状态（非 data 数组内的值），需传给 `extraData` 确保重渲染。
