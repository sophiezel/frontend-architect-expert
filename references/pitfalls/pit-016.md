# pit-016: Promise 未处理拒绝被静默吞掉

## 环境
- Node.js ≤ 14（`unhandledRejection` 不终止进程）
- 浏览器中某些情况下 Promise rejection 被静默吞掉

## 症状
`fetch` 请求失败后没有任何错误提示，页面白屏或无响应，控制台无错误输出（Node.js 14 及以下），导致故障排查极其困难。

## 根因
Promise 链中未添加 `.catch()` 或未使用 `try/catch`（async/await），rejection 没有被处理。Node.js 14 及之前版本对此只打印一个 `UnhandledPromiseRejectionWarning`，进程继续运行但状态未知。Node.js 15+ 改为终止进程。

## 诊断方法

```bash
# 1. Node.js 全局监听
node -e "
process.on('unhandledRejection', (reason, promise) => {
  console.error('未处理的 Promise 拒绝：', reason);
});
// 模拟未处理的 rejection
new Promise((_, reject) => reject(new Error('测试错误')));
"

# 2. 浏览器全局监听
# 在控制台执行：
window.addEventListener('unhandledrejection', (event) => {
  console.error('未处理的 Promise 拒绝：', event.reason);
});

# 3. 查找未处理的 Promise
grep -rn "\.then(" src/ --include="*.ts" --include="*.js" | grep -v "\.catch"
# 查找 .then() 但没有 .catch() 的 Promise 链

grep -rn "async function" src/ --include="*.ts" --include="*.js" | while read line; do
  file=$(echo "$line" | cut -d: -f1)
  # 检查 async 函数内部是否有 try/catch
  grep -A 50 "$line" "$file" | grep -q "catch" || echo "可能缺少错误处理: $line"
done

# 4. Node.js 版本检查
node -e "console.log(process.version)"
# 如果是 v14 或更早 → 需特别注意未处理 rejection
```

## 修复

```diff
// ❌ 问题：fetch 未处理错误
async function loadUserData(userId) {
-  const response = await fetch(`/api/users/${userId}`);
-  const data = await response.json();
-  return data;
}

// ✅ 修复：完整的错误处理链
async function loadUserData(userId) {
+  try {
+    const response = await fetch(`/api/users/${userId}`);
+    if (!response.ok) {
+      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
+    }
+    return await response.json();
+  } catch (error) {
+    console.error('加载用户数据失败：', error);
+    throw error;
+  }
}
```

```diff
// ❌ 问题：Promise 链中遗漏 catch
function processItems(items) {
-  items.forEach(item => {
-    fetch('/api/process', { body: JSON.stringify(item) })
-      .then(res => res.json())
-      .then(result => updateUI(result));
-      // 没有 .catch() → rejection 被吞掉
-  });
}

// ✅ 修复：Promise.allSettled
async function processItems(items) {
+  const results = await Promise.allSettled(
+    items.map(item =>
+      fetch('/api/process', { body: JSON.stringify(item) })
+        .then(res => res.json())
+    )
+  );
+  results.forEach((result, i) => {
+    if (result.status === 'rejected') {
+      console.error(`处理 item ${i} 失败：`, result.reason);
+    } else {
+      updateUI(result.value);
+    }
+  });
}
```

## 原理
Promise 有三种状态：pending、fulfilled、rejected。当 Promise 进入 rejected 状态但没有 `.catch()` 或 `await` 处理时，rejection 成为"未处理"。浏览器触发 `unhandledrejection` 事件，Node.js 发出 `UnhandledPromiseRejectionWarning`。在 Node.js 14 及之前，未处理的 rejection 不会终止进程，导致问题被掩盖。从 Node.js 15 开始，行为改为终止进程。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Promise rejection 被吞掉的架构根因是**错误处理策略未全局化、未分层**：
- **每个模块自定错误处理策略，无全局兜底**：组件的 `useEffect` 中调用 `fetch`，catch 了错误但没有上报；另一个模块直接裸用 `fetch` 没有 catch。应用没有统一的"错误处理中件间"——所有异步操作的错误最终都应该经过一个统一的错误处理管道（Error Boundary + 全局 unhandledrejection + Sentry）。
- **错误被当作实现细节而非架构关注点**：错误处理被认为是"开发时注意写 try/catch 就行"的实现细节，实际上错误处理是架构的质量属性（Quality Attribute）——可观测性（Observability）、韧性（Resilience）、可调试性（Debuggability）都依赖统一的错误策略。
- **异步操作的错误传播链断裂**：`Promise.catch()` 如果不重新 `throw`，错误就被"吞掉"了——后续链不知道出错了。架构缺少"强制错误传播"机制，类似 Rust 的 `?` 操作符或 Go 的 `if err != nil`。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 强制 Promise 错误处理
rules: {
  'promise/catch-or-return': 'error',
  'promise/always-return': 'error',
  '@typescript-eslint/no-floating-promises': 'error',
  '@typescript-eslint/no-misused-promises': 'error',
  // 禁止裸 fetch 调用（必须通过封装的数据层）
  'no-restricted-imports': ['error', {
    paths: [{
      name: 'fetch',
      message: '请使用 @app/data/fetchClient (内置错误处理/重试/日志)'
    }]
  }]
}
```

```ts
// TypeScript: Result 类型强制错误处理
type Result<T, E = Error> = { ok: true; value: T } | { ok: false; error: E };

// async 函数强制返回 Result
async function loadUser(id: string): Promise<Result<User>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) return { ok: false, error: new Error(`HTTP ${res.status}`) };
    return { ok: true, value: await res.json() };
  } catch (e) {
    return { ok: false, error: e as Error };
  }
}
// 调用方必须处理 ok 分支
const result = await loadUser('1');
if (!result.ok) { /* 处理错误 */ } // TypeScript 强制
```

**架构防护**：
- **全局错误处理管道 (Error Pipeline)**：
```typescript
// src/boot/error-pipeline.ts
window.addEventListener('unhandledrejection', (event) => {
  // 1. 记录到监控平台
  captureException(event.reason, { level: 'fatal' });
  // 2. 展示全局错误提示
  showGlobalError('系统异常，请刷新页面');
  // 3. 阻止默认行为（浏览器 console.error）
  event.preventDefault();
});

// React Error Boundary — 组件级错误捕获
<ErrorBoundary 
  fallback={<ErrorPage />}
  onError={(error, info) => captureException(error, { extra: info })}
>
  <App />
</ErrorBoundary>
```
- **数据获取层封装**：禁止组件直接使用 `fetch`/`axios`，必须通过 `@app/data` 层，该层内建重试、超时、错误日志、缓存策略。
- **错误优先级定义**：
  - Level 1 (可恢复): 单个请求失败 → toast 提示 + 重试按钮
  - Level 2 (需刷新): 认证过期 → 跳转登录
  - Level 3 (致命): 页面崩溃 → 刷新或展示维护页

**流程防护**：
- Code Review 检查项："所有 Promise 链是否有 `.catch()` 或外层的 try/catch？未被处理的 rejection 是否有全局监听器兜底？"

### 跨层影响链

```
上游影响 (消费者):
  - 用户看到白屏但不知道出了什么问题
  - 用户操作无反馈（点击保存按钮后无响应也不报错）
  
下游影响 (依赖):
  - Sentry/LogRocket 没有捕获到错误（因为被吞掉了）
  - 业务逻辑基于 undefined/falsy 值继续执行 → 级联错误
  - 状态管理库中的脏数据（部分成功的操作未被回滚）

部署影响:
  - 线上故障排查时间指数级增加（无错误日志）
  - Node.js 14 → 16 升级后，原本被吞掉的错误突然 crash 进程
  - SSR 中未处理的 rejection 导致整个 Node 进程内存泄漏

监控影响:
  - Sentry: unhandledrejection 事件必须接入
  - 监控指标: 全局 unhandledrejection 发生率
  - 告警: unhandledrejection 事件数 > 0/分钟 → 紧急告警
  - RUM: 用户遇到白屏的比例
```

### 相关架构模式
- **Event Bus（反模式）** (`patterns.md` 第六节) — 全局 Event Bus 的"无法追踪"问题与未处理 rejection 的"无法追踪"同构，都是因为缺少集中的事件管道。
- **构建架构 — Node.js 版本** (`decisions.md` 第四节) — Node.js 15+ 的 fail-fast 策略优于旧版本的静默吞掉，应明确设定项目的最低 Node.js 版本。
- **组件库决策** (`decisions.md` 第八节) — 第三方组件库可能在内部发起网络请求（如自动更新检查），需要全局 unhandledrejection 来捕获这些不可控的异步错误。

## 预防

1. **全局监听 `unhandledrejection`** 作为兜底。
2. **ESLint 规则**：`no-floating-promises` (TypeScript)、`promise/catch-or-return`、`promise/always-return`。
3. **使用 `Promise.allSettled`** 代替 `Promise.all` 当部分失败可接受时。
4. **强制 try/catch**：自定义 ESLint 规则，禁止裸 `await`（除了顶层）。
5. **Node.js 升级到 15+**：未处理 rejection 自动终止进程，fail-fast。
