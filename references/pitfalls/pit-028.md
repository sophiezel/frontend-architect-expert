# pit-028: Next.js 缓存策略导致数据过期

## 环境
- Next.js 13+ App Router (Server Components + Client Components)
- Next.js 12+ Pages Router (getStaticProps / getServerSideProps)
- 部署在 Vercel / 自建 Node.js 服务器 + CDN

## 症状
1. **页面显示旧数据**：发布新文章后首页仍显示昨天内容，用户看到过期的产品库存/价格。
2. **ISR revalidate 不生效**：在 Pages Router 中设置了 `revalidate: 60`，但 60 秒后页面没有更新。
3. **CDN 缓存与 Next.js 缓存冲突**：CDN 缓存策略（`Cache-Control` header）与 Next.js 内置缓存的 TTL 不一致，数据过期时间不确定。
4. **`router.refresh()` 未调用**：Client Component 中执行 `fetch` 更新数据后，Server Component 的数据没有刷新。
5. **`fetch` cache 默认 `force-cache`**：Server Component 中的 `fetch` 默认缓存，即使数据源已变更也返回旧结果。

## 根因
Next.js 有**四层缓存**：
1. **React Cache (`fetch` 请求级)** — Server Component 中 `fetch` 默认 `cache: 'force-cache'`
2. **Next.js Data Cache** — `fetch` 响应持久化到 `.next/cache`
3. **Full Route Cache** — 静态/ISR 页面的 HTML/RSC Payload 缓存
4. **Router Cache** — 客户端内存中缓存 RSC Payload（30s 或 5min）

任一层的缓存策略与预期不一致都会导致数据过期。最常见的是：`fetch` 默认 force-cache + ISR `revalidate` 与 CDN TTL 不一致 + `router.refresh()` 未调用。

## 原理
Next.js 的四层缓存模型：
- **Layer 1** — React `fetch` 请求级缓存：App Router 中 `fetch` 默认缓存（在 React Component Tree 渲染期间抑制重复请求）
- **Layer 2** — Next.js Data Cache：`fetch` 响应持久化缓存，`force-cache`（默认）永久缓存直到手动失效，`no-store` 不缓存
- **Layer 3** — Full Route Cache：HTML 和 RSC Payload 在构建时或 ISR 重建时缓存
- **Layer 4** — Router Cache（客户端内存）：浏览器导航时缓存 RSC Payload，30s（动态）/ 5min（静态）

`revalidateTag` / `revalidatePath` 清除 Layer 2 和 Layer 3。`router.refresh()` 清除 Layer 4。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Next.js 缓存问题的架构根因是**多层缓存一致性策略缺失**：

- **缓存层级之间缺少明确的失效协议**：Next.js 有四层缓存 + CDN 缓存 + 浏览器缓存，但项目中没有一个文档说明"当数据变更时，哪些层需要失效、按什么顺序、每一层的 TTL 是多少"。默认值隐含了假设（`force-cache` 永不过期），但这些假设是一厢情愿的。
- **数据"新鲜度"未在架构中做语义分级**：并非所有数据都需要同样的缓存策略——库存数据需要实时（`no-store`）、文章内容可以容忍 60s 延迟（`revalidate: 60`）、静态页面可以构建时缓存。但架构中缺少"数据新鲜度 分级"（Freshness Tier）声明，导致所有数据都被同一默认策略处理。
- **`router.refresh()` 是反直觉的手动失效**：Next.js 要求 Client Component 在 mutation 后调用 `router.refresh()` 来刷新 Router Cache，但这在 API 设计中是"容易被遗忘的步骤"。架构没有通过 wrapper（如自定义 `useAction` hook）自动处理这个步骤。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 禁止 Server Component 中 fetch 不带缓存声明
rules: {
  'no-restricted-syntax': ['warn', {
    selector: 'CallExpression[callee.name="fetch"]:not(:has(Property[key.name="cache"])',
    message: 'fetch 必须显式声明 cache 策略: no-store | force-cache | next.revalidate'
  }],
}
```

**CI 防护**：
```bash
# CI 中验证缓存头
test('Cache headers are correct', async () => {
  const res = await fetch('https://staging.example.com/posts');
  const cacheControl = res.headers.get('cache-control');
  // 动态页面不应有 public cache
  expect(cacheControl).toContain('private');
  expect(cacheControl).toContain('no-cache');
});

# 构建输出检查: 确认标记为 dynamic 的页面不生成静态 HTML
npx next build 2>&1 | grep 'ƒ /posts'  # 确保 /posts 是 dynamic
```

**架构防护**：
- **定义数据新鲜度分级（Freshness Tier）**：
```typescript
// @app/cache/freshness-tiers.ts
export const FreshnessTier = {
  REALTIME:  { cache: 'no-store' as const },
  NEAR_REALTIME: { next: { revalidate: 10 } },
  STANDARD:  { next: { revalidate: 60 } },
  STATIC:    { cache: 'force-cache' as const },
};
// 使用:
fetch(url, FreshnessTier.STANDARD)
```
- **封装 mutation + refresh 为原子操作**：
```tsx
// @app/hooks/useAction.ts
function useAction<T>(action: (...args: any[]) => Promise<T>) {
  const router = useRouter();
  return useCallback(async (...args: any[]) => {
    const result = await action(...args);
    router.refresh(); // 自动刷新 Router Cache
    return result;
  }, [action, router]);
}
```
- **CDN 缓存对齐文档**：在部署文档中明确声明 CDN 的 TTL 策略应与 Next.js 的 revalidate 一致。

**流程防护**：
- Code Review: "此页面中 fetch 的 cache 策略是什么？是否符合数据的新鲜度要求？"
- 架构决策：每个路由必须声明其新鲜度级别（REALTIME / STANDARD / STATIC），在路由清单中记录。

### 跨层影响链

```
上游影响 (消费者):
  - 用户看到过期价格 → 下单后价格变动 → 投诉/退款
  - 管理员发布内容后看不到更新 → 重复发布
  - A/B 实验：缓存导致实验版本不一致 → 实验数据无效

下游影响 (依赖):
  - 下游微服务被缓存屏蔽 → 无法感知真实流量
  - 第三方 CMS（Contentful/Strapi）的 webhook 触发 revalidation 失败
  - 搜索引擎抓取到过期内容（SEO 损害）

部署影响:
  - ISR 重建失败 → 页面停留在旧版本直到下一次 rebuild
  - CDN purge API 与 Next.js cache 不一致 → 部分用户看到旧版本
  - 灰度部署中缓存键未区分版本 → 新旧版本数据混合

监控影响:
  - 监控指标: x-nextjs-cache HIT/MISS/STALE 比率
  - 告警: revalidate 失败次数
  - RUM: 数据新鲜度延迟（用户看到的数据时间戳 vs 服务器当前时间戳）
```

### 相关架构模式
- **渲染模式选型** (`decisions.md` 第一节) — SSG / ISR / SSR 的选择决定默认缓存行为。
- **pit-029: 流式 SSR 中断** — SSR 中的 `no-store` fetch 失败与流式渲染的交互。
- **CQRS 前端化** (`patterns.md` 第三节) — mutation (Command) 后自动触发 query 失效 (Query)，`revalidateTag` + `router.refresh()` 是此模式在 Next.js 中的实现。

## 预防
1. 所有 `fetch` 调用明确声明 `cache` 策略（`no-store` / `force-cache` / `next.revalidate`）。
2. 使用数据新鲜度分级定义（REALTIME / STANDARD / STATIC），禁止裸 `fetch`。
3. Mutation Action 中必须调用 `revalidateTag` / `revalidatePath` + Client 端调用 `router.refresh()`。
4. CDN 缓存 TTL 与 Next.js revalidate 对齐，在部署文档中声明。
