# pit-057: Taro 编译时与运行时不一致

## 环境
- Taro 3.x (所有版本)
- 微信小程序 + H5 双端编译
- React 或 Vue 作为开发框架

## 症状
同一份源码在不同平台产生不一致的行为：
- 微信小程序端正常渲染，H5 端空白或样式错乱
- 动态渲染的组件在小程序端不显示，H5 端正常
- 条件渲染 (`&&` / 三元) 在小程序端逻辑与预期不符
- `map` 遍历的动态列表在小程序端数量错误或顺序颠倒
- 复杂 JS 表达式 (`arr.filter().map()`) 在小程序端静默失败
- 三目表达式嵌套在小程序端只渲染第一个分支

## 根因
Taro 的编译时静态分析无法将**所有合法的 JSX/模板语法**转换为小程序 WXML 模板。当编译器遇到不能安全静态化的表达式时，有两条路径：

1. **降级为 data 绑定**：把表达式计算结果放入 `setData` 中，在小程序端通过 `{{}}` 模板表达式插值。但这受限于小程序模板表达式的表达能力（不支持方法调用、不支持复杂运算符链）。
2. **回退到运行时虚拟 DOM**：在逻辑层维护虚拟 DOM 树，diff 后通过 `setData` 同步。但这受 `setData` 256KB 单次限制和序列化约束影响。

两条路径的边界条件不同，导致"源码 → H5 (React DOM)"和"源码 → 小程序 (WXML + setData)"的分叉行为。

## 修复

```diff
// ❌ 问题1：链式数组操作 — 小程序模板无法表达 .filter().map()
function UserList({ users }) {
-  const activeUsers = users.filter(u => u.active).map(u => (
-    <View key={u.id}>{u.name}</View>
-  ));
-  return <View>{activeUsers}</View>;
+  // 修复：将复杂变换提前到变量声明，保持 JSX 表达式简单
+  const activeUsers = useMemo(
+    () => users.filter(u => u.active),
+    [users]
+  );
+  return (
+    <View>
+      {activeUsers.map(u => <View key={u.id}>{u.name}</View>)}
+    </View>
+  );
}
```

```diff
// ❌ 问题2：嵌套三元表达式 — 小程序模板无短路求值，逻辑与 JS 不同
function StatusBadge({ status }) {
-  return (
-    <View className={
-      status === 'online' ? 'green' :
-      status === 'away' ? 'yellow' :
-      status === 'busy' ? 'red'   : 'gray'
-    }>{status}</View>
-  );
+  // 修复：将条件逻辑提取到独立函数或变量
+  const statusClass = {
+    online: 'green',
+    away:   'yellow',
+    busy:   'red',
+  }[status] || 'gray';
+  return <View className={statusClass}>{status}</View>;
}
```

```diff
// ❌ 问题3：JSX 中的逻辑运算符链 — 小程序端可能只执行第一个条件
function Dashboard({ user, config }) {
-  return (
-    <View>
-      {user && user.permissions && user.permissions.admin && <AdminPanel />}

+    <View>

+    </View>
```

## 原理
Taro 编译器的核心是将 JSX 表达式树映射到 WXML 模板语言的语法树上。但 WXML 的表达能力远弱于 JavaScript：不支持函数调用、不支持 `Array.prototype` 方法、不支持逻辑运算符短路、不支持动态组件引用、不支持 `...` 展开运算符。当栈上出现编译器无法静态求值的表达式时，编译器必须做以下两件事之一：

1. **静态降级**：将表达式结果预先计算为 data 字段，在 WXML 中用 `{{data.field}}` 引用。但这导致 data 膨胀（每个计算字段都需一次 `setData`）。
2. **运行时回退**：放弃静态编译，在运行时维护虚拟 DOM，通过递归的 `setData` 同步 diff 结果。但这引入了 `setData` 的 256KB 单次上限和异步时序问题。

更隐蔽的问题是：H5 端使用标准的 React/Vue DOM 渲染，而小程序端走 Taro 运行时。两套渲染管线的语义存在微妙差异（如事件冒泡顺序、样式层叠优先级、列表更新策略），导致"代码相同、行为不同"。

## 架构视角 (Architecture Perspective)

### 架构层面根因
编译时与运行时不一致的表面原因是"JSX 表达能力 > WXML 模板表达能力"，但架构层面根因是**双渲染管线缺乏契约验证层**：
- Taro 的设计假设"编译时能覆盖大部分场景，剩余回退到运行时是透明的"，但这个透明性从未被形式化验证。没有契约规定"哪些 JSX 模式一定能被安全编译"，导致开发者凭直觉写代码，在 CI 阶段才暴露不一致。
- **平台抽象泄漏**：Taro 的适配器模式虽然封装了 API 差异，但渲染层始终有两套语义系统（React DOM vs WXML），适配器只处理了 API 层面，没有处理渲染语义层面。这是典型的"抽象层次不足"——适配器在 API 层，但不一致发生在语义层。
- **编译边界无声**：当编译器遇到无法处理的表达式时，降级行为是隐式的（静默回退到运行时），没有向开发者发出任何警告。这导致不一致 Bug 在开发阶段不可见，到集成/回归阶段才暴露，修复成本呈指数增长。

### 系统级预防策略

**静态防护**：
```js
// ESLint 自定义规则: 禁止小程序端不安全的 JSX 表达式
// .eslintrc.js (仅对 Taro 项目)
rules: {
  'taro/no-jsx-inline-filter': 'error',    // 禁止 JSX 内 .filter()
  'taro/no-jsx-nested-ternary': 'error',   // 禁止嵌套三元
  'taro/no-dynamic-component-ref': 'error',// 禁止动态组件引用
  'taro/no-jsx-logical-chain': 'warn',     // 警告链式逻辑运算符
}
```

**CI 防护**：
```bash
# CI 中添加双端渲染一致性检查
# 1. 编译产物分析
npx taro build --type weapp
npx taro build --type h5
# 2. 对比双端渲染快照
npx jest --testMatch "**/cross-platform/**" --ci
# 3. WXML 复杂度检查: 单个页面 template 引用数 > 30 则 CI 阻塞
find dist/weapp -name "*.wxml" | while read f; do
  count=$(grep -c 'template is=' "$f")
  if [ "$count" -gt 30 ]; then
    echo "ERROR: $f 运行时回退过多 (${count} templates)"
    exit 1
  fi
done
```

**架构防护**：
- 引入**渲染契约层**：在组件开发阶段声明"该组件的目标平台和渲染策略"，通过 CI 验证声明与编译产物的匹配度。
- **安全 JSX 子集**：定义"跨端安全 JSX 表达式白名单"（仅允许原始值绑定、单一条件、简单 map），在白名单外的表达式必须在代码审查中人工确认。
- **编译时警告增强**：Taro 编译器应在降级发生时（如回退到运行时 template 渲染）输出 `warning` 级别日志，而非静默。

**流程防护**：
- Code Review 清单项："此 JSX 表达式是否在 WXML 模板中有等价语法？"
- 每个 Taro 组件必须通过双端快照测试才能合并。
- 复杂 UI 逻辑提取到 `useMemo` / `computed` 中，保持 JSX 表达式为纯展示。

### 跨层影响链

```
上游影响 (业务层):
  - 产品验收时 H5 通过、小程序不通过，或反之 → 验收周期加倍
  - 运营活动页面双端不一致 → 用户投诉 (如优惠券金额不同)
  - A/B 测试数据污染 (双端行为不同但共享同一实验标识)

下游影响 (平台):
  - 小程序审核被拒 (运行时回退过多 → 性能不达标 → 审核不通过)
  - 微信小程序性能评分下降 → 搜索权重降低
  - H5 SEO 受影响 (渲染异常导致爬虫无法正确索引)

部署影响:
  - 需要为同一功能发布双端补丁 → 发布协调成本 2x
  - 紧急回滚时可能只有一端成功 → 用户体验碎片化
  - CI 矩阵爆炸：每个组件 × 双端 = 双倍测试时间

监控影响:
  - 错误上报需要区分平台 (TARO_ENV) 来分析不一致频次
  - Funnel 分析中双端转化率差异可能被误判为平台用户偏好
  - Sentry issue grouping 可能将同一根因的不一致 Bug 分到不同 group
```

### 相关架构模式
- **适配器模式 (Adapter)** (`patterns.md`) — Taro 的核心模式，但当前适配器在 API 层而非渲染语义层；需要升级为"语义适配器"。
- **安全子集模式 (Safe Subset)** — 定义跨端 JSX 白名单，类似于 Web 开发中的"渐进增强"策略，将最安全的表达式作为基线。
- **编译时契约** — 参考 Rust 的编译时所有权检查：编译器应明确告知"此处无法安全编译"，而非静默回退。

## 预防

1. **编写跨端安全的 JSX**：不在 JSX 中做复杂计算，将逻辑提取到 `useMemo` 或组件的顶层变量。
2. **避免嵌套三元和链式条件**：使用查找表 (lookup object) 或 `if-return` 模式替代嵌套三元。
3. **禁止 JSX 中的 `.filter().map()`**：拆分为两步——先 filter 得到数组，再在 JSX 中 `.map()`。
4. **启用 Taro ESLint 插件**：安装 `@tarojs/plugin-eslint`，使用内置规则检测不安全模式。
5. **每个组件写双端快照测试**：不是可选项，是跨端项目的架构要求。

```bash
# 在该组件中添加 Taro 配置，强制使用运行时渲染:

mini: {
  compile: {
    exclude: [
      // 强制该模块使用运行时渲染，避免编译时不一致的紧急方案

    ]
  }
}

# 方案B：精简组件直到双端一致
# 将一个复杂组件拆分为两个简单组件，分别验证
git diff src/components/ > /tmp/component_diff.patch
# 回退该组件后重新实施，一次只改一个表达式

# 方案C：平台特定实现（最后手段）
# 使用 Taro 的跨平台文件扩展名:
// UserList.h5.tsx — H5 特定实现 (可使用复杂表达式)
// UserList.weapp.tsx — 小程序特定实现 (使用安全子集)
# 删除有问题的统一实现，以双端特定文件替代
```
