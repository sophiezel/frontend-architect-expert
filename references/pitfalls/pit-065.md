# pit-065: CSP 配置错误导致整个应用不可用 — strict-dynamic + nonce 配置不当

## 环境
- Nginx/Apache/CDN 配置 CSP 响应头
- 使用了 `'strict-dynamic'` + nonce-based CSP
- SPA 应用（React/Vue/Svelte），可能有 Webpack/Vite 动态脚本注入

## 症状
部署 CSP 后，用户打开页面白屏——所有 JavaScript 被浏览器阻止执行。Chrome DevTools Console 满屏红色 CSP violation 报错：

```
Refused to execute inline script because it violates the following Content Security Policy directive:
"script-src 'strict-dynamic' 'nonce-abc123'".
Either the 'unsafe-inline' keyword, a hash ('sha256-...'), or a nonce is required to enable inline execution.
```

或者 CSS 和图片也被阻止，页面只有无格式文本。线上应用完全不可用，用户投诉激增，紧急回滚 CSP 配置。

## 根因
CSP `strict-dynamic` 模式的核心依赖是**每个页面请求动态生成唯一 nonce 值**，并注入到所有 `<script>` 标签上。配置错误集中在三个环节：

1. **nonce 未注入到所有 script 标签**：Webpack/Vite 内联的 `<script>` 没有 nonce 属性
2. **nonce 不是每次请求唯一**：错误地用固定 nonce（如 `nonce-fixed-value`）或缓存了 nonce
3. **strict-dynamic 与旧版浏览器/插件冲突**：浏览器插件的 script 没有 nonce 被阻止
4. **缺少 `'unsafe-inline'` 作为老旧浏览器回退**（strict-dynamic 下不安全但可作 fallback）
5. **CSS `<style>` 标签也需要 nonce**：如果 `style-src` 也启用了 strict 模式

## 诊断方法

```bash
# 1. 查看浏览器 Console CSP violation
# Chrome → F12 → Console → 过滤 "Content Security Policy"
# 记录 violated-directive 和 blocked-uri

# 2. 确认当前页面的 CSP 头
curl -I https://yourapp.com | grep -i content-security-policy

# 3. 检查 HTML 中的 nonce 注入
curl -s https://yourapp.com | grep -o 'nonce="[^"]*"' | sort | uniq
# 如果 nonce 是固定值或缺失 → 配置错误

# 4. 检查响应头 nonce 与 HTML nonce 是否一致
NONCE_HEADER=$(curl -sI https://yourapp.com | grep -o "nonce-[^';]*")
NONCE_HTML=$(curl -s https://yourapp.com | grep -o 'nonce="[^"]*"' | head -1)
echo "Header: $NONCE_HEADER"
echo "HTML: $NONCE_HTML"
# 两者必须一致

# 5. 检查 Webpack/Vite 构建产物中的 script 标签
grep -r "nonce" dist/  # 确认构建工具是否感知 nonce
# Vite: __webpack_nonce__ 或 document.currentScript.nonce
# Webpack: __webpack_nonce__ = window.__NONCE__;
```

## 修复

```diff
// ═══════ 修复1：后端：每个请求生成唯一 nonce ═══════
// Express 中间件
+ import crypto from 'crypto';
+
+ app.use((req, res, next) => {
+   // 每个请求生成唯一 nonce
+   res.locals.nonce = crypto.randomBytes(16).toString('base64');
+   next();
+ });

// ═══════ 修复2：HTML 模板注入 nonce ═══════
// 模板（EJS/Pug/Handlebars）
- <script src="/static/js/main.js"></script>
- <script>window.__CONFIG__ = {...}</script>
+ <script nonce="<%= nonce %>" src="/static/js/main.js"></script>
+ <script nonce="<%= nonce %>">window.__CONFIG__ = {...}</script>
+ <style nonce="<%= nonce %>">/* critical CSS */</style>

// ═══════ 修复3：Webpack 配置注入 nonce ═══════
// webpack.config.js
+ const HtmlWebpackPlugin = require('html-webpack-plugin');
+
+ // 方式A：通过全局变量
+ // HTML 模板中先设置:
+ // <script nonce="<%= nonce %>">window.__CSP_NONCE__ = "<%= nonce %>"</script>
+ // webpack 入口:
+ // __webpack_nonce__ = window.__CSP_NONCE__;

+ // 方式B：HtmlWebpackPlugin 注入
+ new HtmlWebpackPlugin({
+   template: 'index.html',
+   nonce: '<%= nonce %>', // 在服务端渲染时替换
+ })

// ═══════ 修复4：Nginx CSP 头 ═══════
- add_header Content-Security-Policy "script-src 'self' 'unsafe-inline';";
+ # 注意：Nginx 中 nonce 需要从上游（应用服务器）获取
+ # 如果 Nginx 直出静态文件，需要应用服务器代理
+ # 或者用 lua/ngx_http_sub_module 做替换
+ add_header Content-Security-Policy "
+   script-src
+     'strict-dynamic'
+     'nonce-{NONCE_FROM_APP}'
+     'unsafe-inline'
+     https:;
+   style-src
+     'nonce-{NONCE_FROM_APP}'
+     'unsafe-inline'
+     https:;
+   object-src 'none';
+   base-uri 'none';
+ ";

// ═══════ 修复5：Report-Only 模式先验证 ═══════
// 先用 Report-Only 模式部署 1 周，确认无 violation 后再切换到 enforce
+ add_header Content-Security-Policy-Report-Only "
+   script-src 'strict-dynamic' 'nonce-{NONCE}';
+   report-uri /api/csp-report;
+ ";
```

## 原理
CSP `strict-dynamic` 的工作机制：
1. 浏览器接收到带有 `nonce-abc` 的 CSP 头
2. 只执行 `nonce="abc"` 的 `<script>` 标签
3. 该脚本通过 `document.createElement('script')` 动态创建的脚本**继承信任**
4. 完全忽略 URL allowlist（`https:` 仅在 nonce 不匹配时作为老旧浏览器回退）

nonce 必须是不可预测的（密码学随机），否则攻击者可以猜测 nonce 绕过 CSP。每次请求使用新 nonce 同样重要——如果用固定 nonce，攻击者可以通过 XSS 读取 DOM 拿到 nonce 来注入脚本。

`'unsafe-inline'` 在 `strict-dynamic` 存在时会被现代浏览器忽略（Chrome 52+, Firefox 52+），因此可以作为老旧浏览器回退而不会打开安全漏洞。

## 架构视角 (Architecture Perspective)

### 架构层面根因
CSP 配置错误的表面原因是"nonce 没配对"，架构根因是**前端构建体系缺少 CSP 感知机制**：
- **Webpack/Vite 的 runtime-injected script 没有 nonce 通道**：构建工具默认不感知 CSP，它们注入的内联脚本（如 HMR runtime、code-split chunk loader）没有 nonce，导致整个应用白屏。
- **静态资源与动态 nonce 的矛盾**：SSR 服务端可以注入 nonce，但 SPA 的静态 HTML 部署到 CDN 后无法动态 nonce——需要 CDN edge worker 或 service worker 介入。
- **第三方脚本管理缺失**：GA、Sentry、Intercom 等第三方脚本的注入方式不同（script tag / createElement / iframe），需要逐一适配 nonce 策略。

### 系统级预防策略

**静态防护**：
```js
// 构建时注入 CSP 感知代码
// 1. Vite 配置
export default defineConfig({
  plugins: [react()],
  // 让 Vite 的 HMR script 带有 nonce
  // 使用 html.transformIndexHtml 钩子
  transformIndexHtml(html) {
    return html.replace(
      /<script(.*?)>/g,
      `<script$1 nonce="__NONCE_PLACEHOLDER__">`
    );
  }
});

// 2. Webpack: 利用 __webpack_nonce__ 变量
__webpack_nonce__ = document.querySelector('meta[property="csp-nonce"]')?.content;
```

**CI 防护**：
```yaml
# CI 中用 Puppeteer/Playwright 验证 CSP
- name: Test CSP
  script: |
    npx playwright test tests/csp/ --config=csp.config.ts
    # test('no CSP violation on homepage', async ({ page }) => {
    #   const violations: string[] = [];
    #   page.on('console', msg => {
    #     if (msg.text().includes('Content Security Policy')) violations.push(msg.text());
    #   });
    #   await page.goto('/');
    #   expect(violations).toHaveLength(0);
    # });
```

**架构防护**：
- **CDN Edge Worker 动态注入 nonce**：CloudFlare Worker / Lambda@Edge 在边缘节点为每个请求生成 nonce 并注入 HTML。
- **CSP 管理平台化**：将 CSP 策略和 nonce 生成抽成内部平台/service，各前端项目统一接入，避免各自配置错误。
- **nonce 通过 `<meta>` 标签传递**：服务端在 HTML 中输出 `<meta property="csp-nonce" content="abc123">`，构建工具的 runtime 读取此 meta 获取 nonce。

**流程防护**：
- CSP 变更强制走 Report-Only → 观察 1 周 → grayscale 10% → grayscale 50% → 全量。
- CSP 变更需要安全工程师 Code Review 审批。
- 每次发布后检查 CSP report endpoint，新增 violation 触发告警。

### 跨层影响链

```
上游影响:
  - CSP 全量开启 → 线上所有 JS 被阻止 → 全站白屏
  - 紧急回滚 CSP 配置 → 部署窗口被占用 → 其他 hotfix 延迟

下游影响:
  - 用户白屏 → 跳出率 100% → 业务指标归零（持续到修复）
  - 客户投诉/客服工单暴增 → 运营团队过载
  - SEO 受冲击（爬虫抓取到空白页面） → 搜索排名下降

组织影响:
  - CSP 被污名化为"危险配置" → 未来安全推动受阻
  - 安全团队与前端团队信任裂痕
  - 可能需要 CEO/CTO 级别的 Post-Mortem Review
```

### 相关架构模式
- **CSP 渐进部署策略** (`frontend-security.md` 第2.2节) — Report-Only → Grayscale → Full
- **Trusted Types** (`frontend-security.md` 第1.5节) — 与 nonce CSP 互补的 DOM XSS 防御
- **Edge-side nonce injection** (CloudFlare Workers / Lambda@Edge)

## 预防

1. **所有 CSP 变更先走 Report-Only 模式至少 1 周**：收集 violation 报告再切换 enforce。
2. **灰度发布**：先 10% → 50% → 100% 流量，每阶段观察 violation 日志。
3. **自动化 CSP 测试**：CI 中用浏览器自动化测试验证核心功能无 violation。
4. **nonce 生成确认**：确保每次 HTTP 请求拿到全新 nonce（密码学随机）。
5. **构建工具 CSP 适配**：配置 Webpack `__webpack_nonce__` 或 Vite nonce placeholder。
