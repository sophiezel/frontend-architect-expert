# pit-019: 数组 map 中遗忘 return 导致 undefined

## 环境
- 所有 JavaScript 环境

## 症状
对一个数组使用 `.map` 转换后，结果数组中包含大量 `undefined`。期望 `['Alice', 'Bob']` 却得到 `[undefined, undefined]`。

## 根因
`Array.map` 的回调如果使用花括号 `{}` 包裹函数体，必须显式使用 `return` 语句。开发者经常在从单行箭头函数改为多行时忘记添加 `return`，导致函数隐式返回 `undefined`。

## 诊断方法

```bash
# 1. 打印 map 结果
const users = [{name: 'Alice'}, {name: 'Bob'}];
const names = users.map(user => {
  user.name; // ← 没有 return
});
console.log(names); // [undefined, undefined]

# 2. 对比正确写法
const names2 = users.map(user => user.name);
console.log(names2); // ['Alice', 'Bob']

# 3. 代码搜索
grep -rn "\.map.*=>\s*{" src/ --include="*.ts" --include="*.js" -A 3 | grep -v "return"
# 查找 map 回调使用了花括号但可能没有 return 的代码

# 4. ESLint 诊断
npx eslint src/ --rule 'array-callback-return: error'
# 检查是否有 map/filter/reduce 回调缺少返回值
```

## 修复

```diff
// ❌ 问题：花括号忘记 return
const names = users.map(user => {
-  user.name; // 没有 return → undefined
});

// ✅ 修复
const names = users.map(user => {
+  return user.name;
});

// 或者使用隐式返回（推荐简单场景）
+ const names = users.map(user => user.name);

// 如果需要对象字面量，用括号包裹
+ const userDTOs = users.map(user => ({
+   id: user.id,
+   name: user.name,
+ }));
```

## 原理
JavaScript 箭头函数有两种语法形式：
- **简洁体（Concise body）**：`items.map(x => x.name)` — 表达式结果自动返回，无需 `return`。
- **块体（Block body）**：`items.map(x => { return x.name; })` — 必须显式 `return`，否则函数返回 `undefined`。

这是 JavaScript 函数的基础语法规则：所有函数如果不遇到 `return` 语句，默认返回 `undefined`。箭头函数的简洁体是唯一的例外。

常见混淆点：从简洁体 `x => x.name` 改为多行时添加了花括号 `{}`，但忘记同时添加 `return`。

## 架构视角 (Architecture Perspective)

### 架构层面根因
map 缺少 return 的架构根因是**转换表达式与转换语句未在语法层面强制区分**：
- **TypeScript 严格模式未充分利用**：TypeScript 在严格模式 + `noImplicitReturns` 下可以检测缺少 return 的路径，但很多项目的 `tsconfig` 中未启用。这是架构层面的决策失误——选择了 TypeScript 但没有启用它的安全特性。
- **代码风格规范未提升为架构约束**：团队可能有"优先使用隐式返回"的风格建议，但没有在工具链上强制执行。如果架构规定"**所有转换操作必须使用表达式语法**"（通过 ESLint 的 `arrow-body-style` 规则），就不存在"添加花括号忘记 return"的问题。
- **缺少数组转换的类型安全抽象**：数组的 `map` 是一个低阶操作，结果类型是 `U[]`，不区分"转换成功"和"转换失败返回 undefined"。架构可以提供高阶的"安全 map"——`safeMap` 要求回调必须返回非空值。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 多维度防护
rules: {
  // 检测 map/filter/reduce 中缺少 return
  'array-callback-return': ['error', { 
    checkForEach: false,  // forEach 不需要 return
    allowImplicit: true   // 允许隐式返回
  }],
  // 强制箭头函数使用最简形式（可选）
  'arrow-body-style': ['warn', 'as-needed'],
  // TypeScript 规则
  '@typescript-eslint/no-unnecessary-condition': 'error',
}

// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitReturns": true,         // 所有代码路径必须有 return
    "noFallthroughCasesInSwitch": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true
  }
}
```

**CI 防护**：
```bash
# CI 类型检查
npx tsc --noEmit  # 必须通过
npx eslint src/ --rule 'array-callback-return: error' --max-warnings 0

# CI 测试: 验证 map 结果不含 undefined
test('map 结果不应包含 undefined', () => {
  const result = users.map(u => u.name);
  expect(result).toHaveLength(2);
  expect(result.every(Boolean)).toBe(true);
});

# 自动化代码审查检查
grep -rn '\.map.*=>\s*{' src/ --include="*.ts" --include="*.tsx" | 
  while read line; do
    # 检查该行之后的 5 行内是否有 return
    # 如果没有 → 潜在的 bug
  done
```

**架构防护**：
- **安全 map 抽象**：
```typescript
// @app/utils/safeArray.ts
// 安全 map：过滤掉 undefined/null
export function compactMap<T, U>(
  items: T[],
  fn: (item: T, index: number) => U | null | undefined
): NonNullable<U>[] {
  return items.map(fn).filter((v): v is NonNullable<U> => v != null);
}

// 严格 map：如果任何一项返回 undefined 则抛出错误
export function strictMap<T, U>(
  items: T[],
  fn: (item: T, index: number) => U
): U[] {
  const result: U[] = [];
  for (let i = 0; i < items.length; i++) {
    const value = fn(items[i], i);
    if (value === undefined) {
      throw new Error(`strictMap: callback returned undefined for index ${i}`);
    }
    result.push(value);
  }
  return result;
}
```
- **强制执行简洁箭头函数风格**：通过 `arrow-body-style: ['error', 'as-needed']` 和 Prettier 配置，自动化地将 `x => { return expr; }` 格式化为 `x => expr`，消除"忘记 return"的语法可能性。

**流程防护**：
- Code Review 检查项："map 回调中是否有花括号？如果是，是否每个路径都有 return？"
- 新人入职培训：强调"花括号 → return"的规则，要求使用隐式返回。

### 跨层影响链

```
上游影响 (消费者):
  - 父组件传入的处理后数组包含 undefined → 子组件渲染异常
  - 数组的 .filter() 链中 undefined 被当作 falsy 过滤掉 → 数量错误

下游影响 (依赖):
  - 后续 .map() / .reduce() 在 undefined 上操作 → TypeError
  - React 列表渲染中 undefined 项 → React key warning 或空白项
  - 后端 API 提交包含 undefined → JSON.stringify 将其转为 null 或删除

部署影响:
  - TypeScript 编译通过但运行时错误 → 单元测试必须覆盖
  - 线上数据量小时（3 项）未触发 → 数据量大时大量 undefined

监控影响:
  - Sentry: `Cannot read properties of undefined` 错误聚合
  - 自定义断言: 对关键数据管道的输出验证 containsNoUndefined
  - E2E 测试: 检查列表渲染的 item 数量是否符合预期
```

### 相关架构模式
- **反模式: 过早抽象** (`patterns.md` 第六节) — 当 map 回调只有 2 行时用隐式返回；当需要 5 行以上逻辑时，应提取为命名函数而非内联回调，减少语法错误。
- **TypeScript 严格模式** (`decisions.md`) — 项目的 `tsconfig` 必须启用 `strict: true` + `noImplicitReturns: true`，将此类问题从运行时前移到编译时。
- **组件组合模式** (`patterns.md` 第二节) — 将复杂的 map 转换封装为自定义 Hook（如 `useTransformedUsers`），内部保证转换正确性，组件层只消费结果。

## 预防

1. **ESLint `array-callback-return`**：对 `map`/`filter`/`reduce` 强制 return。
2. **代码风格约定**：
   - 单行转换 → 隐式返回 `items.map(x => x.id)`
   - 多行逻辑 → 显式 return，且在花括号后立即写 `return`
3. **TypeScript**：如果声明了返回类型，编译器会捕获此问题。
4. **Code Review**：搜索 `map.*=>\s*{` 并确认有对应的 `return`。
