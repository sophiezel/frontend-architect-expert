# pit-001: useEffect 无限循环

## 环境
- React 16.8+（Hooks 引入后）
- 任何使用 useEffect 的组件

## 症状
组件渲染后 CPU 飙升到 100%，浏览器卡死或控制台不断刷出日志。React DevTools 显示组件在疯狂重渲染。

## 根因
useEffect 的依赖数组中包含了每次渲染都会重新创建的引用类型（对象、数组、函数），导致 effect 每次执行后又触发新的渲染，形成死循环。

核心机制：React 使用 `Object.is` 对依赖数组做浅比较，引用类型每次渲染都是新引用 → 永远"不相等" → 反复执行。

## 诊断方法

```bash
# 1. 在 effect 内部打印日志，观察执行频率
# 在组件代码中添加:
useEffect(() => {
  console.log('[DEBUG] effect fired at', new Date().toISOString());
}, [someObj]);

# 2. 使用 React DevTools Profiler 录制
# Chrome → React DevTools → Profiler → 开始录制 → 操作页面 → 停止
# 观察火焰图中组件是否反复提交

# 3. 检查依赖数组中的引用类型
# 搜索模式：依赖数组内是否有对象字面量 { }、数组字面量 [ ]、箭头函数 ()
grep -n "useEffect.*\[.*{" src/**/*.{tsx,jsx}
grep -n "useEffect.*\[.*\[" src/**/*.{tsx,jsx}
```

## 修复

```diff
// ❌ 问题代码：每次渲染 data 都是新对象
function Profile({ user }) {
-  const data = { name: user.name, age: user.age };
-
-  useEffect(() => {
-    fetch('/api/update', { body: JSON.stringify(data) });
-  }, [data]); // data 每次渲染都是新引用 → 无限循环
+  useEffect(() => {
+    fetch('/api/update', {
+      body: JSON.stringify({ name: user.name, age: user.age })
+    });
+  }, [user.name, user.age]); // 只依赖原始值

  return <div>{user.name}</div>;
}
```

```diff
// ❌ 问题代码：内联函数作为依赖
function Search() {
  const [query, setQuery] = useState('');
-
-  useEffect(() => {
-    const handler = (e) => console.log(e);
-    window.addEventListener('resize', handler);
-    return () => window.removeEventListener('resize', handler);
-  }, [handler]); // handler 每次都是新函数
+  useEffect(() => {
+    const handler = (e) => console.log(e);
+    window.addEventListener('resize', handler);
+    return () => window.removeEventListener('resize', handler);
+  }, []); // 空数组：仅在挂载时执行一次
}
```

## 验证

```bash
# 1. 移除 effect 内的日志（如果有），然后在组件外添加计数器
# 在组件中临时添加：
# let renderCount = 0; // 组件外
# 组件内: renderCount++; console.log('render:', renderCount);
# 如果数字在几秒内暴涨（>100），说明仍在无限循环

# 2. React DevTools → ⚛️ Components → 选中组件 → 右侧面板查看 "Rendered" 次数
# 正常：仅挂载时 1 次，或随合法 state/props 变化有限次数
# 异常：每秒数十次以上

# 3. Chrome Performance 面板录制 3 秒
# 如果看到密集的 "React commit" 或组件名重复出现 → 仍有循环
# 正常：只有零星几次渲染任务

# 4. 终极验证：在 useEffect 清理函数中打印
useEffect(() => {
  console.log('effect mounted');
  return () => console.log('effect cleanup — 如果频繁出现说明在反复重建');
}, [dep]);
```

## 原理
React 的 useEffect 依赖比较使用 `Object.is()` 浅比较。JavaScript 中每次执行 `{}` 或 `[]` 或 `() => {}` 都会创建全新的引用，`Object.is(obj1, obj2)` 对引用类型比较的是内存地址，因此永远返回 false。这导致 effect 在每次渲染后都认为"依赖变了"，重新执行 effect，如果 effect 内部又触发了 setState，就形成了死循环。

## 架构视角 (Architecture Perspective)

### 架构层面根因
无限循环的表面原因是"依赖引用不稳定"，但架构层面的根因是**数据流设计缺陷**：
- **派生数据被当作独立状态存储**：`data` 对象从 `user.name` 和 `user.age` 派生，但没有被建模为派生值（`useMemo`），而是被存储为每次渲染都重新创建的对象，破坏了 React 单向数据流的可预测性。
- **副作用边界未定义**：组件承担了数据同步（将 props 同步到后端）的责任，但未声明同步策略（何时同步、同步什么、同步顺序）。缺少一个明确的"同步策略模块"，导致副作用散落在组件中，形成连锁 effect 瀑布。
- **模块边界错误**：组件在渲染层同时处理了业务逻辑（API 调用）和 UI 展示，违反了单一职责。应将数据同步逻辑提取到自定义 Hook 或数据层，由明确的接口契约驱动。

### 系统级预防策略

**静态防护**：
```js
// ESLint 规则: react-hooks/exhaustive-deps (必须开启，设为 error 级别)
// .eslintrc.js
rules: {
  'react-hooks/exhaustive-deps': 'error',
  // 禁止在 useEffect 依赖中使用对象/数组字面量
  'no-restricted-syntax': ['error', {
    selector: 'CallExpression[callee.name="useEffect"] ArrayExpression:not(:has(Identifier))',
    message: 'useEffect 依赖数组中禁止数组字面量，请使用原始值'
  }]
}

// TypeScript: 将 API 返回类型与组件 props 严格分层
interface UserSyncConfig {
  userId: string;  // 原始值，稳定引用
}
// 禁止在 effect 内部从 props 构建新对象
```

**CI 防护**：
```bash
# CI 中添加渲染次数检测（Jest + React Profiler）
# __tests__/render-budget.test.tsx
test('组件在稳定 props 下不应重复渲染超过 3 次/秒', async () => {
  const { rerender } = render(<Component userId="1" />);
  // 连续三次 rerender 同一 props，观察渲染次数
  // 若每次 props 不变但渲染 > 1 次 → 循环风险
});

# Bundle 分析：检测是否存在连锁 effect 模式
# 使用 dependency-cruiser 定义规则：
# "组件文件不应同时包含 useState + useEffect + fetch 调用"
```

**架构防护**：
- 引入**副作用声明机制**：使用 `useSyncExternalStore` 或 Redux Toolkit Query 声明数据同步策略，禁止组件内裸 `useEffect` 做数据获取。
- **模块边界规则**：组件文件（`.tsx`）不允许直接调用 `fetch`/`axios`，只能通过数据层 Hook 获取数据。通过 `dependency-cruiser` 强制执行：
```bash
npx depcruise --include-only 'src/components' --exclude 'src/services|src/hooks' \
  --forbid 'fetch|axios\.get|axios\.post'
```

**流程防护**：
- Code Review 清单项："此 useEffect 的依赖数组中是否存在引用类型？如果是，它是否被 useMemo/useCallback 包裹？"
- 对每个新增的 useEffect 要求编写注释说明："触发条件、执行内容、清理逻辑"三要素。

### 跨层影响链

```
上游影响 (消费者):
  - 使用该组件的父组件被无限重渲染拖垮
  - 同一页面的兄弟组件被抢占渲染资源
  - React Concurrent Mode 的时间切片被无限循环破坏

下游影响 (依赖):
  - effect 内的 fetch 调用变成 DDoS 攻击后端
  - 数据库连接池耗尽（每次渲染触发 N 个请求）
  - 浏览器 GC 压力剧增，最终 Tab 崩溃

部署影响:
  - Sentry/LogRocket 产生海量报错日志，费用暴涨
  - 用户体验评分（LCP/INP/CLS）全面退化
  - 用户投诉量激增 → 紧急回滚 → 部署流程被破坏

监控影响:
  - 需要监控指标: 组件渲染频率 > 阈值
  - APM 告警: useEffect 执行时长 > 50ms 持续 5 秒
  - 自定义事件: useEffect 清理函数调用频率异常
```

### 相关架构模式
- **CQRS 前端化** (`patterns.md` 第三节) — 将"读取 props"和"写入后端"分离为独立的 Query 和 Command 模块，组件只消费预计算的数据。
- **反模式: useEffect 地狱** (`patterns.md` 第六节) — 连锁 effect 触发是典型的反模式，正确的架构是将多个 effect 合并为单一数据流管道。
- **状态管理选型** (`decisions.md` 第二节) — 涉及服务端数据同步时，应使用 React Query / SWR 而非 useEffect + useState 组合，它们内建了请求去重、缓存失效和竞态处理。

## 预防

1. **依赖原语值而非引用值**：将对象属性拆解为 `user.name` 层级放入依赖数组。
2. **使用 useMemo / useCallback 稳定引用**：如果必须传递引用类型，用它们包裹。
3. **安装 eslint-plugin-react-hooks**：`react-hooks/exhaustive-deps` 规则会静态分析依赖完整性并报 warning。
4. **Code Review 检查清单**：每次看到 `useEffect` 先看依赖数组是否有对象/数组/函数字面量。
