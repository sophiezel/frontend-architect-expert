# pit-002: useEffect 闭包陷阱（Stale Closure）

## 环境
- React 16.8+
- 尤其当依赖数组遗漏或为空时

## 症状
useEffect 或 useCallback 内部读取的 state / props 始终是初始值，无论外部如何更新。定时器内 count 永远是 0，事件监听器中的 roomId 永远是挂载时的值。

## 根因
JavaScript 闭包捕获的是变量在当前作用域的快照。当 effect 在第一次渲染时创建闭包，后续渲染不会更新已创建的闭包中的变量值——除非依赖数组正确声明，使 effect 重新创建。

## 诊断方法

```bash
# 1. 在 effect 内用 console.log 打印变量，对比组件渲染时的值
const [count, setCount] = useState(0);
useEffect(() => {
  const timer = setInterval(() => {
    console.log('[DEBUG] count in closure:', count); // 始终打印 0
  }, 1000);
  return () => clearInterval(timer);
}, []);

# 2. 使用 useRef 做"逃生舱"验证
const countRef = useRef(count);
countRef.current = count; // 每次渲染更新
useEffect(() => {
  const timer = setInterval(() => {
    console.log('[DEBUG] count ref:', countRef.current); // 始终最新
    console.log('[DEBUG] count state:', count);           // 始终初始值
    // 如果两者不一致 → 确认闭包过时
  }, 1000);
  return () => clearInterval(timer);
}, []);

# 3. React DevTools 查看 state 实际值
# Components → 选中组件 → hooks 区域 → 查看 state 当前值
# 与 effect 内 console.log 对比
```

## 修复

```diff
// ❌ 问题代码：定时器内 count 永远是 0
function Timer() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const timer = setInterval(() => {
-      console.log(count); // 闭包捕获初始值 0
-      setCount(count + 1);
+      setCount(c => c + 1); // 函数式更新：读取最新 state
    }, 1000);
    return () => clearInterval(timer);
  }, []);
}
```

## 原理
JavaScript 闭包捕获的是词法作用域中的变量绑定。当 `useEffect(fn, [])` 传入空依赖数组时，fn 只在组件挂载时创建一次，此后组件的每次重渲染都会创建新的作用域和新的 state 变量，但 fn 仍然引用第一次渲染时的旧变量。React Hooks 的每个渲染都有自己独立的 state 和 props 快照。

## 架构视角 (Architecture Perspective)

### 架构层面根因
闭包陷阱的架构根因是**生命周期耦合设计错误**：
- **长期订阅与短期 props 的绑定方式不当**：WebSocket 连接（长生命周期资源）与 `roomId` prop（短生命周期值）采用了"重建式绑定"（effect 依赖变化 → 销毁 → 重建），而不是"参数更新式绑定"（连接保持，仅更新参数）。这暴露了架构中缺少"可更新参数的长生命周期资源管理器"抽象。
- **状态时间线建模缺失**：闭包陷阱本质上是"读取过去的值"——系统需要 `current` 值但拿到了 `captured` 值。架构层面缺少对"值的时间语义"的明确建模：哪些值是"此刻"语义（用 ref）、哪些值是"渲染帧"语义（用 state）。两种语义混用导致歧义。
- **缺少资源管理层**：WebSocket/EventSource/Timer 等长生命周期资源应由统一的"资源管理器"控制生命周期，组件只通过稳定的回调接口接收数据，不应每个组件自己管理连接。

### 系统级预防策略

**静态防护**：
```js

**CI 防护**：

```

**架构防护**：
- 使用**长生命周期资源管理器模式**：将 WebSocket/Timer/EventSource 封装为独立的 resource manager，通过 `subscribe(callback)` 提供稳定的回调注册，内部通过 ref 保持最新参数：
```typescript
// 架构模式：可更新参数的长生命周期 Hook
function useStableWebSocket(url: string) {
  const urlRef = useRef(url);
  urlRef.current = url;  // 稳定更新，不触发重连
  
  useEffect(() => {
    const ws = new WebSocket(urlRef.current);
    // 只建立一次连接，回调通过闭包 ref 读取最新值
    return () => ws.close();
  }, []);  // 仅在挂载/卸载时执行
}
```
- **依赖注入边界**：组件不应直接持有 WebSocket 引用，而应通过 Context 注入连接对象。

**流程防护**：
- Code Review 检查项："此 useEffect 的依赖数组是否为空？如果是，它内部是否读取了任何 state/props？" 若答案为是 → 存在闭包陷阱风险。
- 对每个 useEffect 要求回答："此 effect 的生命周期与哪些值的生命周期绑定？"

### 跨层影响链

```
上游影响 (消费者):
  - 实时数据展示错误：消息发送到错误的 room
  - 用户操作基于过期数据：点击按钮后发现操作的是旧版本数据
  - 级联错误：闭包中的旧数据通过 callback 传播到其他模块

下游影响 (依赖):
  - WebSocket 发送消息到错误的频道（安全风险）
  - 定时器基于错误的状态做决策（如自动保存保存了旧内容）
  - 第三方 SDK 回调使用过时的配置

部署影响:
  - 难以重现的 bug（依赖特定操作时序）
  - 线上事故排查困难（日志显示的是"正确"的最新值，但实际执行的是旧值）
  - 回滚无效（闭包 bug 已存在于之前多个版本）

监控影响:
  - 需要监控：useRef 值与闭包值的一致性（开发环境断言）
  - Sentry 自定义事件：检测到 state !== ref.current 的异步场景
  - E2E 测试必须覆盖：切换上下文后原有订阅是否使用新上下文
```

### 相关架构模式
- **useRef 逃生舱模式** (`patterns.md` 第一节 状态管理模式) — 对于长生命周期回调，使用 ref 保持最新值而不触发 effect 重建，是"参数更新式绑定"的参考实现。
- **CQRS 前端化** (`patterns.md` 第三节) — 将 WebSocket 消息处理作为 Command 层，读取最新状态作为 Query 层，两者通过 ref 而非闭包衔接。
- **状态管理选型 — Zustand/Jotai 脱离 React 访问** (`decisions.md` 第二节) — Zustand 的 `getState()` 可以在 WebSocket handler 中获取最新状态而不受闭包限制，是架构级解决方案。

## 预防

1. **函数式更新**：`setState(prev => prev + 1)` 无需依赖当前 state。
2. **useRef 逃生舱**：对于事件监听等不想频繁重建的场景，用 `ref.current` 保持最新值。
3. **eslint-plugin-react-hooks** 务必开启，任何遗漏依赖都会报 warning。
4. **useCallback / useMemo 同理**：依赖数组必须完整。
