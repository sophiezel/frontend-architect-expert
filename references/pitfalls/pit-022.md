# pit-022: Android WebView 兼容性问题 — 碎片化 / Chromium 版本差异 / JS 引擎不一致

## 环境
- Android 5.0 ~ 14 (API 21 ~ 34)
- Android System WebView（系统内置）
- 厂商定制 WebView（华为/小米/OPPO/vivo/三星）
- 混合应用（混合式 App / React Native WebView / Flutter WebView）
- 微信 / 支付宝 / 小程序内置 WebView
- Google Chrome（独立 Chromium 引擎，作为对照）

## 症状
同一个 H5 页面在 Chrome 浏览器中渲染正常、交互正常，但在同设备的 WebView 中：

1. **白屏**：页面完全空白，`webView.loadUrl()` 不报错但也不渲染。
2. **样式异常**：Flexbox 布局错乱、CSS Grid 不生效、`gap` 属性被忽略、`backdrop-filter` 不显示。
3. **JS 报错**：`Uncaught TypeError: Array.prototype.flat is not a function`、`Promise.allSettled is not defined`、`BigInt` 语法错误。
4. **TLS/SSL 证书错误**：HTTPS 接口被 WebView 拦截，显示 `ERR_CERT_AUTHORITY_INVALID`，但 Chrome 中正常。
5. **渲染性能极差**：CSS 动画掉帧严重，`requestAnimationFrame` 回调间隔 >100ms，页面滚动卡顿。
6. **ES Module 加载失败**：`<script type="module">` 无法加载，或 dynamic `import()` 抛异常。

## 根因
Android WebView 的根本问题是**没有独立浏览器引擎**，直接依赖系统 `Android System WebView` APK：

1. **版本碎片化严重**：Android 5~6 设备搭载 Chromium 37~51，不支持 Flexbox gap/`Object.fromEntries`/可选链 `?.`。Android 7+ 后 WebView 独立于系统 OTA 更新，但实际更新率低——大量设备 WebView 版本比 Chrome 落后 2~3 个大版本。
2. **厂商定制 WebView 行为差异**：华为在 EMUI 上使用自研内核（非 Chromium），小米 MIUI 修改了 WebView 渲染管线，OPPO ColorOS 禁用了部分 Web API。同一套 Web 标准在不同厂商 WebView 中表现不一致。
3. **TLS/SSL 证书链验证差异**：系统 WebView 使用 Android 系统证书存储（而非 Chrome 自己的），部分根证书缺失或过期。Let's Encrypt 的 `ISRG Root X1` 在 Android 7.0 以下的 WebView 中不受信任。
4. **JS 引擎差异**：部分厂商定制 WebView 使用了非 V8 引擎（如华为方舟引擎早期版本），导致 `Proxy`/`Reflect`/`WeakRef` 等 API 行为异常。
5. **WebView 特性开关未启用**：`WebSettings.setJavaScriptEnabled(true)` 默认启用，但 `setDomStorageEnabled`/`setMixedContentMode`/`setAllowFileAccessFromFileURLs` 等默认关闭，导致 localStorage 不可用或混合内容被静默拦截。

```typescript
// src/boot/capability-detector.ts
// 功能检测器：运行时检测 WebView 实际能力

export interface WebViewCapabilities {
  webViewVersion: string;
  chromeVersion: string | null;
  androidVersion: string;
  vendor: string;
  supports: Record<string, boolean>;
  missing: string[];
}

export function detectWebViewCapabilities(): WebViewCapabilities {
  const ua = navigator.userAgent;
  
  // 解析 User-Agent
  const isWebView = /wv|WebView/i.test(ua);
  const chromeMatch = ua.match(/Chrome\/(\d+)/);
  const androidMatch = ua.match(/Android\s([\d.]+)/);
  const vendorMatch = ua.match(/;\s(\w+)Build\//);
  
  const chromeVersion = chromeMatch ? chromeMatch[1] : null;
  const chromeMajor = parseInt(chromeVersion || '0', 10);
  
  // 功能检测矩阵（基于 Chromium 版本）
  const supports: Record<string, boolean> = {
    // ES2019+
    flatMap: typeof [].flatMap === 'function',
    flat: typeof [].flat === 'function',
    objectFromEntries: typeof Object.fromEntries === 'function',
    // ES2020+
    optionalChaining: checkSyntax('window?.x'),
    nullishCoalescing: checkSyntax('0 ?? 1'),
    bigInt: checkSyntax('1n'),
    promiseAllSettled: typeof Promise.allSettled === 'function',
    globalThis: typeof globalThis !== 'undefined',
    // ES2021+
    replaceAll: typeof ''.replaceAll === 'function',
    // CSS
    cssGrid: CSS.supports('display', 'grid'),
    cssGap: CSS.supports('gap', '1px'),
    cssFlexboxGap: CSS.supports('gap', '1px') && CSS.supports('display', 'flex'),
    backdropFilter: CSS.supports('backdrop-filter', 'blur(1px)'),
    cssVariables: CSS.supports('--x', '0'),
    // DOM
    intersectionObserver: typeof IntersectionObserver !== 'undefined',
    mutationObserver: typeof MutationObserver !== 'undefined',
    // Web API
    esModule: checkScriptTypeModule(),
    dynamicImport: typeof import === 'function', // 不可靠
    webp: checkWebP(),
  };
  
  const missing = Object.entries(supports)
    .filter(([_, v]) => !v)
    .map(([k]) => k);
  
  return {
    webViewVersion: isWebView ? `WebView/Chrome-${chromeVersion}` : 'Browser',
    chromeVersion,
    androidVersion: androidMatch ? androidMatch[1] : 'unknown',
    vendor: vendorMatch ? vendorMatch[1] : 'unknown',
    supports,
    missing,
  };
}

function checkSyntax(code: string): boolean {
  try { new Function(`return ${code}`)(); return true; } catch { return false; }
}

function checkScriptTypeModule(): boolean {
  return 'noModule' in document.createElement('script');
}

function checkWebP(): boolean {
  const c = document.createElement('canvas');
  c.width = 1; c.height = 1;
  return c.toDataURL('image/webp').startsWith('data:image/webp');
}
```

## 原理
Android WebView 的核心矛盾是**浏览器引擎与操作系统的绑定**。与 iOS 统一使用 WKWebView 不同，Android 允许 OEM 替换 WebView 实现。Google 自 Android 7.0 起将 WebView 发布为独立 APK（`com.google.android.webview`），可通过 Play Store 更新，但厂商预装设备常锁定 WebView 版本。

`@supports` 是 CSS 标准的特性查询规则，允许浏览器在解析时跳过不支持的声明块，实现无 JavaScript 的渐进增强。Polyfill 注入 + `@supports` = 编译时 + 运行时双层防护。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Android WebView 兼容性问题的架构根因是**多端兼容策略缺少统一的 Capability API 层**：

- **将"浏览器"当作统一抽象，忽略了平台 WebView 的碎片化现实**：前端代码中直接使用 `Array.prototype.flatMap` / CSS Grid `gap` / `dynamic import()` 等 API，隐式假设所有运行时环境都支持。但实际上 WebView 的能力矩阵差异巨大——同一套代码在 Chrome 120 上运行完美，在 WebView Chrome 51 上直接崩溃或白屏。
- **能力检测（feature detection）与 UA 检测（user-agent sniffing）混淆**：许多项目用 UA 字符串判断"是不是 Android WebView"，然后整体降级。这种粗粒度的二进制决策（"是 WebView 就降级"）导致高性能设备也无法享受现代 API。正确的做法是按能力粒度检测——"不支持 flatMap 就注入 polyfill"。
- **功能降级矩阵未在架构中显式建模**：前端架构通常定义了路由/状态/数据流，但**没有定义"功能降级矩阵"**——哪些特性在哪些环境下可用、不可用时如何处理、是否有 fallback。没有这个矩阵，每次遇到兼容问题都是应急修复。

### 系统级预防策略

**静态防护**：

// src/platform/capability-matrix.ts
// 功能降级矩阵：定义每个特性在各环境下的可用性和降级策略
export const CAPABILITY_MATRIX = {
  // JS API
  'Array.prototype.flatMap': {
    minChromeVersion: 69,
    fallback: 'polyfill-core-js',
    severity: 'breaking', // 缺失则 JS 报错
  },
  'Optional Chaining (?.)': {
    minChromeVersion: 80,
    fallback: 'babel-transform', // 编译时转译
    severity: 'breaking',
  },
  'BigInt': {
    minChromeVersion: 67,
    fallback: 'jsbi-polyfill', // 运行时 polyfill
    note: '有性能代价，建议业务侧避免频繁使用',
    severity: 'warning',
  },
  // CSS
  'Flexbox gap': {
    minChromeVersion: 84,
    fallback: 'margin-spacing',
    severity: 'visual', // 缺失则布局异常但不崩溃
    cssFallback: '@supports (gap: 1px) { ... }',
  },
  'CSS Grid': {
    minChromeVersion: 57,
    fallback: 'block-layout',
    severity: 'visual',
  },
  'backdrop-filter': {
    minChromeVersion: 76,
    fallback: 'opaque-overlay',
    severity: 'cosmetic', // 缺失只影响视觉效果
    cssFallback: '@supports (backdrop-filter: blur(1px)) { ... }',
  },
  // Web API
  'IntersectionObserver': {
    minChromeVersion: 51,
    fallback: 'polyfill-intersection-observer',
    severity: 'degraded', // 缺失则懒加载等功能降级
  },
  'CSS env(safe-area-inset-*)': {
    minChromeVersion: 'N/A (iOS+Android P)',
    fallback: 'constant() + JS',
    severity: 'visual',
  },
} as const;

// 环境能力评分
export function evaluateCapabilityScore(
  caps: WebViewCapabilities
): { score: number; risk: 'low' | 'medium' | 'high'; recommendations: string[] } {
  const breaking = caps.missing.filter(
    f => CAPABILITY_MATRIX[f]?.severity === 'breaking'
  );
  const visual = caps.missing.filter(
    f => CAPABILITY_MATRIX[f]?.severity === 'visual'
  );
  
  const score = 100 - breaking.length * 15 - visual.length * 5;
  
  return {
    score,
    risk: score < 60 ? 'high' : score < 80 ? 'medium' : 'low',
    recommendations: [
      ...breaking.map(f => `${f}: 需 polyfill(${CAPABILITY_MATRIX[f]?.fallback})`),
      ...visual.map(f => `${f}: 需 CSS fallback(${CAPABILITY_MATRIX[f]?.cssFallback})`),
    ],
  };
}

**运行时防护**：

```typescript
// src/boot/webview-guard.ts
// 应用启动时的 WebView 能力检测与适配
export async function initWebViewGuard(): Promise<void> {
  const caps = detectWebViewCapabilities();
  const { risk, score, recommendations } = evaluateCapabilityScore(caps);
  
  // 1. 注入必要 polyfill
  const polyfillPromises = caps.missing
    .filter(f => CAPABILITY_MATRIX[f]?.fallback?.startsWith('polyfill'))
    .map(f => loadPolyfill(f));
  await Promise.all(polyfillPromises);
  
  // 2. 高风险环境 → 上报告警
  if (risk === 'high') {
    reportError('webview_capability_high_risk', {
      score,
      missing: caps.missing,
      webViewVersion: caps.webViewVersion,
      recommendations,
    });
  }
  
  // 3. 设置全局环境标识
  window.__WEBVIEW_CAPS__ = caps;
  window.__WEBVIEW_RISK__ = risk;
  
  // 4. 向原生层同步能力报告
  if (window.WebViewBridge) {
    window.WebViewBridge.send('capability-report', { score, risk, missing: caps.missing });
  }
  
  // 5. 低风险环境降级不注入大量 polyfill，减少包体积
  console.info(`[WebView] capability score: ${score}, risk: ${risk}`);
}
```

**架构防护**：
- **WebViewCapabilityDetector**：在应用初始化时（`beforeRender`），运行功能检测并将结果注入全局状态。业务代码只消费 `caps.supports.xxx`，不直接检测 `navigator.userAgent`。
- **功能降级矩阵**：将 `CAPABILITY_MATRIX` 作为架构文档的一部分在团队中维护。新增 API 使用时必须查矩阵确认最低支持版本并注册 fallback 策略。
- **Polyfill 按需注入系统**：不预打包所有 polyfill。根据 `detectWebViewCapabilities()` 的结果，动态 `import()` 缺失的 polyfill。减小 80% 设备的包体积浪费（大部分较新设备不需要 polyfill）。
- **双层 CSS 防护**：编写 CSS 时强制使用 fallback（`display: flex; gap: 16px` 写成 fallback + `@supports` 包裹），通过 Stylelint 规则自动化检查。

**流程防护**：
- 新 API 引入：Code Review 必须检查 `CAPABILITY_MATRIX` 中是否已注册该 API，severity 是否正确，fallback 策略是否合理。
- 发布清单：Android 5/6/7 的云真机冒烟（登录 → 主要页面 → 核心交互），最低覆盖 Android 7 WebView Chrome 51。
- RUM 监控：按 `webViewVersion` 维度分组统计 JS 错误率和白屏率，自动告警特定 WebView 版本的异常。

### 跨层影响链

```
上游影响 (消费者):
  - WebView 中白屏/JS 崩溃 → 用户无法使用 → 混合应用日活下降
  - 微信小程序 WebView 不兼容 → 社交裂变流量无法承接
  - 低端设备用户（Android 5~7 仍占 ~15% 市场份额）完全被排除

下游影响 (依赖):
  - 第三方 SDK（支付/地图/统计）也可能在 WebView 中崩溃
  - 前端路由懒加载使用 dynamic import() → 低版本 WebView 无法加载子页面
  - BFF/网关层返回的 BigInt ID → WebView 解析失败导致数据异常

部署影响:
  - Android System WebView 更新策略不可控 → 能力矩阵版本分布持续变化
  - CDN 缓存的 polyfill bundle 需要按版本分段 → 缓存策略比标准浏览器更复杂
  - SSR 中 WebView 检测代码需要在服务端安全执行 → `typeof window` 守卫

监控影响:
  - RUM: platform=AndroidWebView 单独分组，不合并到 Android Chrome
  - 关键指标: WebView 版本分布热力图 × JS 错误率 × 白屏率
  - 告警: 某 WebView 版本错误率突增 → 回滚或紧急 polyfill
  - E2E: 云真机矩阵覆盖主流厂商 × Android 版本
```

### 相关架构模式
- **pit-020: WKWebView postMessage 通信** — iOS 对应的 WebView 兼容问题。Android 和 iOS WebView 的跨端通信机制差异（`@JavascriptInterface` vs `WKScriptMessageHandler`）需要在统一的 Bridge 层抽象。
- **pit-021: iOS 软键盘弹出** — 同为 WebView 特有问题。Android WebView 在键盘弹出时调整 `window.innerHeight` 的行为与 iOS 相反，视口管理层需同时处理两种策略。
- **pit-023: Safe Area / 安全区域** — Android P+ 全面屏的 Safe Area 适配与 WebView 版本检测逻辑可共享功能检测机制。
- **Babel/TypeScript 编译策略** (`decisions.md`) — 编译目标应与 `CAPABILITY_MATRIX` 对齐。例如声明最低支持 Android 7 WebView (Chrome 51)，则 Babel target 应设为 Chrome 51 而非 "defaults"，避免误转译已可用的 API。
- **Polyfill 加载架构** — 按需 polyfill 系统需与路由懒加载、代码分割协同设计，避免 polyfill 在子页面加载后才生效（导致子页面首次打开报错）。

## 预防
1. **编译时**：Babel `@babel/preset-env` 设置 `targets: { chrome: '51', android: '7' }`（或项目声明的最低 WebView 版本），自动转译和注入 polyfill。
2. **运行时**：按需 polyfill 注入系统（`detect → import → init`），不预打包所有 polyfill。
3. **CSS**：所有新 CSS 属性使用 `@supports` 包裹并提供 fallback；通过 Stylelint 插件 `stylelint-plugin-supports` 自动化检查。
4. **原生端**：Android 团队使用统一的 `WebViewConfigurator` 初始化 WebView；在 CI 中检测 `WebSettings` 配置是否完整。
5. **测试**：Android 云真机矩阵（至少覆盖 API 24/29/34）× 主流厂商（华为/小米/OPPO）的 WebView 冒烟。
