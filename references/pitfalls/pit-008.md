# pit-008: Flexbox 嵌套宽度塌陷

## 环境
- 所有现代浏览器
- 常见于表格、卡片列表、面包屑等文本截断场景

## 症状
多层 Flex 嵌套下，内部元素的文本溢出省略（`text-overflow: ellipsis`）失效，或元素宽度变为 0 不显示。文本没有出现 "..." 而是溢出容器。

## 根因
Flexbox 子项的默认 `min-width` 是 `auto`（基于内容的最小宽度）。当外层 flex 容器收缩时，子元素不会缩到比内容更小，但如果内层也是 flex 且设置了 `overflow: hidden` 或 `text-overflow: ellipsis`，`min-width: auto` 会阻止收缩，导致溢出或布局错乱。

## 诊断方法

```bash
# 1. Chrome DevTools Computed 面板
# 选中溢出的元素 → Computed → 搜索 "min-width"
# 如果 min-width 显示为 auto 且值大于容器宽度 → 确认问题

# 2. 红色边框诊断
# 在控制台执行，给所有 flex 子元素加边框：
document.querySelectorAll('.flex-container > *').forEach(el => {
  el.style.outline = '2px solid red';
});
# 观察哪个元素的实际渲染宽度超出容器

# 3. 逐步注释排查
# 从最内层 flex 开始，逐个取消 display: flex
# 观察问题在哪一层消失 → 该层需要 min-width: 0

# 4. CSS 诊断脚本
# 在控制台执行，查找所有缺少 min-width: 0 的 flex 截断元素：
$$('[style*="text-overflow: ellipsis"], .truncate').forEach(el => {
  const style = getComputedStyle(el);
  if (style.minWidth === 'auto' && style.display.includes('flex')) {
    console.warn('需要 min-width: 0:', el);
  }
});
```

## 修复

```css
/* ❌ 问题：嵌套 flex 中 ellipsis 不生效 */
.breadcrumb {
  display: flex;
}
.breadcrumb-item {
  display: flex;
  align-items: center;
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
  /* min-width 默认为 auto → 不会收缩到内容以下 */
}

/* ✅ 修复：显式设置 min-width: 0 */
.breadcrumb-item {
  display: flex;
  align-items: center;
  min-width: 0; /* 关键：允许 flex 子元素收缩到比内容更小 */
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

```css
/* ❌ 更复杂的情况：多层嵌套 + 省略 */
.card-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
}
.card {
  display: flex;
  flex-direction: column;
}
.card-title {
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
  /* flex column 的子元素，宽度受限于父级，ellipsis 可能不触发 */
}

/* ✅ 修复：逐层设置 min-width: 0 或 overflow: hidden */
.card {
  display: flex;
  flex-direction: column;
  min-width: 0; /* 关键 */
  overflow: hidden;
}
.card-title {
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

```bash
# 缩小浏览器窗口，观察文本是否出现 "..." 省略
# 修复前：文本溢出容器或宽度为 0
# 修复后：文本正确截断并显示省略号

# 2. DevTools Computed 面板
# 选中截断元素 → 查看 min-width
# 修复后：min-width 应为 0

# 3. 响应式测试
# Chrome DevTools → 设备工具栏 → 选择窄屏设备（如 iPhone SE）
# 检查卡片/面包屑文本是否正确截断

# 4. 多浏览器验证
# Chrome、Firefox、Safari 分别测试
# Safari 对 min-width: 0 的行为有时与 Chrome 不同
# 如果仍不生效，尝试同时加 overflow: hidden
```

## 原理
Flexbox 规范规定 flex 子元素的 `min-width` 默认值为 `auto`，表示"不小于内容的最小尺寸"。这是为了防止内容意外被截断。但对于需要文本截断的场景，这个保护机制反而成了障碍。`min-width: 0` 告诉浏览器"此元素可以收缩到零"，从而允许 `overflow: hidden` 和 `text-overflow: ellipsis` 生效。

Grid 布局同理：`min-width: auto` 会阻止 grid 子项缩小到内容以下。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Flexbox 嵌套塌陷的架构根因是**截断策略与布局容器之间缺少显式契约**：
- **截断是布局策略而非样式细节**：文本截断（ellipsis）是一种布局策略决策——"当空间不足时优先保持布局稳定，截断文本"。但这个决策没有被上升为布局系统的显式规则，而是作为零散的 CSS 属性分散在组件样式中，导致嵌套时不生效。
- **缺少"可收缩容器"抽象**：应用需要一种"自适应截断容器"——即声明一个容器"当空间不足时可以收缩到 0，优先截断内容而非破坏外部布局"。Flexbox 的 `min-width: 0` 是实现手段，但架构层面应将其封装为可复用的布局原语（Primitive）。
- **CSS 布局与组件边界的错配**：组件的 DOM 结构（由 React 组件树决定）与 CSS 布局的 Flex 嵌套不一定对齐。当组件开发者不感知上层容器的布局约束时，就会出现"内层需要截断但外层不允许收缩"的矛盾。架构需要建立"布局约束向下传递"的机制。

### 系统级预防策略

**静态防护**：
```css
/* 设计系统：封装截断原语 */
.truncate {
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
  min-width: 0; /* 作为截断容器的必要前提 */
}

/* 将 flex 子元素的 min-width: 0 纳入 CSS reset */
*, *::before, *::after {
  /* 默认允许 flex/grid 子元素收缩 */
  min-width: 0;
  min-height: 0;
}
```

```js
// ESLint/tsx 检查：text-overflow 必须配合 overflow: hidden
// stylelint 规则
rules: {
  'declaration-block-no-duplicate-properties': true,
  // 自定义插件：如果使用了 text-overflow: ellipsis，必须同时有 overflow: hidden
}
```

**CI 防护**：
```bash
# CI 视觉回归测试：窄视口下文本截断验证
# Playwright 测试
test('卡片文字在窄视口下正确截断', async ({ page }) => {
  await page.setViewportSize({ width: 320, height: 800 });
  await page.goto('/cards');
  
  const cards = await page.$$('.card-title');
  for (const card of cards) {
    const isClipped = await card.evaluate(el => el.scrollWidth > el.clientWidth);
    const hasEllipsis = await card.evaluate(el => {
      const style = getComputedStyle(el);
      return style.textOverflow === 'ellipsis' && style.overflow === 'hidden';
    });
    // 如果内容超出宽度，必须有省略号效果
    expect(isClipped ? hasEllipsis : true).toBeTruthy();
  }
});
```

**架构防护**：
- **布局原语库**：建立一组经过验证的布局原语（Primitives），内部处理所有 Flex/Grid 截断细节：
```tsx
// 架构层原语，开发者不需要知道 min-width: 0
<FlexRow>
  <FlexCell shrink>  {/* 内部设置 min-width: 0 + overflow: hidden */}
    <Truncate>长时间不换行的文本内容</Truncate>
  </FlexCell>
  <FlexCell fixed={120}>固定操作区</FlexCell>
</FlexRow>
```
- **布局约束声明**：在组件接口上声明"我需要截断支持"或"我需要最小宽度保证"，由布局引擎自动注入正确的 CSS。

**流程防护**：
- Code Review 检查项："所有 ellipsis 截断的元素是否在 Flex/Grid 子元素中？如果是，父级和自身是否设置了 min-width: 0？"
- UI 审查：在每个响应式断点检查文本截断是否正确。

### 跨层影响链

```
上游影响 (消费者):
  - 父级 Grid/Flex 容器的列宽计算基于错误的子元素尺寸
  - max-width 限制失效 → 容器被内容撑爆 → 页面横向滚动

下游影响 (依赖):
  - CSS ellipsis 失效导致文本溢出 → 本文周边的元素布局被破坏
  - 面包屑/标签等内联元素溢出导致行高异常
  - Shadow DOM 内的组件无法被外部 CSS 修正

部署影响:
  - CDN 缓存了旧版 CSS → 修复后部分用户仍看到溢出（缓存失效未处理）
  - 新增语言（如德语/俄语长单词）后触发新的截断问题

监控影响:
  - RUM: 监控页面横向滚动事件（用户意外触发了 x 轴滚动）
  - 视觉回归测试: Percy/Chromatic 在各种语言/视口宽度下截图
  - CSS 审计: 统计使用了 text-overflow 但缺少 min-width: 0 的元素
```

### 相关架构模式
- **CSS 方案选型** (`decisions.md` 第三节) — 无论使用 Tailwind/CSS Modules/Vanilla Extract，"截断"都应作为设计系统的基础 Token/Utility 存在（如 Tailwind 的 `.truncate`）。
- **反模式: CSS-in-JS 运行时性能陷阱** (`patterns.md` 第六节) — 动态生成截断样式同样适用于此反模式，应使用静态工具类而非运行时计算。
- **虚拟列表** (`patterns.md` 第四节) — 虚拟列表中每一行的截断同样受此问题影响，行高计算依赖内容宽度，截断失效会导致行高计算错误。

## 预防

1. **Flex 子元素需要文本截断 → 设置 `min-width: 0`**。
2. **Grid 子元素同理** → `min-width: 0` + `overflow: hidden`。
3. **ESLint**：可以考虑自定义规则检测 `overflow: hidden` + `text-overflow: ellipsis` 但缺少 `min-width: 0` 的 flex 子元素。
4. **组件库封装**：将文本截断抽取为 `<Truncate>` 组件，内部自动处理 `min-width: 0`。

## 回退

```bash
# 如果 min-width: 0 导致内容完全消失（宽度为 0）
# 方案A：使用 min-width: fit-content 作为折中

# 方案B：在更外层设置固定宽度
# 给父容器一个明确的宽度值或 flex-basis

# 方案C：使用 CSS clamp 限制最小/最大宽度

```
