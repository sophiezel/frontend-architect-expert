# pit-020: map 遗漏 return 导致 undefined 数组

## 环境
- JavaScript ES6+
- 任何使用 Array.map 的场景

## 症状
使用 `Array.map` 处理后得到的数组元素全是 `undefined`，或者部分元素为 `undefined`。

预期是转换后的值，实际拿到 `[undefined, undefined, ...]`。

## 根因
`Array.map` 中箭头函数使用了花括号 `{}` 但忘记写 `return` 语句。花括号会创建一个块作用域，没有 `return` 则隐式返回 `undefined`。

另一种情况是 `if` 分支有 `return` 但 `else` 分支没有，部分路径返回 `undefined`。

## 诊断方法

```bash
# 在 map 回调中打印每个元素
items.map((item, i) => {
  const result = processItem(item);
  console.log(`[${i}] input:`, item, 'output:', result);
  return result; // 确认 return 存在
});

# 使用 ESLint 检测
npx eslint --rule 'array-callback-return: error' src/
```

## 修复

```diff
// ❌ 花括号没有 return → 全是 undefined
const names = users.map(user => {
-  user.firstName + ' ' + user.lastName;
});

// ✅ 方案1: 箭头函数隐式返回（无花括号）
+ const names = users.map(user => user.firstName + ' ' + user.lastName);

// ✅ 方案2: 花括号 + 显式 return
+ const names = users.map(user => {
+   return user.firstName + ' ' + user.lastName;
+ });

// ✅ 方案3: 有 if 分支时要覆盖所有路径
const statusTexts = orders.map(order => {
+  if (order.status === 'completed') return '已完成';
+  if (order.status === 'pending') return '待处理';
+  return '未知'; // ← 必须有兜底 return
});
```

## 验证

```bash
# 运行单元测试确认 map 结果正确
npx jest --testPathPattern="map" --verbose
# 或检查 ESLint
npx eslint --rule 'array-callback-return: error' src/**/*.{ts,tsx,js,jsx}
# 期望: 0 errors
```

## 原理
`Array.map` 对每个元素执行回调，用返回值构建新数组。箭头函数有两种写法：
- `(x) => expr` — 隐式返回 expr
- `(x) => { statements }` — 必须显式 return

JavaScript 函数没有 return 语句时默认返回 `undefined`，所以有花括号就必须有 return。

## 架构视角 (Architecture Perspective)

### 架构层面根因
此 pitfall 与 pit-019 同源（map 缺少 return），但突出了**分支返回不完整**这个更隐蔽的场景。

架构层面根因包括：
- **代码路径完整性未在类型系统表达**：`if (condition) return A;` 这种代码只覆盖了一条路径，但 TypeScript 的 `noImplicitReturns` 只在函数声明的返回类型存在时才检查。如果回调没有声明返回类型（`items.map((item) => { ... })`），编译器和 ESLint 都难以检测"部分路径无返回"。
- **map 回调的语义被弱化**：`map` 被当作"循环 + 副作用"使用（如 `items.map(item => { processItem(item); })`），而不是"纯转换函数"。这是对 `map` 语义的滥用——`map` 是为转换设计的，副作用应用 `forEach`。架构应区分"转换"和"副作用"的操作，禁止在 `map` 中执行副作用。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 与 pit-019 相同的规则集，额外增强
rules: {
  'array-callback-return': ['error', { 
    allowImplicit: true,
    checkForEach: false
  }],
  // 禁止在 map 中不返回值（即禁止把 map 当 forEach 用）
  'no-restricted-syntax': ['error', {
    selector: 'CallExpression[callee.property.name="map"] ArrowFunctionExpression[body.type="BlockStatement"]',
    message: 'map 回调中使用了块语句，请确认所有分支都有 return 语句或改用隐式返回'
  }]
}

// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitReturns": true,
    // 额外：对数组回调的隐式类型推断更严格
    "strictFunctionTypes": true
  }
}
```

**CI 防护**：
```bash
# CI: 自动化代码审查检查 map 回调的分支完整性
# 检查所有 .map() 回调中的 if 语句数量与 return 语句数量
# 如果 if 数量 > return 数量 → 可能存在未覆盖的分支
```

**架构防护**：
- **禁止在 map 中执行副作用**：作为编码规范，`map` 仅用于数据转换，副作用操作（如 `console.log`、状态修改）使用 `forEach`。
- **引入 `match` / `when` 等模式匹配抽象**，替代 `if/else` 链：
```typescript
const statusTexts = orders.map(order => 
  match(order.status, {
    'completed': '已完成',
    'pending': '待处理',
    '_': '未知'  // 必须提供默认分支
  })
);
```

**流程防护**：
- Code Review 检查项："map 回调中是否有条件分支（if/else/switch）？每个分支是否都有 return？"

### 跨层影响链

与你 pit-019 相同，额外增加：
```
下游影响 (依赖):
  - 部分路径返回 undefined 的 map → .filter(Boolean) 可能过滤掉合法值
  - 表格/列表的分列计算中，个别列显示空白 → 数据完整性被质疑
  
监控影响:
  - 按路径覆盖率：map 回调中未覆盖的代码分支
  - Istanbul 覆盖率报告中 map 回调未覆盖的分支 → 潜在 undefined 风险
```

### 相关架构模式
与 pit-019 完全相同，额外引用：
- **TDD (Test-Driven Development)** — 当 map 回调有多个分支时，TDD 流程应要求为每个分支编写测试用例，覆盖所有条件路径。

## 预防
1. ESLint 规则 `array-callback-return: error`
2. 优先使用隐式返回（无花括号）的箭头函数
3. Code Review 检查 map 回调中是否有花括号无 return
4. TypeScript 中为 map 回调声明返回类型: `users.map((u): string => ...)`
