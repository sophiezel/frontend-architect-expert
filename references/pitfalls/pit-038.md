# pit-038: Express/Koa 中间件顺序陷阱

## 环境
- Express 4.x / Koa 2.x
- Node.js 16+
- 任何使用了中间件链的后端服务

## 症状
API 返回空响应（客户端一直 pending 直到超时）；错误被静默吞掉，日志中没有任何 stack trace；"响应已发送后不能设置 header"的 `ERR_HTTP_HEADERS_SENT` 错误频繁出现。更难排查的是：中间件 A 给请求附加了 `req.user`，中间件 B 却读不到——因为 B 被放在了 A 前面。

典型场景：
- 错误处理中间件 `(err, req, res, next)` 没有放在所有路由之后 → 抛出的异常无人捕获
- `async` 中间件没有包裹 try-catch → rejected Promise 被 Express 4 无声吞掉
- 在 `next()` 调用之后还写了 `res.json()` → 一个请求被响应两次

## 根因
Express/Koa 的中间件执行模型是洋葱模型（onion model）：请求从外向内穿过所有中间件，到达路由处理函数，然后响应从内向外返回。这个模型的正确性依赖两个前提：
1. 每个中间件要么响应请求，要么调用 `next()` 将控制权交给下一个
2. `next()` 之后的代码在响应返回后执行，不能再修改响应

常见违反：
- `next()` 之后写响应逻辑（"已经响应了"）
- async 中间件抛出的异常没有被 Express 4 自动捕获（Express 4 不处理 Promise rejection）
- 中间件排列顺序不满足依赖关系（认证中间件放在日志中间件后面）

## 修复

```diff
// ❌ 问题 1: 错误处理中间件位置错误
const app = express();
-app.use(errorHandler);           // ← 错误：放在路由之前
-app.use(authenticate);
+app.use(authenticate);
 app.use('/api', apiRoutes);
+app.use(errorHandler);           // ✅ 在所有路由之后

// ❌ 问题 2: async 中间件未捕获错误 (Express 4)
-app.get('/users/:id', async (req, res) => {
-  const user = await db.findUser(req.params.id); // 若 reject → 进程 hang
-  res.json(user);
-});

+// ✅ 方案 A: 包装异步处理器
+function asyncHandler(fn: RequestHandler): RequestHandler {
+  return (req, res, next) => {
+    Promise.resolve(fn(req, res, next)).catch(next);
+  };
+}
+app.get('/users/:id', asyncHandler(async (req, res) => {
+  const user = await db.findUser(req.params.id);
+  res.json(user);
+}));

+// ✅ 方案 B: 使用 express-async-errors (Express 4)
+import 'express-async-errors'; // 在 app.ts 第一行导入
+// 这使得 Express 4 自动捕获 async 中间件的异常

// ❌ 问题 3: next() 之后写响应
app.use((req, res, next) => {
  console.log(`${req.method} ${req.path}`);
  next();           // 交出控制权
-  res.status(500); // ← 错误：响应已经在路由中发送过了
});

+app.use((req, res, next) => {
+  const start = Date.now();
+  res.on('finish', () => {
+    console.log(`${req.method} ${req.path} ${res.statusCode} ${Date.now() - start}ms`);
+  });
+  next(); // next() 之后只监听 finish 事件，不操作 res
+});

// ❌ 问题 4: 依赖顺序错误
-app.use(logRequest);     // 先日志（但需要 user 信息）
-app.use(authMiddleware); // 后认证（设置 req.user）

+app.use(authMiddleware); // 先认证
+app.use(logRequest);     // 后日志——此时 req.user 已可用
```

## 原理
Express 中间件的洋葱模型可以直观理解：
```
          ┌──────────────────────────┐
Request → │ mid1 → mid2 → route → mid2 → mid1 │ → Response
          └──────────────────────────┘
```

`next()` 相当于把控制权交给内层中间件。调用 `next()` 后当前中间件的代码暂停执行，等内层全部完成后才继续执行 `next()` 之后的代码。

Express 4 的同步错误处理器可以捕获 `throw`，但 async 函数的 rejected Promise 不是 throw——它是一个微任务中的 rejection，Express 4 的 try-catch 框架无法感知。这就是 `express-async-errors` 的工作：它 monkey-patch Express 的 `Layer.handle` 方法，检测返回值是否为 Promise。

## 架构视角 (Architecture Perspective)

### 架构层面根因
中间件顺序陷阱的架构根因是**中间件编排缺少声明式依赖管理**：
- **Express/Koa 的中间件是"命令式组装"**：顺序由 `app.use()` 的书写顺序决定，没有任何机制表达"B 依赖 A"的语义。当中间件数量增长到几十个时，开发者只能靠"注释"或"命名约定"来管理依赖顺序——这是无法扩展的。
- **错误处理是二等公民**：Express 将错误处理中间件定义为四参数函数 `(err, req, res, next)`，与普通中间件使用不同的函数签名，但没有在 API 层面强制"错误处理中间件必须在最后"，也没有类型系统提示。
- **"中间件"概念的边界被滥用**：认证、日志、CORS、压缩、静态文件服务、Body Parser 都被称为"中间件"，但它们有不同的依赖要求和执行约束。架构缺少对中间件类型的分类和编排规则。

### 系统级预防策略

**静态检查**：
```bash
# TypeScript 类型约束：强制 asyncHandler 包装
// types/express.d.ts
declare global {
  namespace Express {
    interface RequestHandler {
      // 覆盖 RequestHandler 使其默认为 async-safe
    }
  }
}

# ESLint 规则
rules: {
  '@typescript-eslint/no-misused-promises': ['error', {
    checksVoidReturn: { arguments: false, attributes: false },
  }],
  '@typescript-eslint/require-await': 'off', // 允许非 async 的 asyncHandler
}
```

**CI 门禁**：

**架构约束**：
- 使用声明式中间件注册系统（而非裸 `app.use()`）：
```typescript
// src/middleware/registry.ts
interface MiddlewareDef {
  name: string;
  handler: RequestHandler;
  after?: string[];   // 显式声明依赖
  before?: string[];
  type: 'auth' | 'log' | 'transform' | 'error' | 'static';
}

const registry: MiddlewareDef[] = [
  { name: 'bodyParser', handler: bodyParser.json(), type: 'transform' },
  { name: 'auth', handler: authMiddleware, type: 'auth' },
  { name: 'requestLog', handler: requestLogger, after: ['auth'], type: 'log' },
  { name: 'errorHandler', handler: errorHandler, type: 'error' },
];
// 拓扑排序自动解析顺序，并在 CI 中验证无循环依赖
```
- 禁止在 Express 4 中使用裸 async 路由处理函数（通过 ESLint）
- 强制每个 API 路由处理函数使用 `asyncHandler` 包装或返回 Promise

**流程约束**：
- 新增中间件必须在 registry 中声明依赖关系
- Code Review 检查："该中间件在洋葱模型中的位置是否正确？是否依赖其他中间件的副作用？"

### 跨层影响链

```
上游影响 (API 消费者):
  - 请求 hung → 客户端超时 → 用户体验差 → 重试加剧服务端压力
  - 错误被吞 → 前端拿到空响应或 200 空 body → 数据不一致

下游影响 (数据库/微服务):
  - 认证中间件位置错误 → 未认证请求打到数据库 → 安全漏洞
  - 响应重复发送 → 数据库事务被提交两次

部署影响:
  - ERR_HTTP_HEADERS_SENT 在生产环境大量出现 → 内存泄漏（连接未正确关闭）
  - Node.js 进程 hung → PM2 无法重启（进程未退出）→ 需要手动 kill

监控影响:
  - APM: 监控 async_hooks 中未完成的 Promise 数量
  - RUM: API 响应时间的 p99（hung 请求持续拉高）
  - 自定义指标: express middleware 每个环节的耗时分布
```

### 相关架构模式
- **Chain of Responsibility 模式** (`patterns.md` 第十四节) — Express 中间件的本质是责任链，但缺少声明式顺序管理。Java 的 Servlet Filter 可以通过 `@Order` 注解声明顺序。
- **洋葱模型与 AOP** (`decisions.md` 第十一节) — 中间件是面向切面编程（AOP）在 Node.js 的体现，但切面的织入顺序需要显式管理。

## 预防
1. 用 `express-async-errors` 或 Koa 原生 async 支持消除 async 错误吞掉的问题
2. 错误处理中间件始终在 `app.ts` 最后注册（在所有路由和普通中间件之后）
3. 中间件注册使用拓扑排序而非手写顺序
4. `next()` 之后绝对不操作 `res`（除监听 `finish` 事件外）
