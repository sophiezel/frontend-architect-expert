# pit-024: Bundle 过大 / JS Bundle Size 爆炸

## 环境
- Webpack 5 / Vite / Rspack 构建的项目
- SPA 或 MPA 前端应用
- 未启用代码分割或懒加载的项目

## 症状
1. **首屏加载时间 > 5 秒**：Lighthouse 报告 "Reduce unused JavaScript"，用户白屏等待。
2. **Lighthouse Performance 评分 < 30**："Avoid enormous network payloads" 告警，JS 传输量 > 1 MB。
3. **低端设备/3G 网络下页面不可用**：首屏 JS 解析阻塞主线程，用户流失。
4. **CI 构建时间异常增长**：Bundle 体积逐月膨胀，无预警。
5. **node_modules 膨胀**：全量引入 moment.js（带 70+ 个 locale）、lodash（全量而非按需）、Ant Design 全量引入。

## 根因
未做代码分割/懒加载、node_modules 巨型依赖未 tree-shake、moment.js 等带 locale 的库全量引入。

核心机制：Webpack 将所有代码打包为单一 bundle（无 splitChunks），或代码虽分割但路由未配置 `React.lazy`，导致首屏加载了所有页面代码。bundled moment.js 包含所有 locale 文件即使只用到 `zh-cn`。

## 诊断方法

```bash
# 1. webpack-bundle-analyzer 审计
npm install --save-dev webpack-bundle-analyzer
npx webpack-bundle-analyzer dist/stats.json

# 2. 检查大依赖 (moment/lodash)
npx depcheck  # 找到未使用的依赖
npx source-map-explorer dist/**/*.js  # 按模块查看体积占比

# 3. Lighthouse 审计
npx lighthouse http://localhost:3000 --view --preset=desktop

# 4. 检查未 tree-shake 的引入方式
grep -rn "import \* as" src/ --include="*.ts" --include="*.tsx"
grep -rn "from 'lodash'" src/ --include="*.ts"  # 应改为 lodash/get
grep -rn "from 'moment'" src/ --include="*.ts"   # 应改为 date-fns
```

## 修复

```diff
// ❌ 问题代码（路由篇）：所有页面同步导入
- import Home from './pages/Home';
- import About from './pages/About';
- import Settings from './pages/Settings';
+ import { lazy, Suspense } from 'react';
+ const Home = lazy(() => import('./pages/Home'));
+ const About = lazy(() => import('./pages/About'));
+ const Settings = lazy(() => import('./pages/Settings'));

// ❌ 全量引入 moment.js
- import moment from 'moment';
- moment(date).format('YYYY-MM-DD');

// ✅ 替换为 date-fns (tree-shakeable)
+ import { format, parseISO } from 'date-fns';
+ format(parseISO(date), 'yyyy-MM-dd');

// ❌ 全量引入 lodash
- import _ from 'lodash';
- _.get(user, 'profile.name');

// ✅ 按需引入
+ import get from 'lodash/get';
+ get(user, 'profile.name');
```

```js
// ✅ Webpack splitChunks 配置
// webpack.config.js
module.exports = {
  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendor',
          priority: 10,
          chunks: 'initial',
        },
        common: {
          minChunks: 2,
          priority: 5,
          reuseExistingChunk: true,
        },
      },
    },
  },
  performance: {
    maxAssetSize: 244 * 1024,    // 单个 chunk < 244 KB
    maxEntrypointSize: 500 * 1024, // 入口 < 500 KB
    hints: 'error',               // CI 中设为 error 阻断
  },
};
```

## 原理
Webpack 的 Tree Shaking 依赖 ES Module 静态分析——`import { debounce } from 'lodash'` 只引入 debounce 及其依赖，而 `import _ from 'lodash'` 引入整个 lodash 对象无法静态分析。moment.js 的 locale 通过副作用（`moment.locale('zh-cn')`）注册，即使未直接 import 也会被打包。

`React.lazy` + `<Suspense>` 实现路由级代码分割，Webpack 为每个 `import()` 创建独立 chunk，浏览器按需加载。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Bundle 过大的架构根因是**缺少 Bundle 预算 CI 门禁机制**：

- **体积增长是渐进的、不可见的**：每次 PR 增加 2 KB 未引起关注，但 6 个月后累积膨胀到 2 MB。缺少 CI 中的体积预算门禁使得膨胀无法被及时发现和阻断。
- **代码分割策略未在架构层面声明**：哪些模块在首屏、哪些懒加载、哪些预加载，应由架构文档明确声明，而非各开发者自行决定。缺少"加载策略架构"导致任意模块都可能被打入首屏 bundle。
- **第三方依赖引入无审核机制**：`npm install moment` 未被 Code Review 拦截，因为缺少依赖体积评估流程（bundlephobia 预检）。

### 系统级预防策略

**静态防护**：
```js
// .eslintrc.js — 禁止已知体积陷阱的引入方式
rules: {
  'no-restricted-imports': ['error', {
    paths: [
      { name: 'moment', message: '请使用 date-fns 替代 moment.js' },
      { name: 'lodash', message: '请使用 lodash/get 按需引入' },
    ],
    patterns: ['lodash/*', '!lodash/get', '!lodash/debounce'],
  }],
}
```

**CI 防护**：
```bash
# CI 中 webpack performance hints 设为 error
# webpack: performance: { hints: 'error', maxEntrypointSize: 500000 }

# bundlewatch / bundlesize 体积门禁
npx bundlewatch --config .bundlewatch.config.js

# 依赖体积预检 (PR 中安装新依赖时触发)
npx bundlephobia-cli check --package=package.json
```

**架构防护**：
- **加载策略架构**：明确定义三层加载策略
  - Critical（首屏同步）：核心布局 + 首屏数据
  - Deferred（路由懒加载）：`React.lazy` 页面
  - Prefetch（空闲预加载）：高频访问的次屏
- **依赖铁律**：新增 npm 依赖必须通过 bundlephobia 体积评估，>50KB 需要架构审批。

**流程防护**：
- PR 模板：`## Bundle Impact` 字段，标注此 PR 对首屏 JS 体积的影响（对比 base branch）。

### 跨层影响链

```
上游影响 (消费者):
  - 首次访问用户白屏 > 5s → 跳出率 +53%（Google 数据）
  - 低端设备解析长任务阻塞 → UI 无响应 → 用户以为页面挂了
  - Google Core Web Vitals 劣化 → SEO 排名下降

下游影响 (依赖):
  - CDN 带宽成本线性增长
  - Service Worker 缓存体积膨胀 → 更新缓慢
  - CI 构建时间变长（大的代码库 + 大的依赖图）

部署影响:
  - 大 chunk 部署失败率高（网络中断风险）
  - 灰度发布时新旧 chunk 不一致概率增加
  - 回滚时缓存失效范围扩大

监控影响:
  - RUM: LCP / FID / TBT 持续劣化
  - Lighthouse CI 断言失败 → 阻断部署
  - 按页面维度监控 JS 传输量 trend
```

### 相关架构模式
- **性能预算模式** (`patterns.md` 第四节) — webpack `performance` + Lighthouse CI assertions 构成双层门禁。
- **加载策略架构** — Critical / Deferred / Prefetch 三层分离。
- **构建工具选型** (`decisions.md` 第四节) — Vite 的按需编译 + Rollup tree-shaking vs Webpack splitChunks 各自优势。

## 预防
1. 路由级代码分割：所有页面组件使用 `React.lazy(() => import(...))`。
2. webpack `performance.maxAssetSize` + `maxEntrypointSize` 设置合理阈值（250KB / 500KB）。
3. 禁止全量引入 moment 和 lodash，使用 `date-fns` 和按需引入替代。
4. CI 集成 Lighthouse CI 或 bundlesize 门禁阻断超预算 PR。

## 回退
```bash
# 如果懒加载引入导致页面切换闪烁
# 方案A: 回退单个页面的 lazy 加载
git checkout -- src/pages/Checkout.tsx  # 恢复为同步 import

# 方案B: 添加 Suspense fallback 优化体验
<Suspense fallback={<PageSkeleton />}>
  <Checkout />
</Suspense>

# 方案C: 使用 prefetch 而非 lazy 缓解闪烁
const Checkout = lazy(() => import(
  /* webpackPrefetch: true */ './pages/Checkout'
));
```
