# pit-062: TypeScript any 泄漏 — 类型安全被 any 逐步侵蚀

## 环境
- TypeScript 3.0+
- 任何使用 TypeScript 的中大型项目
- ESLint + @typescript-eslint

## 症状
原本经过严格类型标注的模块，在添加了某个功能后，IDE 的类型提示逐渐变成 `any`。编译不报错但运行时偶现 `undefined is not a function` / `TypeError`。类型覆盖率从 90% 逐步下滑到 60% 以下，团队对类型系统失去信心，"加 any 能跑就行"成为惯性。

## 根因
`any` 类型在 TypeScript 中具有**传染性**。一个返回 `any` 的函数，其调用者推导为 `any`，调用者的调用者继续 `any`，形成"any 黑洞"。常见的 any 泄漏路径：
1. `JSON.parse()` 返回 `any`
2. `fetch(...).then(res => res.json())` 返回 `Promise<any>`
3. `document.getElementById()` 返回 `HTMLElement | null`——无天然类型参数
4. 第三方库缺失类型声明，`declare module 'foo'` 隐式 `any`
5. 代码审查不严格，"先 any 后改"变成"先 any 不改"

## 诊断方法

```bash
# 1. 统计项目中 any 的数量和分布
npx type-coverage --detail
# 输出: "type coverage 67.23%" — 显示类型覆盖率
# --detail 显示每个文件覆盖率

# 2. 找出 any 最多的文件（排序）
npx type-coverage --detail 2>&1 | grep -E "^[0-9]" | sort -t'%' -k1 -n | head -20

# 3. ESLint 扫描 any 显式使用
grep -rn ": any" src/ | head -30
grep -rn "as any" src/ | head -30

# 4. TypeScript 编译器报告隐式 any
# tsconfig.json: "noImplicitAny": true
# 编译时找出所有未标注类型的参数
npx tsc --noImplicitAny --noEmit 2>&1 | grep "implicitly has an 'any' type"

# 5. 追踪 any 传染路径
# 在调用链上添加临时类型检查
function traceType<T>(val: T, label: string): T {
  console.log(label, typeof val, val);
  return val;
}
// 如果 T 被推导为 any → 链路上游存在 any 泄漏
```

## 修复

```diff
// ❌ 问题1：JSON.parse 返回 any 传染
- const config = JSON.parse(localStorage.getItem('config') || '{}');
- config.theme; // any
- config.timeout; // any

+ interface AppConfig {
+   theme: 'light' | 'dark';
+   timeout: number;
+ }
+ function parseConfig(raw: string | null): AppConfig {
+   if (!raw) return { theme: 'light', timeout: 5000 };
+   const parsed: unknown = JSON.parse(raw);
+   if (typeof parsed === 'object' && parsed !== null && 'theme' in parsed) {
+     return parsed as AppConfig; // 显式类型断言 + 运行时校验
+   }
+   throw new Error('Invalid config');
+ }
+ const config = parseConfig(localStorage.getItem('config'));

// ❌ 问题2：fetch 响应 any 传染
- async function getUsers() {
-   const res = await fetch('/api/users');
-   return res.json(); // Promise<any>
- }

+ interface User {
+   id: string;
+   name: string;
+ }
+ async function getUsers(): Promise<User[]> {
+   const res = await fetch('/api/users');
+   if (!res.ok) throw new Error(`HTTP ${res.status}`);
+   const data: unknown = await res.json();
+   // 运行时类型守卫
+   if (Array.isArray(data) && data.every(isUser)) return data;
+   throw new Error('Invalid response shape');
+ }
+ function isUser(v: unknown): v is User {
+   return typeof v === 'object' && v !== null && 'id' in v && 'name' in v;
+ }

// ❌ 问题3：第三方库缺少类型
- declare module 'legacy-lib'; // 隐式 any

+ declare module 'legacy-lib' {
+   export function init(config: { apiKey: string }): void;
+   export function track(event: string, data: Record<string, unknown>): Promise<void>;
+ }
```

## 原理
TypeScript 的类型系统是一个有向图：每个表达式节点有类型，节点之间通过类型推导连接。`any` 是此图中的"传染节点"——它同时兼容所有类型（赋值目标）并接受所有操作（调用源）。当图中一个节点变成 `any`，它下游所有依赖它的节点丢失具体类型信息。这解释了为什么单个 `as any` 会污染整个模块。

`unknown` 是 `any` 的类型安全替代品。`unknown` 仅兼容 `unknown | any`，不接受任何操作（除非先收窄）。因此 `JSON.parse()` 返回 `unknown` 而非 `any` 才是类型安全的。

## 架构视角 (Architecture Perspective)

### 架构层面根因
any 泄漏表面上是"偷懒没写类型"，架构层面根因是**类型边界未分层定义**：
- **I/O 边界无契约**：API 响应、localStorage、URL 参数等外部输入进入系统时没有经过类型解码层（Decoder），导致 `unknown`/`any` 跨层传播。
- **缺少架构层面的类型守卫层**：没有在数据流入端（API client, storage adapter）强制解码，导致类型验证被推迟到消费组件，而组件开发者使用 `any` 绕过。
- **类型声明与运行时不一致**：`.d.ts` 声明了 `User` 类型，但后端实际返回的字段名不同，前端为了"快速修复"直接 `as any`，破坏了类型系统信任基础。

### 系统级预防策略

**静态防护**：
```js
// .eslintrc.js — 零容忍 any 规则
rules: {
  '@typescript-eslint/no-explicit-any': 'error',
  '@typescript-eslint/no-unsafe-assignment': 'error',
  '@typescript-eslint/no-unsafe-member-access': 'error',
  '@typescript-eslint/no-unsafe-call': 'error',
  '@typescript-eslint/no-unsafe-return': 'error',
}

// 可选：允许 any 但要求写注释解释原因
'@typescript-eslint/no-explicit-any': ['warn', { fixToUnknown: true, ignoreRestArgs: true }]
```

**CI 防护**：
```bash
# 合并前检查：类型覆盖率不得下降
# .github/workflows/type-check.yml
- name: Type Coverage
  run: |
    npx type-coverage --at-least 95
    # 覆盖率低于 95% 则 CI 失败

# 检测新增 .d.ts 中的隐式 any
- name: Check declaration files
  run: |
    find dist -name '*.d.ts' -exec grep -l ': any' {} \; | wc -l
    # 输出不为 0 则失败
```

**架构防护**：
- 引入 **I/O Decoder 层**（io-ts / zod）：所有外部数据入口强制运行时验证 + 类型推导。
```typescript
import { z } from 'zod';
const UserSchema = z.object({ id: z.string(), name: z.string() });
type User = z.infer<typeof UserSchema>;
// API 层必须: const data = UserSchema.parse(await res.json());
// 禁止在其他层裸用 JSON.parse
```
- **模块边界规则**：`dependency-cruiser` 规则：`src/components/**` 不允许直接调用 `JSON.parse`，必须通过 `src/decoders/**`。

**流程防护**：
- Code Review 清单项："此 PR 是否新增了 `any`？如果是，注释说明为什么必须使用 any。"
- 每月类型覆盖率报告作为工程健康指标纳入团队周报。

### 跨层影响链

```
上游影响:
  - IDE 智能提示退化 → 开发效率下降 → 团队抱怨 TypeScript "拖慢开发"
  - 重构时无法使用 rename symbol / find all references → 回归 bug 增多

下游影响:
  - 运行时错误从编译时捕获退化到生产环境暴露
  - Sentry 中 TypeError 占比上升 → oncall 负担加重
  - 类型安全的 discriminator union 等高级模式无法应用

服务影响:
  - 类型覆盖率下降 → 新人 onboarding 成本上升
  - "TypeScript 项目"退化为 "AnyScript 项目"
  - 最终团队可能放弃 TypeScript（误解为 TypeScript 无用）
```

### 相关架构模式
- **I/O Decoder 模式** (`patterns.md`) — 外部数据边界运行时验证
- **Branded Types** (`typescript-engineering.md` 第1.4节) — 防止基本类型混淆
- **Exhaustiveness Checking** (`typescript-engineering.md` 第2.4节) — 用 never 替代 any 做兜底

## 预防

1. **开启 strict 模式**：`tsconfig` 中 `strict: true` 是最低门槛。
2. **禁止生产代码出现 any**：`@typescript-eslint/no-explicit-any: error` 配合 CI 阻断。
3. **用 unknown 替代 any**：`JSON.parse` 返回标注为 `unknown`，调用处必须收窄。
4. **引入运行时解码库**：zod / io-ts / typebox 作为数据入口的强制关卡。
5. **Code Review 红线**："新增 any 必须有注释解释原因且经过审批"。
