# pit-051: setData 性能灾难 — 高频调用导致页面卡顿/渲染延迟

## 环境
- 微信小程序（基础库 1.0+）
- 涉及 onPageScroll / onPullDownRefresh / 定时器回调等高频场景
- 页面数据量大（列表 > 50 项或单对象 > 10KB）

## 症状
页面滚动时出现明显卡顿（掉帧），scroll-view 滑动不跟手，输入框输入延迟 200ms+，按钮点击后无响应直至页面"缓过来"。微信开发者工具的性能面板显示 setData 耗时 > 50ms 且调用频率极高。真机上滑动列表时 JS 线程持续繁忙，帧率从 60fps 降至 20fps 以下。

## 根因
setData 的完整链路是：逻辑层序列化数据 → JSBridge 传递到 Native → Native 转发到 WebView 渲染层 → 渲染层反序列化并触发 diff + patch。**每次 setData 调用完整走这条链路，且渲染层的 JS 线程是单线程的**。当 onPageScroll 以 ~16ms/次的频率触发并在回调中直接 setData 时，上次的渲染还没完成，下一次 setData 数据已经到达，形成调用堆积。WebView 的 JS 线程被连续的 setData 阻塞，无法及时响应用户交互。

此外，setData 传输大数据时，JSBridge 的 `evaluateJavascript` 会阻塞 WebView 的 JS 线程。传输 64KB 数据可阻塞 WebView 渲染线程 200ms+。

## 原理
小程序双线程架构中，逻辑层 setData → 渲染层更新是非阻塞异步的，但渲染层的 JS 线程只有一个。当短时间内连续调用 setData，Native 会将数据排队发送给 WebView，WebView 内部依次处理。如果数据量大或频率过高，WebView 的 JS 线程被序列化/反序列化和 diff 操作占满，用户的触摸事件、动画帧、定时器回调都被排队等待。这是典型的"通信风暴"反模式：不是计算慢，是通信阻塞了渲染。

WXS 的存在就是为了解决这个问题——WXS 代码直接运行在 WebView 渲染层，可以直接修改组件的 class/style，完全绕过逻辑层→渲染层的 JSBridge 通信链路。对于纯 UI 的计算（如滚动位置→样式映射），WXS 比 setData 快 10~100 倍。

## 架构视角 (Architecture Perspective)

### 架构层面根因
setData 性能灾难的表面原因是"调用频率过高"和"数据量过大"，但架构根因是**渲染层与逻辑层的通信契约未定义**。小程序的双线程架构本意是隔离（安全 + 性能），但开发者将 Web 开发的习惯（"修改 DOM 是廉价的"）直接平移到了小程序中，没有意识到每次 setData 都是一次跨进程通信。这反映了两个架构缺陷：

1. **缺少数据传输 SLA**：没有定义"一次 setData 应传输多少数据、多快频率、最大延迟"，导致开发者无约束地调用
2. **渲染状态与逻辑状态混同**：把需要渲染的数据（scrollTop）和逻辑数据（用户信息）都放在 data 中通过 setData 传输，而没有区分哪些数据适合渲染层自行处理（WXS）

### 系统级预防策略

**静态防护**：
```js
// ESLint 自定义规则: 禁止在 onPageScroll 中直接 setData
// .eslintrc.js
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'FunctionExpression[key.name="onPageScroll"] CallExpression[callee.property.name="setData"]',
    message: 'onPageScroll 中禁止直接 setData，请使用节流函数包装'
  }]
}
```

**架构防护**：
```js
// 封装安全的 setData 中间件 — 合并 + 节流 + 大小限制
function createSafeSetData(page) {
  let pendingData = {};
  let timer = null;
  const MAX_SIZE = 5 * 1024; // 5KB 上限

  return function safeSetData(data, immediate = false) {
    Object.assign(pendingData, data);
    
    // 数据量检查
    const size = JSON.stringify(pendingData).length;
    if (size > MAX_SIZE) {
      console.warn('[arch] setData too large:', size, 'bytes — consider chunking');
    }

    if (immediate) {
      page.setData(pendingData);
      pendingData = {};
      return;
    }

    clearTimeout(timer);
    timer = setTimeout(() => {
      page.setData(pendingData);
      pendingData = {};
    }, 16); // 合并到下一帧
  };
}

// 使用
Page({
  onLoad() {
    this.$setData = createSafeSetData(this);
  },
  onPageScroll(e) {
    this.$setData({ scrollTop: e.scrollTop }); // 自动节流合并
  }
});
```

### 跨层影响链

```
上游影响 (用户体验):
  - 页面滑动不跟手 → 用户感知"慢/卡" → 跳出率飙升
  - iOS 上掉帧尤其明显（WebView 的 JS 线程与合成线程分离不如 Android）
  - 输入框延迟 → 用户以为没输入 → 重复点击 → 表单数据错乱

下游影响 (后端):
  - setData 阻塞导致网络回调延迟处理 → 请求堆积
  - 页面卡顿期间用户重复提交 → 后端收到重复请求

监控影响:
  - 小程序后台性能监控中 "setData 耗时" 和 "JS 线程繁忙率" 两个核心指标
  - setData 平均耗时 > 30ms → P0 告警
  - JS 线程繁忙率 > 80% → P1 告警
```

### 相关架构模式
- **双线程通信预算模式** — 为 setData 设定频率和数据量预算，类似 pit-050 的三级预算
- **WXS 前置计算模式** — 渲染层能做的计算不推到逻辑层
- **数据分层模式** (`patterns.md`) — 区分渲染数据（scrollTop）和业务数据（userInfo），不同层处理
- **反模式: setData 风暴** — 高频 setData 是双线程架构下的经典反模式

## 预防
1. **铁律**：onPageScroll/onScroll 中 setData 必须加节流（≤ 100ms）
2. **铁律**：单次 setData 数据量 < 5KB，超过则分帧
3. **铁律**：能用 WXS 在渲染层处理的，绝不用 setData 回到逻辑层
4. **使用自定义组件**：组件级 setData 影响范围小，天然降低通信量
5. **代码审查**：每次 PR 中新增 setData 调用须标注频率预期和数据量
