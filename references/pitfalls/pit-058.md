# pit-058: React Native Bridge 性能瓶颈

## 环境
- React Native < 0.70 (旧 Bridge 架构)
- iOS 和 Android 双端
- 高频交互场景：手势拖拽、实时动画、无限滚动列表、WebSocket 数据推送

## 症状
- 手势拖拽卡顿，手指跟随明显延迟（>100ms）
- FlatList 快速滚动时出现空白块（未渲染的白框），然后突然填充内容
- 动画掉帧：JS 驱动的 `Animated` 动画在 JS 线程繁忙时完全停止
- WebSocket 高频数据（如实时行情）导致 UI 无响应
- 多指触控时帧率从 60fps 降至 10-20fps
- 页面切换时的转场动画卡顿（JS 线程被 Navigator 逻辑占用）

## 根因
React Native 旧架构使用**异步 JSON Bridge** 作为 JS 线程与 Native 线程的唯一通信通道。每次通信经历：JS 对象 → JSON.stringify → Bridge 队列 → JSON.parse → Native 对象。这个过程的三大瓶颈：

1. **全异步**：所有通信都是异步的，无法做同步调用。对于需要即时反馈的场景（如手势跟随），异步延迟（通常 16-50ms per message）会累积。
2. **JSON 序列化开销**：大数据结构（如列表 dataSource、图片 base64）的序列化/反序列化消耗大量 CPU。100KB 的 JSON 序列化约需 2-5ms on JS thread + 2-5ms on Native thread。
3. **单通道争抢**：所有 Native Module 调用（相机、蓝牙、存储、动画、网络）共享同一个 Bridge 队列。一个缓慢的 Native 操作会阻塞 Bridge 队列中后续所有通信。

在高频场景下（如 FlatList 快速滚动：每 16ms 需要一次 onScroll 回调 + 新的 item 渲染），Bridge 通道严重饱和，形成性能悬崖。

## 原理
React Native 旧架构的核心问题是**跨语言调用的异步性**。JS 运行在 JavaScriptCore/Hermes 引擎中，Native 运行在各自的 UI 线程中，两者之间通过 C++ 层的 Bridge 模块通信。Bridge 维护一个消息队列，所有从 JS 到 Native 的调用都经历：序列化为 JSON → 入队 → 等待 Native 线程处理 → 出队 → 反序列化。这个架构有三个天生的性能悬崖：

1. **批处理延迟（Batching Latency）**：Bridge 不是逐条处理消息，而是批量处理（通常每 5ms 一批）。这意味着即使单条消息的序列化只需 0.5ms，它也要等待下一个批处理窗口。
2. **序列化爆炸（Serialization Blowup）**：JSON.stringify 的复杂度是 O(n)，其中 n 是数据大小。当传输图片 base64（通常 1-10MB）时，序列化本身占用 JS 线程数十毫秒，在此期间 UI 完全冻结。
3. **队头阻塞（Head-of-Line Blocking）**：Bridge 队列是 FIFO 的。如果一个慢速 Native Module 调用（如 SQLite 查询，耗时 200ms）进入队列，后续所有调用（包括高优先级的 UI 更新）都必须等待。

React Native 新架构通过 JSI（JavaScript Interface）替代 Bridge，实现了直接同步调用和对象共享（零拷贝），从根本上解决了上述三个瓶颈。对于仍在使用旧架构的项目，优化策略围绕"减少 Bridge 通信频率 × 单次通信数据量"展开。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Bridge 性能瓶颈的表面原因是"异步序列化通道的物理限制"，但架构层面根因是**通信边界建模错误**：
- JS 线程与 Native 线程的交互被建模为"远程过程调用"（RPC）模式，但实际上这是一个**紧耦合的本地进程内通信**。RPC 模式假设高延迟、不可靠的网络，而本地通信场景恰恰相反——延迟极低、可靠性极高。Bridge 的异步批处理设计是为了应对"网络 RPC"场景的，但在本地通信上造成了不必要的排队开销。
- **UI 更新优先级缺失**：Bridge 没有消息优先级的概念。滚动事件、动画帧、数据查询、存储读写——所有这些被平等对待。这违反了"UI 更新 > 数据查询"的优先级排序原则，导致用户可感知的 UI 卡顿。
- **数据所有权模糊**：JSON 序列化迫使每次通信都进行"数据拷贝"，但实际上很多数据（如列表数据、图片 buffer）可以由 JS 和 Native 共享同一块内存。这不是 Bridge 的 bug，而是架构假设不成立：RPC 模型中的"数据隔离"在本地通信中是反模式。

### 系统级预防策略

**静态防护**：
```js
// ESLint 规则: 禁用非必要的 onScroll 高频事件
rules: {
  'react-native/no-unnecessary-scroll-listener': 'error',
  // 禁止 scrollEventThrottle < 100 (除非提供正当理由)
  'react-native/min-scroll-throttle': ['error', { min: 100 }],
  // 强制 useNativeDriver: true
  'react-native/no-js-driven-animation': 'error',
}
```

**CI 防护**：
```bash
# CI 中添加 Bridge 通信量基准测试
# 启动 App → 录制 30 秒交互 → 统计 Bridge 消息数
# 如果消息数超过历史均值的 150%，CI 阻塞

# react-native-performance-monitor 集成
npx react-native-measure-messages \
  --scenario e2e/main-flow \
  --max-messages-per-sec 150 \
  --max-bytes-per-sec 51200  # 50KB/s
```

**架构防护**：
- **Bridge 消息预算制度**：为每个模块（动画、滚动、数据、网络）设定每秒 Bridge 消息配额，超标时在开发阶段报 warning。
- **Native 优先原则**：所有可移至 Native 层的逻辑（图片处理、数据加解密、动画序列）一律在 Native 层完成，JS 只接收最终状态。
- **升级至新架构**：RN 0.70+ 逐步启用新架构（JSI + Fabric + Turbo Modules），这是根本性解决方案。旧架构项目的每一个 Bridge 优化都是对症状的缓解，而非治愈。

**流程防护**：
- Code Review 检查清单："新增的 onScroll / animation / 定时器是否使用了 useNativeDriver？"
- 每个涉及 Bridge 通信的 PR 必须附带消息量估算（msg/s）。
- 性能回归测试作为合并检查 (merge gate)，而非发布后才跑。
- CodePush 热更新的 bundle 必须附带 Bridge 性能基准测试报告。

### 跨层影响链

```
上游影响 (UI 层):
  - 手势延迟 > 100ms → 用户感知"不跟手" → 评分/留存下降
  - 动画掉帧 → 品牌印象受损（尤其是启动动画和转场动画）
  - 列表白色块 → 用户认为内容未加载 → 提前离开页面

下游影响 (Native 层):
  - Bridge 拥塞导致 Native 模块回调延迟 → 相机/蓝牙操作超时
  - JSON 序列化消耗 CPU → iOS 热降频/Android thermal throttling
  - Native 线程被 JS 线程堵塞 → 系统 ANR/看门狗 (Watchdog) 杀死 App

部署影响:
  - 性能问题到线上才暴露（开发环境 JS 线程负载低 ≠ 生产负载高）
  - CodePush 热更新修复不了 Bridge 瓶颈（Bridge 在 Native 层）
  - 只能通过 App Store/Google Play 发版修复 → 修复周期 3-7 天

监控影响:
  - 需要接入 RN 性能监控 SDK 来区分 JS/Native 瓶颈
  - Sentry 无法捕获 Bridge 延迟问题（这不是异常，是性能退化）
  - 需要自定义性能指标: JS→Native message latency p50/p95/p99
```

### 相关架构模式
- **CQRS 前端化** (`patterns.md`) — 将"查询 Bridge 状态"和"向 Bridge 发送命令"分离，避免查询调用污染命令通道。
- **Observable 模式 + Native Driver** — 将高频数据流（如 WebSocket tick）通过 Native Module Observable 直接推送给 Animated.Value，完全绕过 Bridge。
- **升级决策** (`decisions.md`) — 从旧 Bridge 到新架构（JSI/Fabric）的迁移策略，属于架构层面的技术债务清理，而非功能开发。

## 预防

1. **所有动画使用 `useNativeDriver: true`**：唯一的例外是动画涉及非 Native 属性（如 `width`、`height`、`backgroundColor`），此时使用 `LayoutAnimation` 替代。
2. **`scrollEventThrottle` 最小设为 100（10次/秒）**：除非正在实现需要帧级精度的功能，否则 16 的默认值过高。
3. **大数据在 Native 层处理**：图片、文件、数据库操作避免经过 Bridge。使用 Native Modules 在 Native 端完成操作，JS 只接收小体积的结果。
4. **使用 `InteractionManager` 延迟非关键操作**：新页面进入时优先保证转场动画流畅，数据加载延后。
5. **启用 Hermes 引擎**：Hermes 减少了 JS 线程的 CPU 占用，间接降低 Bridge 调用频率。
