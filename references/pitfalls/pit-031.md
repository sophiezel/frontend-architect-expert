# pit-031: Vue watch/watchEffect 无限循环

## 环境
- Vue 3.0+
- 使用 `watch()` / `watchEffect()` 的项目
- 复杂表单、实时搜索、数据联动场景

## 症状
页面 CPU 占用异常升高，浏览器逐渐卡死。控制台被不断刷出的日志淹没。Vue DevTools 显示组件频繁重渲染且 watch 回调被反复触发。常见于：
- 在 `watch` 回调中修改了被监听的数据
- `watchEffect` 内部副作用修改了依赖的响应式值
- `watch` 使用 `deep: true` 监听大型对象/数组，且内部又修改了嵌套属性

## 根因
watch/watchEffect 的设计假设是"观察数据 → 产生副作用"，数据与副作用应分属不同的响应式域。当两者交叉时形成闭环：

1. **watch 内修改监听源**：`watch(source, (newVal) => { source.value = newVal + 1 })` — 经典死循环
2. **watchEffect 自触发**：`watchEffect(() => { state.a = state.b + 1 })` — 如果其他地方触发了 `state.b` 的更新，可能导致连锁反应
3. **deep:true 的规模陷阱**：largeObject 任意深层属性变化都触发回调，回调内若修改同一 largeObject 的任何属性 → 再次触发 → 循环

## 诊断方法

```bash
# 1. 在 watch 回调中加入循环检测
let watchCallCount = 0;
const MAX_WATCH_CALLS = 20;
watch(source, (newVal, oldVal) => {
  watchCallCount++;
  if (watchCallCount > MAX_WATCH_CALLS) {
    console.error('[WATCH LOOP DETECTED]', {
      callCount: watchCallCount,
      newVal: JSON.stringify(newVal).slice(0, 200),
      oldVal: JSON.stringify(oldVal).slice(0, 200),
      stack: new Error().stack
    });
    return; // 跳出循环
  }
  // 正常逻辑
});

# 2. 搜索可疑模式
grep -rn "watch(" src/ --include="*.vue" --include="*.ts" | while read line; do
  file=$(echo "$line" | cut -d: -f1)
  echo "=== $file ==="
  # 显示 watch 内部是否修改了自身监听的数据
  grep -A 10 "watch(" "$file" | head -20
done

# 3. 性能面板录制（Chrome DevTools → Performance）
# 看是否密集出现 "Vue update" 或组件名相关的函数调用块

# 4. Vue DevTools → Timeline：如果 Events 列表在短时间内涌入大量 "effect" 事件 → 循环
```

## 修复

```diff
// ❌ 问题 1: watch 中修改被监听的值
const count = ref(0);
-watch(count, (newVal) => {
-  if (newVal > 10) count.value = 0; // 又改了 count → 触发自身
-});

+watch(count, (newVal) => {
+  if (newVal > 10) {
+    // 方案 A: 使用 flush: 'sync' + 条件判断避免重复
+    nextTick(() => { count.value = 0; }); // 推迟到下一帧
+  }
+});

// ❌ 问题 2: watchEffect 内部修改依赖
const a = ref(0);
const b = ref(0);
-watchEffect(() => {
-  a.value = b.value + 1; // 若其他地方改了 a → 无限循环
-});

+watch(b, (newB) => {
+  a.value = newB + 1;
+}); // 明确监听 b，只改 a —— 单向

// ❌ 问题 3: deep:true 嵌套修改
const form = ref({ user: { name: '', age: 0 } });
-watch(form, (val) => {
-  // 验证逻辑
-  if (!val.user.name) val.user.name = '默认名'; // 修改触发自身
-}, { deep: true });

+// 拆分为两个 watcher 或使用 computed 处理派生值
+const userName = computed(() => form.value.user.name || '默认名');
+watch(() => form.value.user.age, (newAge) => {
+  // 仅对具体字段进行副作用
+});

// ✅ 通用防御：在 watch 内使用 isFlushing 标记
import { ref, watch } from 'vue';
const isFlushing = ref(false);
watch(source, (val) => {
  if (isFlushing.value) return;
  isFlushing.value = true;
  // ... 副作用逻辑
  nextTick(() => { isFlushing.value = false; });
});
```

## 验证

```bash
# 1. 单元测试：检测 watch 循环
import { ref, watch, nextTick } from 'vue';

test('watch 不应无限循环', async () => {
  const source = ref(0);
  let callCount = 0;
  
  watch(source, (val) => {
    callCount++;
    if (callCount > 10) throw new Error('Watch loop detected');
    if (val < 5) source.value++;
  });
  
  source.value = 1;
  await nextTick();
  // 期望：当 source 达到 5 时停止（最多 5 次回调，而非无限）
}, 5000); // 超时 5s，无限循环会触发 timeout

# 2. 手动回归检查清单
# - 打开有 watch 的页面，保持 10 秒不动 → CPU 不超过 5%
# - 快速连续修改被监听的值（如快速输入搜索词）→ UI 不卡顿
# - Chrome Performance 录制 5s → 无持续密集的 JS 执行

# 3. E2E 中监听 JS 执行时间
# Cypress: cy.window().its('performance').then(p => p.now())
```

## 原理
Vue 3 的响应式调度器使用 `queueFlush` 机制：多个同步的响应式变更会被合并到一个更新队列，在下一次微任务（`nextTick`）中批量执行。但这不能防止"在微任务中再次触发新的响应式变更"，新的变更会重新进入队列，形成跨 tick 的循环。

`watchEffect` 特殊之处在于它自动追踪内部所有响应式依赖，开发者可能不清楚依赖了什么，导致无意中修改了依赖源。

`deep: true` 的实现原理是对对象做深度遍历并追踪每一个嵌套属性——大型对象的 deep watch 在每次回调中触发新的深度追踪，如果回调修改了对象属性，新的追踪会立即生效，形成递归。

## 架构视角 (Architecture Perspective)

### 架构层面根因
watch 无限循环的架构根因是**副作用与数据流的边界混淆**：
- **watch 被滥用作状态机**：watch 的回调被当作"当 A 变化时，修改 B"，但多个这种单向规则串联后会形成隐形有向图，一旦图中存在环路（A→B→C→A），就形成无限循环。应用缺少一个集中的"派生状态规则引擎"来检测环路。
- **数据流未分层**：原始状态（用户输入/API 响应）、派生状态（computed）、副作用（DOM 操作/请求发送）三层在代码中混在一起。正确的架构是派生状态永远用 `computed` 表达（无副作用），副作用放在明确的"命令处理函数"（事件回调）中，watch 仅用于"无法用 computed 表达的非同步副作用"（如 localStorage 持久化）。
- **deep:true 是"懒惰设计"的标志**：当开发者需要监听对象的任意变化时，说明"变化了什么"的语义已经在设计中丢失——应该用具体的业务事件（如 `onFieldChange('age')`）而非通配的 `deep:true`。

### 系统级预防策略

**静态检查**：
```bash
# ESLint 自定义规则：检测 watch/flush:sync 与内部修改
# 规则逻辑：若 watch 回调内修改了与 getter 相同的变量 → 警告
# 可以通过 no-restricted-syntax 和 AST 检查实现基础版本

# grep 辅助检测
grep -rn "watch(" src/ -A 20 | grep -E "\.value\s*=" | head -30
```

**CI 门禁**：
```bash
# scripts/ci/check-watch-loops.sh
# 对每个组件文件，分析 watch 回调 AST，检测循环模式

# 单元测试中检测：对每个包含 watch 的 composable，
# 将监听源在循环中修改 500 次，断言回调执行次数 << 500
find src/composables -name "*.spec.ts" -exec npx vitest run {} \;
```

**架构约束**：
- **派生状态只用 computed**，禁止 `watch(source, cb) { target.value = ... }` 模式
- **watch 仅用于命令式副作用**：DOM 操作、localStorage、API 调用、第三方库初始化
- **deep:true 必须 Review 审批**：在 PR 描述中说明"为什么 computed 不能满足需求"
- 在 composable 中提供 `useDerived(source, transform)` 而非让开发者手写 watch+赋值

**流程约束**：
- Code Review 清单："此 watch 回调是否修改了任何响应式数据？若有，是否可能形成循环？"
- 每个带 `deep:true` 的 watch 必须附带 perf benchmark 证明首屏加载不劣化

### 跨层影响链

```
上游影响 (用户交互):
  - 表单联动（省份→城市→区县）中任一环节循环 → 整个表单链断裂
  - 实时搜索建议无法给出结果（输入被覆盖或反复重置）

下游影响 (数据持久化):
  - watch 写 localStorage 的循环 → 存储被反复写入过量数据 → 导致配额超限
  - API 被循环调用 → 后端限流/封 IP → 用户账号被误判为恶意

部署影响:
  - Vite HMR 在开发时可能掩盖循环（组件重挂载打断循环），生产环境暴露
  - SSR 中 watch 只在客户端激活时运行 → hydration mismatch 风险

监控影响:
  - RUM：JS 执行时长 > 5s → 判定为页面挂死
  - 自定义事件：watch 回调执行频率 histogram，检测异常高频率
  - Error Boundary 无法捕获（无异常，只是 CPU 100%）
```

### 相关架构模式
- **Redux 单向数据流在 Vue 中的映射** (`patterns.md` 第二节) — Vue 的 computed 相当于 Redux 的 selector，watch 相当于 saga/listener。应遵循 reducer → selector → component 的单向路径，避免 listener 间相互触发。
- **Event Sourcing 思想** (`decisions.md` 第七节) — 将所有状态变更建模为事件序列（而非互相 watch），可以天然避免循环依赖（事件没有"回去"的路径）。
- **pit-001: useEffect 无限循环** — 同源的副作用循环问题，React 的解决方案（依赖原始值、useMemo 稳定引用）对 Vue 的浅层 watch 同样适用。

## 预防
1. **能用 computed 就不用 watch**：派生数据永远用 computed
2. 在 watch 回调中修改数据前，先判断 `if (newVal === oldVal) return` 和业务条件
3. 给每个 watch 添加 `maxCallCount` 防御（非生产环境），超过阈值抛异常而非静默死循环
4. `deep: true` 默认不开——仅在有充分理由时才开启，且明确指定 `immediate: false`
