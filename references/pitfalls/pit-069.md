# pit-069: TypeScript 类型声明与运行时脱节 — .d.ts 与实现不一致

## 环境
- TypeScript 项目使用 `.d.ts` 声明文件
- 手动编写的类型声明（非自动生成）
- API 类型定义与后端实现分离维护

## 症状
IDE 自动补全显示某个方法存在且接收 3 个参数，但运行时调用报错：`TypeError: obj.method is not a function`。或者 TypeScript 编译零错误，但生产环境报 `Cannot read properties of undefined`。

调试发现：`.d.ts` 文件声明了 `getUser(id: string): User`，但实际 API 返回的形状是 `{ data: { user_info: {...} } }`——嵌套层不同、字段名也不同。前端代码按照类型声明解构导致 `undefined`。

或者：打包后的 `.d.ts` 发布为 `@myorg/utils`，消费者按照类型使用，却遇到运行时不存在的导出。

## 根因
TypeScript 的类型声明（`.d.ts`）是一种编译时契约，不产生运行时代码。当声明与实现脱节时，编译器无法检测到不一致。常见场景：

1. **手动维护 `.d.ts` 与手动维护 `.js`**：两套代码，同步靠人
2. **API 类型声明滞后于后端变更**：后端字段改名/加嵌套，前端 `.d.ts` 未更新
3. **declare module 的临时补丁变成永久**：最初为解决第三方库类型缺失而写，后来库更新了但声明未更新
4. **发布类型与实现版本不匹配**：`@myorg/utils@1.2.0` 的 `.d.ts` 是 1.1.0 的
5. **overloads 声明顺序错误**：最后一个 overload 是具体实现签名但参数顺序不对

## 诊断方法

```bash
# 1. 检查 .d.ts 中的函数签名是否对应实际导出
# 工具：ts-api-guardian
npx ts-api-guardian --verifyStableApi --old packages/pkg/api-report.api.md

# 2. 检查是否有 declare module 补丁残留
grep -rn "declare module" src/types/ | wc -l
# 逐个检查是否仍有必要（库版本升级后可能不再需要）

# 3. 对比 .d.ts 导出与 .js 实际导出
# 生成当前版本的 API report
npx api-extractor run
# 对比新旧 API report 差异

# 4. 运行时检测类型与实现不一致
# 编写类型验证测试
test('runtime matches types', () => {
  const user = api.getUser('1');
  // 用 zod 验证运行时数据符合 TS 类型
  const result = UserSchema.safeParse(user);
  expect(result.success).toBe(true);
});

# 5. 检查 .d.ts 与 npm package 版本是否一致
npm pack --dry-run 2>&1 | grep "d.ts"
# 确保发布的 .d.ts 来自同一构建版本

# 6. 利用 TypeScript isolatedModules 检查重导出
npx tsc --noEmit --isolatedModules
# 如果 .d.ts 中的类型仅用于类型（未 import type），会报错
```

## 修复

```diff
// ❌ 问题1：过时的 declare module 补丁
// types/legacy-lib.d.ts（2年前写的）
- declare module 'legacy-lib' {
-   export function init(config: any): void;
-   export function track(event: string, data: any): void;
- }
// 实际上 legacy-lib v3 已有自己的类型定义，且 init 签名已变为 (config: LegacyConfig): Promise<void>

+ // 删除过时的 declare module，使用库自带的类型
+ import { init, track } from 'legacy-lib';
+ // 如果库无类型，使用精确声明
+ declare module 'legacy-lib' {
+   export function init(config: { apiKey: string; env: 'dev' | 'prod' }): Promise<void>;
+   export function track(event: string, data: Record<string, unknown>): Promise<void>;
+ }

// ❌ 问题2：API 响应类型与后端不一致
- interface User {
-   name: string;
-   email: string;
- }
// 实际后端响应: { data: { user_name: "...", email_addr: "..." } }

+ // ✅ 定义网络层 DTO（数据从网络来时的形状）
+ interface UserDTO {
+   user_name: string;
+   email_addr: string;
+ }
+
+ // ✅ 定义领域层 Model（应用内部使用的形状）
+ interface User {
+   name: string;
+   email: string;
+ }
+
+ // ✅ 提供转换函数
+ function toUser(dto: UserDTO): User {
+   return {
+     name: dto.user_name,
+     email: dto.email_addr,
+   };
+ }

// ❌ 问题3：package 发布 .d.ts 版本不匹配
// 通过 monorepo project references + CI 验证
+ // tsconfig.json
+ {
+   "compilerOptions": {
+     "composite": true,
+     "declaration": true,
+     "declarationMap": true,
+     "stripInternal": true,  // 未导出的类型不出现在 .d.ts
+     "declarationDir": "./dist/types"
+   }
+ }

// ❌ 问题4：函数 overload 声明与实现不一致
- export function fetchData(url: string): Promise<unknown>;
- export function fetchData(url: string, options: FetchOptions): Promise<unknown>;
- export function fetchData(url: string, options?: FetchOptions, callback?: () => void): Promise<unknown> {
-   // 实现签名多了 callback 参数但前两个 overload 声明未包含
- }

+ export function fetchData(url: string): Promise<unknown>;
+ export function fetchData(url: string, options: FetchOptions): Promise<unknown>;
+ // 实现签名
+ export function fetchData(
+   url: string,
+   options?: FetchOptions
+ ): Promise<unknown> {
+   // 实现
+ }
```

## 原理
TypeScript 的类型声明文件 `.d.ts` 只影响编译阶段（类型检查），不影响运行时行为。编译器假定 `.d.ts` 声明反映了实际运行时的形状——但如果手动编写 `.d.ts` 时出错，编译器无法知道。

这类似于 C/C++ 中的头文件（`.h`）和实现文件（`.c`）分离——如果头文件声明了 `int add(int, int)` 但实现是 `float add(float, float)`，编译器可能不会报错（取决于编译选项），但链接或运行时会失败。TypeScript 的差异在于——它只做编译时检查，没有任何"链接"阶段来验证声明与实现的一致性。

函数重载的声明与实现签名分离是另一个常见陷阱。TypeScript 的重载分为"重载签名"（对外可见）和"实现签名"（内部实现）。如果实现签名的参数列表不与重载签名兼容，编译器会报错，但如果实现签名额外添加了可选参数而重载签名未声明，则不会报错——而这会导致调用方以为不需要传递该参数。

## 架构视角 (Architecture Perspective)

### 架构层面根因
类型声明脱节表面上是"人写错了"，架构根因是**缺少类型与实现的同源生成机制**：
- **类型手动维护而非代码生成**：API 类型从后端 OpenAPI/GraphQL schema 手动复制到前端 `.d.ts`，人为引入不一致。
- **DTO 与 Domain Model 不分层**：API 返回的蛇形命名（`user_name`）直接被前端类型消费，后端重构字段名时前端类型被迫同步——而往往忘记更新 `.d.ts`。
- **Monorepo 构建产物版本漂移**：`packages/shared` 的 `.d.ts` 与 `.js` 在不同时间构建，CI 没有验证两者来自同一次源码构建。
- **缺少 contract testing**：没有消费者驱动的契约测试来验证前端期望的类型/形状与后端实际返回一致。

### 系统级预防策略

**静态防护**：
```bash
# 1. 从 OpenAPI 自动生成类型（而非手写）
npx openapi-typescript https://api.example.com/openapi.json -o src/types/api.ts
# 每次 CI 构建重新生成，并与已有类型 diff
# 如果 diff 有意外变更 → 阻断（后端破坏了契约）

# 2. ts-api-guardian 锁定公共 API
npx ts-api-guardian --verify api-report.api.md
# api-report 文件提交到 Git，任何公共 API 签名变更需显式更新
```

**CI 防护**：
```yaml
# 类型声明正确性验证
- name: Verify type declarations
  run: |
    # 1. 构建声明文件
    tsc --declaration --emitDeclarationOnly --outDir dist/types
    
    # 2. 验证声明文件能通过编译
    tsc --noEmit --project tsconfig.types-check.json
    
    # 3. 对比本次和上次的 API report
    npx api-extractor run
    git diff --exit-code etc/api-report.api.md
    
    # 4. 运行 contract test
    npx jest --testPathPattern='contract/'
```

**架构防护**：
- **DTO/Domain 严格分层**：网络层 DTO（蛇形命名）+ 转换层 + 领域层 Model（驼峰命名），领域层只依赖自己的类型。
- **类型生成管道 (Type Generation Pipeline)**：OpenAPI/Swagger → `openapi-typescript` / `swagger-typescript-api` → 前端类型文件，完全自动化，零手动。
- **Contract Testing**：Pact 或类似工具确保前端期望与后端返回一致：
```typescript
// 消费者端 contract test
const provider = new Pact({ consumer: 'web', provider: 'api' });
await provider.addInteraction({
  uponReceiving: 'a request for user',
  withRequest: { method: 'GET', path: '/users/1' },
  willRespondWith: {
    status: 200,
    body: Matchers.somethingLike({ id: '1', name: 'Alice' }),
  },
});
// 如果后端返回的形状不匹配 → contract test 失败
```

**流程防护**：
- API 契约变更流程：后端 team 通知前端 team → 更新 OpenAPI schema → 重新生成类型 → 前端类型编译失败 → 适配代码 → 更新 contract test。
- Code Review 清单："手动编写的 `.d.ts` 是否有对应的运行时验证？"
- 定期（每月）检查 `declare module` 残留——如果目标库已有官方类型则删除。

### 跨层影响链

```
上游影响:
  - 类型声明错误 → 开发者信任类型系统 → 跳过运行时边界检查
  - API 字段改名 → .d.ts 未同步 → 前端代码按旧类型解构 → undefined 传播

下游影响:
  - 运行时 TypeError → Sentry 告警 → 紧急修复
  - 消费者项目依赖错误类型 → 消费者也出现运行时错误 → 连锁故障
  - 类型安全信任崩塌 → 团队开始加 as any → 恶性循环（参见 pit-062）

生态影响:
  - 发布的类型包有误 → npm 上的消费者受影响 → GitHub issues 增多
  - 类型错误如果进入流行的 @types/* 包 → 影响大量项目（如 @types/react 早期的问题）
```

### 相关架构模式
- **DTO/Domain 分层** (`typescript-engineering.md` 第3章) — 防止 API 形状直接污染前端类型
- **API 客户端类型推导** (`typescript-engineering.md` 第2.3节) — 从路由定义自动推导请求/响应类型
- **I/O Decoder 模式** (`patterns.md`) — 运行时验证编译时类型假设
- **Contract Testing** (Pact / Spring Cloud Contract) — 消费者驱动的 API 契约验证

## 预防

1. **类型从源头生成**：API 类型从 OpenAPI/GraphQL schema 自动生成，不手写。
2. **运行时类型验证**：关键数据入口使用 zod / io-ts 验证运行时数据符合类型。
3. **DTO / Domain Model 分层**：网络层形状和领域层形状分离，独立演化。
4. **API Extractor + API Report**：将公共 API 签名锁定为 `.api.md` 文件，变更需显式审批。
5. **Contract Testing**：消费者驱动的契约测试验证前端期望与后端返回一致。
6. **定期清理 declare module**：移除过时的手动类型补丁。

```bash
# 方案A：如果是手动 declare module 过时

# 从最新 OpenAPI 重新生成类型

# 根据类型编译错误定位需要适配的代码

# 方案C：如果是发布包类型不一致

npm run build:types
npm publish

# 方案D：紧急情况下，用 zod schema 代替 .d.ts 作为类型真相
# 先修复运行时（解析时做校验），再后续修复类型声明

  captureException(result.error);
}
```
