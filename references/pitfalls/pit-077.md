# pit-077: WASM 加载阻塞主线程 — 大 WASM 文件同步加载

## 环境
- Web 应用使用 WebAssembly (WASM) 模块
- 大 WASM 文件 (> 2MB) 同步编译/实例化
- 主线程加载，无 Web Worker 分流

## 症状
页面打开后白屏 2-5 秒或明显卡顿。Chrome Performance 面板显示主线程被一个长任务 (Long Task > 50ms) 阻塞，调用栈指向 `WebAssembly.compile` 或 `WebAssembly.instantiate`。用户点击、滚动、输入完全无响应。Lighthouse 报告的 TBT (Total Blocking Time) 高达 3-8 秒，INP (Interaction to Next Paint) 严重恶化。

更隐蔽的症状：
- 移动端/低端设备远比桌面端严重（低端设备编译 WASM 可能需 500ms-2s）
- 竞争条件：如果 WASM 加载期间用户触发了依赖 WASM 功能的操作（如图片滤镜、PDF 渲染），功能静默失败
- `WebAssembly.instantiateStreaming` 仍在主线程执行编译阶段，只是 fetch 和 compile 可以重叠
- 多个 WASM 模块的编译在队列中排队，主线程可能被连续阻塞多次

典型 Performance 录制结果：
```
Main Thread:
[████████████████████ 1800ms] WebAssembly.compile (长任务)
[   █████████████████ 1500ms] WebAssembly.instantiate
[                            ] idle (但用户操作无响应)
```

## 根因
**WASM 编译是 CPU 密集型同步操作，阻塞主线程的事件循环**。核心机制：

1. **WASM 编译必须完成才能实例化**：`WebAssembly.compile()` 将 `.wasm` 字节码验证并编译为目标机器码。这是纯 CPU 操作，必须同步完成，无法拆分为微任务。

2. **主线程阻塞 = 用户交互失效**：事件循环被 `compile()` 独占，`requestAnimationFrame`、`click` 事件、`scroll` 事件全部等待 compile 完成后才能执行。用户感受到的是"页面假死"。

3. **`instantiateStreaming` 的隐藏开销**：虽然使用了流式编译（下载和编译并行），但**编译部分仍在主线程**。如果 WASM 非常大（如 `ffmpeg.wasm` ~25MB），编译时间可达数秒。

4. **无渐进增强**：`<script type="module">` 或 `import()` 加载的 WASM 模块如果没有显式的 Worker 分流，默认在主线程编译。

5. **浏览器差异**：Chrome 的 Liftoff (baseline) 编译器快但生成慢代码；Firefox 在编译时可能暂停事件循环；Safari 的 WASM 性能通常最慢。

```javascript
// ❌ 问题模式: 主线程同步编译
import init, { process } from './heavy.wasm'; // 隐式调用 compile + instantiate
// 或
const wasm = await WebAssembly.instantiateStreaming(
  fetch('/heavy.wasm'), imports
); // compile 和 instantiate 都在主线程

// 浏览器内部:
// fetch → compile (阻塞主线程!) → instantiate → 返回
```

## 修复

```diff
// ❌ 问题代码 1: 主线程加载
- import init, { process } from './ffmpeg.wasm';
- await init();
- const result = process(input);

// ✅ 修复 1: 使用 Web Worker 分流（推荐）
// main.js
+ const worker = new Worker(new URL('./wasm-worker.js', import.meta.url));
+ worker.postMessage({ type: 'process', input });

+ worker.onmessage = (e) => {
+   const { result } = e.data;
+   // 使用结果更新 UI
+ };

// wasm-worker.js (独立 Worker 线程)
+ import init, { process } from './ffmpeg.wasm';
+ self.onmessage = async (e) => {
+   await init();
+   const result = process(e.data.input);
+   self.postMessage({ result });
+ };
```

```diff
// ✅ 修复 2: 使用 Comlink 封装 Worker 通信

// main.js
```

```javascript
// ✅ 修复 3: Lazy Load + 预编译缓存
// 不是每个用户都需要 WASM，按需加载
async function loadWasmOnDemand() {
  // 低优先级加载（避免影响首屏）
  const link = document.createElement('link');
  link.rel = 'preload';
  link.as = 'fetch';
  link.href = '/heavy.wasm';
  link.crossOrigin = 'anonymous';
  document.head.appendChild(link);

  // 等到空闲时再编译
  const compileWhenIdle = () => {
    if ('requestIdleCallback' in window) {
      return new Promise((resolve) => {
        requestIdleCallback(async () => {
          const module = await WebAssembly.compileStreaming(fetch('/heavy.wasm'));
          resolve(module);
        }, { timeout: 2000 });
      });
    }
    // Fallback to Worker
  };
}
```

## 原理
WebAssembly 模块的编译过程：`.wasm` 二进制 → 验证 → 解码 → 编译为机器码 → 实例化。`compile` 和 `instantiate` 是同步操作（即便使用 `Streaming` 版本仅优化了 fetch 和 compile 的并行，compile 本身仍是同步的）。主线程的同步操作阻塞事件循环——没有 `await` 点，没有微任务插入，完全占有 CPU。唯一的解是将编译移到 Worker 线程。

Web Worker 有独立的事件循环和内存空间，WASM 在 Worker 中编译完全不影响主线程的渲染和交互。唯一的代价是通信开销（`postMessage` + 结构化克隆），但对于非高频交互场景（如图片处理、文档渲染），这个开销可以忽略。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **依赖注入设计缺陷**：WASM 模块被设计为全局单例且在应用初始化时同步加载，而非按需懒加载 + Worker 隔离。
- **性能预算未包含 WASM 编译成本**：只计算了下载大小，但未计算编译时间（在移动端可能 > 1s）。
- **无渐进增强的 WASM 功能降级**：WASM 功能不可用时应该有纯 JS fallback 或优雅降级，而非白屏。

### 系统级预防策略
- **WASM Worker Pool**：预创建 Worker 实例池，复用 Worker 减少创建开销。
- **性能预算包含 CPU**：首屏不允许 WASM 编译，必须是交互后才懒加载。
- **Lighthouse CI 门禁**：TBT < 300ms，INP < 200ms，阻断不满足的 PR。

### 跨层影响链
- 用户：白屏等待 → 跳出 → 损失转化
- 低端设备用户：完全不可用（WASM 编译超时）
- SEO：高 TBT → Core Web Vitals 不达标 → 搜索排名下降

## 预防

1. **WASM 只在 Worker 中加载**：永远不在主线程做 `compile`。
2. **预加载，懒编译**：用 `<link rel="preload">` 预下载 `.wasm`，在用户交互时才编译。
3. **WASM 文件大小预算**：首屏相关的 WASM < 500KB；非首屏可 > 1MB 但必须 Worker + lazy。
4. **功能降级**：WASM 加载期间提供 JS fallback 或明确 loading UI。

## 回退

```bash
# 如果 Worker 方案出现通信问题，临时回退

}, { timeout: 5000 });

# 方案B: 完全 JS fallback

  loadWasmInWorker();
}
```
