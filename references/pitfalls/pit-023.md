# pit-023: Safe Area / 安全区域适配

## 环境
- iOS 11+（引入 Safe Area）
- iPhone X 及以上（刘海屏/灵动岛）
- iPad Pro（四边等宽 Safe Area）
- Android 全面屏（挖孔/水滴/屏下摄像头）
- WKWebView / Safari / 混合应用 WebView

## 症状
1. 底部固定导航栏、提交按钮、Toast 被 Home Indicator 遮挡，用户无法点击。
2. 页面标题、返回按钮被刘海/灵动岛切口遮挡。
3. `fixed; bottom: 0` 的元素在键盘弹出后位置错乱，或键盘收起后留下无法消除的空白。
4. 全屏弹窗/视频播放器中关键控件（关闭按钮、进度条）延伸到 Safe Area 外部而不可操作。
5. 横竖屏切换时 Safe Area 突变：横屏时左右 Safe Area 变大，底部缩小，布局未重新计算。

## 根因
未使用 CSS 环境变量 `safe-area-inset-*`，或 WebView 中 `viewport-fit` 未设为 `cover`。

核心机制：iOS 11+ / Android P+ 定义了 Safe Area。默认 viewport（`viewport-fit=contain`）将网页限制在 Safe Area 内，`env(safe-area-inset-*)` 不生效。混合应用中，原生层未将 `safeAreaInsets` 传递给 WebView，导致 CSS 变量返回 0。

## 诊断方法

```bash
# 检查环境变量是否生效
# Safari Web Inspector 控制台:
console.log('safe-area-inset-bottom:',
  getComputedStyle(document.documentElement)
    .getPropertyValue('env(safe-area-inset-bottom)'));

# 检查 meta viewport 配置
grep -rn "viewport" public/index.html

# 如果返回 0px，检查 viewport-fit=cover 和原生 insets 传递
```

## 修复

```diff
- <!-- ❌ 默认 viewport：Safe Area 变量不生效 -->
- <meta name="viewport" content="width=device-width, initial-scale=1">

+ <!-- ✅ viewport-fit=cover 启用 Safe Area -->
+ <meta name="viewport"
+   content="width=device-width, initial-scale=1, viewport-fit=cover">
```

```css
/* ✅ CSS 自定义属性统一管理 */
:root {
  --safe-area-top: env(safe-area-inset-top, 0px);
  --safe-area-bottom: env(safe-area-inset-bottom, 0px);
  --safe-area-left: env(safe-area-inset-left, 0px);
  --safe-area-right: env(safe-area-inset-right, 0px);
}

/* iOS < 11.2 降级 constant() */
@supports (padding-top: constant(safe-area-inset-top)) {
  :root {
    --safe-area-top: constant(safe-area-inset-top);
    --safe-area-bottom: constant(safe-area-inset-bottom);
  }
}

.header { padding-top: var(--safe-area-top); }
.fixed-bottom-bar { padding-bottom: var(--safe-area-bottom); }
.fullscreen-overlay {
  padding: var(--safe-area-top) var(--safe-area-right)
           var(--safe-area-bottom) var(--safe-area-left);
}
```

```tsx
// ✅ 封装 <SafeAreaView> 组件而非逐个页面处理
interface SafeAreaProps {
  position: 'top' | 'bottom' | 'both';
  children: React.ReactNode;
}
function SafeAreaView({ position, children }: SafeAreaProps) {
  const style = {
    paddingTop: (position === 'top' || position === 'both')
      ? 'var(--safe-area-top)' : undefined,
    paddingBottom: (position === 'bottom' || position === 'both')
      ? 'var(--safe-area-bottom)' : undefined,
  };
  return <div style={style}>{children}</div>;
}
```

## 原理
CSS Environment Variables（`env()`）从系统 API 获取真实物理像素值并映射为 CSS 逻辑像素。`viewport-fit=cover` 将 viewport 扩展到全屏是 `env()` 生效的前提。`constant()` 是 iOS 11.0-11.2 的旧语法，作为降级。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Safe Area 问题的架构根因是**多终端布局抽象层缺失**——应将 Safe Area 当作全局布局约束在系统顶层统一处理：

- **Safe Area 被当作"个别页面的 CSS 问题"而非"全局布局约束"**。Safe Area 是物理设备的不可变属性，类似屏幕尺寸，但代码中适配逻辑散落在各页面局部 CSS 中，没有统一的 `<SafeAreaView>` 抽象。
- **WebView 容器与 Web 内容之间缺少 insets 传递协议**。混合应用中，原生层 `safeAreaInsets` 需通过 bridge 传递给 WebView，但多数项目未定义此传递契约。
- **全屏场景（播放器/弹窗）与常规场景的 Safe Area 策略未区分**。全屏内容需延伸到 Safe Area 外（沉浸式），但交互控件仍需在 Safe Area 内（可操作），要求层级策略支持。

### 系统级预防策略

**静态防护**：
```bash
# CI: meta viewport 必须包含 viewport-fit=cover
grep -rn 'viewport' public/index.html | grep -v 'viewport-fit=cover' \
  && echo "ERROR: 缺少 viewport-fit=cover" && exit 1
```

**CI 防护**：
```bash
# Playwright 多设备 Safe Area 验证
test('Safe Area insets on real devices', async ({ page }) => {
  await page.setViewportSize({ width: 390, height: 844 }); // iPhone 14 Pro
  const insetBottom = await page.evaluate(() =>
    parseFloat(getComputedStyle(document.documentElement)
      .getPropertyValue('--safe-area-bottom')));
  // 如果 viewport-fit=cover + 运行在真机/模拟器，应 > 0
  expect(insetBottom).toBeGreaterThanOrEqual(0);
});
```

**架构防护**：
- `SafeAreaView` 作为全局布局原语：所有页面必须通过 SafeAreaView 包裹。
- 混合应用 insets 传递协议：原生层在 WebView 初始化时注入 insets 到 CSS 变量。
```swift
let insets = view.safeAreaInsets
webView.evaluateJavaScript("""
  document.documentElement.style.setProperty(
    '--safe-area-bottom', '\(insets.bottom)px');
""")
```

**流程防护**：
- Code Review: "页面是否包含底部/顶部固定元素？是否使用 SafeAreaView 包裹？"
- 设计稿审查: UI 设计稿是否标注了 Safe Area 边界？

### 跨层影响链

```
上游影响 (消费者):
  - 用户无法点击底部主按钮 → 支付/提交流程中断 → 转化率下降
  - 顶部标题被刘海遮挡 → 品牌可见性受损
  - 全屏广告关闭按钮不可见 → 用户投诉

下游影响 (依赖):
  - 第三方 UI 库（Modal/Drawer/Toast）未适配 Safe Area → 异形屏异常
  - 视频播放器进度条/全屏按钮被遮挡
  - 引导页"下一步"按钮不可点击

部署影响:
  - 新 iPhone 发布 → Safe Area 值可能变化 → 需回归测试
  - Android 厂商碎片化 → 不同机型 Safe Area 高度不同
  - Hybrid App 需原生 + 前端协同更新

监控影响:
  - RUM: 按设备型号统计底部按钮点击率
  - 自定义事件: Safe Area CSS 变量值上报
  - 视觉回归: 新设备发布后自动化截图对比
```

### 相关架构模式
- **pit-006: iOS 100vh** — 同源的视口管理问题。Safe Area 和 100vh 共享视口适配层，统一使用 CSS 变量管理。
- **设计系统中的布局原语** — SafeAreaView 应与 `<Page>` `<BottomBar>` `<FullScreenModal>` 一起构成布局原语库。
- **CSS 方案选型** (`decisions.md` 第三节) — `env()` 和 `constant()` 的降级策略需要 CSS 方案支持。

## 预防
1. 所有移动端页面必须通过 `<SafeAreaView>` 包裹，禁止页面内裸写 `padding-top: env(safe-area-inset-top)`。
2. WebView 初始化时原生层必须注入 Safe Area insets 到 CSS 变量。
3. 设计系统底部组件（BottomBar/FAB/Toast）默认包含 `padding-bottom: var(--safe-area-bottom)`。
4. 测试矩阵覆盖 iPhone X+/iPad Pro/Android 全面屏机型。
