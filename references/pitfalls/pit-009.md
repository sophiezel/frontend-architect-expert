# pit-009: backdrop-filter 引发的性能灾难

## 环境
- 使用 `backdrop-filter` 或 `-webkit-backdrop-filter` 的页面
- 影响 Chrome、Safari（尤其严重）、Firefox
- 低端设备（Moto G4 等）表现更差

## 症状
页面滚动卡顿、动画掉帧（FPS 降到 30 以下）、合成层过多导致内存爆炸。尤其在低端设备和 Safari 上表现严重。

## 根因
`backdrop-filter` 强制浏览器为元素创建合成层（GraphicsLayer），且每帧都需要对底层内容做 GPU 纹理采样 + 滤镜计算。如果应用在频繁变化的背景上（滚动列表、视频、动画），GPU 开销极大。Safari 的实现尤其脆弱，大面积的 backdrop-filter 会导致整个页面被提升为合成层。

## 诊断方法

```bash
# 1. Chrome DevTools → Rendering → Layer borders
# 右上角 ⋮ → More tools → Rendering
# 勾选 "Layer borders" → 橙色框 = 合成层
# 观察是否有大面积或过多的橙色框

# 2. Chrome Performance 面板
# Performance → 开始录制 → 滚动页面 → 停止
# 查看帧率（FPS）和 GPU 耗时
# 如果 FPS < 30 且 GPU 条很长 → 确认性能问题

# 3. Safari Web Inspector → Layers
# Develop → Web Inspector → Layers 标签
# 查看 "Backdrop Filter" 标注的层大小

# 4. Chrome GPU 状态
# 地址栏输入 about:gpu
# 检查 "Canvas" 和 "Compositing" 相关状态

# 5. 诊断代码
# 在控制台统计 backdrop-filter 元素
const bfElements = document.querySelectorAll('*');
let count = 0;
let totalArea = 0;
bfElements.forEach(el => {
  const style = getComputedStyle(el);
  if (style.backdropFilter !== 'none' || style.webkitBackdropFilter !== 'none') {
    count++;
    const rect = el.getBoundingClientRect();
    totalArea += rect.width * rect.height;
  }
});
const vpArea = window.innerWidth * window.innerHeight;
console.log(`backdrop-filter 元素: ${count} 个, 总面积: ${(totalArea/vpArea*100).toFixed(1)}% 视口`);
```

## 修复

```css
/* ❌ 问题：大面积毛玻璃效果 */
.hero {
  backdrop-filter: blur(20px);
  /* 整个 hero 区域（可能 100vw × 80vh）每帧都需要 GPU 采样 */
}

/* ✅ 修复一：缩小模糊区域 */
.hero-overlay {
  position: absolute;
  inset: 0;
  backdrop-filter: blur(20px);
  pointer-events: none; /* 不影响底层交互 */
}

/* ✅ 修复二：使用 will-change 提示浏览器 */
.hero-overlay {
  backdrop-filter: blur(20px);
  will-change: backdrop-filter;
}

/* ✅ 修复三：静态背景用 SVG 滤镜或伪元素模糊代替 */
.hero::before {
  content: '';
  position: absolute;
  inset: 0;
  background: inherit;
  filter: blur(20px); /* 对自身内容模糊，非底层——性能更好 */
  z-index: -1;
}
```

```css
/* ❌ 问题：在可滚动容器内使用 backdrop-filter */
.scrollable-list {
  overflow-y: auto;
}
.list-header {
  position: sticky;
  top: 0;
  backdrop-filter: blur(10px); /* 每次滚动都重新计算 */
}

/* ✅ 修复：用半透明背景代替 */
.list-header {
  position: sticky;
  top: 0;
  background: rgba(255, 255, 255, 0.95); /* 牺牲毛玻璃效果，换性能 */
}
```

## 原理
`backdrop-filter` 的工作原理：浏览器将元素下层的内容渲染到离屏纹理，然后将该纹理传给 GPU，GPU 应用滤镜（模糊/亮度/对比度等），最后将结果合成到屏幕上。每一步都需要 GPU 显存和计算。大面积模糊（如 blur(20px)）的 GPU 采样半径大，开销更高。滚动的每一帧都需要重新采样底层内容。

## 架构视角 (Architecture Perspective)

### 架构层面根因
backdrop-filter 性能问题的架构根因是**视觉效果未纳入性能预算管理体系**：
- **视觉效果选择缺少性能成本建模**：架构决策中选择了"毛玻璃"视觉语言，但没有同步评估其 GPU 成本。每个视觉效果的引入应该有对应的"渲染成本标签"（如 `blur(20px)` = 高成本，`blur(5px)` = 中成本，`opacity: 0.9` = 低成本），由架构师在设计评审中权衡。
- **缺少设备能力分层策略**：同一套 CSS 在所有设备上加载，没有根据设备 GPU 能力做差异化。高端设备可以承担大面积 backdrop-filter，但低端设备（Moto G4、低端 Android）需要降级方案。架构缺少"设备能力探测 → 渐进增强"的机制。
- **合成层策略无治理**：每个 backdrop-filter 创建一个合成层，每个合成层消耗 GPU 内存。但没有架构级的合成层预算——"整个页面最多 X 个合成层，总合成层面积不超过 Y% 视口"。开发者可以无限制地叠加合成层，最终导致页面崩溃。

### 系统级预防策略

**静态防护**：
```css
/* 设计系统: backdrop-filter 限制使用 */
/* 在全局 CSS 中使用 @supports 和媒体查询做渐进增强 */
.glass-effect {
  /* 降级方案 */
  background: rgba(255, 255, 255, 0.85);
}

/* 仅在条件满足时启用毛玻璃 */
@supports (backdrop-filter: blur(1px)) {
  @media (min-device-pixel-ratio: 2) and (min-memory: 4GB) {
    .glass-effect {
      backdrop-filter: blur(10px);
    }
  }
}
```

```js
// 运行时: 低端设备自动禁用 backdrop-filter
// src/boot/performance-polyfill.ts
if (navigator.deviceMemory && navigator.deviceMemory < 4) {
  document.documentElement.classList.add('low-memory');
}
// CSS: .low-memory .glass-effect { backdrop-filter: none; }
```

**CI 防护**：

**架构防护**：
- **视觉效果委员会**：引入新的视觉效果（如 backdrop-filter、mix-blend-mode、clip-path 动画）前，必须在低端设备（Moto G4、iPhone 8）上通过 30fps 性能测试。
- **合成层预算**：设定架构级的合成层预算——页面同时存在的合成层不超过 10 个，总合成面积不超过视口面积的 40%。通过 Chrome DevTools Layers 面板在 CI 中监控。
- **设备能力探测**：根据 `navigator.deviceMemory`、`navigator.hardwareConcurrency`、屏幕像素比动态调整视觉效果等级。在应用启动时注入 `data-performance-tier` 属性到 `<html>`。

**流程防护**：
- Code Review 检查项："新增的 backdrop-filter 是否在低端设备上测试过？是否有降级方案？"
- 性能评审：每次发布前执行 Lighthouse 审计，关注 "Avoid enormous network payloads" 和 "Avoid large layout shifts" 背后的 GPU 内存影响。

### 跨层影响链

```
上游影响 (消费者):
  - 页面 FPS 下降 → 所有交互（点击/滚动/动画）响应延迟
  - GPU 内存耗尽 → 浏览器合成器崩溃 → 页面闪白

下游影响 (依赖):
  - iframe/微前端子应用的渲染也被拖慢（共享 GPU 资源）
  - Canvas/WebGL 内容因 GPU 资源竞争导致帧率下降
  - Web Animations API 的动画无法达到 60fps

部署影响:
  - 低端设备用户流失率上升（页面卡顿无法使用）
  - CDN 虽然缓存了静态 CSS，但 GPU 性能无法通过缓存解决
  - App Store/Google Play 评分受 WebView 性能影响

监控影响:
  - RUM: 按设备分组监控 FPS/INP (Interaction to Next Paint)
  - 自定义指标: `navigator.deviceMemory` 分布与性能相关性
  - E2E 测试: CI 中加入低端设备模拟的性能基准测试
  - 告警: 合成层数量超过阈值时触发
```

### 相关架构模式
- **CSS 方案选型** (`decisions.md` 第三节) — 设计系统必须包含"视觉效果性能等级"定义，不同的 CSS 属性对应不同的性能成本。
- **渲染优化模式 — Skeleton/SSR Streaming** (`patterns.md` 第四节) — 视觉效果不应阻塞首屏渲染，骨架屏和 SSR Streaming 可以在视觉效果计算完成前展示内容。
- **CSS-in-JS 运行时性能陷阱（反模式）** (`patterns.md` 第六节) — 运行时动态生成 backdrop-filter 样式同样会触发合成层重建，应使用静态 CSS 类名。

## 预防

1. **限制 backdrop-filter 面积**：尽量只在小元素上使用（导航栏、小卡片）。
2. **性能预算**：在低端设备（Moto G4 等）上测试，设置"超过 3 个 backdrop-filter 或总面积 > 20% 视口 → 报警"。
3. **优雅降级**：使用 `@supports not (backdrop-filter: blur(1px))` 提供降级方案。
4. **减少滤镜复杂度**：`blur(5px)` 比 `blur(20px)` 快很多。
