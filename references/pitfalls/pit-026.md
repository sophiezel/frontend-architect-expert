# pit-026: 定时器/WebSocket 未清理

## 环境
- React 16.8+ / Vue 3+
- SPA 单页应用
- 任何使用 `setInterval` / `setTimeout` / `WebSocket` / `EventSource` 的组件

## 症状
1. **离开页面后 setInterval 仍在执行**：原本每秒更新一次的仪表盘，切换到其他页面后控制台仍持续打印日志。
2. **WebSocket 保持连接**：切换到其他页面后 WebSocket 未断开，服务器推送的消息被丢弃或尝试更新已卸载组件 → 控制台报 `state update on unmounted component`。
3. **多次进出页面导致多个定时器/连接叠加**：从 A → B → A，现在有两个 setInterval 在运行，数据更新速度翻倍。
4. **应用空闲时 CPU 仍较高**：Performance Monitor 显示 JS heap 和 CPU 持续占用，即使页面在后台。
5. **移动端电量消耗异常**：后台定时器阻止设备进入深度休眠。

## 根因
`setInterval` / `setTimeout` / `WebSocket` 在组件卸载时未清理。React 的 useEffect cleanup 是唯一容器保证在组件卸载时执行清理——当开发者忘记从 useEffect 返回清理函数，定时器/连接"逃逸"成为孤儿资源。

核心机制：JavaScript 引擎有一个 `TimerList`（定时器队列）和多个 `WebSocket` 连接。这些资源独立于组件生命周期存在——即使组件被卸载、DOM 被移除，定时器和连接仍在运行，直到手动 `clearInterval` / `socket.close()`。

## 修复

```tsx
// ❌ 问题代码：setInterval 逃逸
function LiveClock() {
  const [time, setTime] = useState(new Date());
  useEffect(() => {
-    setInterval(() => {
-      setTime(new Date());
-    }, 1000);
  }, []); // 组件卸载后 setInterval 仍在运行

  return <div>{time.toLocaleTimeString()}</div>;
}

// ✅ 方案1: cleanup 清理
function LiveClock() {
  const [time, setTime] = useState(new Date());
  useEffect(() => {
+    const timer = setInterval(() => setTime(new Date()), 1000);
+    return () => clearInterval(timer);
  }, []);
  return <div>{time.toLocaleTimeString()}</div>;
}

// ✅ 方案2: 使用 useRef 持有 timer ID + 防多次挂载
function DataPoller({ url }: { url: string }) {
  const timerRef = useRef<number>();
  useEffect(() => {
    timerRef.current = window.setInterval(() => {
      fetch(url).then(/* ... */);
    }, 5000);
    return () => clearInterval(timerRef.current);
  }, [url]); // 依赖 url 变化时重新创建定时器
}
```

## 原理
JavaScript 的定时器（`setInterval`/`setTimeout`）和 WebSocket 是宿主环境（浏览器/Node.js）提供的 Web API，独立于 DOM 生命周期。定时器由浏览器的事件循环 Timer Phase 管理，WebSocket 由网络线程管理。React 的组件卸载只移除 DOM 节点和组件 fiber，不会自动取消这些宿主级资源——必须通过 useEffect cleanup 显式断开。

`useRef` 持有 timer ID 而非 `useState`，因为 timer ID 的变化不应触发重渲染。

## 架构视角 (Architecture Perspective)

### 架构层面根因
定时器/WebSocket 泄漏的架构根因是**长连接/长生命周期资源管理层缺失**：

- **组件直接持有宿主资源引用**：`new WebSocket()` 和 `setInterval()` 直接写在组件中，使得这些资源与组件生命周期强耦合但不受组件生命周期保护——React 只管理 DOM 和 state，不管 host API 资源。
- **缺少 `useInterval` / `useWebSocket` 抽象**：定时器逻辑（间隔、回调、依赖变化重建）在数十个组件中重复实现，每个实现都有一点点不同，其中一些忘了 cleanup。抽象为 hook 将"正确实现"集中到一个地方。
- **长连接的连接池/复用策略缺席**：多个组件可能需要同一个 WebSocket 连接，但每个组件各自 `new WebSocket()` 造成重复连接。缺少"连接管理器"统一管理连接的创建、复用和销毁。

**架构防护**：
- **封装 `useInterval` / `useWebSocket` hook 统一生命周期**：
```tsx
// @app/hooks/useInterval.ts
function useInterval(callback: () => void, delay: number | null) {
  const savedCallback = useRef(callback);
  savedCallback.current = callback; // 始终最新

  useEffect(() => {
    if (delay === null) return;
    const id = setInterval(() => savedCallback.current(), delay);
    return () => clearInterval(id);
  }, [delay]);
}

// @app/hooks/useWebSocket.ts
function useWebSocket(url: string | null) {
  const [lastMessage, setLastMessage] = useState(null);
  useEffect(() => {
    if (!url) return;
    const ws = new WebSocket(url);
    ws.onmessage = (e) => setLastMessage(JSON.parse(e.data));
    // 自动重连逻辑...
    return () => ws.close(1000, 'hook cleanup');
  }, [url]);
  return lastMessage;
}
```
- **WebSocket 连接池**：多组件共享同一 WS 连接时，通过 Context + ref-counting 管理连接生命周期（最后一个 unsubscribe 才 close）。
- **模块边界规则**：组件文件禁止直接 `new WebSocket()`，必须通过 `useWebSocket` hook。

**流程防护**：
- Code Review: "此 useEffect 是否包含 setInterval / setTimeout / WebSocket / EventSource？返回的 cleanup 是否覆盖了所有？"

### 跨层影响链

```
上游影响 (消费者):
  - 仪表盘数据刷新频率异常（多个定时器叠加）
  - 实时通知重复推送（多个 WebSocket 连接）
  - 移动端电量快速耗尽、设备发热

下游影响 (依赖):
  - WebSocket 服务端维护大量僵尸连接 → 连接数超限
  - 数据库：定时器驱动的轮询请求在后台持续消耗后端资源
  - 第三方 SDK（推送/实时协作）的连接未释放

部署影响:
  - WebSocket 服务端连接数监控阈值误报
  - 灰度发布时新旧版本 WebSocket 协议不兼容 → 连接泄漏加剧
  - 移动端 App Store 审核因功耗被拒

监控影响:
  - APM: WebSocket 连接数时间序列
  - Sentry: "Can't perform state update on unmounted" 错误
  - RUM: JS Heap Size 趋势
  - 自定义指标: 页面级别的 active timer count
```

### 相关架构模式
- **反模式: 组件内裸宿主 API 调用** — 与 pit-025 同源：`new WebSocket()` / `setInterval()` 直接写在组件中造成资源泄漏。
- **资源管理层 (Resource Manager)** (`patterns.md` 第一节) — 类似 Zustand store 管理 WebSocket 连接，组件通过 store.subscribe() 接收消息，生命周期由 store 控制。
- **CQRS 前端化** (`patterns.md` 第三节) — 定时轮询是 Command（触发数据刷新），store 更新是 Query（提供数据），分离后组件的 subscribe/unsubscribe 自动清理。

## 预防
1. 封装 `useInterval` / `useTimeout` / `useWebSocket` hook，统一管理生命周期和清理。
2. useEffect 中创建的任何宿主 API 资源必须在返回的 cleanup 函数中释放。
3. 代码审查检查项覆盖定时器和 WebSocket 的清理逻辑。
4. TypeScript strict mode 避免因动态属性名导致的引用缺失。
