# pit-067: 第三方 CDN 脚本被篡改 — 未使用 SRI

## 环境
- 使用 `<script src="https://cdn.example.com/...">` 引入第三方库
- 未设置 `integrity` 属性（Subresource Integrity / SRI）
- CDN 为第三方运营（如 jsDelivr / cdnjs / Google Hosted Libraries）

## 症状
用户反馈页面出现异常行为：按钮点击没反应，或者有奇怪的弹窗。部分用户看到"您的浏览器已过期"的钓鱼提示。安全团队调查发现 CDN 上的 jQuery 文件被替换为包含恶意代码的版本——攻击方式可能是 CDN 被入侵、中间人攻击（HTTP）、或 DNS 劫持。

由于未使用 SRI，浏览器无条件信任并执行了被篡改的脚本。影响所有加载该 CDN 资源的用户。

## 根因
CDN 脚本加载的安全模型建立在"信任 CDN 提供商"之上。但这个假设会在以下场景被打破：
1. **CDN 被入侵**：攻击者替换 CDN 上的文件
2. **中间人攻击**：如果页面通过 HTTPS 但 script src 使用了 HTTP（或者用户被 SSL strip 攻击）
3. **DNS 劫持**：`cdn.example.com` 被解析到恶意服务器
4. **CDN 服务变更/停服**：CDN 提供商修改了文件（非恶意），导致应用行为变化
5. **包版本模糊**：`jquery@3` 这种模糊版本号可能指向不同实际版本

Subresource Integrity (SRI) 让浏览器在加载资源前验证其哈希，如果不匹配则拒绝执行。

## 诊断方法

```bash
# 1. 找出所有缺少 integrity 的 CDN <script>/<link>
curl -s https://yourapp.com | grep -E '<script[^>]*src="https?://' | grep -v 'integrity='
curl -s https://yourapp.com | grep -E '<link[^>]*href="https?://' | grep -v 'integrity='

# 2. 检查 CDN 脚本是否有 SRI 哈希
# 使用 sri-toolbox:
npx sri-toolbox generate https://cdn.example.com/jquery@3.7.1.min.js

# 3. 对比本地预期版本与 CDN 实际版本
wget -q https://cdn.example.com/jquery.min.js -O /tmp/cdn-jquery.js
diff <(md5sum node_modules/jquery/dist/jquery.min.js) \
     <(md5sum /tmp/cdn-jquery.js)
# 如果哈希不同 → 文件可能已被篡改

# 4. 检查 CDN 是否可访问且返回正确内容
curl -sI https://cdn.example.com/jquery@3.7.1.min.js | head -10

# 5. CSP 中是否限制了 script-src
curl -sI https://yourapp.com | grep -i 'content-security-policy'
# 检查 script-src 是否允许该 CDN 域名
# 如果 CSP 限制了但无 SRI → 双重风险：CSP 可能被绕过 + 文件无完整性验证
```

## 修复

```diff
- <!-- ❌ 无 SRI：信任 CDN 不验证内容 -->
- <script src="https://cdn.jsdelivr.net/npm/jquery@3.7.1/dist/jquery.min.js"></script>

+ <!-- ✅ 方案A：添加 SRI + crossorigin -->
+ <script
+   src="https://cdn.jsdelivr.net/npm/jquery@3.7.1/dist/jquery.min.js"
+   integrity="sha384-kenU8dZ/iPAZwDwl0yXB6wF8BlcFmgkAqL+1k+0poPUC4JhxB2CxHZcD7eHRuFE"
+   crossorigin="anonymous"
+ ></script>

+ <!-- ⚠️ crossorigin="anonymous" 必须设置 -->
+ <!-- 即使是同源 CDN，SRI 也要求 CORS 或同源，缺此属性 SRI 不生效 -->

- <!-- ❌ 同样的，CSS 也需要 SRI -->
- <link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/bootstrap@5.3/dist/css/bootstrap.min.css">

+ <!-- ✅ CSS 也添加 integrity -->
+ <link
+   rel="stylesheet"
+   href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css"
+   integrity="sha384-9ndCyUaIbzAi2FUVXJi0CjmCapSmO7SnpJef0486qhLnuZ2cdeRhO02iuK6FUUVM"
+   crossorigin="anonymous"
+ >

// ═══════ 方案B：自托管（最佳实践，免 SRI） ═══════
// 不再依赖第三方 CDN，将库打包进构建产物
- // index.html: <script src="https://cdn.example.com/jquery.min.js"></script>
+ // 代码中直接 import，打包工具处理
+ import jQuery from 'jquery';
+ // webpack/vite 会将 jquery 打包进 bundle，无需 CDN

// ═══════ 方案C：CDN fallback（防御性加载） ═══════
// 带 CDN + SRI + 本地 fallback
+ <script
+   src="https://cdn.jsdelivr.net/npm/react@18.2.0/umd/react.production.min.js"
+   integrity="sha384-XiynDWvBaGYS9+IVZOZvBQuavCH8LyQqR0BYsVDiBBqZVoBlxQtUqFZbSxN+Fsj"
+   crossorigin="anonymous"
+ ></script>
+ <script>
+   // 如果 CDN 加载失败（integrity 不匹配），回退到自托管
+   if (typeof React === 'undefined') {
+     document.write('<script src="/vendor/react.18.2.0.min.js"><\/script>');
+   }
+ </script>
```

## 验证

```bash
# 1. 手动计算 SRI 哈希验证
curl -s https://cdn.example.com/jquery@3.7.1.min.js \
  | openssl dgst -sha384 -binary | openssl base64 -A
# 与 HTML 中 integrity="sha384-..." 对比，确保一致

# 2. 浏览器验证：Chrome DevTools → Network → 选中资源 → Response 标签
# 如果 integrity 不匹配 → 资源状态为 "(failed)" 且 Console 有报错

# 3. 自动化 SRI 测试（Playwright）
# test('all CDN scripts have valid integrity', async ({ page }) => {
#   const violations: string[] = [];
#   page.on('response', response => {
#     if (response.status() === 0 && response.url().includes('cdn')) {
#       violations.push(response.url());
#     }
#   });
#   await page.goto('/');
#   expect(violations).toHaveLength(0);
# });

# 4. 版本一致性检查
# 确保 integrity 与 package.json 中的版本匹配
PACKAGE_VERSION=$(node -e "console.log(require('./node_modules/react/package.json').version)")
echo "React version: $PACKAGE_VERSION"
# 然后确保 CDN URL 中也使用相同版本

# 5. Crossorigin 属性检查
curl -s https://yourapp.com | grep -oP 'integrity="[^"]+"' | while read -r line; do
  # 确认同一行有 crossorigin="anonymous"
  echo "Check crossorigin for: $line"
done
```

## 原理
SRI 的验证流程：
1. 浏览器发起请求获取 `https://cdn.example.com/lib.js`
2. 收到响应后计算其 SHA-384 哈希
3. 将计算出的哈希与 HTML 中 `integrity` 属性的值比较
4. 匹配 → 执行脚本；不匹配 → 拒绝执行并在 Console 报错
5. 如果脚本被拒绝，浏览器会触发 `window.onerror`（而非 script 的 error 事件）

`crossorigin="anonymous"` 是必须的，因为即使 CDN 与页面同源，没有此属性的请求不包含 Origin 头，且响应的 CORS 头不被处理。SRI 规范要求请求必须是 CORS-same-origin 或 CORS-cross-origin，因此必须声明 `crossorigin`。

使用 SHA-384（而非 SHA-256）作为哈希算法是业界共识，因为 SHA-256 存在理论上的碰撞风险（尽管现实中尚未被用于攻击 SRI）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
CDN 脚本缺失 SRI 的表面原因是"没有加 integrity 属性"，架构根因是**外部资源治理策略缺失**：
- **没有区分"可信资源"和"不可信资源"**：从 CDN 拉取的脚本和自托管脚本在代码审查中被一视同仁，但安全模型完全不同。
- **版本锁定链条断裂**：`package-lock.json` 锁定了 npm 依赖版本和完整性，但 HTML 中 `src="jquery@3"` 是模糊版本——实际上等于没有版本锁定。
- **构建流程不统一**：部分库通过 npm + import 加载（安全），部分通过 CDN `<script>` 加载（不安全），因为团队没有统一的"资源引入策略"。
- **SRI 维护成本的心理障碍**：团队认为手动计算和更新 SRI 哈希很麻烦，因此选择不加——忽略了自动化工具的存在。

### 系统级预防策略

**静态防护**：
```js
// ESLint 自定义规则：所有 <script src> 必须有 integrity
// 使用 eslint-plugin-html 或 eslint-plugin-security
rules: {
  // 或自定义 script
}

// 构建时自动注入 SRI
// webpack.config.js — HtmlWebpackPlugin 插件
const { SubresourceIntegrityPlugin } = require('webpack-subresource-integrity');
plugins: [
  new SubresourceIntegrityPlugin({
    hashFuncNames: ['sha384'],
    enabled: true,
  }),
]
// 自动为所有 webpack 生成的 bundle 添加 integrity
```

**CI 防护**：
```yaml
# CI 中检查所有外部资源是否有 SRI
- name: Check SRI
  run: |
    # 提取所有 CDN script src
    curl -s https://staging.example.com | grep -oP 'src="https?://[^"]*"' | \
    while read -r src; do
      if echo "$src" | grep -qv 'integrity'; then
        echo "⚠️ Missing integrity: $src"
        exit 1
      fi
    done
```

**架构防护**：
- **构建时 SRI 自动生成**：Webpack `SubresourceIntegrityPlugin` 或 Vite `vite-plugin-sri`，自动为构建产物中的 CDN 引用生成 integrity 哈希，消除手动维护成本。
- **资源引入策略标准化**：制定团队规范——所有第三方库优先通过 npm + bundle，禁止使用 CDN URL 直接引入（除非有充分理由如 perfromance/caching）。
- **CSP 兜底**：CSP `script-src` 即使允许 CDN 域名，也要配合 nonce/hash，确保即使 CDN 被篡改，非预期修改仍被 CSP 阻止。

**流程防护**：
- Code Review 清单项："HTML 中是否新增了 CDN 脚本？是否有 integrity + crossorigin？"
- 每季度 "外部依赖审计"：扫描所有 CDN 引用，确认其仍在使用且 SRI 哈希有效。

### 跨层影响链

```
上游影响:
  - CDN 被入侵 → 加载该 CDN 的所有站点受影响（面攻击）
  - 单个站点的 SRI 缺失不影响其他站点，但该站点所有用户受影响

下游影响:
  - 恶意脚本窃取登录凭证/支付信息 → 用户资金损失
  - 恶意脚本注入挖矿 → 用户 CPU 飙升、电池耗尽
  - 恶意脚本做 SEO spam → 插入无关链接 → 搜索引擎降权
  - 恶意脚本做 DDoS 攻击（利用浏览器）→ 站点被标记为恶意

合规影响:
  - PCI DSS 合规要求对支付页面的所有脚本有完整性控制
  - SRI 是支付页面强制的安全措施（PCI DSS 6.4.3 / 11.6.1）
```

### 相关架构模式
- **SRI（Subresource Integrity）** (`frontend-security.md` 第3.2节) — CDN 资源完整性验证
- **CSP strict-dynamic** (`frontend-security.md` 第1.4节) — 即使无 SRI，CSP 也能阻止部分恶意脚本
- **供应链安全** (`frontend-security.md` 第3章) — 端到端的依赖安全

## 预防

1. **所有 CDN `<script>` 和 `<link>` 必须有 `integrity` 属性**：使用 SHA-384。
2. **优先自托管**：将库通过 npm 引入并由打包工具打包到 bundle，无需依赖 CDN。
3. **构建工具自动 SRI**：Webpack 的 `SubresourceIntegrityPlugin` 或 Vite 的 `vite-plugin-sri`。
4. **CSP `script-src` + SRI 双层防御**：即使 SRI 绕过，CSP 限制可执行的脚本源。
5. **定期校验 SRI 哈希**：CI 中与 `node_modules` 文件对比，确保 CDN 版本匹配。

## 回退

```bash
# 如果 SRI 配置错误导致脚本加载失败（白屏/功能异常）：
# 方案A：先回退 HTML 中的 integrity 属性
git checkout HEAD~1 -- index.html
# 然后重新计算正确的 SRI 哈希
npx sri-toolbox generate node_modules/jquery/dist/jquery.min.js

# 方案B：如果 CDN 文件版本与锁定的哈希不一致
# 更新 CDN URL 到正确的版本
# 旧: jquery@3.7.1.min.js（但哈希是 3.7.0 的）
# 新: jquery@3.7.1.min.js + 重新计算哈希

# 方案C：紧急切换为自托管
# 在 HTML 中直接使用本地 script
<script src="/vendor/jquery.min.js"></script>
# 无需 integrity（同源脚本天然可信）
```
