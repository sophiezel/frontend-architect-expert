# pit-025: 事件监听未清理导致内存泄漏

## 环境
- React 16.8+ / Vue 3+ / 原生 JS SPA
- SPA 单页应用（页面切换不刷新浏览器）
- 使用 `addEventListener` / `document.addEventListener` 的组件

## 症状
1. **SPA 页面切换后内存持续增长**：在 A 页面和 B 页面之间切换 10 次，Chrome Task Manager 内存从 50 MB 涨到 300 MB。
2. **切换次数多了 Tab 崩溃**：长时间使用后 Chrome 弹出 "Aw, Snap!" 或页面变空白。
3. **控制台持续报错**：离开页面后，之前注册的 scroll/resize 事件仍触发并试图更新已卸载组件的 state，React 报 `Can't perform a React state update on an unmounted component`。
4. **Chrome DevTools Memory 面板**：Class 级内存快照对比显示 EventListener 数量持续增长，Detached DOM 树无法被 GC 回收。

## 根因
`addEventListener` 未在组件卸载时对应的 `removeEventListener`。闭包持有 DOM 引用，DOM 被移除后监听器未解绑，闭包 → DOM → 事件链阻止 GC 回收。

核心机制：JavaScript 的 GC 基于引用计数 + 标记清除。事件监听器通过闭包持有组件 state/ref → 即使用了旧的 DOM 被移除，监听器仍然持有对 state 的引用 → GC 无法回收。

## 诊断方法

```bash
# 1. Chrome DevTools → Memory → Take Heap Snapshot
# 在 A 页面拍快照 → 切换到 B → 切回 A → 再拍快照
# 对比两次快照：按 "Detached" 过滤，如果出现 Detached HTMLElement → 泄漏

# 2. Chrome DevTools → Performance Monitor
# 观察 JS heap size 在页面切换中是否单调增长

# 3. 代码搜索：查找未配对 removeEventListener 的 addEventListener
grep -rn "addEventListener" src/ --include="*.tsx" --include="*.ts" -A 10 |
  grep -v "removeEventListener"

# 4. 检查 useEffect 是否有 cleanup 函数
grep -rn "useEffect.*addEventListener" src/ -A 5 | grep -v "return"
```

## 修复

```tsx
// ❌ 问题代码：resize 监听未清理
function Dashboard() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
-    window.addEventListener('resize', () => {
-      setWidth(window.innerWidth);
-    });
  }, []); // 组件卸载后事件仍在触发 state 更新

  return <div>Width: {width}</div>;
}

// ✅ 方案1: useEffect cleanup 移除监听
function Dashboard() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
+    const handler = () => setWidth(window.innerWidth);
+    window.addEventListener('resize', handler);
+    return () => window.removeEventListener('resize', handler);
  }, []);

  return <div>Width: {width}</div>;
}

// ✅ 方案2: AbortController 统一管理多个监听
function InteractivePanel() {
  useEffect(() => {
+    const controller = new AbortController();
+    const { signal } = controller;
+
+    document.addEventListener('scroll', handleScroll, { signal });
+    window.addEventListener('keydown', handleKeydown, { signal });
+    // signal 被 abort 时自动移除所有监听器
+
+    return () => controller.abort();
  }, []);
}
```

## 验证

```bash
# 1. Chrome DevTools Memory → 拍快照 → 切换页面 → 手动 GC → 再拍快照
# 在 Memory 面板点击垃圾桶图标（手动 GC）后 Detached DOM 应归零

# 2. Performance Monitor 监控
# 页面切换 20 次 → JS heap size 应呈锯齿状（升→GC→降）
# 异常：单调增长不回落

# 3. Jest 测试验证事件清理
test('should not leak event listeners', () => {
  const addSpy = jest.spyOn(window, 'addEventListener');
  const removeSpy = jest.spyOn(window, 'removeEventListener');
  const { unmount } = render(<Dashboard />);
  unmount();
  // 每个 addEventListener 必须对应一个同名的 removeEventListener
  expect(removeSpy).toHaveBeenCalled();
});
```

## 原理
浏览器的事件系统维护一个 `EventListenerMap`（事件类型 → 目标元素 → 监听器列表）。`addEventListener` 注册监听器并建立闭包引用链：`window → EventListenerMap → handler → [state/ref via closure]`。`removeEventListener` 的唯一依据是函数引用相等（`===`），因此必须传入与注册时完全相同的函数引用——这就是为什么不能用匿名函数而需要具名引用。

AbortController 的 `{ signal }` 选项（Chrome 88+ / Firefox 92+）允许一次性 abort 移除所有关联监听器，是管理多个事件的最佳实践。

## 架构视角 (Architecture Perspective)

### 架构层面根因
事件监听泄漏的架构根因是**全局事件总线反模式的滥用**：

- **全局 `window`/`document` 事件被当作"公共通信总线"**：组件 A 注册 `document.addEventListener('custom-update')`，组件 B dispatch 该事件，但 B 不知道 A 是否已卸载。这构成了隐藏的模块耦合——B 隐式地依赖 A 的存在。
- **事件生命周期与组件生命周期未绑定**：`addEventListener` 的注册是原生 DOM API，不受 React/Vue 虚拟 DOM 生命周期管理。React 的 useEffect cleanup 是唯一的防护点，但容易被遗漏。
- **缺少统一的事件管理层**：项目中散落着数十个 `addEventListener` 调用，没有统一的"事件注册器"确保所有注册都有对应的清理路径。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 强制 useEffect 中的 addEventListener 有对应 cleanup
// @typescript-eslint 规则 + 自定义规则
rules: {
  // 确保 useEffect 返回 cleanup 函数
  'react-hooks/exhaustive-deps': 'error',
}
// 自定义 no-bare-addeventlistener 规则:
// detect addEventListener inside useEffect without corresponding removeEventListener in returned function
```

**CI 防护**：
```bash
# 内存泄漏 E2E 检测 (Puppeteer)
test('no memory leak after 10 page switches', async () => {
  const metrics = [];
  for (let i = 0; i < 10; i++) {
    await page.click('[data-nav="dashboard"]');
    await page.click('[data-nav="settings"]');
    const { JSHeapUsedSize } = await page.metrics();
    metrics.push(JSHeapUsedSize);
  }
  // 前 5 次应趋于稳定（GC 回收），不应单调增长
  const trend = metrics[9] - metrics[4];
  expect(trend).toBeLessThan(10 * 1024 * 1024); // < 10 MB growth
});
```

**架构防护**：
- **使用 Context + hook 封装的事件系统替换全局事件总线**：
```tsx
// @app/events/useGlobalEvent.ts
function useGlobalEvent<K extends keyof WindowEventMap>(
  type: K, handler: (e: WindowEventMap[K]) => void
) {
  useEffect(() => {
    window.addEventListener(type, handler, { passive: true });
    return () => window.removeEventListener(type, handler);
  }, [type, handler]);
}

// 使用: const handleResize = useCallback((e) => { ... }, []);
//      useGlobalEvent('resize', handleResize);
```
- **事件注册器中间件**：所有 `addEventListener` 必须通过统一的 `EventRegistry` 注册，该注册器维护注册表并在路由变化时自动清理上一页面的所有事件。
- **模块边界规则**：组件文件(`.tsx`)不得直接调用 `document.addEventListener`，必须通过 `useGlobalEvent` hook。

**流程防护**：
- Code Review 检查项："此 useEffect 中是否有 addEventListener / setInterval / WebSocket？是否有对应的 cleanup 函数？"
- 每个 PR 需附 Chrome Memory 面板截图证明无内存泄漏。

### 跨层影响链

```
上游影响 (消费者):
  - 用户长时间使用后页面卡顿 → 标签页崩溃 → 数据丢失
  - 内存占用导致设备发热 → 电池快速耗尽 → 体验差（尤其在移动端）

下游影响 (依赖):
  - 第三方组件内的 addEventListener 同样泄漏（如轮播图/富文本编辑器）
  - 全局事件被多个已卸载组件同时响应 → 逻辑错乱
  - 内存压力导致浏览器 GC 频率增加 → 页面帧率下降

部署影响:
  - 泄漏累积是时间相关的 → 测试环境短时间运行不会发现
  - 生产环境用户长时间停留在 SPA 中 → 泄漏逐渐暴露
  - A/B 测试中泄漏版本的监控指标逐渐劣化

监控影响:
  - Sentry: 捕获 "unmounted component state update" warning
  - RUM: JS Heap Size 时间序列监控
  - 自定义指标: EventListener 数量（通过 `getEventListeners(window)`）
```

### 相关架构模式
- **反模式: 全局事件总线** (`patterns.md` 第六节) — `document.dispatchEvent` 是不可追踪、不可取消、不可类型安全的通信方式，应替换为 Context + Hook 的事件系统。
- **CQRS 前端化** (`patterns.md` 第三节) — resize/scroll 事件 → Command 层处理（计算尺寸）→ Query 层提供（组件订阅）。分离后 query 层可以通过 `useSyncExternalStore` 自动管理订阅生命周期。
- **依赖注入边界** — 事件源（`window`/`document`）作为依赖注入，而非组件直接引用，使得测试时可替换。

## 预防
1. 每个 `addEventListener` 必须对应一个 `removeEventListener`，且在 useEffect cleanup 中执行。
2. 使用 AbortController 管理多事件生命周期。
3. 封装 `useGlobalEvent` hook 统一事件注册/清理。
4. CI 集成页面切换内存泄漏检测。
