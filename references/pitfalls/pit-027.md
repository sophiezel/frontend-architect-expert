# pit-027: 移动端软键盘弹出导致布局异常

## 环境
- Android Chrome / WebView (Keyboard resize 行为不同于 iOS)
- iOS Safari / WKWebView (visualViewport 行为)
- 移动端 H5 / PWA / 混合应用

## 症状
1. **Android 软键盘弹出时 viewport 被压缩**：整个页面被挤压，`height: 100vh` 的元素变形，fixed 定位元素随键盘上移但超出可视区。
2. **iOS 键盘收起后留空白**：键盘弹出时页面被推上去，键盘收起后页面未弹回原位，顶部出现白条。
3. **Android 键盘弹出/收起触发 window.resize**：导致依赖 resize 的布局重计算异常，页面闪烁。
4. **`position: fixed; bottom: 0` 的行为不一致**：Android 上元素跟随键盘上移但可能被键盘遮挡，iOS 上元素不跟随键盘（留在原位被键盘覆盖）。
5. **键盘收起后 `input` 失焦但页面未滚动回来**：`window.scrollY` 异常偏移。

## 根因
不同 OS 对 `visualViewport` vs `layoutViewport` 行为不同，未使用 VisualViewport API。

核心机制：
- **Android**：键盘弹出时 resize `layoutViewport`（压缩 viewport），`window.innerHeight` 改变，所有 `vh` 单位重新计算。fixed 元素随 layoutViewport 底部移动。
- **iOS**：键盘弹出时不 resize `layoutViewport`（`window.innerHeight` 不变），但缩小 `visualViewport`（可视区域被覆盖）。fixed 元素不跟随 keyboard 移动——因为 fixed 相对于 layoutViewport 而非 visualViewport 定位。

## 修复

```css
/* ✅ 使用动态 viewport height 替代 100vh */
:root {
  --app-height: 100%;
  --app-height: 100dvh; /* dynamic viewport height (Safari 15.4+) */
}

@supports not (height: 100dvh) {
  :root {
    --app-height: -webkit-fill-available;
  }
}

.page-container {
  min-height: var(--app-height);
}
```

```tsx
// ✅ VisualViewport API 动态管理视口
function useVisualViewport() {
  const [viewportHeight, setViewportHeight] = useState(window.innerHeight);

  useEffect(() => {
    const handleResize = () => {
      const height = window.visualViewport?.height || window.innerHeight;
      setViewportHeight(height);
      document.documentElement.style.setProperty(
        '--viewport-height', `${height}px`);
    };

    window.visualViewport?.addEventListener('resize', handleResize);
    window.visualViewport?.addEventListener('scroll', handleResize);

    return () => {
      window.visualViewport?.removeEventListener('resize', handleResize);
      window.visualViewport?.removeEventListener('scroll', handleResize);
    };
  }, []);

  return viewportHeight;
}
```

```css
/* ✅ 键盘弹出时 fixed 底部元素的策略 */
.fixed-bottom-input-area {
  position: fixed;
  bottom: 0;
  /* 使用 JS 动态设置的 --viewport-offset-bottom */
  transform: translateY(calc(-1 * var(--viewport-offset-bottom, 0px)));
  padding-bottom: env(safe-area-inset-bottom, 0px);
}
```

## 原理
`visualViewport` API 是 W3C 规范，用于解决 `layoutViewport`（`window.innerHeight`）在键盘场景下的不足。`visualViewport.height` 返回当前实际可见区域高度（扣除键盘覆盖部分），`visualViewport.offsetTop` 返回当前可见区域相对 layoutViewport 的偏移。

`100dvh` (dynamic viewport height) 是 CSS Values Level 4 规范中的新单位，会根据当前 viewport 状态（含/不含键盘）动态调整。iOS Safari 15.4+、Chrome 108+ 支持。

## 架构视角 (Architecture Perspective)

### 架构层面根因
移动端键盘问题的架构根因是**移动端键盘行为未系统化处理**：

- **键盘弹出/收起是跨页面的全局行为，但被当作"输入框的局部问题"**：键盘改变的是整个视口（影响所有 fixed 元素、全屏布局、CSS vh 单位），但这些影响需要由全局 `<MobileViewportManager>` 统一处理，而非分散在各个输入框组件的 `onFocus` / `onBlur` 中。
- **Android 和 iOS 的键盘行为差异未在架构中建模**：同一个 Web 应用运行在两个平台，但代码中将它们当作"同一个浏览器"。Android 压缩 layoutViewport（触发 resize + vh 重算），iOS 仅调整 visualViewport（不触发 resize），需要不同的适配策略但代码未区分。
- **fixed 元素与键盘的交互策略缺失**：设计系统未定义"底部固定元素在键盘弹出时的行为"——是跟随键盘上移（聊天输入框）、保持在原位（底部导航栏隐藏）、还是隐藏自身。

**CI 防护**：
```bash
# Playwright 移动端键盘 E2E 测试
test('keyboard does not break layout', async ({ page }) => {
  await page.setViewportSize({ width: 412, height: 915 }); // Pixel 7
  await page.goto('/form');
  const input = page.locator('input').last();
  await input.focus();
  await page.waitForTimeout(500);
  // 输入框应在可视区域内
  const inView = await input.evaluate(el => {
    const rect = el.getBoundingClientRect();
    const vh = window.visualViewport?.height || window.innerHeight;
    return rect.bottom <= vh && rect.top >= 0;
  });
  expect(inView).toBe(true);
});
```

**架构防护**：
- **MobileViewportManager 统一管理**：初始化时注册 visualViewport 监听，键盘状态变化时更新全局 CSS 变量（`--viewport-height` / `--keyboard-offset` / `--keyboard-visible`），所有组件通过 CSS 变量响应而非各自监听。
```tsx
// @app/mobile/MobileViewportManager.tsx
function MobileViewportManager({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    const onViewportChange = () => {
      const height = window.visualViewport?.height || window.innerHeight;
      const offsetTop = window.visualViewport?.offsetTop || 0;
      const root = document.documentElement;
      root.style.setProperty('--viewport-height', `${height}px`);
      root.style.setProperty('--keyboard-offset', `${offsetTop}px`);
      root.classList.toggle('keyboard-visible',
        height < window.innerHeight - 100);
    };
    window.visualViewport?.addEventListener('resize', onViewportChange);
    window.visualViewport?.addEventListener('scroll', onViewportChange);
    return () => {
      window.visualViewport?.removeEventListener('resize', onViewportChange);
      window.visualViewport?.removeEventListener('scroll', onViewportChange);
    };
  }, []);
  return <>{children}</>;
}
// 在 App 根组件包裹
```
- **键盘行为声明**：设计系统中为布局组件（`<BottomBar>` / `<ChatInput>` / `<FixedFooter>`）声明键盘行为枚举：`follow-keyboard` / `fixed-to-layout` / `hide-on-keyboard`。

### 跨层影响链

```
上游影响 (消费者):
  - 登录/注册页输入框被键盘遮挡 → 用户放弃注册
  - 聊天页输入框行为异常 → 核心功能不可用
  - 支付密码输入框不可见 → 交易失败

下游影响 (依赖):
  - 第三方组件（DatePicker/Select）的弹出层与键盘叠加冲突
  - fixed 底部导航在键盘场景下遮挡内容
  - 富文本编辑器工具栏被键盘遮挡

部署影响:
  - Android 厂商碎片化: 小米/OPPO/vivo 键盘 resize 策略不同
  - iOS 系统版本升级可能改变 visualViewport 行为
  - WebView 容器的键盘策略受 App 原生配置影响

监控影响:
  - RUM: 按平台统计 keyboardVisible 时的表单完成率
  - 自定义事件: visualViewport 变化序列（检测异常偏移）
  - E2E: 必须在真实移动设备上执行（模拟器的键盘行为不一致）
```

## 预防
1. 使用 `window.visualViewport` API 替代 `window.innerHeight` 做移动端视口计算。
2. 使用 `100dvh` 或 JS 动态设置的 `--viewport-height` CSS 变量替代 `100vh`。
3. 封装 `<MobileViewportManager>` 在根组件中统一处理键盘状态。
4. 为设计系统中的底部固定组件声明键盘行为，而非每个页面自行处理。
