# pit-013: HMR 热更新导致组件状态丢失

## 环境
- Webpack Dev Server + HMR（Hot Module Replacement）
- React / Vue 等框架

## 症状
修改代码后页面无刷新自动更新，但组件内的 state（表单输入、计数器、展开/折叠状态）全部重置为初始值。每次保存文件就要重新填写表单。

## 根因
HMR 按模块边界更新。如果修改的模块恰好是状态所在模块，HMR 会用新模块替换旧模块，React state 和 DOM state 随之丢失。原因包括：
1. 组件定义文件被修改 → 组件函数体替换 → React 认为是新组件 → 卸载旧组件。
2. `module.hot.accept` 配置不完整，导致整棵树重新挂载。
3. React Fast Refresh 对某些模式不兼容（如 class 组件、导出非组件值）。
4. 修改了 Hooks 依赖链上游的模块。

## 诊断方法

```bash
# 1. 观察 HMR 日志
# Webpack Dev Server 控制台 → 查看 [HMR] 前缀日志
# "Updated" = 热更新成功（保留状态）
# "Reloaded" = 全量刷新（状态丢失）

# 2. 组件挂载/卸载日志
# 在组件中添加：
useEffect(() => {
  console.log('[HMR DEBUG] 组件挂载:', Date.now());
  return () => console.log('[HMR DEBUG] 组件卸载');
}, []);
# 修改代码后如果看到"卸载"→"挂载"→ 状态丢失

# 3. React DevTools 检查
# Components → 修改代码后观察组件树
# 如果组件 key 或 type 发生变化 → 导致卸载重建

# 4. 检查文件导出
grep -n "export" src/components/ProblemComponent.tsx
# 如果文件同时导出组件和常量/工具函数 → 可能干扰 Fast Refresh

# 5. 检查 Babel/Webpack 配置
cat babel.config.js | grep "react-refresh"
cat webpack.config.js | grep "ReactRefresh"
# 确认 React Fast Refresh 插件已配置
```

## 修复

```diff
// ❌ 问题：组件文件同时导出组件和工具函数
// useCounter.ts
export function useCounter() {
  const [count, setCount] = useState(0);
  return { count, increment: () => setCount(c => c + 1) };
}
- export const MAX_COUNT = 100; // 非 React 导出 → 可能干扰 Fast Refresh

// ✅ 修复：分离 React 组件/Hook 与非组件导出
// useCounter.ts
export function useCounter() {
  const [count, setCount] = useState(0);
  return { count, increment: () => setCount(c => c + 1) };
}
+ // constants.ts
+ export const MAX_COUNT = 100;
```

```diff
// ❌ 问题：命名导出非组件值被修改时触发全量刷新
// Button.tsx
export const Button = (props) => <button {...props} />;
- export const buttonVariants = ['primary', 'secondary'];

// ✅ 修复：保持每个文件只导出 React 组件
// Button.tsx
export const Button = (props) => <button {...props} />;
+ // buttonVariants.ts
+ export const buttonVariants = ['primary', 'secondary'];
```

```js
// babel.config.js
module.exports = {
  plugins: [
    process.env.NODE_ENV === 'development' && 'react-refresh/babel',
  ].filter(Boolean),
};
```

## 原理
HMR（Hot Module Replacement）通过 Webpack 的 `module.hot.accept` API 在运行时替换模块。React Fast Refresh 在此基础上进一步优化：它能识别 React 组件签名，在替换组件模块时保留组件内的 state 和 DOM state。

Fast Refresh 的限制：
- 不保留 class 组件的 state（仅函数组件）
- 不保留非 React 导出的变化（会触发全量刷新）
- 匿名默认导出 `export default function() {}` 不被支持

## 架构视角 (Architecture Perspective)

### 架构层面根因
HMR 状态丢失的架构根因是**模块组织策略未考虑开发时体验**：
- **文件职责混合**：组件文件和常量/工具文件职责混合，打破了 HMR 的模块更新边界。从架构角度看，一个文件应只承担一种角色——"视图组件"或"逻辑 Hook"或"常量定义"或"类型声明"。混合文件破坏了模块的热替换粒度，因为修改任何一部分都会影响整个文件。
- **模块边界未按"变更频率"划分**：常量和工具函数的变更频率与 UI 组件的变更频率不同。将它们放在同一个文件中意味着低频变更（常量）的修改也会触发高频变更（组件）的状态丢失。架构应按照"变更频率"作为模块划分的第二维度（第一维度是职责）。
- **缺少开发模式下的架构保障**：Fast Refresh 的限制（如不支持匿名默认导出、不支持 class 组件）实际上是架构约束的信号——说明这些模式在模块热替换场景下是不稳定的。架构应将这些约束纳入编码规范，而非等待开发者在开发中遇到问题再修复。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 强制命名导出和单一职责文件
rules: {
  // 禁止匿名默认导出（Fast Refresh 要求）
  'import/no-anonymous-default-export': ['error', {
    allowArray: false,
    allowArrowFunction: false,
    allowAnonymousClass: false,
    allowAnonymousFunction: false,
    allowLiteral: false,
    allowObject: false,
  }],
  // 禁止在同一文件中混合组件导出和非组件导出
  'import/exports-last': 'warn',
}

// 自定义 ESLint 规则: 检测 .tsx 文件同时导出非 React 值
// rules/custom/no-mixed-exports.js
// 如果 .tsx 文件 export 了既非组件也非 Hook 的值 → 警告
```

**CI 防护**：
```bash
# CI 检测: 所有组件必须有命名标识（Fast Refresh 兼容性）
grep -rn "export default function(" src/components/ --include="*.tsx" &&
  echo "WARNING: 匿名默认导出不被 Fast Refresh 支持" 

# 测试: HMR 状态保留
# Playwright + HMR 测试
test('HMR 应保留组件状态', async ({ page }) => {
  await page.goto('http://localhost:3000');
  // 填写表单
  await page.fill('input[name="email"]', 'test@example.com');
  // 触发 HMR（通过修改源码或模拟）
  await page.evaluate(() => window.__HMR_SIMULATE__('Button.tsx'));
  await page.waitForTimeout(500);
  // 表单内容应保留
  expect(await page.inputValue('input[name="email"]')).toBe('test@example.com');
});
```

**架构防护**：
- **文件组织规范强制执行**：
```
src/
  components/
    Button/
      Button.tsx          // 只导出 React 组件
      Button.test.tsx     // 测试
      Button.stories.tsx  // Storybook
      Button.module.css   // 样式
      index.ts            // barrel export
      variants.ts         // 非组件常量（分离文件）
  constants/
    MAX_COUNT.ts          // 全局常量，独立于组件
```
- **模块热替换协议**：为每个文件定义 `module.hot.accept` 的精确行为——哪些导出是可热替换的，哪些需要全量刷新。通过构建工具插件自动生成正确的 `module.hot` 模板块。

**流程防护**：
- Code Review 检查项："组件文件是否只导出组件/Hook？常量/类型/工具函数是否在独立文件中？"
- 架构评审：新增模块时，确认模块的热替换策略（full reload vs state-preserving vs accept）。

### 跨层影响链

```
上游影响 (消费者):
  - 修改自定义 Hook → 所有使用该 Hook 的组件状态丢失
  - 修改 Context Provider → 所有 Consumer 的局部状态丢失

下游影响 (依赖):
  - 开发效率下降：每次保存需要重新操作到之前的状态
  - 复杂交互（多步骤表单、拖拽排序）的开发调试极其困难
  - Storybook 的 HMR 同样受影响

部署影响:
  - N/A（仅影响开发环境）

监控影响:
  - 开发团队反馈"保存后状态丢失"的频率
  - HMR 日志中 "Reloaded" vs "Updated" 的比例
  - 团队开发效率指标（表单调试时间、上下文恢复时间）
```

### 相关架构模式
- **组件组合模式** (`patterns.md` 第二节) — Compound Components 模式中父组件状态通过 Context 传递给子组件，HMR 更新子组件时状态保留依赖于父组件不重新挂载。
- **反模式: 过早抽象** (`patterns.md` 第六节) — 开发阶段频繁修改的代码如果被过早抽象为共享模块，HMR 的任何修改都会影响多个消费组件。HMR 模块边界应遵循"稳定的边界早抽象，变化的边界晚抽象"。
- **Monorepo 管理** (`patterns.md` 第五节) — Monorepo 中修改共享包（packages/shared）会触发所有消费它的应用 HMR 刷新，应限制开发阶段的跨包依赖。

## 预防

1. **使用 React Fast Refresh**（`@pmmmwh/react-refresh-webpack-plugin` / Vite 内置）。
2. **文件组织规范**：每个文件只导出一个组件或 Hook，避免混入常量/工具函数。
3. **避免匿名默认导出**：`export default function MyComponent() {}` 而非 `export default function() {}`。
4. **使用 `useRef` 保留关键状态**：对于需要在 HMR 后保留的状态，考虑手动持久化。
