# pit-075: Edge 与 Origin 数据不一致 — KV 最终一致性导致

## 环境
- Cloudflare Workers KV 或其他最终一致存储 (Vercel KV)
- Edge Function 写入数据后立即读取
- 功能依赖写入后立即可见的数据状态

## 症状
用户提交表单后点击"查看结果"页面，显示"数据为空"或旧状态。刷新 2-3 次后数据出现了。更严重的是：用户操作 A 的副作用（如"标记已读"）在后续请求中不可见，导致用户重复操作或认为操作失败。多个地理位置（不同 Edge 节点）的用户看到不同的数据——亚洲用户看到"库存: 5"，美洲用户看到"库存: 3"。

典型场景：
- 购物车：添加商品后跳转到购物车页，商品不在列表中（已添加但 KV 未传播）
- 发布内容：发布文章后跳转到文章列表，文章未出现
- 配置更新：管理员更新配置项后，其他用户仍看到旧配置
- 计数器：点赞后的计数不更新（因为读写分离到不同边缘节点）

## 根因
**最终一致性系统中"写后读"的时间窗口不一致**。核心机制：

1. **KV 是异步写入 + 最终一致传播**：Cloudflare Workers KV 的 `put()` 在调用后立即返回 Promise resolved，但数据实际到全球所有边缘节点的传播**最长可达 60 秒**。写入到 edge-asia 的 KV 数据在 edge-us 上可能 30 秒后才可见。

2. **读取命中"陈旧"边缘节点**：`kv.get()` 从最近的边缘节点读取，如果该节点的数据尚未同步，返回旧值或 null。

3. **不保证 Read-Your-Writes**：同一个用户 / 同一 cookie 会话的连续请求可能路由到不同的边缘节点，导致用户刚写入的数据在下一个请求中不可见。

4. **技术细节**：CF Workers KV 的底层是**中心化存储 + 边缘 CDN 缓存**。写入到中心存储是异步的（最多几秒），从中心同步到各边缘 CDN 节点又是一轮（取决于 TTL 和刷新策略）。

5. **Vercel KV**：基于 Redis + 最终一致性，虽然延迟低很多（通常 < 1s），但同样不保证严格 Read-Your-Writes。

## 原理
CAP 定理在边缘 KV 中的体现：`Consistency`（一致性）、`Availability`（可用性）、`Partition Tolerance`（分区容错）。边缘 KV 选择 AP 模型——保证可用性和分区容错，牺牲强一致性。数据写入中心后，通过异步复制传播到全球边缘节点。在传播期间（通常 < 10s，极端可达 60s），不同节点可能返回不同版本的数据。

这与传统数据库的"写后读"完全不同。传统 RDBMS 写入返回后，任何后续读取都保证看到新数据。KV 写入返回后，后续读取**可能**看到新数据，**但不确定**。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **数据一致性要求未分级**：所有数据都放进 KV，但购物车需要强一致（Durable Objects），配置可以最终一致（KV）。
- **读写模式假设错误**：继承自传统 Web 框架的"save → redirect → read"模式在边缘环境中被打破。
- **客户端缓存策略加剧不一致**：浏览器缓存了一个陈旧响应，叠加 KV 的最终一致性，用户看到更旧的数据。

### 系统级预防策略
- **数据层分层**：强一致数据 → Durable Objects；最终一致数据 → KV；不可变数据 → R2。
- **写入即返回模式**：write 操作直接返回写入值，不依赖后续 read。
- **版本号 / 事务 ID**：每次写入附加递增版本号，读取时检查版本是否满足客户端期望。
- **客户端重试 + 指数退避**：对读请求检测"新鲜度"，不满足则退避重试。

### 跨层影响链
- 用户：重复操作（以为未成功）→ 重复订单/重复扣款
- 业务流程：状态机依赖的"写入 → 读取确认"流程断裂
- 客服：处理大量"我操作了但没反应"的投诉

## 预防

1. **原则：写入后不要读 KV**。如果必须读，用 Durable Objects。
2. **使用 D1（边缘 SQLite）处理关系数据**：D1 提供更强的一致性保证。
3. **对关键路径使用 Durable Objects**：购物车、实时协作、计数器。
4. **监控 KV 陈旧读取率**：在日志中记录 `kv.get()` 返回 null 或版本过旧的比例。

## 回退

```bash
# 紧急：将 KV 操作回退到 Origin
# Edge 中只做路由，KV 操作全部在 Origin 中完成
export default {
  async fetch(request, env) {
    if (request.method === 'POST' || request.method === 'PUT') {
      return fetch(request); // 直接转发到 Origin
    }
    // 只读 KV（不写）
    const cached = await env.KV.get(new URL(request.url).pathname);
    if (cached) return new Response(cached);
    return fetch(request);
  }
};
```
