# pit-046: RUM 性能数据采集失真 — CLS value/delta 误用导致数据无效

## 环境
- 使用 `web-vitals` 或等效 RUM SDK 的前端应用
- 接入 Datadog RUM / Grafana Faro / 自建性能采集管线
- 后端起聚合存储（ClickHouse / Prometheus / InfluxDB）

## 症状
性能看板上 CLS (Cumulative Layout Shift) 聚合值持续偏高，显示 p75 达到 0.5~1.2，远超 Google 推荐的 "good" 阈值 0.1。产品经理质疑前端质量，但用户调研和客服渠道均未收到布局抖动相关投诉。前端团队反复排查页面无明显布局偏移问题。更诡异的是：同一 session 内 CLS 时间序列图呈单调递增斜线——页面已完全渲染后 CLS 仍在持续"增长"，看起来像是泄漏了。

## 根因
`web-vitals` 库提供了两个关键字段：`metric.value`（累计值）和 `metric.delta`（增量值）。CLS 是一个在整个页面生命周期内只增不减的指标——每当发生新的布局偏移，CLS 的累积值累加，但 `metric.value` 始终返回截至当前的累计值，而 `metric.delta` 返回自上次回调以来的增量。

如果 RUM 上报逻辑中错误地使用了 `metric.value`，且配置了 `reportAllChanges: true`（或多次手动调用上报），则每次回调都会发送从头累计到当前的完整 CLS 值。后端聚合（取平均/分位）时，这些"虚高"的重复累计值被当作独立样本参与计算，导致最终数值严重偏高——通常是真实值的 5~10 倍。

```typescript
// web-vitals 内部行为（简化）
// CLS 在整个生命周期中:
//   第1次 shift 0.05 → cumulative = 0.05, delta = 0.05
//   第2次 shift 0.03 → cumulative = 0.08, delta = 0.03
//   第3次 shift 0.02 → cumulative = 0.10, delta = 0.02
// 
// 如果用 value (cumulative) 上报三次:
//   后端收到: 0.05, 0.08, 0.10
//   p75 = (0.08 + 0.10) / 2 = 0.09  ← 看起来还行
//   但如果用户做了若干交互后又有几次微小 shift:
//   cumulative 继续累加到 0.15, 0.18, 0.22...
//   后端聚合时 p75 就被严重拉高
```

此外，还有一个变体根因：在 `visibilitychange` 触发时（页面进入后台）上报终态值，但上报的是 `metric.value` 而非 `metric.delta`。由于 CLS 在页面后台时继续累积（某些浏览器行为），终态上报的 value 又一次包含了已上报过的部分。

## 修复

```diff
// ❌ 问题代码: 使用 metric.value 上报 CLS
import { onCLS } from 'web-vitals';

onCLS((metric) => {
  // BUG: value 是累计值，每次上报都在变大
-  monitor.send('web_vitals', {
-    name: 'CLS',
-    value: metric.value,
-    rating: metric.rating,
-  });
});

// ❌ 更糟糕: reportAllChanges + value 组合
onCLS((metric) => {
-  monitor.send('web_vitals', {
-    name: 'CLS',
-    value: metric.value, // 每次回调都发送更大的累计值
-  });
-}, { reportAllChanges: true });

+// ✅ 修复 1: 始终使用 metric.delta
+onCLS((metric) => {
+  monitor.send('web_vitals', {
+    name: 'CLS',
+    value: metric.delta,   // 仅本次回调的增量
+    rating: metric.rating,
+    id: metric.id,          // 保留 metric.id 用于后端按 session 聚合
+  });
+});

+// ✅ 修复 2: 终态上报策略 (推荐)
+// 不配置 reportAllChanges，仅在页面隐藏时上报最终 CLS
+let clsValue = 0;
+
+onCLS((metric) => {
+  clsValue = metric.value; // 跟踪累计值
+});
+
+// 在页面进入后台时一次性上报最终 CLS
+document.addEventListener('visibilitychange', () => {
+  if (document.visibilityState === 'hidden' && clsValue > 0) {
+    monitor.send('web_vitals', {
+      name: 'CLS',
+      value: clsValue,      // 此时用 value 是安全的——仅上报一次
+      rating: clsValue <= 0.1 ? 'good' : clsValue <= 0.25 ? 'needs-improvement' : 'poor',
+    });
+  }
+});

+// ✅ 修复 3: 如果需要实时追踪，上报 delta 并在后端 SUM 聚合
+onCLS((metric) => {
+  monitor.send('web_vitals', {
+    name: 'CLS',
+    value: metric.delta,    // 发送增量
+    id: metric.id,
+    metric_type: 'delta',   // 标注类型，后端根据此字段决定聚合方式
+  });
+}, { reportAllChanges: true });
+// 后端聚合: SELECT sum(cls_delta) as total_cls FROM events GROUP BY session_id, metric_id
```

## 原理
CLS 指标由 Layout Instability API 驱动。每个 `LayoutShift` 事件有一个 `value` 属性（该次偏移的分数：impact fraction × distance fraction）。浏览器将 session 内所有 `LayoutShift.value` 累加形成 session CLS 累计值。

`web-vitals` 库在此基础上封装：
- `metric.value`：截至当前累积的 CLS（每次 shift 后递增，永不减少）
- `metric.delta`：`metric.value` 与上一次回调时的差值
- `metric.id`：唯一标识一次 CLS 测量（session 内不变）

当 `reportAllChanges: true` 时，每次新的 layout shift 都会触发回调。如果每次回调都上报 `metric.value`，后端会收到一串递增数字。聚合时 p50/p75/p95 被严重抬高，因为"同一个用户的同一 session 的同一个 CLS 事件"被计入了多次。

正确做法取决于后端存储模型：
- **增量模型**（推荐）：上报 `delta`，后端按 `metric_id + session_id` 做 SUM 得到 session 级 CLS
- **终态模型**：仅在 `visibilitychange` (hidden) 时上报一次 `value`，后端直接使用

## 架构视角 (Architecture Perspective)

### 架构层面根因
RUM 数据失真的根因不是"用错了字段名"，而是**采集层与聚合层之间缺少语义契约**。`web-vitals` 的 API 设计已经区分了 `value` 和 `delta`，但采集 SDK 的开发者和后端聚合管线的开发者对"每个指标的数据模型"没有达成一致理解。

核心架构缺陷：**缺少 Metrics Semantics Validator**——一个在采集层校验每个上报事件是否符合该指标语义规则的程序。CLS 必须单调递增、LCP 不可超过 TTFB + 资源加载时间理论下限、INP 必须是 interaction 延迟而非 navigation 延迟。这些语义规则应当在采集 SDK 的上报管线中做 assertion（仅在 dev/staging 环境），捕获不符合语义的上报并在 CI 中报警。

更根本的，这是**可观测性数据建模**的缺失。每个 Web Vital 都有自己的数据模型：CLS 是累积型、LCP 是取最大型、INP 是取最差型、FCP 是首次型。没有哪种通用聚合策略（avg、p75、max）适合所有指标。架构上应该在指标注册时就声明其聚合策略。

### 系统级预防策略

**静态检查**：
```bash
# ESLint 规则：禁止在上报函数中直接使用 metric.value（CLS 场景）
# .eslintrc.js
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'CallExpression[callee.name="onCLS"] Identifier[name="value"]' +
              '[property.name="value"]',
    message: 'CLS 上报必须使用 metric.delta，禁止使用 metric.value。参考 pit-046'
  }]
}
```

```typescript
// TypeScript 类型约束: 指标语义标注
type MetricSemantics = 'cumulative' | 'maximum' | 'worst' | 'first';

interface MetricDefinition {
  name: string;
  semantics: MetricSemantics;
  aggregation: 'delta_sum' | 'last_value' | 'max_value';
  validate: (metric: WebVitalsMetric) => boolean;
}

const CLS_DEFINITION: MetricDefinition = {
  name: 'CLS',
  semantics: 'cumulative',
  aggregation: 'delta_sum',
  validate: (m) => m.delta >= 0 && m.delta <= 1, // CLS delta 应在 [0, 1]
};
```

**CI 门禁**：
```bash
# E2E 测试中检查上报的 CLS 数据格式
test('RUM CLS 事件格式正确', async ({ page }) => {
  const rumEvents: any[] = [];
  page.on('request', req => {
    if (req.url().includes('/rum/events')) {
      const body = JSON.parse(req.postData() || '{}');
      if (body.name === 'CLS') rumEvents.push(body);
    }
  });
  await page.goto('/');
  await page.waitForTimeout(5000);
  
  // 验证: CLS delta 不为负且不超过 1
  for (const evt of rumEvents) {
    expect(evt.value).toBeGreaterThanOrEqual(0);
    expect(evt.value).toBeLessThanOrEqual(1);
  }
  
  // 验证: 如果多次上报，delta 之和 ≈ final value
  const totalDelta = rumEvents.reduce((s, e) => s + e.value, 0);
  expect(totalDelta).toBeLessThanOrEqual(1); // 理论上界
});

# Metrics Semantics Validator 集成到 CI
npx metrics-validator --env staging --metrics CLS,LCP,INP,FCP
# 输出: CLS delta monotonic? ✓  LCP ≤ theoretical_max? ✗ (5 events)
```

**架构约束**：
- 所有 Web Vital 指标必须注册其 `semantics` 和 `aggregation` 策略，不可默认
- RUM SDK 上报管线必须包含 `MetricsSemanticsValidator` 中间件（staging 报错，production 采样 1% 记录）
- 后端聚合层必须根据指标注册的聚合策略选择聚合函数（CLS 用 delta sum，LCP 用 max）
- 性能看板必须显示"样本有效性比率"——被 Metrics Semantics Validator 拒绝的事件数 / 总事件数

### 跨层影响链

```
上游影响 (产品决策):
  - CLS 虚高 → 产品认为"前端质量差" → 砍功能优化性能 → 误伤用户体验
  - 性能看板不可信 → PM 对工程团队失去信心 → 用"感觉"而非数据做决策

下游影响 (后端):
  - 虚高数据写入 ClickHouse → 存储空间浪费（同样的真实 event 被重复写入）
  - 聚合计算浪费资源 → 查询"虚高"数据产生的 Dashboard 和告警都是无意义的算力消耗

部署影响:
  - 基于虚高数据的自动回滚 → 好的部署被错误回滚
  - 性能预算门禁误报 → CI 频繁 block 正常 PR → 团队开始忽视 CI 结果

监控影响:
  - RUM 数据质量下降 → 告警阈值失准 → CLS "恶化"告警频繁触发
  - 团队对监控系统的信任崩塌 → 告警被静音 → 真正的性能退化无法发现
  - 与 pit-044 形成连锁: 监控盲区 + 数据失真 = 完全不可观测
```

### 相关架构模式
- **Metrics Contract Pattern** (`patterns.md`) — 每个指标必须声明其数据模型契约（类型/范围/聚合方式/有效期），采集层和聚合层基于同一契约工作。
- **Telemetry Pipeline** (`decisions.md`) — 观测数据应经过 validate → transform → filter → route → aggregate 五阶段管线，而非裸上报。
- **Data Quality SLA** (`decisions.md`) — 性能数据的质量应该有明确 SLA（如"CLS 数据有效率 > 99.9%"），不达标的指标不应出现在生产 Dashboard 中。
- 与 **pit-044 (监控盲区)** 互补——盲区是无数据，失真是假数据，两者叠加导致"有数据的幻觉"是更危险的架构反模式。

## 预防
1. **Metrics Registry**：为每个 Web Vital 建立元数据注册表，包含 semantics / aggregation / validation 规则
2. **SDK 层类型约束**：封装 `web-vitals` 的 `onCLS/onLCP/onINP` 为内部 `reportMetric` 函数，强制传入 `metric.delta` 或明确标注 `metric.value` 的场景
3. **Staging 环境 assertion**：在 staging 环境的 RUM SDK 中开启 Metrics Semantics Validator，严格拒绝违反语义的上报
4. **Documentation**：在 RUM SDK 文档中明确标注每个指标的 `value` vs `delta` 使用指南和聚合策略
5. **Dashboard 标注**：性能看板中每个指标应标注"聚合方式：delta sum / max value"，让消费者理解数据语义
