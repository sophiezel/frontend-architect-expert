# pit-007: z-index 层叠上下文陷阱

## 环境
- 所有浏览器
- 尤其常见于 Modal/Drawer/Dropdown 等浮层组件

## 症状
设置了 `z-index: 9999` 的元素仍被 `z-index: 1` 的元素覆盖。无论把 z-index 调得多大都没用。

## 根因
CSS 层叠上下文（Stacking Context）是独立的层级空间。子元素的 z-index 只在当前层叠上下文内比较。当两个兄弟元素属于不同的层叠上下文，它们的 z-index 不可直接比较——由父级层叠上下文的 z-index 决定。

创建层叠上下文的条件（部分）：
- `position: relative/absolute` + `z-index` 不为 `auto`
- `opacity < 1`
- `transform`, `filter`, `perspective` 不为 `none`
- `will-change` 为上述任一属性
- `isolation: isolate`

## 诊断方法

```bash
# 1. Chrome DevTools → Layers 面板
# 打开 DevTools → 右上角 ⋮ → More tools → Layers
# 查看层叠上下文边界，每个 3D 矩形是一个层叠上下文

# 2. 控制台诊断脚本
# 选中被覆盖的元素（$0 是当前选中元素）
let el = $0;
while (el) {
  const style = getComputedStyle(el);
  if (style.zIndex !== 'auto' || parseFloat(style.opacity) < 1 || style.transform !== 'none') {
    console.log('层叠上下文创建者：', el, {
      zIndex: style.zIndex,
      opacity: style.opacity,
      transform: style.transform
    });
  }
  el = el.parentElement;
}

# 3. 元素覆盖关系检测
# 在 DevTools Elements 面板，hover 覆盖元素和被覆盖元素
# 观察它们在页面上的高亮区域
# 如果被覆盖元素的高亮完全在覆盖元素下方 → 层叠问题确认
```

## 修复

```css
/* ❌ 问题结构：Modal 的 z-index 无论多大都被侧边栏覆盖 */
.sidebar {
  position: relative;
  z-index: 10; /* 创建了层叠上下文 */
}
.sidebar .dropdown {
  z-index: 9999; /* 只在 sidebar 的层叠上下文内有效 */
}
.main-content .modal {
  z-index: 20; /* 主内容区被 sidebar 覆盖 */
}

/* ✅ 修复：打破约束，将浮层提升到更高层级 */
.sidebar {
  position: relative;
  z-index: 1;
}
/* 将 modal 渲染到 body 下，与 sidebar 并列 */
body > .modal-overlay {
  position: fixed;
  z-index: 1000; /* 全局最高层，与 sidebar 在同一层叠上下文比较 */
}
```

```css
/* ❌ 问题：transform 隐式创建层叠上下文 */
.card {
  transform: translateY(0); /* 动画重置 → 创建层叠上下文 */
  position: relative;
}
.card .tooltip {
  z-index: 999; /* 被相邻 .card 的 z-index 限制 */
}

/* ✅ 修复：动画结束后移除 transform */
.card {
  position: relative;
}
.card.animating {
  transform: translateY(-10px);
}
.card .tooltip {
  z-index: 999;
}
/* 更好的方案：用 Portal 把 tooltip 渲染到 document.body */
```

```bash
# 修复后: Layers 面板中浮层应位于独立的顶层
# 不再嵌套在侧边栏的层叠上下文内

# 2. 交互测试
# 打开 Modal/Dropdown，观察是否被其他元素覆盖
# 修复后：浮层应正常显示在最前面

# 3. z-index 数值检查
# 在控制台执行：
Array.from(document.querySelectorAll('*'))
  .filter(el => getComputedStyle(el).zIndex !== 'auto')
  .map(el => ({ el, z: getComputedStyle(el).zIndex }))
  .sort((a, b) => b.z - a.z);
# 修复后：浮层 z-index 应为最高值

# 4. Portal 渲染验证
# 在 React DevTools 中查看 Modal 组件
# 确认其渲染位置在 document.body 直接子级，而非嵌套在某组件内
```

## 原理
CSS 层叠上下文是 HTML 元素沿 Z 轴排列的独立空间。层叠上下文的创建条件包括 `z-index` + 定位、`opacity < 1`、`transform` 等。最关键的理解：层叠上下文是"隔离"的——子元素的 z-index 只在当前上下文中与其他同级元素比较。两个不同层叠上下文中的元素，它们的 z-index 不直接可比。

这类似于两个箱子：箱子 A 放在箱子 B 上面，无论箱子 A 里面的东西位置多低，它永远在箱子 B 里面的所有东西之上。

## 架构视角 (Architecture Perspective)

### 架构层面根因
z-index 陷阱的架构根因是**浮层渲染层缺乏全局层级管理**：
- **DOM 结构决定了 Z 轴层级**：浮层（Modal/Dropdown/Tooltip）被渲染在其触发元素的 DOM 子树中，Z 轴层级受限于祖先的层叠上下文。这是一种"位置耦合"——浮层的渲染位置（React 组件树中的位置）和显示层级（Z 轴）被绑定在一起，而它们应该解耦。
- **全局 Z 轴空间缺乏统一调度**：应用中的各种浮层组件（Modal、Drawer、Dropdown、Tooltip、Toast、Popover）各自管理自己的 z-index，使用 `999`、`9999`、`99999` 等魔数，没有统一的"层级分配器"。当两个浮层同时出现时，谁在上谁在下是不确定的。
- **隐式层叠上下文创建者未纳入设计系统**：`opacity < 1`、`transform`、`will-change` 等 CSS 属性会隐式创建层叠上下文，但设计系统中的动画、过渡组件在应用这些属性时没有考虑层叠影响。架构缺少对"层叠上下文感知"的设计系统规范。

### 系统级预防策略

**静态防护**：
```css
/* 设计系统：全局 z-index 层级定义 */
:root {
  --z-base: 0;        /* 默认内容层 */
  --z-dropdown: 100;  /* 下拉菜单 */
  --z-sticky: 200;    /* 吸顶元素 */
  --z-overlay: 300;   /* 遮罩层 */
  --z-modal: 400;     /* 模态框 */
  --z-toast: 500;     /* 全局提示 */
  --z-tooltip: 600;   /* 工具提示(最高优先级) */
}

/* stylelint: 禁止使用 -var 之外的裸 z-index 数值 */
/* .stylelintrc.js */
rules: {
  'scale-unlimited/declaration-strict-value': [
    'z-index',
    { ignoreValues: ['/^var\\(--z-/'] }
  ]
}
```

**CI 防护**：
```bash
# CI 检测裸 z-index 数值
grep -rPn 'z-index:\s*\d+(?!\d)' src/ --include="*.css" --include="*.scss" | 
  grep -v 'var(--z-' && echo "ERROR: 禁止裸 z-index 数值" && exit 1

# 检测 transform + opacity 隐式层叠上下文创建
# 对每个使用了 transform 的组件，检查其内部是否包含浮层元素
```

**架构防护**：
- **Portal 渲染策略**：所有浮层组件统一使用 Portal 渲染到 `document.body` 下的预定义容器，由架构层统一管理：
```html
<!-- 在 index.html 中预定义层级容器 -->
<div id="dropdown-root"></div>   <!-- z-index: 100 -->
<div id="modal-root"></div>      <!-- z-index: 400 -->
<div id="toast-root"></div>      <!-- z-index: 500 -->
```
- **层级管理器 (Stacking Manager)**：实现一个全局的层级管理器，自动化分配和回收 z-index：
```typescript
class StackingManager {
  private layers: Map<string, number> = new Map();
  private baseValues = { modal: 400, toast: 500, tooltip: 600 };
  
  register(type: string): number {
    const count = this.layers.get(type) || 0;
    this.layers.set(type, count + 1);
    return this.baseValues[type] + count; // 同类型嵌套时递增
  }
  
  unregister(type: string): void {
    const count = this.layers.get(type) || 1;
    this.layers.set(type, count - 1);
  }
}
```

**流程防护**：
- Code Review 检查项："此浮层组件是否使用 Portal 渲染到 document.body？z-index 值是否来自 CSS 变量？"
- 架构评审：新增浮层类型时，必须在设计系统的 z-index 层级定义中分配位置。

### 跨层影响链

```
上游影响 (消费者):
  - 父组件布局变化时，内部浮层位置可能偏移
  - overflow: hidden 的祖先会裁剪浮层（即使 z-index 正确）
  - CSS transform 动画的祖先会影响浮层的 fixed 定位基准

下游影响 (依赖):
  - 浮层内的第三方组件（如 DatePicker 内的日期面板）同样被困在层叠上下文中
  - 可访问性：aria-modal 与 DOM 层级不一致，屏幕阅读器导航混乱
  - 测试：React Testing Library 的查询在 Portal 渲染时需要额外配置

部署影响:
  - 不同 CDN 环境（staging/prod）的 CSS 加载顺序可能影响 z-index 优先级
  - CSS-in-JS 的动态样式注入时机不确定 → 运行时 z-index 可能被覆盖

监控影响:
  - 需要监控: 页面中同时存在的浮层数量
  - E2E 测试: 验证多层浮层叠加时（Modal + Dropdown + Tooltip）的层级正确性
  - 视觉回归测试: Percy/Chromatic 截图对比覆盖关系
```

### 相关架构模式
- **事件总线（反模式）** (`patterns.md` 第六节) — 全局 Event Bus 的"无法追踪"问题与全局 z-index 的"无法追踪"同构。正确的替代方案是集中管理层级（类似 Zustand 管理全局状态）。
- **组件组合模式 — Compound Components** (`patterns.md` 第二节) — 浮层组件应使用此模式（如 `<Dropdown>` + `<Dropdown.Menu>`），隐式管理打开/关闭状态和层级关系。
- **微前端决策阈值** (`decisions.md` 第五节) — 微前端场景下，子应用的浮层可能被另一个子应用的层叠上下文囚禁，需要额外的跨应用层级通信机制。

## 预防

1. **了解层叠上下文触发条件**，特别是 `opacity < 1` 和 `transform` 这两个隐式触发器。
2. **全局 z-index 约定**：使用常量管理（如 0-底座、100-页头、1000-遮罩、2000-弹窗），并使用 CSS 变量或预处理器。
3. **Portal**：Modal、Dropdown、Tooltip 应渲染到 `document.body` 下，避免被父级层叠上下文囚禁。
4. **避免滥用固定 z-index 值**：不要 `z-index: 99999`，这是绝望的信号，说明结构有问题。
