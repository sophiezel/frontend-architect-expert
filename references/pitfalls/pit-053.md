# pit-053: 双线程通信数据丢失 — 传输数据被复制非共享导致的不一致

## 环境
- 微信小程序（双线程架构）
- 页面与自定义组件之间、或在 Worker 与主线程之间传递引用类型数据
- 涉及多次 setData / 连续操作同一数据对象的场景

## 症状
在逻辑层修改了 data 中的数组或对象，但页面渲染结果仍是旧值，看起来"setData 没生效"。或者在组件中收到 props 后修改了对象内部属性，但父组件不知道变化。更隐蔽的情况：console.log 打印出来的是最新值，但渲染的是旧值——开发者误以为 setData 被吞掉了。

## 根因
小程序双线程架构的核心约束：**逻辑层和渲染层之间没有共享内存，所有数据通过 setData/triggerEvent 传递时都被 JSON 深拷贝**。逻辑层中的 `this.data` 和渲染层中的 `data` 是两份完全独立的副本。当开发者在逻辑层修改 `this.data.list[0].name = 'new'` 后不调用 setData，渲染层永远不知道变化。

更深层问题：即使是逻辑层内部的组件间通信（`triggerEvent`），传递的对象也会被序列化后再反序列化——因为组件的数据需要跨线程传输到对应的 WebView 实例。这意味着 `Object.is` / `===` 比较在跨 setData 后永远为 false，引用相等性彻底失效。

+   this.setData({
+     cart: { items: newItems, total: newTotal }
+   });
  }

// ✅ 架构级修复: 引入 Immutable 更新辅助
function produce(data, recipe) {
  const copy = JSON.parse(JSON.stringify(data));
  recipe(copy);
  return copy;
}

Page({
  addItem(item) {
    this.setData(produce(this.data, (draft) => {
      draft.cart.items.push(item);
      draft.cart.total += item.price;
    }));

  }
});

## 原理
小程序的双线程架构天然决定了：逻辑层的 `this.data` 和渲染层的 data 是两套独立的 JavaScript 堆内存。setData 本质上是"通知渲染层：请按照我提供的数据更新你的那份副本"。这意味着：

1. 在逻辑层修改 `this.data`（无论多深）**永远不会自动同步到渲染层**，必须显式 setData
2. setData 传递的是**序列化后的副本**，不是引用，`this.data.list === oldList` 在 setData 后永远为 false（因为逻辑层的 list 还是原引用，但 setData 传的是新值）
3. 子组件 `triggerEvent` 传的数据在父组件 `e.detail` 中也是副本，修改它不影响子组件

这个设计有安全优势（隔离，防止渲染层污染逻辑层），但引入了"引用断裂"的认知负担。Web 开发者习惯的"改对象属性 → DOM 刷新"在这里不成立。

## 架构视角 (Architecture Perspective)

### 架构层面根因
双线程通信数据丢失的架构根因是**缺少数据所有权声明**。在小程序双线程架构中，数据的"真源"(source of truth)应该在哪一层？逻辑层存储了 `this.data`，渲染层也有各自的副本。当多个组件（每个可能在不同的 WebView 实例中）都持有一份数据的副本时，需要明确声明：

1. **谁拥有数据**：逻辑层的 `this.data` 是主源
2. **如何修改**：只能通过 setData 单向推送
3. **如何读取**：子组件通过 properties 获取父组件推送的副本

但小程序 API 设计并未在代码层面强制执行这些约束——开发者可以直接 `this.data.x = y` 而没有任何警告，直到渲染层没反应才发现问题。这本质上是**缺少声明式数据契约**的架构缺陷。

### 系统级预防策略

**静态防护**：
```js
// ESLint 规则
rules: {
  // 禁止直接修改 this.data
  'no-restricted-syntax': ['error', {
    selector: 'AssignmentExpression[left.object.property.name="data"]',
    message: '禁止直接修改 this.data，请使用 setData'
  }],
  // 禁止在 Component 中直接修改 this.properties
  'no-restricted-syntax': ['error', {
    selector: 'AssignmentExpression[left.object.property.name="properties"]',
    message: '禁止直接修改组件 properties，请使用 triggerEvent 通知父组件'
  }]
}
```

**运行时防护**：
```js
// 在开发环境下用 Proxy 保护 data
if (typeof __wxConfig !== 'undefined' && __wxConfig.envVersion === 'develop') {
  const PageHook = Page;
  Page = function(options) {
    const origOnLoad = options.onLoad;
    options.onLoad = function(...args) {
      // 用 Object.freeze 防止意外修改（深度冻结代价高，仅顶层）
      Object.keys(this.data).forEach(key => {
        const val = this.data[key];
        if (typeof val === 'object' && val !== null) {
          this.data[key] = deepFreeze(val);
        }
      });
      origOnLoad && origOnLoad.apply(this, args);
    };
    return PageHook(options);
  };
}
```

### 跨层影响链

```
上游影响 (开发者体验):
  - 数据不一致 bug 极难排查——console.log 看到的是对的，渲染是错的
  - 新人容易产生"setData 不生效"的困惑 → 信任降低 → 防御式编程

下游影响 (业务):
  - 表单数据丢失: 用户填写了表单，组件内部改了但父组件不知道 → 提交时数据缺失
  - 列表操作异常: 删除/移动项目后界面残留 → 用户以为操作失败 → 重复操作

监控影响:
  - setData 调用次数 vs 实际渲染更新次数不匹配 → 数据一致性异常
  - 需要建立 data diff 监控：每次 setData 回调后校验渲染层数据与预期一致
```

### 相关架构模式
- **单向数据流模式** (`patterns.md`) — 数据从父组件流向子组件，子组件通过事件通知父组件修改
- **CQRS 前端化** (`patterns.md` 第三节) — 逻辑层是 Command 端（修改），渲染层是 Query 端（读取）
- **Immutable 更新模式** — 每次修改创建新引用，避免引用断裂
- **反模式: 共享可变状态** — 跨双线程修改同一对象引用

## 预防
1. **铁律**：永远不直接修改 `this.data`，必须通过 `setData` 更新
2. **铁律**：组件间数据传递遵循单向数据流——子组件不修改 props，通过 `triggerEvent` 通知父组件
3. **使用 Immutable 辅助函数**：封装 `produce` 确保每次更新创建新引用
4. **ESLint 规则**：禁止 `this.data.xxx = yyy` 和 `this.properties.xxx = yyy`
5. **开发环境启用 data Proxy**：拦截直接修改并报警
