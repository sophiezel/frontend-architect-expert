# pit-063: 类型体操过度 — 复杂 conditional type 导致 IDE 卡顿/编译变慢

## 环境
- TypeScript 4.x / 5.x
- VSCode / WebStorm / NeoVim + tsserver
- 项目中定义了深度递归或大型联合分发类型

## 症状
IDE 的类型提示出现 2-5 秒延迟、鼠标悬停等待、`tsc --noEmit` 从 10 秒膨胀到 60+ 秒。VSCode 右下角持续显示 "Initializing JS/TS language features"。在 Activity Monitor 中 `node` (tsserver) 进程 CPU 占用长期 > 80%，内存占用 > 2GB。

开发者体验退化："写代码快，等类型检查慢"。

## 根因
TypeScript 类型系统是图灵完备的，但编译器有性能上限。三类典型问题：
1. **深度递归类型**：递归超过 ~50 层时编译器退化到全量展开
2. **大联合类型分发**：`T extends any ? ... : ...` 对 100+ 成员的联合类型分发，计算量 O(n²)
3. **交叉类型爆炸**：多层 `&` 交叉导致编译器递归检查每个属性的兼容性
4. **非尾递归**：TS 5.0 之前非尾递归类型无优化

## 诊断方法

```bash
# 1. 找出编译最慢的文件
npx tsc --noEmit --generateTrace trace
# 生成 trace/trace.json 和 trace/typesTrace.json
# 在 chrome://tracing 中打开 trace.json
# 过滤 "checkSourceFile" 事件 → 找出耗时最长的文件

# 2. tsserver 性能日志
# VSCode → Cmd+Shift+P → "TypeScript: Open TS Server Log"
# 观察 <semantic-diagnostics> 耗时
# 如果单个文件 > 3000ms → 存在性能问题

# 3. 识别大型联合类型
grep -rn "extends.*infer" src/ | wc -l     # 条件类型数量
grep -rn "extends any ?" src/ | head -20   # 联合类型分发

# 4. 递归深度测试
# 创建一个测试文件，逐步增加数组嵌套深度
type Test<T> = T extends { children: infer C } ? Test<C> : T;
type Deep = Test<{children:{children:{children:{children:{children:{}}}}}}>;
// 如果嵌套 10 层就编译超时 → 递归过深

# 5. tsc --diagnostics 总览
npx tsc --noEmit --diagnostics
# 输出:
# Files:           1200
# Lines:         320000
# Identifiers:  1500000
# Check time:     45.2s  ← 如果 > 30s → 需要优化
```

## 修复

```diff
// ❌ 问题1：深层 tree 递归类型
- type DeepReadonly<T> = {
-   readonly [K in keyof T]: T[K] extends object
-     ? T[K] extends Function ? T[K] : DeepReadonly<T[K]>
-     : T[K];
- };
// 对嵌套 > 20 层的 tree 结构，编译时间指数增长

+ // 方案A：使用接口分层（终止递归爆炸）
+ interface ReadonlyTreeNode {
+   readonly children: readonly ReadonlyTreeNode[];
+   readonly value: string;
+ }
+ // 明确限定递归深度到已知结构

+ // 方案B：使用映射表替代递归
+ type Primitive = string | number | boolean | null | undefined;
+ // 只展开一层，要求调用方显式嵌套
+ type ShallowReadonly<T> = { readonly [K in keyof T]: T[K] };

// ❌ 问题2：联合类型大分发
type ApiResponse = 
  | { type: 'user'; data: User }
  | { type: 'order'; data: Order }
  | { type: 'product'; data: Product }
  // ... 50+ 种类型

- type AllData = ApiResponse extends { data: infer D } ? D : never;
- // 对 50 个成员分发 → 性能退化

+ // 方案：用 mapped type + 索引替代条件类型分发
+ type ResponseMap = {
+   user: User;
+   order: Order;
+   product: Product;
+ };
+ type ApiResponse<K extends keyof ResponseMap = keyof ResponseMap> = {
+   [P in K]: { type: P; data: ResponseMap[P] };
+ }[K];
// 用对象映射替代联合分发，编译器无需遍历每个成员

// ❌ 问题3：深层交叉类型
- type Mega = A & B & C & D & E & F & G & H & I & J;
// 交叉类型过多时，编译器需检查每对属性的兼容性

+ // 方案：用 interface extends 替代交叉
+ interface Mega extends A, B, C, D, E, F {}
+ // interface extends 使用名义合并，比交叉类型性能好
```

## 原理
TypeScript 编译器的类型检查阶段 (`checkSourceFile`) 是最耗时的。当类型定义包含递归或大联合分发时，编译器需要为每个实例化点展开类型。例如，`DeepReadonly<{a:{b:{c:...}}}` 会展开为 `{readonly a: {readonly b: ...}}`，每一层嵌套需要一次条件判断和一次映射。20 层嵌套意味着 20 次递归展开，但每个中间类型都被缓存，所以时间复杂度接近 O(n)。

但联合类型分发更严重：`T extends any ? Transform<T> : never` 对 50 个成员的联合，编译器逐个成员应用 `Transform`（50 次），如果 `Transform` 本身也包含分发，则变成 O(n×m)。如果在泛型上下文中，每次泛型实例化都会重新分发——没有缓存。

TS 5.4+ 的 `NoInfer<T>` 可以阻止某些不必要的泛型推导，减少分发次数。

## 架构视角 (Architecture Perspective)

### 架构层面根因
类型体操过度的表面原因是"追求类型安全极致"，架构根因是**类型复杂度与业务复杂度未解耦**：
- **类型定义缺少抽象层次**：所有类型（DTO、ViewModel、Domain Model）在一个文件中或深度交叉引用，导致修改一个类型触发全局重新检查。
- **公共类型包体积膨胀**：monorepo 的 shared types 包导出过多互相依赖的类型，使用方只需一个 `User` 却被迫加载整个类型图。
- **类型体操替代了运行时验证**：用复杂的 conditional type 在编译时约束输入，不如用 zod/yup 在运行时验证，前者是"零运行时成本"的幻觉——代价转嫁到了 IDE 和 CI。

### 系统级预防策略

**静态防护**：
```bash
# 自定义 ESLint 规则：限制条件类型深度
# .eslintrc.js
rules: {
  // 禁止超过 3 层嵌套的 conditional type
  // 使用 @typescript-eslint 的 scope 分析
}

# 用 ts-ast-viewer 审计类型 AST 节点数
# 单个类型定义超过 200 个 AST 节点 → 拆分
```

**CI 防护**：
```yaml
# .github/workflows/type-perf.yml
- name: Type Check Performance
  run: |
    # 记录编译时间，超过阈值失败
    /usr/bin/time -f "%e" -o /tmp/tsc_time.txt npx tsc --noEmit
    TSC_TIME=$(cat /tmp/tsc_time.txt)
    if (( $(echo "$TSC_TIME > 30" | bc -l) )); then
      echo "⚠️ tsc 编译超过 30 秒（当前: ${TSC_TIME}s）"
      exit 1
    fi
```

**架构防护**：
- **类型分层**：DTO（API 形状）→ Domain（业务逻辑）→ ViewModel（UI 渲染），每层独立定义，用 mapper 函数转换。
- **按需导出类型**：类型包的 `index.ts` 只 re-export 稳定的公共接口，内部工具类型放在 `internal/` 目录不对外暴露。
- **类型复杂度预算**：参考 CSS 选择器特异性预算，设定类型定义的"复杂度预算"（条件类型层数 ≤ 3，递归深度 ≤ 5）。

**流程防护**：
- Code Review 清单项："此 PR 是否新增了条件类型或递归类型？如果是，请提供编译时间的 before/after 对比。"
- 每季度类型性能审计：用 `--generateTrace` 分析，淘汰不必要的复杂类型。

### 跨层影响链

```
上游影响:
  - IDE 卡顿 → 开发者切换文件等待 → 打断心流 → 生产力下降
  - tsserver 内存溢出 → VSCode 频繁弹窗 "Restart TS Server" → 开发者疲劳

下游影响:
  - CI 编译时间膨胀 → PR merge 延迟 → 交付周期延长
  - 新成员机器发热 → 入职体验差 → 抱怨"为什么不用 JavaScript"

组织影响:
  - 团队对 TypeScript 产生负面认知："TS 降低了开发速度"
  - 为求快关闭 strict 模式 → 更多运行时 bug → 恶性循环
  - 最终可能激进迁移到非 TS 方案，丢失类型安全收益
```

## 预防

1. **设定递归深度上限**：递归类型最多 3 层，超过则拆分为显式接口。
2. **避免大联合类型分发**：对 >20 成员的联合类型使用映射表模式。
3. **interface extends 优先于 & 交叉**：名义类型合并性能更好。
4. **CI 编译时间监控**：每次 PR 记录 `tsc --noEmit` 耗时，设置阈值告警。
5. **类型复杂度 Code Review**：新增条件类型/递归类型需解释必要性。
