# pit-073: RTL 布局破坏 — 阿拉伯语/希伯来语布局错乱

## 环境
- 多语言 Web 应用支持阿拉伯语 (ar)、希伯来语 (he) 等 RTL 语言
- 使用 CSS 物理属性 (left/right/margin-left/padding-right) 而非逻辑属性 (start/end/inline)
- 未使用 CSS Logical Properties 或 postcss-rtlcss / rtlcss 等构建工具

## 症状
切换到阿拉伯语后整个页面布局崩溃：
- Sidebar 从左侧"飞到"右侧但内部文字仍左对齐
- 表格列头顺序反转但列内容未反转
- 带有箭头的按钮图标朝向错误（向前箭头指向了后方）
- Modal 的关闭按钮跑到了左侧而非右上角
- 列表项的图标与文字重叠
- Flexbox 容器方向正确但绝对定位元素偏移相反方向
- `box-shadow` 和 `text-shadow` 的偏移量指向错误方向

更隐蔽的症状：
- LTR 中"截断 + ..."在 RTL 中变成了"反向截断"（头部被截断而非尾部）
- `transform: translateX()` 动画方向相反
- SVG 图标未做 `transform: scaleX(-1)` 导致指向错误

用户反馈："你们的阿拉伯语版本是从镜子里面看的吗？"

## 根因
**物理 CSS 属性在 RTL 环境下语义错误**——开发者使用 LTR 的物理方向 (`left`/`right`) 而非书写模式无关的逻辑方向 (`inline-start`/`inline-end`)。核心问题分解：

1. **CSS 属性硬编码物理方向**：`margin-left: 20px` 在 RTL 中应该变成右边距，但物理属性不会翻转。这是最常见的原因——占 RTL bug 的 70%+。

2. **绝对定位方向写死**：`right: 0` 将元素固定在右侧，在 RTL 中应该固定在左侧（因为"起始边"从左侧翻转到右侧）。

3. **transform 方向不翻转**：`translateX(-100%)` 在 RTL 中仍向左移动，而不是视觉上的"向起始方向"。

4. **图标朝向硬编码**：使用 `→` / `←` 字符或带方向的 SVG，在 RTL 中应该翻转。

5. **Flexbox direction 覆盖**：手动设置 `flex-direction: row-reverse` 与页面的 `dir="rtl"` 产生双重翻转。

6. **text-align 写死**：`text-align: left` 在 RTL 中破坏了自然的右对齐。

## 诊断方法


## 修复

```diff
// ❌ 问题代码 1: 物理 margin/padding
.sidebar {
-  margin-left: 20px;
-  padding-right: 10px;
+  margin-inline-start: 20px;
+  padding-inline-end: 10px;
}

// ❌ 问题代码 2: 绝对定位
.badge {
  position: absolute;
-  right: -8px;
-  left: auto;
+  inset-inline-end: -8px;
+  inset-inline-start: auto;
}

// ❌ 问题代码 3: transform 方向
.slide-panel {
-  transform: translateX(-100%);
+  transform: translateX(calc(var(--direction-factor, 1) * -100%));
}
+:root[dir="rtl"] { --direction-factor: -1; }

// ❌ 问题代码 4: text-align
.title {
-  text-align: left;
+  text-align: start;
}

// ❌ 问题代码 5: border
.divider {
-  border-left: 1px solid #eee;
+  border-inline-start: 1px solid #eee;
}
```

```javascript
// ✅ React 组件中的 RTL 感知
function ArrowButton({ direction = 'forward' }) {
  const isRTL = document.dir === 'rtl';
  const shouldFlip =
    (isRTL && direction === 'forward') ||
    (!isRTL && direction === 'backward');

  return (
    <span style={shouldFlip ? { transform: 'scaleX(-1)', display: 'inline-block' } : {}}>
      →
    </span>
  );
}

// ✅ 或使用 CSS 变量方案
// postcss.config.js 使用 postcss-rtlcss 自动转换
module.exports = {
  plugins: [
    require('postcss-rtlcss')({ mode: 'override' })
    // 这会自动将 left → right, margin-left → margin-right 等
    // 并生成 [dir="rtl"] 选择器
  ]
};
```

```css
/* ✅ 完整 RTL 基础设施 */
/* base.css */
:root {
  --text-align-start: start;
  --text-align-end: end;
}
[dir="rtl"] {
  --direction-multiplier: -1;
}
[dir="ltr"] {
  --direction-multiplier: 1;
}

/* 为不支持逻辑属性的旧浏览器提供 fallback */
.box {
  margin-left: 20px;        /* fallback */
  margin-inline-start: 20px; /* 标准 */
}

/* 使用 @supports 检测 */
@supports not (margin-inline-start: 0) {
  [dir="rtl"] .box {
    margin-left: 0;
    margin-right: 20px;
  }
}
```

## 原理
HTML 的 `dir` 属性通知浏览器文档的书写方向。浏览器会自动处理：
- 文本对齐（text-align）默认值从 `left` 变为 `right`
- Flexbox/Grid 的默认方向
- 表格列顺序
- 滚动条位置
- `dir="auto"` 时可自动检测文本方向

但**浏览器不会自动翻转 CSS 物理属性**。`margin-left: 20px` 始终是 20px 的左边距，无论 `dir` 是什么值。这是因为 CSS 的设计原则——物理属性永远指向屏幕的物理方向。要获得书写方向感知的行为，必须使用 CSS Logical Properties（`*-inline-*` 和 `*-block-*`）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **RTL 不是后期追加的功能**：项目在 LTR 完成后"加上" RTL 支持，物理属性已经遍布代码库，需要大量重构。
- **设计系统缺少 RTL 原语**：Buttons、Layout、Modal 等基础组件没有内建 RTL 感知能力，每个使用处都需要手动处理方向。
- **构建工具未接入 RTL 转译**：postcss-rtlcss 可在编译时自动生成 RTL 版本，但未集成到工具链。

### 系统级预防策略
- **组件库内建 RTL**：每一个基础组件从第一天就使用 CSS Logical Properties。
- **postcss-rtlcss 自动转译**：编译时自动将物理属性转为 RTL-safe 版本。
- **视觉回归测试 (Percy/Chromatic)**：对每个 PR 执行 RTL 语言的截图对比。
- **禁止物理属性**：ESLint + Stylelint 规则在 CI 中禁止 `margin-left` 等，强制使用 `margin-inline-start`。

### 跨层影响链
- 用户：RTL 用户看到破碎的 UI → 品牌信任下降 → 流失到竞品
- 团队：新增功能需要在 RTL 下单独修复 → 开发效率下降
- 市场：中东/北非 (MENA) 市场是增长极，RTL Bug 直接损失收入

## 预防

1. **使用 CSS Logical Properties 作为默认**：`margin-inline-start` 替代 `margin-left`。
2. **postcss-rtlcss 自动转换**：构建时自动生成 RTL 版本。
3. **禁止物理属性规则**：Stylelint / ESLint 配置。
4. **组件库强制 RTL**：所有组件必须通过 RTL 视觉测试才能合入。

## 回退

