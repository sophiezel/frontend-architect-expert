# pit-040: Node.js 集群/进程管理陷阱

## 环境
- Node.js 14+（cluster 模块 / worker_threads）
- PM2 / Docker / Kubernetes 部署
- 有状态服务（Session、WebSocket、内存缓存）

## 症状
开启 PM2 cluster 模式后，用户频繁被踢出登录——每次刷新页面都可能跳到登录页。WebSocket 连接随机断开。内存缓存（如 `new Map()` 缓存的 API 响应）在不同 worker 中不一致，用户看到的数据反复跳变。

更严重的情况：一个未捕获的异常导致整个 Node.js 进程退出，虽然 PM2 自动重启了进程，但在这几秒的空窗期中请求全部失败。Rolling restart（PM2 `--update-env`）期间服务整体不可用。

典型场景：
- `pm2 start app.js -i max` 启动 4 个 worker，session 存在内存中 → 请求到不同 worker 看到不同 session
- Worker Threads 被用来做 CPU 密集型计算，但创建线程的开销比计算本身还大
- 未捕获异常 `uncaughtException` 触发后进程退出，`process.on('uncaughtException')` 的 handler 中又抛了异常

## 根因
1. **Cluster 模式的内存隔离**：Node.js cluster 的每个 worker 是独立进程，拥有独立的 V8 堆。内存中的 session、缓存、WebSocket 连接池在不同 worker 间完全隔离。
2. **PM2 的"假集群"**：PM2 cluster 模式只是 Node.js `cluster` 模块的 wrapper，不做请求路由、不做 session 亲和性（sticky session）。
3. **Worker Threads 的创建成本**：每个 Worker Thread 需要独立的 V8 isolate 和 libuv 事件循环引用——创建成本约 5ms-50ms，不适合频繁创建/销毁（应使用线程池复用）。
4. **uncaughtException 的危险性**：`uncaughtException` 事件表示程序进入了未定义状态。即使捕获了异常，进程内部状态可能已损坏。Node.js 官方文档明确建议：捕获后应优雅退出。

## 原理
Node.js cluster 模块通过 `fork()` 创建子进程，每个子进程有独立的事件循环和 V8 实例。主进程通过 IPC（进程间通信）将 socket 文件描述符分发给 worker。不同 worker 的 `global`、`process`、内存堆完全隔离。

Worker Threads 共享同一个进程的内存空间（通过 `SharedArrayBuffer`），但拥有独立的 V8 isolate 和 libuv 引用。创建成本包括：分配 V8 isolate（~3MB）、初始化 libuv（小但不可忽略）、线程栈（通常 1MB）。

`uncaughtException` 是 V8 的"最后一站"——当 JavaScript 调用栈中没有任何 try-catch 且 Promise rejection 无人处理时触发。此时 Node.js 内部可能处于不一致状态（如文件描述符未关闭、Buffer 部分写入）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
集群/进程管理陷阱的架构根因是**应用的有状态性未在架构中显式建模**：
- **所有状态被默认为"本地内存状态"**：应用启动时，session、缓存、限流计数器、WebSocket 连接都被自然地存在内存中（`new Map()`）。在单进程环境下这是安全和高效的，但在多进程/多实例环境下，本地内存状态就是"状态分裂"的根源。架构需要从第一天就将所有状态分类为：无状态（每次请求独立计算）、有状态但可迁移到外部存储、有状态且必须本地（如 WebSocket 连接）。
- **进程生命周期管理缺少声明式配置**：PM2 的 `--max-memory-restart`、`--cron-restart` 等配置可以防止内存泄漏积累，但没有在项目的架构文档中明确"重启策略"和"回退时间"。进程的启动/优雅停止/异常退出/重启是一个完整的生命周期状态机，缺了任何一个状态的处理都会导致故障。
- **线程与进程的选型缺少决策框架**：CPU 密集型（Worker Threads）、I/O 密集型（Cluster）、混合型（两者结合）——项目中缺少"任务类型 → 并发模型"的映射表。

# Worker Thread 内存泄漏测试
test('线程池在 1000 次调度后无内存泄漏', async () => {
  const start = process.memoryUsage().heapUsed;
  const pool = new StaticPool({ size: 2, task: './task.js' });
  for (let i = 0; i < 1000; i++) {
    await pool.exec({ data: 'test' });
  }
  await pool.destroy();
  global.gc?.();
  const end = process.memoryUsage().heapUsed;
  expect(end - start).toBeLessThan(5 * 1024 * 1024); // 5MB
});

**架构约束**：
- 状态分类清单（写入 ARCHITECTURE.md）：
  | 状态类型 | 存储方式 | 示例 |
  |---------|---------|------|
  | 请求级 | 本地变量 | 请求参数、临时计算结果 |
  | 会话级 | Redis/DB | 登录态、购物车 |
  | 缓存 | Redis/Memcached | API 响应缓存、限流计数器 |
  | 连接级 | 本地 + 重连机制 | WebSocket、SSE |
- uncaughtException 必须触发 graceful shutdown + exit（不得继续运行）
- Worker Threads 必须通过线程池使用（禁止裸 `new Worker()`）

**流程约束**：
- 任何新增的有状态存储必须在 PR 中注明"在 cluster 模式下如何共享"
- PM2/Docker 配置必须作为代码提交到 repo（非人工在服务器上手写）

### 跨层影响链

```
上游影响 (用户体验):
  - Session 不同步 → 登录态丢失 → 用户在支付时被踢出 → P0 事故
  - WebSocket 断开 → 实时通知丢失 → 用户收到过期的聊天消息

下游影响 (数据库/Redis):
  - 未使用外部 session store → 迁移到 Redis 时需要数据库团队配合
  - 内存缓存不一致 → 打到后端数据库的请求量翻倍

部署影响:
  - PM2 rolling restart 期间 50% 的 worker 同时重启（默认行为，非逐个）
  - Docker 中需要配置 `stop_grace_period` 确保优雅停止（默认 10s 可能不够）
  - Kubernetes 的 liveness probe 可能在 graceful shutdown 期间误杀 pod

监控影响:
  - 监控: 每个 worker 的内存使用差异（差异大 → 内存泄漏或负载不均）
  - 告警: worker restart 频率（> 5次/小时 → 异常）
  - RUM: session 丢失率（用户操作中途被要求重新登录的频率）
```

### 相关架构模式
- **十二要素应用 - 第六要素：无状态进程** (`patterns.md` 第十一节) — "任何需要持久化的数据都必须存储在后端服务中（数据库、共享缓存）"。内存状态是反模式。
- **Graceful Shutdown 模式** (`decisions.md` 第十二节) — SIGTERM → 停止接收新请求 → 等待进行中请求 → 关闭连接 → 退出。K8s 的 `terminationGracePeriodSeconds` 需要大于 shutdown 超时。
- **BFF 层设计** (`decisions.md` 第十三节) — 与 pit-045 相关，BFF 作为有状态聚合层时尤其需要注意 cluster 模式。

## 预防
1. 所有需要跨请求/跨 worker 共享的状态必须存储于外部（Redis/DB），内存仅做本地缓存
2. PM2 cluster 模式提前规划 sticky session 或外部 session store
3. Worker Threads 必须使用线程池（`node-worker-threads-pool` / `piscina`）
4. 实现 `gracefulShutdown()` 函数并注册 `SIGTERM`/`SIGINT`/`uncaughtException` 处理
