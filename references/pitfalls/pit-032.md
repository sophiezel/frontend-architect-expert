# pit-032: Vue 组件通信反模式

## 环境
- Vue 3.0+（Composition API + `<script setup>`）
- 中大型单页应用，组件树深度 > 4 层
- 使用 provide/inject 或 mitt（EventBus）的项目

## 症状
修改父组件的数据后子组件不更新；或者子组件 A 的改动意外影响了没有直接关系的兄弟组件 B。调试时数据流无法追踪——你不知道数据是哪一层改的、经历了多少次中间传递。典型表现：
- provide 的值变了但 inject 的组件没反应
- EventBus 事件满天飞，同一个事件被多个组件同时监听，修改顺序不可预测
- props 从根组件一级级钻到叶子组件，中间 5 层组件只需要透传，却都要声明 props

## 根因
Vue 提供了三种跨层级通信机制，每种在不当使用时都会变成反模式：

1. **provide/inject 滥用**：将 provide/inject 当作全局状态管理使用。provide 的值如果不是响应式的（未用 ref/reactive 包裹），inject 方无法感知变化。而且 provide/inject 的依赖关系是隐式的——组件声明了 inject('key') 但不知道是谁提供的，形成"幽灵依赖"。

2. **EventBus 遗毒**（mitt 等）：事件发送方和接收方完全解耦，没有任何类型约束。一个 `emit('update')` 不知道被哪些组件接收、以什么顺序执行、是否会影响其他事件——完全的"喊话式通信"，无法被静态分析。

3. **Props 深度传递（prop drilling）**：中间组件被迫声明自己不需要但必须传递的 props，导致组件与上层强耦合。Props 变更触发整条链路重渲染。

## 诊断方法

```bash
# 1. 统计 provide/inject 使用密度
grep -rn "provide(" src/ --include="*.vue" --include="*.ts" | wc -l
grep -rn "inject(" src/ --include="*.vue" --include="*.ts" | wc -l
# 若 provide/inject 总数 > 10 且 key 命名分散（非统一前缀）→ 滥用信号

# 2. 检测 EventBus 使用
grep -rn "mitt\|emitter\.on\|emitter\.emit\|EventBus" src/ --include="*.ts" --include="*.vue"

# 3. 检测深层 prop drilling
# 查找组件中声明的 props 数量与模板中直接使用的 props 数量差异
# 若 props > 模板使用的 props → 存在透传 props

# 4. 使用 Vue DevTools 检查组件树
# 选中中间层组件 → 查看 props 列表 → 哪些 props 在模板中没出现？
```

## 修复

```diff
// ❌ 反模式 1: provide 非响应式值 + inject 方不知道来源
// 父组件
-const config = { theme: 'dark', lang: 'zh' };
-provide('config', config); // 非响应式，后续修改不更新

+const config = reactive({ theme: 'dark', lang: 'zh' });
+provide('config', readonly(config)); // 响应式 + 子组件只读

+// 使用 Symbol 作为 injection key 防止命名冲突
+// types/injection-keys.ts
+export const CONFIG_KEY = Symbol('app-config') as InjectionKey<AppConfig>;

// ❌ 反模式 2: EventBus 代替状态管理
// form.vue
-emitter.emit('user:updated', userData);
-// list.vue
-emitter.on('user:updated', refreshList);
-// header.vue
-emitter.on('user:updated', updateAvatar);

+// 统一状态管理
+// store/user.ts (Pinia)
+const useUserStore = defineStore('user', () => {
+  const currentUser = ref(null);
+  async function updateUser(data) {
+    currentUser.value = data;
+  }
+  return { currentUser, updateUser };
+});
+// 所有消费方统一从 store 读取
+const userStore = useUserStore();
+watch(() => userStore.currentUser, (user) => { refreshList(); updateAvatar(); });

// ❌ 反模式 3: 深度 props 传递
// App.vue → Layout.vue → Sidebar.vue → MenuItem.vue
// permissions 需要从 App 传到 MenuItem，中间经过 Layout 和 Sidebar
-App.vue: <Layout :permissions="permissions">
-Layout.vue: <Sidebar :permissions="permissions">
-Sidebar.vue: <MenuItem :permissions="permissions">

+// 使用 provide/inject 跨越中间层（仅此一个合法场景）
+// App.vue
+provide(PERMISSIONS_KEY, readonly(permissions));
+// MenuItem.vue
+const permissions = inject(PERMISSIONS_KEY);
+// Layout.vue 和 Sidebar.vue 不再需要声明 permissions prop
```

## 验证

```bash
# 1. 类型检查：确保 injection key 类型安全
npx vue-tsc --noEmit
# 如果有 inject 使用了未定义的 key → 类型错误

# 2. E2E 测试通信链路
# test: 父组件修改数据 → 深层子组件 UI 更新
test('跨层级通信正常', async ({ page }) => {
  await page.click('[data-testid="change-theme"]');
  await expect(page.locator('[data-testid="sidebar"]')).toHaveClass(/theme-dark/);
});

# 3. 组件单元测试：中间层组件不应声明透传 props
import { shallowMount } from '@vue/test-utils';
test('Layout 不应声明 permissions prop', () => {
  const wrapper = shallowMount(Layout);
  // Layout 的 props 定义中不应有 permissions
  expect(Object.keys(wrapper.props())).not.toContain('permissions');
});

# 4. 架构审计：统计通信模式分布
# - Props 传递: N 个（跨层级 > 2 层的记为反模式）
# - Provide/Inject: M 个（无 Symbol key 的记为反模式）
# - EventBus: K 个（期望为 0）
# - Pinia Store: P 个（推荐）
```

## 原理
Vue 的组件树本质是一棵树，但应用的数据依赖关系通常是图。用树来传递图的数据必然会出现"穿层"——即 prop drilling。provide/inject 解决了树传递的痛点，但牺牲了显式性。EventBus 解决了任意两点间的通信，但牺牲了可追踪性。

架构上正确的解法是：**状态管理库（Pinia）处理全局/多组件共享状态，provide/inject 处理可控范围内的跨层级依赖，props 处理严格的父子单向数据流。** 三种机制各有边界，边界混用就是反模式。

## 架构视角 (Architecture Perspective)

### 架构层面根因
组件通信反模式的根因是**未定义通信拓扑的分层策略**：
- **缺少"数据所有权"概念**：应用中每份数据应该有唯一的 owner（store 或顶层组件），但实际开发中数据所有权模糊——父组件觉得"子组件应该自己拿数据"，子组件觉得"父组件应该传下来"，于是 EventBus 成了避难所。
- **组件树的"物理结构"与数据的"逻辑图"耦合**：UI 树的层级关系决定了 props 的传递路径，但数据的逻辑关系（如 theme 被数十个组件消费）与 UI 层级无关。当物理结构与逻辑图不匹配时，强制 props 传递就是削足适履。
- **隐式依赖是技术债务的温床**：inject('xxx') 不声明提供方，类似于全局变量 `window.xxx`。新成员接手代码时无法从组件签名中知道它依赖什么——必须全局搜索。

**静态检查**：

**架构约束**：
- 通信机制分层决策树（写入 CONTEXT.md）：
  ```
  数据是否被 1 个直接子组件消费？ → props
  数据是否需要跨越 >=2 层但仍在同一个功能域内？ → provide/inject (with Symbol key)
  数据是否被多个不相关组件消费？ → Pinia store
  是否需要非父子间的事件通知（如全局通知）？ → Pinia action + watch
  是否跨微前端子应用？ → 自定义事件（CustomEvent）+ 类型约束
  ```
- 禁止在 production 代码中使用 EventBus 库（mitt、tiny-emitter）

**流程约束**：
- Code Review："这个 provide 的 key 是 Symbol 吗？inject 方是否靠近 provide 方（<3 层）？"
- PR 模板中增加"通信方式"字段："此 PR 涉及的数据传递使用 [ ] props [ ] provide/inject [ ] Pinia [ ] 其他"

### 跨层影响链

```
上游影响 (组件设计):
  - 深层 prop drilling 导致中间组件无法独立复用（强依赖上层 props 结构）
  - 组件单元测试需要 mock 所有透传 props → 测试维护成本线性增长

下游影响 (重构):
  - EventBus 导致的隐式依赖使组件无法安全重构——你不知道改了 emit 会影响谁
  - provide/inject key 的命名冲突在大型项目中成为"定时炸弹"

部署影响:
  - 通信链路问题在 dev 环境因 HMR 可能偶然正常，生产环境稳定复现
  - 微前端场景下跨子应用的 EventBus 引用不同的 mitt 实例 → 完全不互通

监控影响:
  - 无法追踪数据变更来源（从 provide 到 inject 的链路不可见）
  - 需要自定义 Vue DevTools plugin 可视化 provide/inject 依赖图
```

### 相关架构模式
- **Flux/Redux 单向数据流** (`patterns.md` 第一节) — 全局状态管理的设计思想：所有变更通过 action 显式触发，可追踪、可回溯。Pinia 是该模式在 Vue 生态的标准实现。
- **依赖注入容器模式** (`patterns.md` 第七节) — Vue 的 provide/inject 本质是组件树范围的 DI。正确的用法是像 Angular 的服务注入一样，定义清晰的 token 和作用域边界。
- **组件分层** (`decisions.md` 第四节) — UI 组件（纯展示）与容器组件（连接数据）的分层策略决定了 props/api 的流向。

## 预防
1. 团队规范文档中明确通信方式选择决策树，Code Review 强制执行
2. 所有 provide/inject 使用 Symbol 作为 key，集中在 `src/types/injection-keys.ts`
3. 禁止安装 mitt 等 EventBus 库（在 `package.json` 的 `overrides` 中可禁止特定依赖）
4. 组件 props 数量硬性上限：每个组件 props 不超过 6 个（通过 ESLint `vue/max-props` 规则）
