# pit-036: Vite 依赖预构建问题

## 环境
- Vite 3.x / 4.x / 5.x
- 使用 CommonJS 依赖或大型 monorepo 的项目
- pnpm / yarn workspaces

## 症状
`vite dev` 启动时卡在 "Pre-bundling dependencies" 阶段数分钟。大型依赖（如 `monaco-editor`、`ag-grid`、`ant-design-vue`）预构建特别慢，每次重启 dev server 都要重新构建。更严重的是预构建缓存损坏导致神秘的运行时错误——某个第三方包的导出是 `undefined`，或样式丢失。

典型场景：
- 引入了一个 CommonJS 格式的依赖（如 `lodash`、`axios` 旧版本），Vite 需要将其转为 ESM
- monorepo 中 workspace 包互相引用，预构建无法正确解析 `link:` 协议
- `node_modules/.vite/deps/` 缓存被 CI 或 git 操作破坏

## 根因
Vite 的依赖预构建（Pre-bundling）有三个目标：
1. 将 CommonJS/UMD 模块转为 ESM（Vite 只服务于 ESM）
2. 将内部模块较多的 ESM 包（如 `lodash-es` 有 600+ 模块）合并为单个模块，减少 HTTP 请求
3. 缓存转换结果以避免重复工作

问题出现于：
- CJS → ESM 转换无法处理动态 require、条件 require
- monorepo 的符号链接（symlink）导致 Vite 无法确定正确的依赖根目录
- 缓存使用文件哈希来判断是否需要重新构建，但某些文件系统（Docker volume、网络挂载）的时间戳不可靠

## 诊断方法

```bash
# 1. 查看预构建耗时
vite dev --debug
# 或设置环境变量
DEBUG=vite:deps vite dev

# 2. 检查预构建缓存
ls -la node_modules/.vite/deps/
cat node_modules/.vite/deps/_metadata.json

# 3. 找出所有 CommonJS 依赖
find node_modules -name "package.json" -maxdepth 2 | while read f; do
  type=$(node -e "console.log(require('$f').type || 'commonjs')")
  if [ "$type" = "commonjs" ]; then echo "$f → CJS"; fi
done

# 4. 检测 monorepo 的符号链接
ls -la node_modules/ | grep "^l"  # 符号链接列表

# 5. 强制清除缓存后重试
rm -rf node_modules/.vite
vite dev --force
```

## 修复

```diff
// vite.config.ts — 优化预构建配置

import { defineConfig } from 'vite';

export default defineConfig({
  optimizeDeps: {
    // ❌ 默认自动扫描，可能遗漏或包含过多
+    // ✅ 显式声明需要预构建的依赖
+    include: [
+      'lodash',                    // CJS 转 ESM
+      'axios',                     // CJS 转 ESM
+      'dayjs',
+      'some-cjs-only-package',
+    ],
+    
+    // ✅ 排除不需要预构建的（已经是 ESM 且内部模块少）
+    exclude: [
+      'vue',                       // Vite 已内置处理
+      '@vueuse/core',              // 已经是 ESM
+      'some-esm-package',
+    ],
+
+    // ✅ monorepo: 排除 workspace 包
+    // 这些包在 monorepo 中已经是源码，不需要预构建
+    exclude: [
+      ...(process.env.MONOREPO_PACKAGES || '').split(','),
+    ],
+
+    // ✅ 处理大型依赖：使用 esbuild 插件定制转换
+    esbuildOptions: {
+      // 处理特定 CJS 包的动态 require
+      plugins: [
+        // 对特定包进行转换
+      ],
+    },
+  },

  // ✅ 处理 monorepo 的 symlink 解析
+  resolve: {
+    preserveSymlinks: false, // 默认值，确保 Vite 解析真实路径
+    dedupe: ['vue', 'vue-router'], // 确保全局只使用一个实例
+  },
});
```

## 原理
Vite 的依赖预构建使用 esbuild 将依赖转换为 ESM 并合并。过程如下：
1. 扫描源码中的 `import` 语句，收集依赖列表
2. 对每个依赖执行 esbuild 的 `build`，将入口文件及其内部模块打包为单个 ESM 文件
3. 将结果存入 `node_modules/.vite/deps/`，用哈希文件记录依赖版本和构建选项

缓存失效条件：依赖的 `package.json` 变化、`vite.config.ts` 中 `optimizeDeps` 选项变化、Vite 版本变化。

monorepo 中的符号链接问题：当 workspace 包通过 symlink 连接时，esbuild 在解析依赖时可能找不到嵌套的 `node_modules`，导致转换失败。

## 架构视角 (Architecture Perspective)

### 架构层面根因
依赖预构建问题的架构根因是**模块格式标准化的过渡期管理缺失**：
- **CommonJS 遗留依赖未在架构层面标记和隔离**：项目中使用了 CJS 依赖但没有在架构文档中标记它们是"需要特殊处理的过渡期依赖"。架构应该为这类依赖建立"准入/隔离/迁移"的三阶段生命周期管理——评估→隔离在适配层→计划迁移到 ESM 替代品。
- **Monorepo 的包边界与构建工具的解析边界不一致**：monorepo 定义了包的逻辑边界，但 Vite 的预构建需要知道"哪些是外部依赖（需预构建）、哪些是内部源码（需原样保留）"。这个信息散落在每个子项目的 `vite.config.ts` 中，缺少统一的配置生成策略。
- **开发环境与 CI 环境的差异未收敛**：开发者在本地有温缓存（体验好），CI 中总是冷缓存，暴露了预构建的性能问题。但团队在"开发体验"评审时只测本地环境。

### 系统级预防策略

**静态检查**：
```bash
# 审计脚本：统计项目中的 CJS 依赖
# scripts/audit-cjs-deps.sh
find node_modules -name "package.json" -maxdepth 2 | while read f; do
  type=$(node -e "const p=require('$f'); console.log(p.type || 'commonjs')")
  if [ "$type" = "commonjs" ]; then
    echo "CJS: $(dirname $f)"
  fi
done > cjs-deps.txt
echo "CJS 依赖总数: $(wc -l < cjs-deps.txt)"
# 若 > 10，建议制定 CJS → ESM 迁移计划
```

**CI 门禁**：
```bash
# CI 中测量预构建时间，超过阈值则报警
timeout 120s vite build --mode preflight || echo "BUILD TIMEOUT"

# 检测重复依赖实例
npx vite-bundle-visualizer --open=false --output=bundle-report.html
# 如果 vue、react 等核心依赖出现两份 → dedupe 配置有误

# 缓存一致性测试：两次构建产物 hash 应一致
vite build && mv dist dist1
vite build && mv dist dist2
diff -r dist1/assets/ dist2/assets/ || echo "WARNING: non-reproducible build"
```

**架构约束**：
- CJS 依赖准入制度：新增 CJS 依赖必须在架构评审中说明"为何没有 ESM 替代品"，并记录迁移计划
- 所有 CJS 依赖必须显式列入 `optimizeDeps.include`（不依赖自动扫描）
- monorepo 中每个子项目配置文件必须通过共享的 `createViteConfig()` 工厂函数生成
- `optimizeDeps.exclude` 中的 workspace 包列表由 root 的 `package.json` workspaces 字段自动推导

**流程约束**：
- 新增依赖时在 PR 中标注模块格式（ESM/CJS/Dual），CJS 需要额外审批
- monorepo 新增 workspace 包时，检查所有子项目的 `optimizeDeps.exclude` 是否需要更新

### 跨层影响链

```
上游影响 (开发体验):
  - 启动慢 → 开发者切换分支后等待时间 > 30s → 频繁切换分支成本高
  - 缓存损坏 → 出现诡异的 "undefined is not a function" → 浪费调试时间

下游影响 (用户):
  - 预构建产物 volume 过大 → 生产 bundle 包含不必要的重复依赖
  - CJS 转换遗漏 → 生产环境报错（如 require is not defined）

部署影响:
  - CI 构建时间显著增加 → 部署 pipeline 变慢 → 回滚时间变长
  - Docker 构建中每次都是冷缓存 → 镜像构建时间膨胀

监控影响:
  - 预构建失败率应作为 CI 关键指标
  - 监控 node_modules/.vite/deps/ 的磁盘占用（异常大 → 依赖冗余）
```

## 预防
1. 所有 CJS 依赖显式加入 `optimizeDeps.include`，所有 workspace 包加入 `exclude`
2. 使用共享的 `createViteConfig()` 工厂函数统一管理 monorepo 子项目配置
3. 每个季度审计 CJS 依赖清单，推动迁移到 ESM 替代品
4. 在 `package.json` 的 `scripts` 中添加 `"preflight": "vite build --mode preflight"` 用于 CI 快速验证
