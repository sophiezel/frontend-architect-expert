# pit-059: uni-app 条件编译遗漏

## 环境
- uni-app 2.x / 3.x
- 多端发布：微信小程序 + H5 + App (Android/iOS)
- 使用 `#ifdef` / `#ifndef` 条件编译

## 症状
- 某个平台（如支付宝小程序）出现特定 Bug，其他平台正常
- 代码在 H5 平台编译通过但功能缺失（如分享按钮消失）
- App 端正常运行的 API 在小程序端直接报错（`xxx is not a function`）
- 样式在微信小程序正常，在 H5 端出现布局错乱
- `pages.json` 中配置的页面只在部分平台注册，导致 404
- `manifest.json` 的平台特定配置未生效

## 根因
uni-app 的条件编译是通过**编译时宏替换**实现的：构建工具根据目标平台，在预处理阶段删除不匹配的 `#ifdef` / `#ifndef` 块。遗漏发生在以下场景：

1. **条件覆盖不完整**：一个 API 在 3 个平台中有 2 个可用，但开发者只用了一个 `#ifdef MP-WEIXIN` 包裹，导致 H5 和 App 端执行了预期外的代码路径。
2. **`#ifndef` 逻辑陷阱**：`#ifndef H5` 意味着"除 H5 外所有平台"，但如果新平台（如快应用）被加入构建目标，它也会执行这段代码——尽管开发者从未在新平台测试过。
3. **条件编译在非 `.vue` 文件中不生效**：uni-app 的 `#ifdef` 仅在 `.vue`、`.js`（通过 uni-app 构建处理的）、`pages.json`、`manifest.json` 中生效。在 `utils/` 下的纯 JS 文件、`static/` 下的资源、`native_modules/` 下不生效。
4. **样式中的条件编译格式特殊**：CSS 中的 `#ifdef` 必须用 `/* #ifdef PLATFORM */ ... /* #endif */` 格式（注释包裹），`//` 和 `<!-- -->` 都不生效但没有报错。
5. **pages.json 的 `#ifdef` 语法差异**：不支持 `#ifndef`（某些版本），不支持嵌套，且 `||` 和 `&&` 行为可能不符合预期。

      // #ifdef APP-PLUS

        type: 0,
        title: '分享',
        href: 'https://app.example.com/share/xxx',
      });
      // #endif
    }
  }
}
</script>

## 原理
uni-app 的 `#ifdef` / `#ifndef` 是在编译时由 `@dcloudio/uni-cli-shared` 中的 preprocessor 处理的。它不涉及宏系统的全部语义——它只是一个**简单的文本包含/排除过滤器**：

1. 构建工具读取目标平台标识（如 `MP-WEIXIN`）。
2. 扫描每个源文件，当遇到 `#ifdef PLATFORM` 时，如果 `PLATFORM` 等于当前目标平台 → 保留该块；否则 → 从产物中完全删除。
3. `#ifndef PLATFORM` 逻辑相反：当前目标平台等于 `PLATFORM` 时删除，否则保留。

这意味着 `#ifdef` 是一个**排除式**而非**包含式**的条件系统。当你写 `#ifdef MP-WEIXIN`，你只是在说"微信小程序需要这段代码"，而没有声明"其他平台不需要这段代码"。当新平台（如快应用、QQ 小程序）被添加到构建目标时，所有的 `#ifndef` 块和未覆盖到的平台都会静默地执行某个代码路径——而这个路径从未被测试过。

更隐蔽的是：uni-app 的部分内置组件（如 `<map>`、`<video>`）在各平台的实现差异很大。即使你没有写 `#ifdef`，底层的行为也可能不同。例如 `<map>` 在微信小程序和 H5 端的属性集合完全不同（小程序有 `show-location`，H5 没有），这些差异不在 `#ifdef` 的覆盖范围内，属于框架层面的跨平台不一致。

## 架构视角 (Architecture Perspective)

### 架构层面根因
条件编译遗漏的表面原因是"开发者忘记覆盖某个平台"，但架构层面根因是**平台抽象的分层错误**：
- `#ifdef` 是平台差异直接暴露给业务代码的表现。理想的分层应该是：平台差异由框架/适配器层处理，业务代码只编写平台无关的逻辑。但 uni-app 的实际情况是——太多平台差异无法被适配器完全封装（如分享 API 的触发方式、地图组件的属性集、支付流程的差异），导致业务代码被迫使用 `#ifdef` 来处理框架层未解决的差异。
- **条件编译缺少编译时完备性检查**：编译器知道目标平台集合（用户在 `manifest.json` 中声明），但不会检查 `#ifdef` 块是否覆盖了所有目标平台。这类似于 TypeScript 的 switch-case exhaustiveness check — TypeScript 可以检查 switch 是否覆盖所有枚举值，但 uni-app 不会检查 `#ifdef` 是否覆盖所有目标平台。
- **API 契约的不兼容性被隐藏**：每个平台的 API 表面上是"同名"的（如 `uni.share`），但参数、行为、触发限制完全不同。框架用"同名 API"制造了统一的假象，但实际使用时仍需 `#ifdef` 补丁——这违反了抽象的一致性约定。

### 系统级预防策略

**静态防护**：
```js
// ESLint 自定义规则: 检查 #ifdef 覆盖度
rules: {
  // 禁止在 #ifndef 中使用 `||` (语义不明确)
  'uni-app/no-ifndef-or': 'error',
  // 要求每个 #ifdef 文件末尾有兜底 #endif 或完整的 else 分支
  'uni-app/require-ifdef-exhaustive': 'warn',
  // 警告 CSS 中使用 // 格式的条件编译 (不生效)
  'uni-app/no-line-comment-ifdef-in-style': 'error',
}
```

**CI 防护**：
```bash
# CI 中自动化平台覆盖度检查
# 1. 列出所有 #ifdef 块及其覆盖的平台
npx uni-ifdef-audit --src src/ --platforms mp-weixin,mp-alipay,h5,app-plus

# 2. 构建全平台并检查编译错误
for platform in mp-weixin mp-alipay h5 app-plus; do
  npx cross-env UNI_PLATFORM=$platform npx uni-app build || exit 1
done

# 3. 对比全平台产物大小 — 异常差异可能指示遗漏
find dist/build -name "*.js" | xargs wc -c | sort -n
# 同一文件在不同平台的产物大小差异 > 50% → 人工审查
```

**架构防护**：
- **平台 API 契约层**：定义 `PlatformCapabilities` 接口，声明"哪些 API 在哪些平台可用"。业务代码不使用 `wx.*` / `uni.*`，而是通过 `PlatformCapabilities.share(params)` 间接调用，在编译时做能力检查。
- **条件编译白名单**：将 `#ifdef` 限制在指定的"平台适配文件"中（如 `adapters/share.adapter.ts`），禁止在业务组件中直接使用 `#ifdef`。
- **新增平台影响分析**：当构建目标新增平台时，自动化扫描现有 `#ifdef` 块，标记可能在新平台产生歧义的代码路径。

**流程防护**：
- Code Review 检查清单："每个 `#ifdef` 块是否明确覆盖了所有目标平台？没有覆盖到的平台的行为是什么？"
- 新增平台到构建目标时，必须做全量 E2E 回归测试（而非仅新增平台的测试）。
- 所有 `#ifdef` 块必须有注释说明"为什么这个平台走特殊路径"，否则自动标记为待审查。

### 跨层影响链

```
上游影响 (产品层):
  - 某个平台的分享/支付功能无声失效 → 业务指标异常
  - 新平台上线的"跨端复用"预期落空 → 实际开发时间远超预估
  - A/B 测试中某平台数据异常 → 浪费实验流量

下游影响 (平台层):
  - 小程序端调用 App 端专属 API → 审核被拒 (使用了未声明的权限)
  - H5 端嵌入了小程序专属组件 (如 open-data) → 渲染异常但无报错
  - App 端使用了 Web API (如 navigator.share) → 在部分设备上抛异常

部署影响:
  - 需要在 3+ 个平台分别发布 hotfix → 发布协调复杂度 O(n^2)
  - 热更新仅对 H5 生效，小程序和 App 需要审核 → 不同平台的修复时间差 1-7 天
  - 回滚时可能只有部分平台成功 → 版本碎片化

监控影响:
  - 平台特定的错误被上报为"未知错误" → Sentry issue 难以分组
  - Funnel 分析中各平台转化率差异可能是 Bug 而非用户行为
  - 需要按 platform 维度拆分所有监控指标 → 监控成本 4x
```

### 相关架构模式
- **策略模式 (Strategy Pattern)** (`patterns.md`) — 将每个平台的特定行为封装为独立策略类，通过工厂函数按平台选择策略，替代散落在代码中的 `#ifdef`。
- **适配器模式 — 能力声明版** — 与 Taro 的 PlatformAdapter 类似，但增加了"能力探测"步骤：在编译时已知哪些 API 可用，而非运行时发现。
- **能力契约 (Capability Contract)** — 每个平台声明其能力清单，CI 阶段验证所有 `#ifdef` 路径是否与平台能力匹配。

## 预防

1. **使用 `#ifdef` 白名单策略**：优先使用 `#ifdef PLATFORM1 || PLATFORM2`（明确列出平台），避免使用 `#ifndef`（隐含"所有其他平台"）。
2. **在适配器文件中集中条件编译**：创建 `src/adapters/` 目录，所有 `#ifdef` 集中于此。业务代码调用适配器而非直接判定平台。
3. **使用 `uni.getSystemInfoSync().uniPlatform` 做运行时兜底**：对于编译时条件无法处理的场景，在运行时做最终判断。
4. **CSS 条件编译统一格式**：始终使用 `/* #ifdef PLATFORM */` 而非 `// #ifdef`。
5. **新增平台时做全量回归**：不要假设"新平台的代码路径已经由某个 `#ifdef` 覆盖了"。
