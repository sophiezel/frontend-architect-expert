# pit-060: 跨端样式兼容

## 环境
- 跨端框架 (Taro / uni-app / React Native) 多端渲染
- 涉及平台：iOS Safari / Android Chrome / 微信小程序 / WKWebView / 微信 X5 WebView
- CSS Flexbox / Grid / Position 布局

## 症状
样式在桌面 Chrome 完美还原设计稿，但在目标平台表现异常：
- **Flexbox**：`gap` 属性在小程序中不生效，元素间距塌陷为 0
- **Grid**：微信小程序部分支持 Grid，但 `grid-template-areas` 完全不支持
- **Position**：iOS Safari 中 `position: fixed` + 软键盘弹出后，元素位置偏移或不跟随
- **z-index**：微信小程序中 z-index 的层叠上下文规则与 Web 标准不同
- **overflow: scroll**：Android WebView 中 `-webkit-overflow-scrolling: touch` 导致子元素 `position: sticky` 失效
- **CSS 变量**：微信 X5 WebView 中 CSS 自定义属性 (Custom Properties) 在部分元素上不继承
- **backdrop-filter**：Android 微信浏览器中 `backdrop-filter` 导致严重卡顿或完全黑屏

## 根因
跨端样式兼容问题的根因是**渲染引擎碎片化**：不同的目标平台使用不同的渲染引擎，每个引擎对 CSS 规范的支持程度、实现方式、性能特征各不相同。

核心不一致矩阵：

| CSS 特性 | 桌面 Chrome | iOS Safari | Android Chrome | 微信小程序 | 微信 X5 | WKWebView |
|---------|------------|-----------|---------------|----------|---------|-----------|
| `gap` (flex) | ✅ | ✅ 14.1+ | ✅ | ❌ | ⚠️ (旧版不支持) | ✅ 14.1+ |
| `grid-template-areas` | ✅ | ✅ | ✅ | ❌ | ⚠️ | ✅ |
| `position: fixed` | ✅ | ⚠️ 软键盘 Bug | ✅ | ⚠️ 不支持 in scroll-view | ⚠️ | ⚠️ 同 Safari |
| `overflow: scroll` + `touch` | ✅ | ✅ | ✅ | ⚠️ (scroll-view 替代) | ⚠️ 粘性失效 | ✅ |
| `backdrop-filter` | ✅ | ✅ 9.0+ | ✅ | ❌ | ⚠️ 性能极差 | ✅ 9.0+ |
| `sticky` | ✅ | ✅ + `-webkit-` | ✅ | ⚠️ 仅垂直 | ⚠️ | ✅ + `-webkit-` |
| `aspect-ratio` | ✅ | ✅ 15.0+ | ✅ | ⚠️ | ⚠️ | ✅ 15.0+ |

"一套 CSS，多端运行"的假设在跨端开发中是一种危险的简化。每个渲染引擎的"标准支持度"不等于"相同行为"，细微的渲染差异会累积为视觉 Bug。

## 诊断方法

```bash
# 1. CSS 特性使用审计
# 扫描项目中使用了哪些可能不兼容的 CSS 属性
grep -rn "gap:" src/ --include="*.css" --include="*.scss" --include="*.less" --include="*.vue"
grep -rn "grid-template" src/ --include="*.css" --include="*.scss" --include="*.vue"
grep -rn "position:\s*fixed" src/ --include="*.css" --include="*.scss" --include="*.vue"
grep -rn "backdrop-filter" src/ --include="*.css" --include="*.scss" --include="*.vue"
grep -rn "sticky" src/ --include="*.css" --include="*.scss" --include="*.vue"
grep -rn "aspect-ratio" src/ --include="*.css" --include="*.scss" --include="*.vue"

# 2. caniuse 批量检查
# 使用 caniuse-lite 或 @mdn/browser-compat-data 做自动化检查
npx caniuse-lite --feature=css-gap --browsers="iOS Safari 15,Android Chrome 100" --json

# 3. 微信开发者工具调试
# WXSS 面板 → 查看编译后的样式
# 检查是否有被小程序静默删除的属性（如 backdrop-filter 编译后消失）

# 4. 真机对比测试
# 使用 Playwright 或 Appium 在多设备上截图对比
# 配合 pixelmatch 做视觉差异检测
```

## 修复

```css
/* ❌ 问题1: flex gap 在小程序中不生效 */
.card-list {
  display: flex;
  flex-wrap: wrap;
-  gap: 12px; /* 小程序不支持 */
}

/* ✅ 修复: 使用 margin 兼容方案 */
.card-list {
  display: flex;
  flex-wrap: wrap;
  /* gap: 12px; — 保留以支持现代浏览器，同时提供 margin 降级 */
}
.card-list > * {
  margin-right: 12px;
  margin-bottom: 12px;
}
/* 可选: 使用 @supports 做渐进增强 */
@supports (gap: 12px) {
  .card-list {
    gap: 12px;
  }
  .card-list > * {
    margin-right: 0;
    margin-bottom: 0;
  }
}
```

```css
/* ❌ 问题2: backdrop-filter 在微信/低端Android中卡死 */
.dialog-overlay {
-  backdrop-filter: blur(10px);
-  background: rgba(0, 0, 0, 0.3);
}

/* ✅ 修复: 使用 @supports 检测，不支持的平台用纯色降级 */
.dialog-overlay {
  /* 降级基线: 纯色半透明背景 (所有平台安全) */
  background: rgba(0, 0, 0, 0.6);
}
@supports (backdrop-filter: blur(10px)) {
  .dialog-overlay {
    backdrop-filter: blur(10px);
    background: rgba(0, 0, 0, 0.3); /* 有 blur 时降低背景不透明度 */
  }
}
```

```css
/* ❌ 问题3: position: fixed + iOS 软键盘 */
.bottom-bar {
-  position: fixed;
-  bottom: 0;
-  left: 0;
-  right: 0;
}

/* ✅ 修复: 使用 visualViewport API 动态计算 (详见 pit-021) */
.bottom-bar {
  position: fixed;
  bottom: 0;
  left: 0;
  right: 0;
  /* JS 动态计算: */
  /* const viewport = window.visualViewport; */
  /* bottomBar.style.bottom = `${window.innerHeight - viewport.height + viewport.offsetTop}px`; */
}
/* 或使用 env(safe-area-inset-bottom) (iOS 11+ 支持) */
.bottom-bar {
  padding-bottom: constant(safe-area-inset-bottom); /* iOS 11.0-11.1 */
  padding-bottom: env(safe-area-inset-bottom);      /* iOS 11.2+ */
}
```

```css
/* ❌ 问题4: overflow: scroll + -webkit-overflow-scrolling 导致 sticky 失效 */
.scroll-container {
-  overflow-y: scroll;
-  -webkit-overflow-scrolling: touch;
}
.sticky-header {
-  position: sticky;
-  top: 0;
}

/* ✅ 修复方案A: 移除 -webkit-overflow-scrolling 但接受非弹性滚动 */
.scroll-container {
  overflow-y: auto; /* 用 auto 替代 scroll + touch */
}
.sticky-header {
  position: -webkit-sticky;
  position: sticky;
  top: 0;
  z-index: 1; /* 确保在滚动容器中覆盖其他内容 */
}

/* ✅ 修复方案B: 如果必须弹性滚动，将 sticky 元素移到滚动容器外 */
.page {
  display: flex;
  flex-direction: column;
  height: 100%;
}
.sticky-header {
  flex-shrink: 0;
  position: relative; /* 不依赖 sticky */
}
.scroll-container {
  flex: 1;
  overflow-y: scroll;
  -webkit-overflow-scrolling: touch;
}
```

```scss
/* ❌ 问题5: CSS 变量在 X5 WebView 中不继承 */
:root {
  --primary-color: #1677ff;
}
.button {
-  color: var(--primary-color); /* X5 可能不继承到 :root */
}

/* ✅ 修复: 同时提供硬编码降级值 */
.button {
  color: #1677ff;          /* 降级 */
  color: var(--primary-color, #1677ff); /* 浏览器支持则覆盖 */
}
```

npx playwright test --project='iPhone 14' --project='Pixel 7' \
  --grep 'visual-regression'

# 2. 特性支持检测脚本
# scripts/check-css-compat.js
const browserslist = require('browserslist');
const caniuse = require('caniuse-api');

const TARGET_BROWSERS = [
  'iOS Safari >= 15',
  'Android Chrome >= 100',
  'and_chr >= 90',    // Android WebView
];

function checkFeature(featureName) {
  const supported = caniuse.isSupported(featureName, TARGET_BROWSERS);
  if (!supported) {
    return { feature: featureName, status: 'UNSUPPORTED', fallback: 'required' };
  }
  return { feature: featureName, status: 'supported' };
}

console.table([
  checkFeature('flexbox-gap'),
  checkFeature('css-backdrop-filter'),
  checkFeature('css-sticky'),
  checkFeature('css-grid'),
  checkFeature('css-aspect-ratio'),
]);

## 原理
CSS 兼容性本质上是**渲染引擎实现差异**的累积。每个浏览器引擎（Blink、WebKit、Gecko）和每个非标准环境（微信小程序 WXSS、微信 X5 WebView）对 CSS 规范的实现时间和质量各不相同。关键机制：

1. **CSS 属性解析阶段**：如果不支持某属性，解析器会静默忽略它（CSS 错误处理模型）。这意味着 `gap: 12px` 在小程序中不报错，只是不生效——开发者需要额外手段检测。
2. **层叠上下文（Stacking Context）差异**：不同引擎对哪些 CSS 属性触发新的层叠上下文有不同实现。例如 `transform: translateZ(0)` 在 Blink 和 WebKit 中都创建层叠上下文，但在 X5 的某些版本中行为异常。
3. **布局引擎差异**：Flexbox 规范经历了多次修订（2009 草案 → 2012 过渡 → 2017 CR）。旧版引擎（如 X5 基于旧版 Blink）可能实现的是过渡期规范，与新版有细微差异。
4. **小程序 WXSS 的限制**：微信小程序的 WXSS 是其自有实现，并非完整的 CSS 解析器。它有意不支持部分 CSS 特性以减少渲染复杂度。不支持的特性被**静默删除**而非忽略，这意味着开发者甚至看不到计算样式中的降级。

## 架构视角 (Architecture Perspective)

### 架构层面根因
跨端样式兼容问题的表面原因是"渲染引擎碎片化"，但架构层面根因是**样式系统缺少平台抽象层**：
- 在 Web 开发中，CSS 直接被浏览器消费。但在跨端开发中，CSS 可能被编译为 WXSS（小程序）、被 RN StyleSheet 映射（React Native）、被 weex 解析（uni-app nvue）——每个目标平台都有一个独特的"CSS 消费管道"。当前的主流跨端框架将 CSS 源码直接暴露给这些管道，没有在中间插入一层"CSS 规范化/转译"层。
- **渐进增强逻辑缺失**：CSS 的 `@supports` 是浏览器运行时特性检测，但在跨端场景下，应该在**构建时**已知目标平台的能力矩阵，并根据能力矩阵生成平台特定的 CSS 产物（移除不支持的属性、插入降级代码）。运行时 `@supports` 在跨端中是不够的（因为它无法处理编译阶段就被删除的属性，如小程序 WXSS）。
- **设计系统的平台令牌未被建模**：间距、颜色、阴影等 Design Token 是平台无关的，但它们的 CSS 实现（`gap` vs `margin`，`backdrop-filter` vs `opacity`，`position: sticky` vs JS sticky）是平台相关的。当前架构将 Token 和实现混在同一个 CSS 文件中，导致平台差异无法被系统化管理。

### 系统级预防策略

**静态防护**：
```js
// stylelint 自定义规则
rules: {
  // 禁止直接使用跨端不安全的 CSS 属性
  'cross-platform/no-unsafe-css': ['error', {
    bannedProperties: [
      'gap',               // 需配合 margin 降级
      'backdrop-filter',   // 需配合 @supports
      'position: fixed',   // 小程序中严禁
      'aspect-ratio',      // 需配合 padding-top hack
    ],
    platforms: ['weapp', 'x5'],
  }],
  // 强制使用 CSS 变量而非硬编码值（便于平台令牌替换）
  'cross-platform/require-design-token': 'warn',
}
```

**CI 防护**：
```bash
# CI 中自动化 CSS 兼容性检查
# 1. 使用 stylelint + caniuse 插件
npx stylelint "src/**/*.{css,scss,less,vue}" \
  --config .stylelintrc.cross-platform.js

# 2. 多平台视觉回归测试
for platform in h5 weapp android-embed ios-embed; do
  npx playwright test --project="$platform" --grep "visual-regression"
done

# 3. CSS 使用统计（首次引入新 CSS 特性时触发审查）
npx css-feature-audit --targets "ios-safari>=15,and_chr>=90" \
  --src src/ --threshold warn
```

**架构防护**：
- **CSS 规范化管道**：在构建流程中插入 CSS 转译步骤，根据目标平台能力矩阵转换 CSS。例如：
  - `gap: 12px` → 小程序端编译为 `> * + * { margin-top: 12px }`
  - `backdrop-filter: blur(10px)` → 不支持平台编译为 `background: rgba(0,0,0,0.6)`
- **平台令牌系统**：定义平台能力变量（`$supports-backdrop-filter`, `$supports-gap`），在设计系统层面做平台适配，而非在每个组件中重复写 `@supports`。
- **样式审查委员会**：首次引入新 CSS 特性时必须经过平台兼容性审查，确认降级方案。

**流程防护**：
- 设计稿评审阶段就标注"跨端风险 CSS 属性"，而非在实现后才发现。
- 每个组件必须通过目标平台的视觉回归测试（截图对比）。
- 多端样式 Bug 的修复必须包含对应 pitfall 的引用（如 `#css` #pit-006, #pit-060）。

### 跨层影响链

```
上游影响 (设计层):
  - 设计稿在桌面端完美，真机上线后效果打折 → 设计-开发信任破裂
  - 设计师被迫学习平台限制 → 创造空间受限
  - "渐进增强"设计理念被"降级修补"替代

下游影响 (渲染层):
  - iOS Safari fixed + 键盘 Bug → 整个页面布局崩坏
  - Android X5 backdrop-filter 黑屏 → 用户认为 App 崩溃
  - 小程序 gap 不生效 → 内容重叠/不可读

部署影响:
  - 同一视觉 Bug 在不同平台需要不同修复策略 → 修复时间 × 平台数
  - 视觉回归测试的 CI 矩阵随平台数增长 → 测试时间可能成为瓶颈
  - CSS-only 的修复无法通过 CodePush 热更新（需要重新编译小程序包）

监控影响:
  - 样式 Bug 无法通过 Sentry 捕获（不抛出异常）
  - 需要专用视觉回归监控（截图对比服务如 Percy/Chromatic）
  - CLS (累积布局偏移) 指标在各平台差异大 → 需要按平台拆分 CWV 监控
```

### 相关架构模式
- **渐进增强模式** (`patterns.md`) — 从最安全的 CSS 基线开始，通过 `@supports` 逐层添加现代特性。
- **平台令牌 (Platform Tokens)** — 将 CSS 属性支持度建模为布尔变量，在设计系统层面路由到正确的 CSS 实现。
- **CSS 编译管道** — 类似 PostCSS 的插件链，但目标不是"添加前缀"，而是"根据平台矩阵生成平台特定的 CSS 文件"。
- **Safe Area 管理模式** (pit-023, pit-021) — `position: fixed` + 键盘的处理方式。

## 预防

1. **使用 CSS 特性检测优先的编写方式**：对所有可能不兼容的属性使用 `@supports` 包裹，并提供降级值。
2. **建立 CSS 兼容性检查 CI**：使用 stylelint + caniuse-lite 在 CI 阶段检查 CSS 属性是否在目标平台安全。
3. **设计系统优先使用安全属性**：在设计 Token 的实现层面使用最保险的 CSS (margin 而非 gap, opacity 而非 backdrop-filter)，在支持的平台上渐进增强。
4. **在 `@supports` 中检测——但不要仅依赖它**：小程序编译时 `@supports` 不生效，需要构建时转译。
5. **真机测试不能省略**：模拟器和桌面浏览器的渲染结果不代表真机。Flexbox gap 在模拟器中"正常"不代表在微信内置浏览器中正常。

## 回退

```bash
# 如果修复后出现新问题:

# 方案A: 移除新 CSS 特性，回退到安全基线
# 将 gap → margin, backdrop-filter → opacity, aspect-ratio → padding-top
git diff src/ -- "*.css" "*.scss" "*.vue" | grep "^+" | grep -E "gap:|backdrop-filter|aspect-ratio"
# 逐一回退到 margin/opacity/padding-top

# 方案B: 增加 @supports 降级而非替换
# 对于已在生产环境使用且部分平台支持的属性，保留现代写法 + 增加降级:
# @supports not (gap: 12px) {
#   .card-list > * { margin: 6px; }
# }

# 方案C: 平台特定 CSS 文件（最后手段）
# 创建 platform-specific.css 文件:
# src/styles/common.css       — 所有平台共享的安全样式
# src/styles/weapp.css        — 小程序特定补丁 (条件编译引入)
# src/styles/modern.css       — 仅现代浏览器 (通过媒体查询或 JS 动态加载)

# 如果引入新 Bug，回退:
git revert HEAD~1
```
