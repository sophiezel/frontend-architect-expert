# pit-066: npm 依赖投毒 — 恶意包通过 supply chain 注入

## 环境
- Node.js 项目使用 npm/yarn/pnpm
- 依赖了数百到数千个传递依赖
- CI/CD 自动构建部署

## 症状
用户报告"登录页行为异常"——输入密码后页面卡住，同时浏览器向陌生域名发送了 POST 请求。安全团队排查发现项目的某个深层传递依赖（如一个不起眼的 `is-even` helper 的依赖的依赖）包含混淆代码，在特定条件下窃取表单数据。

或者：CI 构建日志中出现异常的网络请求（下载额外 payload），构建产物体积异常增大。npm audit 未报警（因为是新型恶意包）。

## 根因
供应链投毒（Supply Chain Poisoning）是攻击者将恶意代码注入合法 npm 包的场景。常见手法：

1. **拼写欺诈 (Typosquatting)**：发布 `@babel/coree`（注意多了一个 e）—— 开发者手误安装
2. **账号劫持**：窃取维护者的 npm 账号，发布带后门的版本
3. **依赖混淆 (Dependency Confusion)**：公司内部包名被发布到公共 registry（`@mycorp/utils`）
4. **恶意维护者**：合法包维护者主动/被收买添加恶意代码
5. **install script 执行**：`preinstall` / `postinstall` 脚本拥有 Shell 执行能力
6. **隐形依赖**：锁版本后不再审计，导致旧已知漏洞长期存在

## 诊断方法

```bash
# 1. 扫描 npm install scripts（最危险的攻击面）
grep -rn '"preinstall"\|"postinstall"\|"install"' node_modules/*/package.json | grep -v '"install":' | head -20
# 检查每个含有 install script 的包是否来自可信来源

# 2. Socket.dev 行为分析
npx @socketsecurity/cli scan package.json --json | jq '.packages[] | select(.score < 0.5)'
# score < 0.5 表示存在可疑行为（网络请求、文件系统访问等）

# 3. 检查依赖中是否存在境外/陌生 registry
grep -rn "resolved.*registry" package-lock.json | grep -v "registry.npmjs.org" | head -20

# 4. 检查包的发布者/维护者是否异常
npm view <package-name> maintainers
npm view <package-name> time --json | jq 'keys | last'  # 最后发布时间
# 如果一个稳定包突然大量更新 → 账号可能被劫持

# 5. npm audit 已知漏洞
npm audit --json | jq '.vulnerabilities | to_entries | map(select(.value.severity == "critical" or .value.severity == "high"))'

# 6. 检查包是否包含混淆代码
# 混淆代码特征：变量名极短、大量数字编码、eval/Function 使用
find node_modules -name '*.js' -exec grep -l "\\\\x[0-9a-f][0-9a-f]" {} \; | head -20
find node_modules -name '*.js' -exec grep -l "atob\|btoa\|Buffer.from.*base64" {} \; | head -20
```

## 修复

```diff
// ═══════ 修复1：移除恶意包 ═══════
// 如果确定某个包是恶意包
- // package.json 中的依赖
- "tiny-helper-utils": "^1.2.3"

+ // 替换为可信替代
+ "lodash.get": "4.4.2"  // 功能相当的已验证包

// ═══════ 修复2：锁定依赖来源 ═══════
// .npmrc
+ registry=https://registry.npmjs.org/
+ // 强制 HTTPS
+ // 如果使用私有 registry，配置 scope
+ @mycompany:registry=https://npm.mycompany.com/

// ═══════ 修复3：禁止 install scripts（安全包） ═══════
// .npmrc
+ ignore-scripts=true
// 注意：这会影响 esbuild/core-js/native 模块的编译
// 可以在 CI 中先 npm ci --ignore-scripts 后单独运行必要的 scripts:
// npm rebuild esbuild

// ═══════ 修复4：npm overrides 强制安全版本 ═══════
// 如果某个传递依赖有已知漏洞
{
  "overrides": {
    "braces": "3.0.3",         // CVE-2024-4068
    "follow-redirects": "1.15.6", // CVE-2024-28849
    "micromatch": "4.0.8"      // CVE-2024-4067
  }
}

// ═══════ 修复5：依赖混淆防护 ═══════
// 确保私有包名在公共 npm 上被注册占位（防止别人抢注）
// 在公共 npm 上发布空的占位包 @mycompany/utils
// 或者在 .npmrc 中配置 scope 指向私有 registry:
// @mycompany:registry=https://npm.mycompany.com/

// ═══════ 修复6：lockfile 完整性验证 ═══════
// CI 中使用 npm ci（而非 npm install）确保 lockfile 被遵守
// package.json scripts:
+ "install": "echo '请使用 npm ci 而非 npm install' && exit 1",
```

## 验证

```bash
# 1. 验证恶意包已完全移除
npm ls tiny-helper-utils 2>&1 | grep "empty"  # 期望输出 "empty"
find node_modules -name "tiny-helper-utils" -type d  # 期望无结果

# 2. 重新执行安全扫描
npm audit --omit=dev --audit-level=moderate
npx @socketsecurity/cli scan package.json

# 3. 检查 network 请求（确保无后门继续通信）
# 启动应用后抓包:
sudo tcpdump -i any -n host not 127.0.0.1 and port not 22
# 或 Chrome DevTools → Network → 过滤 domain != 'your-domain.com'

# 4. 检查构建产物 hash 恢复
# 修复前后对比 dist/ 的 checksum
md5sum dist/**/*.js > /tmp/after-fix.md5
diff /tmp/before-attack.md5 /tmp/after-fix.md5
# 差异项应是修复带来的预期变更

# 5. CI 中添加审计门禁
# .github/workflows/security.yml
# npx @socketsecurity/cli scan package.json --enforce
# exit code != 0 → CI 阻断
```

## 原理
npm 生态的安装模型使每个包都有机会执行任意代码：`package.json` 中的 `scripts.preinstall` 和 `scripts.postinstall` 在安装时被 npm 自动执行。这意味着安装一个依赖瞬间，攻击者就可获得开发者的本地 Shell 权限，进而：
- 窃取 SSH 密钥 / `.env` 文件 / `.gitconfig`
- 修改 source code 注入后门
- 将后门打包进 CI 构建产物

即使没有 install scripts，恶意代码也可通过运行时的 `require` 执行：攻击者在包的 `index.js` 中直接放置混淆代码，消费方 `import` 时即触发。这使得恶意代码可以潜伏很长时间不被发现（尤其是对复杂传递依赖的深层节点）。

依赖混淆的原理更隐蔽：当项目同时依赖公共 registry 和私有 registry 时，如果私有包名 (`@mycorp/utils`) 在公共 registry 也被攻击者发布，npm/pnpm 的解析顺序可能导致优先使用公共 registry 的恶意版本。

## 架构视角 (Architecture Perspective)

### 架构层面根因
依赖投毒的表面原因是"安装了恶意包"，架构根因是**依赖治理体系缺失**：
- **依赖引入无审批流程**：开发者可以随意 `npm install <anything>` 带入新依赖及其数千个传递依赖。
- **依赖去重/扁平化缺乏可见性**：`node_modules` 扁平化后，直接依赖和传递依赖地位相同，无法区分"可信来源"和"间接依赖"。
- **缺少运行时行为基线**：不知道正常应用的网络请求/文件访问模式，无法检测到依赖引入的异常行为。
- **lockfile 信任链断裂**：`package-lock.json` 虽然记录完整性哈希，但哈希对应的包本身可能是恶意的（npm registry 允许发布）。

### 系统级预防策略

**静态防护**：
```bash
# .npmrc
ignore-scripts=true             # 禁止 install scripts
audit-level=high                # 高/严重漏洞阻断安装
fund=false                      # 隐藏 funding 噪音
package-lock=true               # 强制生成 lockfile
lockfile-version=3              # 使用最新 lockfile 格式
```

**CI 防护**：
```yaml
# GitHub Actions supply chain security
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
  with:
    node-version: 20
- run: npm ci --audit=true
  env:
    NPM_CONFIG_AUDIT: true
- run: npm audit --omit=dev --audit-level=high
- run: npx @socketsecurity/cli scan package.json --enforce
- uses: step-security/harden-runner@v2
  with:
    egress-policy: audit
# 阻止 CI 构建步骤访问外网（除已知 registry）
```

**架构防护**：
- **私有 npm mirror/proxy**：所有依赖经过内部代理（Verdaccio / Artifactory），由安全团队审核后才能进入内部 registry。
- **Dependency Firewall**：设定允许依赖白名单，新增依赖需自动化 + 人工审批（类似 GitHub Branch Protection）。
- **SBOM 生成并审计**：每次构建生成 SBOM，对比上次构建的 SBOM 差异，标记新增依赖。
- **依赖最小化原则**：用 `depcheck` 定期清理未使用的依赖，减少攻击面。

**流程防护**：
- "新增 npm 依赖"需在 PR 描述中写清理由和评估（维护者活跃度 / 下载量 / install scripts 数量）。
- 每次依赖变更自动触发 Socket.dev 扫描，高危结果阻断 merge。
- 定期（每 2 周）运行 `npm outdated + npm audit` 并排期升级。

### 跨层影响链

```
上游影响:
  - 开发环境被入侵 → SSH 密钥泄露 → GitHub 仓库被篡改
  - CI 构建产物被注入后门 → 所有用户受影响

下游影响:
  - 用户数据（密码/Token/支付信息）被窃取 → 财务损失 + 合规罚款
  - 浏览器被注入挖矿脚本 → 用户体验恶化 + 品牌受损
  - 后门被发现 → CVE + 安全公告 → 客户信任崩塌

行业影响:
  - 类似 event-stream 事件 (2018) — 著名 npm 包被恶意维护者植入窃取比特币代码
  - 类似 colors/faker 事件 (2022) — 维护者主动破坏包功能
  - 类似 xz utils 后门 (2024) — 国家级供应链攻击
```

### 相关架构模式
- **SBOM + SLSA attestation** (`frontend-security.md` 第3.3节) — 供应链物料清单与来源验证
- **lockfile-lint** (`frontend-security.md` 第3.1节) — lockfile 完整性和注册源验证
- **npm provenance** (`frontend-security.md` 第3.3节) — 发布的包来源可验证

## 预防

1. **启用 `.npmrc` ignore-scripts**：禁止 install scripts 的自动执行（需要 postinstall 的包单独审计）。
2. **私有 npm mirror**：所有外部依赖经过内部代理审计。
3. **CI 门禁**：npm audit + Socket.dev scan + lockfile 完整性检查，三个缺一不可。
4. **依赖准入审批**：新增依赖需要 PR 描述 + 自动扫描 + 人工审批。
5. **定期依赖清理**：每季度用 `depcheck` 移除未使用依赖，减少攻击面。
6. **订阅安全公告**：npm security advisories / GitHub Dependabot / Snyk 持续监控。

## 回退

```bash
# 方案A：回退到已知安全的 lockfile
git checkout HEAD~1 -- package-lock.json
npm ci

# 方案B：使用 overrides 强制安全版本（临时措施）
# package.json
{
  "overrides": {
    "compromised-package": "npm:trusted-alternative@^1.0.0"
  }
}
npm ci

# 方案C：固定到上一安全版本
npm install compromised-package@1.2.0  # 回到已知安全版本
# 在 package.json 中移除 ^ 或 ~，使用精确版本

# 回退后动作：
# 1. 轮换所有泄露的密钥/Token
# 2. 审计 CI 构建日志，确认后门是否已被打包进历史产物
# 3. 如果确认恶意代码已进入生产 → 按安全事件流程上报
# 4. 回滚最近一次部署
```
