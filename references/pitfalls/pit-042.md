# pit-042: Monorepo 管理噩梦

## 环境
- pnpm workspaces / yarn workspaces / npm workspaces / Turborepo / Nx
- 5+ packages 的 monorepo
- 跨包互相依赖

## 症状
CI 构建时间从 5 分钟膨胀到 30 分钟——每次改了 `packages/shared` 的一个文件，所有依赖它的包和下游应用都被重新构建。`package.json` 中声明的依赖在代码中能用（因为有"幽灵依赖"），但在另一台机器或 CI 中 `pnpm install` 后报 `Module not found`。

更难排查的是：同一个包被安装了多个不兼容的版本——`packages/A` 用 `lodash@4.17.21`、`packages/B` 用 `lodash@3.10.1`，但它们交互时传递了 Lodash 包装后的对象，导致运行时类型不匹配。

典型场景：
- `packages/web` 引用了 `vue@3.4`，`packages/shared` 引用了 `vue@3.3` → pnpm 的严格模式下不会提升，各自用各自的版本
- `packages/A` 用了 `lodash` 但没有在 `package.json` 中声明（依赖了 B，B 依赖了 lodash → 幽灵依赖）
- Turbo 的缓存命中率 < 30%（`turbo.json` 中 `inputs` 配置太宽，随便什么文件变化都导致缓存失效）

## 根因
1. **幽灵依赖(Phantom Dependency)**：npm/yarn classic 的扁平化 `node_modules` 将所有依赖提升到顶层，包 A 可以 require 包 B 的依赖。pnpm 使用符号链接（symlink）实现了严格的依赖隔离，暴露了这些隐式依赖。

2. **版本不一致**：monorepo 中多个包各自声明同一依赖的不同版本。如果没有版本统一策略（如 syncpack），这些版本会逐渐漂移。

3. **构建缓存失效**：monorepo 的构建缓存（Turbo/Nx/Bazel）取决于依赖图的哈希。任何一个间接依赖（如 `tsconfig.json`、`.eslintrc.js`）被包含在 `inputs` 中，都会导致整个依赖链缓存失效。

4. **循环依赖在 monorepo 中放大**：packages/A → packages/B → packages/A，这种循环在单 repo 中是模块级问题，在 monorepo 中变成包级——影响整个包的构建图。

## 诊断方法

```bash
# 1. 检测幽灵依赖
# 使用 pnpm 的严格模式检查
pnpm dlx @pnpm/check-peer-deps -- .

# 或直接分析：哪些依赖被用了但未声明
npx depcheck

# 2. 检测版本不一致
npx syncpack list-mismatches

# 3. 检查循环依赖
npx madge --circular --extensions ts packages/

# 4. 检查构建缓存命中率（Turborepo）
turbo run build --dry-run 2>&1 | grep "cache hit"
# 计算 cache hit 比例

# 5. 分析包之间的依赖关系
npx nx graph  # Nx 可视化依赖图
# 或
npx turbo run build --graph  # Turbo 输出依赖图

# 6. 检查包大小
find packages -name "dist" -type d | while read d; do
  du -sh "$d"
done
```

## 修复

```diff
// ❌ 问题 1: 幽灵依赖 — 用了但未声明
// packages/web/src/index.ts
import { debounce } from 'lodash'; // lodash 没在 web/package.json 中声明
// 能跑是因为 packages/shared 声明了 lodash 且 pnpm 隐式链接了

// ✅ packages/web/package.json
{
  "dependencies": {
+    "lodash": "workspace:*"  // 或固定版本
  }
}

// ❌ 问题 2: 版本不一致
// packages/A/package.json: "vue": "^3.3.0"
// packages/B/package.json: "vue": "^3.4.0"
// packages/web/package.json: "vue": "3.2.47"

// ✅ 方案 A: 使用 syncpack 统一版本
// .syncpackrc.js
module.exports = {
  dependencyTypes: ['dev', 'prod', 'peer'],
  versionGroups: [{
    dependencies: ['vue', 'vue-router', 'pinia'],
    packages: ['**'],
    pinVersion: '3.4.21',
  }]
};
// 在 CI 中运行: syncpack fix-mismatches

// ✅ 方案 B: 在 root package.json 中声明所有共享依赖的 catalog (pnpm 9+)
// pnpm-workspace.yaml
packages:
  - 'packages/*'
catalog:
  vue: ^3.4.21
  vue-router: ^4.3.0
  pinia: ^2.1.7
// 子包中使用 "vue": "catalog:"

// ❌ 问题 3: Turbo 构建缓存频繁失效
// turbo.json
{
  "pipeline": {
    "build": {
-      "dependsOn": ["^build"],
+      "dependsOn": ["^build"],
+      "inputs": ["src/**", "tsconfig.json", "!src/**/*.test.ts", "!src/**/*.stories.ts"],
+      "outputs": ["dist/**", ".next/**"]
    }
  }
}

// ❌ 问题 4: 循环依赖
// packages/A/package.json: "dependencies": { "@myapp/B": "workspace:*" }
// packages/B/package.json: "dependencies": { "@myapp/A": "workspace:*" }

// ✅ 提取共享类型到第三个包
+// packages/types/package.json
+{ "name": "@myapp/types", "main": "./src/index.ts" }
+// packages/A 和 packages/B 都依赖 @myapp/types，而不是互相依赖
+// 如果 B 需要调用 A 的功能，通过依赖反转：
+// packages/A 实现 B 定义的接口(interface)
+// 而非 B 直接 import A
```

## 验证

```bash
# 1. 幽灵依赖为 0
npx depcheck
# 或在 pnpm 项目中
pnpm run check:deps  # 需要配置 depcheck 脚本

# 2. 版本一致性
npx syncpack list-mismatches
# 期望：No mismatches found

# 3. 循环依赖为 0
npx madge --circular --extensions ts packages/
# 期望：No circular dependencies found

# 4. CI 构建时间
# 分别测试：全量构建 (no cache) vs 单文件变更构建
time turbo run build --force  # 全量
time turbo run build          # 增量（修改一个文件后）

# 5. 子包可独立发布
cd packages/shared
npm pack --dry-run  # 检查打包的产物是否完整
```

## 原理
pnpm 的 `node_modules` 结构：
```
node_modules/
  .pnpm/                  ← 所有包的硬链接存储（content-addressable）
    vue@3.4.21/
    lodash@4.17.21/
  vue → .pnpm/vue@3.4.21/node_modules/vue  ← 符号链接
  package-A/              ← 符号链接到 packages/A
```

这种结构确保了每个包只能访问自己在 `package.json` 中声明的依赖。当包 A 尝试 `require('lodash')` 但未声明时，pnpm 不会在 A 的 `node_modules` 中创建 lodash 的符号链接 → Module not found。

Turborepo 的缓存机制：对于每个 task，Turbo 计算其 `inputs` 文件集合的哈希值（SHA256）。如果哈希与之前相同，直接复用缓存的输出。`inputs` 配置决定了"文件变化粒度"——越精确，缓存命中率越高。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Monorepo 噩梦的架构根因是**monorepo 被当作"把仓库合并在一起"而非"包与包之间的契约体系"来管理**：
- **依赖声明是"建议"而非"法律"**：在 npm/yarn classic 中，依赖扁平化让开发者养成了"忽略未声明依赖"的习惯。但在 pnpm 严格模式和 monorepo 中，依赖声明是包能正常运行的唯一保证。架构需要将"依赖声明完整性"提升为与"类型安全"同等的红线。
- **包的边界没有建模为"微服务"的边界**：微服务的黄金法则是"每个服务独立部署"。monorepo 中的包也应有同样的原则——任何包应能独立于其他包进行构建、测试、发布。但现实中包之间的耦合度常常与单 repo 中模块之间的耦合度没有区别。
- **版本漂移是"熵增"的自然结果**：没有版本统一策略，依赖版本会随时间自然散开。syncpack/catalog 是"对抗熵增"的工具，但多数项目在 monorepo 建立几个月后才发现版本已严重不一致。

### 系统级预防策略

**静态检查**：
```bash
# pre-commit hook: 检查幽灵依赖
- repo: pre-commit
  hooks:
    - id: depcheck
      name: Check for phantom dependencies
      entry: npx depcheck
      language: node
      files: \.(ts|tsx|js|jsx)$

# 检查版本一致性
    - id: syncpack-mismatches
      name: Check dependency version mismatches
      entry: npx syncpack list-mismatches
      language: node
```

**CI 门禁**：
```bash
# .github/workflows/monorepo-check.yml
- name: Depcheck (phantom deps)
  run: pnpm run check:deps
- name: Version consistency
  run: syncpack list-mismatches
- name: Circular dependency check
  run: madge --circular --extensions ts packages/ --no-spinner
- name: Build cache integrity
  run: |
    turbo run build --force
    touch packages/shared/src/constants.ts
    turbo run build  # 应只重建 shared 及其下游
```

**架构约束**：
- 依赖声明完整性是硬性门禁——depcheck 必须通过才能合并
- 共享的 framework/library（vue/react/antd）使用 catalog 锁定版本
- 每个包必须在 CI 中独立构建（`turbo run build --filter=@myapp/xxx`）
- 包间依赖方向必须遵循"单向无环"：`types ← shared ← feature ← app`（越底层越稳定）
- 新增包时必须明确其"稳定性层级"（types: 最高稳定，app: 最低稳定）

**流程约束**：
- 新增依赖时在 PR 中标注"已确认在正确的 package.json 中声明"
- 版本升级必须通过 syncpack 推动（而非手动各包修改）
- 每周自动运行一次 `syncpack fix-mismatches`（自动化 PR）

### 跨层影响链

```
上游影响 (开发体验):
  - 幽灵依赖 → "在我机器上能跑" → 新人 on-boarding 时间加倍
  - 构建缓存失效 → 每次 push 等 30 分钟 → 开发者频繁切换上下文

下游影响 (部署):
  - 版本不一致 → 生产环境运行时行为与 staging 不同
  - 包未独立验证 → 发布 broken package 到 npm registry

部署影响:
  - npm publish 前的 pre-publish 检查不完整 → 发布了缺少依赖的包
  - changeset 版本管理错误 → 发布了不兼容的 breaking change

监控影响:
  - CI 构建时间趋势（p50/p95）
  - 缓存命中率（应 > 60%）
  - 依赖版本漂移的度量（每月运行 syncpack 统计不一致数）
```

### 相关架构模式
- **Dependency Inversion 在 monorepo 中的应用** (`patterns.md` 第十七节) — 高层包定义接口，低层包实现接口，通过依赖反转打破循环。
- **Changeset 版本管理** (`decisions.md` 第十五节) — 与 semver 结合，管理 monorepo 中多包的版本发布。

## 预防
1. pnpm 严格模式（默认）+ depcheck 检测幽灵依赖
2. syncpack 或 pnpm catalog 统一共享依赖版本
3. Turbo/Nx 的 `inputs` 配置精细化，排除测试文件、stories、文档
4. monorepo 依赖方向约定：`types ← shared ← feature-lib ← app`，禁止反向依赖

## 回退
