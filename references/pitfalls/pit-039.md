# pit-039: Node.js Stream 背压问题

## 环境
- Node.js 14+（Stream API 稳定）
- 使用 `fs.createReadStream` / `fs.createWriteStream` / `pipeline` 的场景
- 大文件处理、日志管道、HTTP 请求代理

## 症状
处理大文件时 Node.js 进程内存持续增长，最终 OOM（Out of Memory）被系统 kill。或者数据处理结果不完整——管道末端的文件比预期小。使用 `pipe()` 时，某个环节出错后整个流断裂但没有错误提示。

典型场景：
- 用 `fs.createReadStream` 读取 2GB 的日志文件写入 HTTP 响应，内存从 50MB 飙升到 2GB+
- `readable.pipe(transform).pipe(writable)` 中 transform 环节抛异常，流提前关闭但 writeStream 还在等待
- CSV 解析流的处理速度远慢于读取速度，积压在内存中

## 根因
Node.js Stream 的背压（backpressure）机制：当 Writable Stream 的处理速度慢于 Readable Stream 的读取速度时，Writable 会通知 Readable "暂停一下"。`pipe()` 方法内置了背压处理，但它有两个致命缺陷：
1. `pipe()` 不转发错误——源流或中间流出错后不会自动销毁目标流
2. `pipe()` 不返回 Promise，无法用 async/await 等待完成

更根本的根因：
- `highWaterMark` 默认 16KB（可读流）和 16KB（可写流），对于高吞吐场景这个缓冲太小
- 开发者自定义 Transform 流时没有正确调用 `callback()` 或在 `_transform` 中操作了 `this.push()` 的时机错误

+const transform = new Transform({
+  transform(chunk, encoding, callback) {
+    try {
+      const result = processChunk(chunk);
+      callback(null, result); // ✅ 通过 callback 传结果，pipeline 会自动处理背压
+    } catch (err) {
+      callback(err);          // ✅ 错误通过 callback 传递

## 原理
Node.js Stream 的背压机制基于内部缓冲区和 `highWaterMark`：
- Readable Stream 内部有一个缓冲区，当 `push()` 的数据超过 `highWaterMark` 时，`push()` 返回 `false`
- Writable Stream 同样有缓冲区，当 `write()` 返回 `false` 时，生产者应暂停写入，等待 `'drain'` 事件
- `pipe()` 自动管理这个握手过程，解除了开发者的心智负担——但代价是错误传播缺失

`pipeline` (Node.js 10+) 解决了 `pipe()` 的两个缺陷：自动转发错误、自动销毁所有流。Node.js 15+ 的 `stream/promises` 提供了 `pipeline` 的 Promise 版本，可以与 async/await 无缝集成。

## 架构视角 (Architecture Perspective)

### 架构层面根因
流背压问题的架构根因是**数据处理管道的"流式思维"缺失**：
- **批处理惯性**：多数后端开发者习惯"读入全部数据 → 处理 → 写出"的批处理模式。当数据规模超出内存时，将批处理代码改为流式处理只是把 `readFileSync` 换成 `createReadStream` + `pipe()`，但没有同步引入背压监控、错误传播、流生命周期管理——流式基础设施被当作"更快的批处理"来用。
- **流管道缺乏可观测性**：一条流管道（read → transform → write）是一个状态机，但在生产环境中这个状态机是完全黑盒的。当前缓冲区大小、暂停次数、处理速率（items/s）都没有暴露为 metrics。
- **多个流的生命周期未协调**：当服务同时打开数百个流（如文件上传代理），每个流无限制地消耗缓冲内存。缺少"全局流并发控制"和"单流内存预算"。

### 系统级预防策略

**静态检查**：
```bash
# ESLint 规则：禁止使用 .pipe()
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'MemberExpression[property.name="pipe"]',
    message: '请使用 stream/promises 的 pipeline() 替代 pipe()'
  }]
}

# TypeScript 类型约束
// 禁止非流式的文件读取
// .eslintrc.js 中禁止 fs.readFileSync（大文件场景）
rules: {
  'no-restricted-imports': ['error', {
    name: 'fs',
    importNames: ['readFileSync'],
    message: '大文件操作请使用 fs.createReadStream + pipeline'
  }]
}
```

**CI 门禁**：
```bash
# CI 中运行流处理的内存测试
test('处理 500MB 数据时内存 < 100MB', async () => {
  const startMemory = process.memoryUsage().heapUsed;
  await processLargeFile(); // 500MB 测试数据

# 流泄漏检测：多次执行后检查 process._getActiveHandles()
test('流程结束后无活跃的流句柄', async () => {
  await pipeline(...);
  const handles = process._getActiveHandles();
  const streamHandles = handles.filter(h => h.constructor.name.includes('Stream'));
  expect(streamHandles.length).toBe(0);
});
```

**架构约束**：
- 项目级强制：所有流组合使用 `pipeline()` 而非 `pipe()`
- 自定义流必须实现 `_destroy()` 方法清理资源
- 对流处理建立配额系统：
```typescript
// src/streams/stream-pool.ts
class StreamPool {
  private activeCount = 0;
  private maxConcurrent = 100;
  
  async acquire(): Promise<void> {
    while (this.activeCount >= this.maxConcurrent) {
      await new Promise(resolve => setTimeout(resolve, 100));
    }
    this.activeCount++;
  }
  
  release(): void { this.activeCount--; }
}
```

**流程约束**：
- Code Review 检查："是否使用了 pipe()？如果是，是否考虑了错误传播和流销毁？"
- 涉及文件/大数据处理的 PR 必须附带内存使用截图

### 跨层影响链

```
上游影响 (API 层):
  - 文件上传 API OOM → 整个 Node.js 进程 crash → PM2 重启 → 丢失所有进行中的请求
  - 日志管道 OOM → 日志丢失 → 丢失可观测性（最讽刺的故障模式）

下游影响 (存储):
  - 流提前断开 → 写入不完整文件 → 数据损坏
  - 无背压 → 磁盘 I/O 被写满 → 影响同机其他服务

部署影响:
  - 需要配置 Node.js 的 --max-old-space-size 作为止血带
  - 容器 OOMKilled 后 K8s 自动重启 → pod 重启循环

监控影响:
  - 必须监控: process.memoryUsage() 中的 external memory（Buffer 占用）
  - 自定义指标: 活跃流数量、流管道吞吐量（bytes/s）
  - 告警: external memory > 阈值 或 活跃流 > 并发上限
```

### 相关架构模式
- **Reactive Streams 规范** (`patterns.md` 第十五节) — Java Reactive Streams 的背压协议（request(n) → onNext → onComplete），是流式处理架构的金标准。Node.js Stream 的背压机制是对其的简化实现。
- **Buffer Pool 模式** — 预分配 Buffer 池避免频繁 GC，与 highWaterMark 配合使用。

## 预防
1. 全局禁止 `pipe()`：ESLint 规则强制使用 `pipeline()` + async/await
2. 对所有自定义可读/可写流实现 `_destroy()` 清理方法
3. 设置合理的 `highWaterMark`：大型文件处理 512KB-1MB，网络流 64KB-256KB
4. 使用 `stream.pipeline()` 或 `stream/promises` 的 `pipeline` 确保错误传播和流清理
