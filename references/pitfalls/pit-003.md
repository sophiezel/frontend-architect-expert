# pit-003: 列表渲染缺少 key 导致协调错误

## 环境
- React 所有版本
- 使用 `Array.map` 渲染列表时

## 症状
列表项错位：输入框内容跑到别的行、动画应用到错误的元素、勾选状态串位。删除/插入/排序操作后 UI 状态完全混乱。

## 根因
React 通过 key 来追踪列表中的每个元素。缺少 key 时 React 使用数组索引作为默认 key。当列表顺序变化（插入/删除/排序），索引移位导致 React 误认为同一个索引位置的元素是"同一"个组件，从而复用错误的 DOM 节点和组件状态。

## 诊断方法

```bash
# 1. 检查 React DevTools 中的 key
# Chrome → React DevTools → Components → 选中列表项
# 右侧面板查看 key 属性值，如果显示为数字索引（0, 1, 2...）即为问题

# 2. 控制台检查 React warning
# 打开浏览器控制台，查找：
# Warning: Each child in a list should have a unique "key" prop.

# 3. 交互测试：给列表项添加状态并检查
# 在每个列表项中添加一个 checkbox 或 input
# 然后删除第一项，观察剩余项的选中状态是否串位

# 4. 代码搜索
grep -rn "\.map.*=>" src/ --include="*.tsx" --include="*.jsx" | grep -v "key="
# 查找 map 返回 JSX 但没有 key 属性的代码
```

## 修复

```diff
// ❌ 问题代码：使用索引作为 key 或缺少 key
function TodoList({ todos }) {
  return (
    <ul>
-      {todos.map((todo, index) => (
-        <TodoItem key={index} todo={todo} /> // 索引不稳定
-      ))}
+      {todos.map((todo) => (
+        <TodoItem key={todo.id} todo={todo} /> // 使用稳定唯一标识
+      ))}
    </ul>
  );
}
```

```diff
// ❌ 更隐蔽的问题：就地生成的随机 key
function List({ items }) {
  return items.map(item => (
-    <div key={Math.random()}> // 每次渲染都变 → 全部卸载重建
-      {item.name}
-    </div>
  ));
}
// ✅ 正确
function List({ items }) {
  return items.map(item => (
+    <div key={item.id}>{item.name}</div>
  ));
}
```

## 原理
React 的 Reconciliation（调和）算法使用 key 来判断元素是否可以复用。当 key 相同时，React 认为元素未变，只更新 props；当 key 不同时，卸载旧元素并挂载新元素。使用索引作为 key 时，插入/删除操作会导致索引偏移，React 就会错误地将不同数据项识别为"同一"组件。

## 架构视角 (Architecture Perspective)

### 架构层面根因
key 缺失的架构根因是**数据模型层缺少稳定身份标识（Stable Identity）**：
- **数据契约不完整**：从后端 API 返回的列表数据没有强制要求包含唯一 ID 字段，导致数据类型定义（TypeScript interface）未将 `id` 标记为必需字段。这是 API Schema 设计层面的问题——列表数据必须包含稳定标识符，否则前端无法建立数据→UI 的稳定映射。
- **视图层与数据层之间的身份映射协议缺失**：组件渲染列表时，React 需要一个"身份协议"来回答"这个 UI 节点代表哪个数据实体"。如果没有显式定义这个协议（通过 key），React 默认使用数组顺序作为身份映射——这是一种脆弱的隐式协议，类似于数据库中依赖插入顺序而非主键。
- **缺少跨模块的数据身份传递通道**：数据从 API → 状态管理 → 组件树的过程中，身份信息在每一层都被保留和传递。但如果在某一层（如状态管理层的 selector）丢失了 ID，后续层面就无法补救。架构需要确保"身份信息贯穿数据流全链路"。

### 系统级预防策略

**静态防护**：
```js
// ESLint 规则: 强制 key
rules: {
  'react/jsx-key': ['error', { 
    checkFragmentShorthand: true,
    checkKeyMustBeforeSpread: true 
  }]
}

// TypeScript: 定义列表项类型必须包含 id
type ListItem = {
  id: string;  // 强制要求身份字段
  [key: string]: unknown;
};

// 禁止使用 Math.random() / Date.now() 作为 key
// 通过 no-restricted-syntax 规则禁止：
'no-restricted-syntax': ['error', {
  selector: 'JSXAttribute[name.name="key"] CallExpression[callee.object.name="Math"][callee.property.name="random"]',
  message: '禁止使用 Math.random() 作为 key'
}]
```

**CI 防护**：
```bash
# CI 中检测 key 模式
# 使用 jscodeshift 检测所有 map 内的 JSX 是否有 key
npx jscodeshift --parser=tsx -t scripts/detect-missing-key.ts src/

# Bundle 分析时检查是否有 index 作为 key 的模式（高风险）
grep -rn 'key={index}\|key={i}\|key={`${index}' src/ && echo "WARNING: index as key found" && exit 1
```

**架构防护**：
- **API Schema 契约层**：在 GraphQL schema 或 REST API 响应类型中，所有列表项的 ID 字段必须为非空、不可变、全局唯一。后端需要在接口文档中明确承诺 ID 的稳定性。
- **数据层归一化（Normalization）**：使用 Redux Toolkit 的 `createEntityAdapter` 或 React Query 的 `select` 将列表数据按 ID 归一化存储，建立 `{ ids: string[], entities: Record<string, T> }` 结构，确保组件层始终能通过 ID 索引数据。
- **组件库规范**：自定义封装的 `<List>` / `<Table>` 组件强制要求 `rowKey` 属性，不接受默认索引，缺失时在开发环境抛出错误。

**流程防护**：
- Code Review 清单项："列表渲染的 key 是否来自数据的稳定唯一 ID（如数据库主键）？"
- 架构评审：新增列表页面时，确认后端接口是否返回稳定 ID，若无则阻塞前端开发。

### 跨层影响链

```
上游影响 (消费者):
  - 父组件传入的数据如果 ID 重复 → 多个 key 冲突 → React 报错
  - 兄弟组件的 zustand selector 返回数据如果丢失 ID → 键传播失败

下游影响 (依赖):
  - DOM 状态（input 值、scroll 位置、focus 状态）串位
  - CSS 动画和 FLIP 动画应用到错误的元素
  - ARIA 无障碍属性（aria-describedby）指向错误的节点
  - React Native FlatList 的视图复用错误（参见 pit-014）

部署影响:
  - 数据驱动 bug（仅在特定列表操作后出现）→ 难以在 staging 复现
  - 用户数据泄露风险：A 用户看到 B 用户的部分 UI 状态
  - 线上紧急修复时，更改 key 策略会导致已缓存的 DOM 状态全量丢失

监控影响:
  - E2E 测试必须覆盖：插入/删除/排序后的 UI 状态一致性
  - 自定义 lint 规则检测 index 作为 key
  - React DevTools 中监控列表渲染的 key 分布是否健康
```

### 相关架构模式
- **组件组合模式** (`patterns.md` 第二节) — Compound Components 模式通过 Context 隐式传递状态，同样需要稳定 key 来保证子组件身份映射正确。
- **虚拟列表（Virtual List）** (`patterns.md` 第四节) — 虚拟列表极度依赖稳定 key 来正确复用 DOM 节点，key 错误会导致渲染串位。
- **CQRS 前端化** (`patterns.md` 第三节) — 在数据读取层使用 `normalizr` 或 `createEntityAdapter` 归一化数据，将身份管理提升为架构级关注点。

## 预防

1. **使用数据中的唯一 ID**（如数据库主键 `item.id`），永远不要用 `Math.random()` 或 `Date.now()`。
2. **索引 key 的例外场景**：仅在列表是静态的、永远不会重新排序或增删时可用，且要确保你不会在未来改变它。
3. **ESLint**：`eslint-plugin-react` 的 `react/jsx-key` 规则强制要求 key。
4. **CR 检查点**：每次看到 `.map` 返回 JSX，立即确认 key 是否来自数据 ID。
