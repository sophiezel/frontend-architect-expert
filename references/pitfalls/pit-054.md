# pit-054: Worker 通信开销超过计算收益 — iOS 无 JIT 时更严重

## 环境
- 微信小程序 Worker 线程
- iOS 平台（无 JIT 编译）的 Worker 表现远差于 Android
- 涉及大量数据传输 + 密集计算的场景（如图像处理、加密、大数据排序）

## 症状
使用 Worker 后计算性能不进反退：原本主线程 400ms 完成的计算，Worker 需要 700ms+。开发者困惑——明明文档说 Worker 用来做耗时计算，为什么更慢了？Android 上表现正常（确实快了），但 iOS 上严重退化，导致线上 iOS 用户反馈页面响应比之前更慢。

更隐蔽的情况：CPU profiling 显示 Worker 的计算时间确实更短，但端到端耗时（postMessage → onMessage）却更长——说明瓶颈不在计算而在通信。

## 根因
小程序 Worker 的"加速"取决于两个变量：计算收益 vs 通信开销。

**Worker 通信链路**：
```
主线程序列化 → Native 转发 → Worker 反序列化 → 计算 → Worker 序列化 → Native 转发 → 主线程反序列化
```

**iOS 无 JIT 的影响**：iOS 的 JavaScriptCore 在非主线程中默认关闭 JIT（Just-In-Time 编译），Worker 内的 JS 代码以解释模式执行，比主线程慢 3~5 倍。这导致"推给 Worker 省计算"的假设在 iOS 上不成立——计算本身反而变慢了。

**通信开销公式**：
```
总传输成本 ≈ 序列化(数据大小 × 0.003ms/KB) + JSBridge 转发(固定 5~15ms) + 反序列化(数据大小 × 0.003ms/KB)
                        
典型 100KB 数据：序列化 0.3ms + 转发 10ms + 反序列化 0.3ms ≈ 11ms
典型 1MB 数据：  序列化 3ms   + 转发 15ms + 反序列化 3ms   ≈ 21ms
```

**决策矩阵**：
```
计算量 < 100ms：几乎总是更慢（通信占主导）
计算量 100~500ms、数据 < 10KB：可能持平或略快
计算量 100~500ms、数据 > 100KB：通信占主导 → 更慢
计算量 > 500ms、数据 < 100KB：Worker 有明确收益（但 iOS 无 JIT 仍需谨慎）
计算量 > 500ms、数据 > 100KB：取决于数据/计算比例
```

```js
// 1. 测量端到端 Worker 耗时（含通信）
async function benchmarkWorker(data, iterations = 10) {
  const worker = wx.createWorker('workers/processor/index.js');
  
  // 主线程基准（直接计算）
  const t0 = Date.now();
  for (let i = 0; i < iterations; i++) {
    directComputation(data);
  }
  const directTime = Date.now() - t0;
  console.log(`[bench] Direct: ${directTime}ms (avg ${directTime/iterations}ms)`);
  
  // Worker 端到端（含通信）
  const t1 = Date.now();
  for (let i = 0; i < iterations; i++) {
    await new Promise((resolve) => {
      worker.onMessage(() => resolve());
      worker.postMessage({ data, id: i });
    });
  }
  const workerTime = Date.now() - t1;
  console.log(`[bench] Worker:  ${workerTime}ms (avg ${workerTime/iterations}ms)`);
  
  console.log(`[bench] Worker overhead: ${workerTime - directTime}ms (${((workerTime/directTime - 1)*100).toFixed(1)}% slower)`);
  
  worker.terminate();
}
```

```diff
// ✅ 修复: Worker 使用决策函数
function decideWorkerStrategy(computation, dataSizeBytes) {
  // 基准测试
  const computeMs = estimateComputeTime(computation);
  const communicateMs = estimateTransferCost(dataSizeBytes);
  const systemInfo = wx.getSystemInfoSync();
  const isIOS = systemInfo.platform === 'ios';
  
  // iOS 无 JIT → 计算成本 × 3
  const adjustedCompute = isIOS && !isExperimentalWorker ? computeMs * 3 : computeMs;
  
+ if (communicateMs > adjustedCompute) {
+   // 通信开销 > 计算收益 → 主线程直接算
+   return { useWorker: false, reason: 'communication dominates' };
+ }
+ 
+ if (adjustedCompute < 50) {
+   // 计算太短 → 不划算
+   return { useWorker: false, reason: 'computation too short' };
+ }
+ 
+ return { useWorker: true };
}

// 估算通信开销
function estimateTransferCost(bytes) {
  // 序列化 + JSBridge + 反序列化
  return (bytes / 1024) * 0.006 + 10; // ~0.006ms/KB + 10ms 固定开销
}
```

## 原理
Worker 不是免费的午餐。在浏览器中 Worker 天然有优势：浏览器主线程需要处理 DOM/渲染/事件/GC，Worker 可以跑在独立线程且 JS 引擎通常有 JIT。但小程序的架构不同：

1. 逻辑层本身就运行在独立的 JsCore 中，没有 DOM 渲染负担
2. iOS 的 JavaScriptCore 对非主线程默认关闭 JIT——这是安全策略而非 bug
3. 小程序 Worker 的通信需要经过 Native 中转，比浏览器的 `structuredClone + postMessage` 多了 JSBridge 这一层

`useExperimentalWorker` 的工作原理：让 Worker 线程也拥有 JIT 编译能力，但这是实验性 API，可能存在稳定性风险。生产环境使用前需充分验证。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Worker 通信开销超过计算收益的架构根因是**计算分层策略缺失**。小程序有三层计算能力：渲染层(WXS)、逻辑层(主线程 JsCore)、Worker 线程(独立 JsCore)。每层各有适用场景，但缺少系统化的计算路由决策机制：

1. **没有建立计算路由**：什么计算在渲染层做（WXS）、什么在逻辑层做（主线程）、什么推给 Worker、什么推给服务端——没有决策树
2. **Worker 的"magic thinking"**：开发者将 Worker 视为"放进去就快"的黑盒，忽略了通信成本和平台差异
3. **缺少平台感知**：代码没有在 Android/iOS 之间做策略差异化

### 系统级预防策略

**架构防护**：
```js
// 计算路由决策引擎
class ComputationRouter {
  static route(task, data) {
    const { computeMs, dataKB } = this.estimate(task, data);
    const platform = this.getPlatform();
    
    // 决策树
    if (computeMs < 16) return 'wxs';           // 渲染层直接算
    if (computeMs < 50) return 'main';           // 逻辑层主线程
    if (dataKB > 100) return 'server';           // 数据量大→服务端
    if (platform === 'ios' && !this.hasJIT()) {
      if (computeMs < 300) return 'main';       // iOS 无 JIT，放主线程
    }
    if (computeMs > 200) return 'worker';        // 用 Worker
    return 'main';                               // 默认主线程
  }
}
```

### 跨层影响链

```
上游影响 (用户体验):
  - iOS 用户感知性能退化 → App Store 评分下降
  - Worker 创建/销毁有内存开销 → 频繁创建导致 OOM 风险

下游影响 (后端):
  - 错用 Worker 导致计算变慢 → 推向服务端计算 → 服务器 CPU 成本上升
  - 但正确推向服务端反而合理——手机不适合做重计算

监控影响:
  - 需建立 Worker 性能仪表盘: 端到端耗时、计算耗时、通信耗时、数据量
  - 告警: Worker 端到端耗时 > 主线程直接计算耗时 × 2
```

### 相关架构模式
- **计算分层模式** (`patterns.md`) — 渲染层(WXS) / 逻辑层 / Worker / 服务端四层计算路由
- **平台感知策略模式** — Android/iOS 差异化计算策略
- **反模式: Worker 万能论** — 认为 Worker 总是能加速的误区

## 预防
1. **使用 Worker 前先做基准测试**：对比主线程和 Worker 端到端耗时
2. **开启 useExperimentalWorker**：iOS 上启用 Worker JIT，但需充分验证稳定性
3. **减少传输数据量**：只传计算必需的字段，精简结果
4. **收效甚微时上移**：计算 > 2 秒的推给服务端（云函数）
5. **平台差异化**：Android 积极用 Worker，iOS 谨慎评估
