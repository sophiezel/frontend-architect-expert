# pit-006: iOS Safari 底部工具栏导致 100vh 出现滚动条

## 环境
- iOS Safari（所有版本）
- 使用 `height: 100vh` 或 `100dvh` 不兼容的旧设备

## 症状
全屏布局在 iOS Safari 中出现纵向滚动条，底部内容被 Safari 底部导航栏遮挡。实际可视区域小于 100vh。

## 根因
iOS Safari 的 `100vh` = 包含底部工具栏的视口高度。当工具栏展开时，`100vh` 比实际可视区域高，导致内容溢出。Safari 工具栏在滚动时还会收缩/展开，`100vh` 是动态变化的。CSS 新单位 `dvh`（dynamic viewport height）解决了这个问题，但旧设备不支持。

## 诊断方法

```bash
# 1. iOS 真机或 Xcode Simulator 中打开页面
# 观察底部是否有意外滚动
# 注意：Chrome DevTools 设备模拟不重现（不模拟工具栏）

# 2. Safari Web Inspector 远程调试
# Mac Safari → Develop → [设备名] → 选中页面
# Elements → 选中根元素 → Computed 面板查看 height
# 对比 window.innerHeight 与 100vh 计算值

# 3. 在控制台执行
# Safari → Develop → Web Inspector → Console:
console.log('innerHeight:', window.innerHeight);
console.log('100vh:', document.documentElement.clientHeight);
// 如果两者不一致且 100vh > innerHeight → 确认问题

# 4. CSS 诊断
# 给根元素添加红色边框
* { outline: 1px solid red; }
# 观察根元素是否超出实际视口
```

## 修复

```css
/* ❌ 问题代码 */
.hero {
  height: 100vh; /* iOS Safari 会溢出 */
}

/* ✅ 修复方案一：使用 dvh + fallback */
.hero {
  height: 100vh;        /* fallback */
  height: 100dvh;       /* 现代浏览器动态视口高度 */
}

/* ✅ 修复方案二：使用 height + min-height 组合 */
.page-wrapper {
  min-height: 100vh;
  min-height: 100dvh;
}

/* ✅ 修复方案三：使用 -webkit-fill-available (iOS Safari 专用) */
.hero {
  min-height: 100vh;
  min-height: -webkit-fill-available;
}
```

```bash
# 在控制台执行：
CSS.supports('height', '100dvh')
# true → dvh 生效
# false → 需要 fallback
```

## 原理
CSS 视口单位 `vh` 基于"布局视口"（Layout Viewport），iOS Safari 的布局视口包含底部工具栏区域。当用户滚动页面时 Safari 会收缩工具栏，此时 100vh 对应的 CSS 像素值会变化，但已渲染的元素不会自动调整尺寸。CSS 工作组在 2022 年引入了三种新视口单位：
- `svh`（Small Viewport）：工具栏展开时的视口高度
- `lvh`（Large Viewport）：工具栏收缩时的视口高度
- `dvh`（Dynamic Viewport）：动态跟随工具栏状态

## 架构视角 (Architecture Perspective)

### 架构层面根因
100vh 问题的架构根因是**跨平台布局策略缺少视口抽象层**：
- **视口语义直接暴露给业务代码**：`100vh` 是 CSS 规范层面的值，其语义在不同平台（iOS Safari / Android Chrome / Desktop）上不同。但业务代码直接使用它，没有通过一个"应用视口抽象"来屏蔽平台差异。正确的架构应该定义 `--app-viewport-height` 变量，由平台适配层注入正确的值。
- **缺乏移动端布局策略声明**：全屏布局（全屏 Hero、全屏 Splash、全屏 Modal）是一种布局策略，应该在设计系统层面定义为可复用的 CSS 工具类或组件，而不是让每个开发者各自使用 `100vh`。架构缺少对"布局策略"的抽象和复用机制。
- **降级策略不系统化**：`dvh`、`-webkit-fill-available`、`visualViewport API` 是三层降级方案，但应用中没有统一的 polyfill 注入机制。架构应该有一个"视口适配中间件"在应用启动时统一注入正确的 CSS 变量。

### 系统级预防策略

**静态防护**：
```css
/* Design Token 层：定义视口高度变量 */
:root {
  --app-viewport-height: 100vh; /* fallback */
  --app-viewport-height: 100dvh; /* 现代浏览器 */
}

@supports not (height: 100dvh) {
  :root {
    --app-viewport-height: -webkit-fill-available;
  }
}

/* 业务层：统一使用变量 */
.hero {
  height: var(--app-viewport-height);
}
```

```js
// JS 层：运行时 polyfill
// src/boot/viewport-polyfill.ts — 应用启动时强制执行
function applyViewportPolyfill() {
  const setVH = () => {
    const vh = window.innerHeight * 0.01;
    document.documentElement.style.setProperty('--vh', `${vh}px`);
  };
  setVH();
  window.addEventListener('resize', setVH);
  window.visualViewport?.addEventListener('resize', setVH);
}
```

**CI 防护**：
```bash
# CI 中添加 CSS 检测：禁止裸用 100vh
# stylelint 自定义规则
# .stylelintrc.js
rules: {
  'declaration-property-value-disallowed-list': {
    height: ['/100vh/'],
    'min-height': ['/100vh/'],
  },
  // 允许在 CSS 变量定义中使用
  // 通过 ignoreValues 排除 var() 用法
}

# 或者使用正则检测:
grep -rn 'height:\s*100vh' src/ --include="*.css" --include="*.scss" && 
  echo "WARNING: 禁止直接使用 100vh，请使用 var(--app-viewport-height)" && exit 1
```

**架构防护**：
- **布局策略组件化**：封装 `<FullScreen>` / `<ViewportHeight>` 组件，内部处理所有平台兼容逻辑：
```tsx
// 架构层组件，开发者不再直接接触 100vh
function FullScreen({ children }: { children: React.ReactNode }) {
  return (
    <div className="app-fullscreen" style={{ height: 'var(--app-viewport-height)' }}>
      {children}
    </div>
  );
}
```
- **设计系统 Token 覆盖**：在 Design Token 中定义 `spacing.viewportHeight`，由构建工具按目标平台注入正确的值。

**流程防护**：
- Code Review 检查项："是否直接使用了 `100vh`？如果使用全屏布局，请用 `<FullScreen>` 组件。"
- 测试清单：每个全屏页面必须在 iOS Safari 真机上验证无意外滚动条。

### 跨层影响链

```
上游影响 (消费者):
  - 设计系统中的全屏组件（Modal/Hero/Splash）依赖此高度值
  - 使用 vh 单位的子元素尺寸计算基于错误的基准

下游影响 (依赖):
  - 基于 vh 的 JS 计算（如滚动位置计算）产生偏差
  - IntersectionObserver 的 threshold 计算基于 100vh，可能触发错误的可见性判断
  - 第三方组件（如 FullPage.js）内部使用 vh，需额外配置

部署影响:
  - CDN 缓存的 CSS 不包含 dvh fallback → 旧设备需要重新部署
  - A/B 测试中不同设备的表现差异大 → 测试结果不可靠

监控影响:
  - 需要监控: 用户页面意外滚动事件（scrollTop > 0 且无用户滚动操作）
  - RUM (Real User Monitoring): 按设备分组统计 CLS
  - E2E 测试: iOS Safari 专项覆盖全屏布局页面
```

### 相关架构模式
- **CSS 方案选型 — CSS Variables** (`decisions.md` 第三节) — 视口高度应作为 Design Token 定义在 `:root` 中，通过 CSS Variables 统一分发，而非各处裸写 `100vh`。
- **图片策略** (`decisions.md` 第七节) — 全屏 Hero 图片同样受 100vh 问题影响，图片尺寸策略需要与视口策略配合。
- **构建架构 — Vite/Rolldown** (`decisions.md` 第四节) — 可通过 Vite 插件在构建时按目标平台注入正确的视口变量值（如 iOS 构建注入 `-webkit-fill-available` polyfill）。

## 预防

1. **使用 `100dvh` 并加 fallback**：这是 CSS 工作组 2022 年正式引入的解决方案。
2. **使用 `-webkit-fill-available`** 作为 iOS Safari polyfill。
3. **布局用 flex/grid + min-height** 代替固定 height。
4. **测试环境**：真实 iOS 设备测试，不要完全依赖 Chrome DevTools 模拟。
