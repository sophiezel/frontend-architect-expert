# pit-055: 小程序 WebView 兼容问题 — CSS/JS 在 Chrome/WebView 正常但小程序 WebView 异常

## 环境
- 微信小程序渲染层（WebView，基于 Chromium 定制版）
- 小程序内嵌 web-view 组件中的 H5 页面
- iOS 的 WKWebView 和 Android 的 Chromium WebView 行为差异

## 症状
样式在开发者工具（Chrome 内核）中显示完美，但真机预览时布局错乱、元素重叠、文字截断、滚动异常。或者 JS 代码在浏览器/Chrome DevTools 中报错但在小程序 WebView 中静默失败。更隐蔽的情况：CSS 动画在开发者工具流畅 60fps，真机掉帧严重甚至不动。

典型表现：`position: sticky` 在微信内置浏览器的 H5 页和开发者工具中都能正常工作，但在小程序 `web-view` 组件的 WKWebView 中完全失效。`overflow: scroll` 在 WXSS 中不生效——小程序根本不支持这个 CSS 属性，必须使用 `scroll-view` 组件。

## 根因
微信小程序的渲染层 WebView 是**定制版的 Chromium 内核**，且不同平台使用不同的 WebView 引擎：
- **iOS**：WKWebView（Apple 的 WebKit，与 Safari 共享但版本和特性有差异）
- **Android**：Chromium WebView（版本由微信客户端内置，可能与系统 Chrome 版本不同）
- **开发者工具**：基于 NW.js（Chromium 较新版本），**与真机差异巨大**

这意味着三端之间 CSS/JS 的表现可能完全不同。此外，WXSS 有特殊的属性白名单——不是所有 CSS 属性都被支持。`overflow: scroll`、`position: fixed`（在 `page` 组件下行为不同）、部分 `@keyframes` 写法都会在小程序中出现与 Web 不同的行为。

## 原理
微信小程序的 WebView 不是标准浏览器，而是一个被微信客户端裁剪和定制的 Web 容器。主要差异来自：

1. **CSS 属性白名单**：WXSS 引擎在编译时只允许白名单中的 CSS 属性。`overflow: scroll` 等不在白名单中 → 直接在编译后被丢弃
2. **WKWebView 的 scroll 行为**：iOS 的 WKWebView 对 `scroll` 有特殊优化（异步滚动/惯性滚动），这导致 `position: fixed` 元素在滚动时可能不跟随预期
3. **微信客户端的 CSS 注入**：微信客户端会注入额外的样式（如底部导航栏的 safe-area padding），可能与业务样式冲突
4. **web-view 的沙箱限制**：`web-view` 组件内的 H5 运行在独立的 WebView 中，受到微信内容安全策略（CSP）的限制，部分 JS API 被屏蔽

## 架构视角 (Architecture Perspective)

### 架构层面根因
小程序 WebView 兼容问题的架构根因是**开发者工具的"谎言"效应**。开发者工具基于相对新的 Chromium 内核，而真机 WebView 是经过裁剪的定制版本。三端差异（开发者工具 / iOS WKWebView / Android Chromium WebView）在架构上缺乏统一的兼容性层来抽象。Web 开发中的"一次编写到处运行"假设在小程序中失效。

更深的架构问题是**平台能力检测缺失**。Web 项目中使用 `@supports` 或 `Modernizr` 做能力检测是标准操作，但小程序的 WXSS 不支持 `@supports`，JS 侧的 `wx.canIUse` 只能检测小程序 API，无法检测渲染层的 CSS/JS API 支持情况。

### 系统级预防策略

**CI 防护**：
```bash
# CI 中的 WXSS 兼容性检查
# 使用自定义 stylelint 插件检测不安全的 CSS
rules: {
  'wechat/no-unsupported-css': ['error', {
    disallowed: [
      'overflow: scroll',
      'overflow: auto',

      'gap',                   // 部分版本不支持
    ]
  }],
  'wechat/require-safe-area': ['warn', {
    properties: ['padding-bottom', 'margin-bottom']
  }]
}
```

**架构防护**：
```js
// 建立平台能力检测中间层
const PlatformCapabilities = {
  supportsSticky: null,
  supportsIntersectionObserver: null,
  
  static async detect() {
    // 在小程序 web-view H5 中:
    if (typeof window !== 'undefined') {
      this.supportsSticky = CSS.supports('position', 'sticky');
      this.supportsIntersectionObserver = 'IntersectionObserver' in window;
    }
    // 在小程序原生中: 通过基础库版本推断
    else {
      const sdk = wx.getSystemInfoSync().SDKVersion;
      this.supportsSticky = compareVersion(sdk, '2.7.0') >= 0;
    }
  }
};
```

### 跨层影响链

```
上游影响 (UI/UX 设计):
  - 设计稿效果在真机上无法实现 → 重新设计 → 项目延期
  - 跨页面的样式不一致 → 用户感知"山寨感"

下游影响 (测试/QA):
  - 测试覆盖矩阵爆炸: 开发者工具 + 2 平台 × N 个系统版本 = QA 噩梦
  - 兼容性 Bug 发现晚 → 修复成本高

监控影响:
  - 需要在 web-view H5 中注入错误收集
  - 监控 JS Error 中与平台相关的错误类型
```

### 相关架构模式
- **渐进增强模式** (`patterns.md`) — 基线功能所有平台可用，增强功能做能力检测
- **平台抽象层模式** — 封装 `scroll-view` / `position: fixed` 等为统一组件
- **反模式: 开发者工具先行** — 以开发者工具为准，真机验证延迟

## 预防
1. **铁律**：任何样式改动必须在真机上验证（至少 iOS + Android 各一台）
2. **铁律**：使用 `scroll-view` 替代 CSS overflow，不支持的原生 CSS 属性不加
3. **CSS 降级双写**：`constant()` + `env()` 双写 safe-area
4. **web-view H5 检测环境**：在 H5 中检测是否在小程序环境，针对性降级
5. **真机自动化测试**：使用 miniprogram-automator 在真机上运行 E2E 测试
