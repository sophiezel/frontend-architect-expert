# pit-034: Pinia Store 架构陷阱

## 环境
- Vue 3.2+ / Pinia 2.x
- 中大型单页应用，store 数量 > 5
- TypeScript 项目

## 症状
应用初始化时出现 `Cannot access before initialization` 错误——两个 store 相互引用导致循环依赖。多处组件修改同一个 store 后出现状态不一致：一个 action 修改了数据，另一个 action 基于旧数据又改了回来，形成"乒乓效应"。使用 Pinia plugin 后 store 行为异常——持久化后恢复的数据被 plugin 覆盖，或 plugin 在 SSR 中泄漏跨请求状态。

典型场景：
- Store A 的 action 调用 Store B 的 action，Store B 的 getter 又依赖 Store A 的状态
- 用户信息和权限分两个 store，但权限 store 的 getter 需要用户 store 的当前角色
- 使用 pinia-plugin-persistedstate 后，页面刷新数据恢复到错误的初始值

## 根因
1. **Store 间循环依赖**：Pinia 的 `useXxxStore()` 在 setup 阶段调用时，store 实例在第一次调用时创建。若 Store A 在 setup 中导入 Store B，Store B 在 setup 中导入 Store A → 循环。Pinia 没有像模块系统那样的依赖图分析和错误提示。

2. **Action 中状态不一致**：多个 action 并行执行时（未 await），读取的 state 可能是中间态。`$patch` 和直接赋值混用导致部分属性被覆盖。

3. **Plugin 作用域陷阱**：Pinia plugin 在所有 store 实例间共享，plugin 内部的状态变量（如计时器、缓存）在 SSR 中是全局单例 → 不同请求的数据互相污染。

## 诊断方法

```bash
# 1. 检测 store 间交叉引用
grep -rn "use\w\+Store()" src/stores/ --include="*.ts"

# 2. 检查是否存在循环导入
npx dpdm --no-warning --no-tree src/stores/*.ts
# dpdm (Dependency Patrol for Modern JS) 输出循环依赖图

# 3. 检查 action 中的异步操作
grep -rn "async\|await" src/stores/ --include="*.ts" -l

# 4. SSR 泄漏检查：在 Nuxt 插件中
export default defineNuxtPlugin(() => {
  const store = useMyStore();
  console.log('[SSR CHECK] store 状态在请求间是否残留:', store.$state);
});
```

## 验证

```bash
# 1. 循环依赖检测
npx dpdm --no-warning src/stores/*.ts
# 期望输出：No circular dependency found

# 2. 状态一致性测试
import { setActivePinia, createPinia } from 'pinia';
test('并发 action 不应产生中间态', async () => {
  const pinia = createPinia();
  setActivePinia(pinia);
  const store = useMyStore();
  
  await Promise.all([
    store.action1(),
    store.action2(),
  ]);
  // 断言 state 满足业务不变量
  expect(store.orders.length + store.pendingCount).toBe(store.totalCount);
});

# 3. SSR 泄漏测试
test('SSR 请求之间状态隔离', async () => {
  const pinia1 = createPinia();
  setActivePinia(pinia1);
  const store1 = useMyStore();
  store1.setValue(42);
  
  const pinia2 = createPinia(); // 第二次请求
  setActivePinia(pinia2);
  const store2 = useMyStore();
  expect(store2.value).not.toBe(42); // 不应泄漏
});
```

## 原理
Pinia 的 store 实例化是惰性的：`useXxxStore()` 第一次调用时创建单例，后续调用返回同一实例。在 Options API 的 `setup()` 中调用是安全的（每个组件实例的 setup 独立），但在另一个 store 的顶层调用时——因为 store 本身就是单例——形成了模块级别的互相引用。

`$patch` 是 Pinia 提供的批量更新方法，它接收一个对象并递归合并到 state，但对于嵌套对象使用 `$patch(state => { state.nested.prop = val })` 形式更安全。

Pinia plugin 的 `context.store` 是当前 store 实例，但 plugin 函数本身（闭包外部变量）是在 `createPinia()` 时注册的，如果缓存存储在 plugin 闭包中，在 SSR 的请求共享同一 Pinia 实例时就会泄漏。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Pinia store 陷阱的架构根因是**缺乏 Store 领域建模**：
- **Store 被视为"装数据的袋子"而非"领域聚合根"**：DDD（领域驱动设计）中每个聚合根有明确的边界和不变量。但 Pinia store 常被当作 "user 数据放 userStore，order 数据放 orderStore"，不考虑 store 间的耦合度。高内聚低耦合原则在 store 设计中缺位。
- **SSR 安全性未作为架构质量属性**：大多数 Pinia 项目从 SPA 演进而来，开发者在 SPA 中心智下写 store（全局单例=安全），迁移到 SSR 时未重新审查所有 plugin 和顶层引用的线程安全性。
- **状态生命周期未建模**：一个 action 执行期间 state 处于"中间态"，但下游 watcher 和组件可能在中间态就触发副作用。缺少 "action 原子性" 的保证——要么全部成功，要么全部回滚。

### 系统级预防策略

**静态检查**：
```bash
# 禁止在顶层调用 useXxxStore()（store 文件顶层）
# 自定义 ESLint 规则
rules: {
  'no-restricted-syntax': [{
    selector: 'Program > VariableDeclaration > VariableDeclarator > CallExpression[callee.name=/^use\w+Store$/]',
    message: '禁止在 store 顶层调用其他 store，请在 action 内部调用'
  }]
}

# 检测 store 间交叉引用
npx dpdm src/stores/*.ts --circular
```

**CI 门禁**：
```bash
# CI 中运行 dpdm 循环依赖检测
npx dpdm src/stores/*.ts --no-warning --exit-code circular:1

# Store 单元测试必须覆盖：
# 1. 并发 action 的状态一致性
# 2. action 失败后的回滚
# 3. SSR 场景（多次 createPinia）

# Vitest 配置 store 测试
import { beforeEach } from 'vitest';
beforeEach(() => {
  setActivePinia(createPinia()); // 每个测试独立的 pinia 实例
});
```

**架构约束**：
- Store 间依赖必须是单向无环的：定义 store 的依赖方向（如 `user → profile → settings`，禁止反向引用）
- 需要双向引用的场景 → 提取共享逻辑到第三个 store 或 composable
- Action 必须满足"可回滚性"：在修改 state 之前保存快照
- Plugin 必须通过工厂函数创建（禁止闭包全局变量），确保每次 `createPinia()` 获得独立 plugin 实例

**流程约束**：
- 每个新增的 store 必须在 PR 描述中画依赖关系图
- Code Review 检查："store 顶层是否有 useXxxStore() 调用？"
- SSR 项目发布前专门跑一轮 "SSR 状态泄漏测试"

### 跨层影响链

```
上游影响 (组件层):
  - Store 循环依赖 → 组件使用任一 store 时报 ReferenceError
  - SSR 状态泄漏 → 用户 A 看到用户 B 的购物车数据（P0 安全事故）

下游影响 (持久化层):
  - 状态不一致被持久化到 localStorage → 刷新后错误数据持续存在
  - API 请求携带中间态数据 → 后端产生脏数据

部署影响:
  - Store 结构重构需要同时发布前后端（API 契约变更）
  - SSR 中 store 泄漏问题在 staging 环境难复现（请求量小）

监控影响:
  - Sentry: 报告 store action 中的异常并携带 action 名称和参数
  - 自定义事件: "store action 执行时长" "action 回滚次数"
  - RUM: 页面初始化时 store 初始化的耗时（统计 plugin 开销）
```

### 相关架构模式
- **DDD 聚合根在 Store 中的映射** (`patterns.md` 第十节) — 将 store 按业务聚合根划分，每个 store 维护自己的不变量（如 orderStore 保证 `items.length > 0 === totalAmount > 0`）。
- **CQRS 模式** (`patterns.md` 第三节) — 分离"查询状态"的 getter 和"修改状态"的 action，避免 getter 内部有副作用。
- **pit-042: Monorepo 管理** — Store 拆分粒度与 Monorepo 包的拆分粒度需要协调设计。

## 预防
1. Store 间交叉引用只能在 action/getter 函数体内（延迟引用），不能在顶层
2. 每个 store 文件必须能用 `npx dpdm` 验证无循环依赖
3. Plugin 一律通过工厂函数创建，禁止在 plugin 文件中使用模块级 `const` 存储可变状态
4. 涉及外部 API 调用的 action 必须实现乐观更新 + 失败回滚模式
