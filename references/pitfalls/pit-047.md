# pit-047: Sentry 上报风暴 — 错误量激增导致配额耗尽/成本爆炸

## 环境
- 使用 Sentry / Datadog Error Tracking / Bugsnag 等错误监控服务
- 前端应用有第三方 SDK 注入（广告、分析、支付、客服等）
- 有月度事件配额限制（免费 tier 5K/月，Team 50K/月，Business 按量付费）

## 症状
部署新版本后 5 分钟内，Sentry 发送告警邮件"您已经消耗了本月 80% 的事件配额"。按量付费的团队发现账单数字暴涨。Sentry Issues 列表中出现大量重复错误——同一个错误被报告了 20,000 次。团队 panic 状态下临时把 Sentry DSN 从生产环境移除，导致真正的错误监控也失明。

更隐蔽的症状：
- "ResizeObserver loop limit exceeded" 被上报了几十万次（浏览器内部错误，不可 actionable）
- 第三方支付 SDK 注入的 iframe 抛出的 `SecurityError` 无法修复但被持续上报
- Chrome 扩展注入的 `TypeError: Cannot read properties of undefined` —— 不是你的代码但占据了 60% 的配额
- 某个 API 返回 500 后，用户反复重试页面导致同一错误 × 访问量级的上报量

## 根因
核心根因是**缺少错误采集的噪音过滤体系**。大多数团队只在 `Sentry.init()` 中传入 `dsn` 就认为"接入完成"了，实际等同于把消防水管对准一个茶杯。

具体根因分解：
1. **未配置 `ignoreErrors`**：未过滤浏览器扩展注入错误、`ResizeObserver loop limit`、`Network request failed` 等已知不可修复的噪音
2. **未配置 `beforeSend` 过滤/采样**：没有基于 fingerprint 的 rate limiting，同一个错误在短时间内被重复上报
3. **未使用 `sampleRate` 按环境分层**：dev/staging/production 使用相同的 100% 采样，dev 环境的热重载把每个错误重复上报 N 次
4. **未设置 `denyUrls`**：第三方脚本的错误被当作自己的错误上报
5. **未使用 fingerprint 去重**：同一个逻辑错误因参数不同被 Sentry 认为是不同 issue，碎片化成上千个 issue

```typescript
// ❌ 典型的"接入即完成"——等于没有噪声过滤
Sentry.init({
  dsn: 'https://xxx@sentry.io/xxx',
  // 什么都没配置...
});
```

## 修复

```diff
// ❌ 原始：裸初始化，零噪音控制
Sentry.init({
  dsn: 'https://xxx@sentry.io/xxx',
  environment: 'production',
});

// ✅ 完整修复：多层噪音过滤管线
Sentry.init({
  dsn: 'https://xxx@sentry.io/xxx',
  environment: 'production',
  
+  // 第一层: 按环境分层采样
+  // dev: 全量捕获用于调试，staging: 全量，production: 10%
+  sampleRate: 1.0,           // 错误采样率（按环境变量动态设置）
+  tracesSampleRate: 0.1,     // Performance 追踪 10%
+  replaysSessionSampleRate: 0.1,  // Session Replay 10%
+  replaysOnErrorSampleRate: 1.0,  // 错误发生时的 Replay 100%
+
+  // 第二层: 已知噪音模式黑名单
+  ignoreErrors: [
+    // 浏览器内部错误 - 不可 actionable
+    'ResizeObserver loop limit exceeded',
+    'ResizeObserver loop completed with undelivered notifications',
+    // 网络中断 - 非代码错误
+    'Network request failed',
+    'NetworkError when attempting to fetch resource',
+    'Failed to fetch',
+    'Load failed',
+    // 浏览器扩展注入
+    'Script error',
+    'Non-Error promise rejection captured with value: undefined',
+    // Chrome 扩展相关
+    /^chrome-extension:/,
+    /^moz-extension:/,
+    // 用户取消操作
+    'AbortError',
+    'Request aborted',
+    // 第三方支付 SDK 常见噪音
+    'Payment request error',
+    /^SecurityError: Blocked a frame/,
+  ],
+
+  // 第三层: 拒绝第三方域名的错误
+  denyUrls: [
+    // 浏览器扩展
+    /^chrome-extension:\/\//i,
+    /^moz-extension:\/\//i,
+    // 第三方广告/分析 SDK
+    /doubleclick\.net/i,
+    /googletagmanager\.com\/.*\/analytics/i,
+    /facebook\.com\/.*\/fbevents/i,
+  ],
+
+  // 第四层: beforeSend 指纹限流
+  beforeSend(event, hint) {
+    // 4a: 基于错误指纹的 rate limiting
+    //      同一 fingerprint 在 60 秒窗口内最多上报 5 次
+    const fingerprint = event.fingerprint?.join('-') || 
+                        event.exception?.values?.[0]?.type + ':' + 
+                        event.exception?.values?.[0]?.value;
+    
+    if (fingerprint) {
+      const key = `sentry_rl:${fingerprint}`;
+      // 使用 sessionStorage 做简单限流（生产级应用用更可靠的方案）
+      const record = JSON.parse(sessionStorage.getItem(key) || '{"count":0,"firstSeen":0}');
+      const now = Date.now();
+      
+      if (now - record.firstSeen < 60000) { // 60 秒窗口内
+        if (record.count >= 5) {
+          // 超过限流阈值，静默丢弃
+          if (record.count === 5) {
+            // 仅记录一次"被限流"事件
+            console.warn(`[Sentry RL] Throttling error: ${fingerprint}`);
+          }
+          record.count++;
+          sessionStorage.setItem(key, JSON.stringify(record));
+          return null; // 丢弃事件
+        }
+        record.count++;
+      } else {
+        record.count = 1;
+        record.firstSeen = now;
+      }
+      sessionStorage.setItem(key, JSON.stringify(record));
+    }
+    
+    // 4b: 移除敏感信息
+    if (event.request?.headers) {
+      delete event.request.headers['Authorization'];
+      delete event.request.headers['Cookie'];
+    }
+    
+    return event;
+  },
+
+  // 第五层: 自定义 fingerprint 防止碎片化
+  beforeBreadcrumb(breadcrumb) {
+    // 去掉 breadcrumb 中的动态参数防止生成不同 fingerprint
+    if (breadcrumb.category === 'fetch' || breadcrumb.category === 'xhr') {
+      breadcrumb.data = { url: breadcrumb.data?.url };
+    }
+    return breadcrumb;
+  },
});
```

## 原理
Sentry 的错误处理模型是"越多越好"——默认情况下任何未捕获异常都会被上报。但生产环境的浏览器是一个混沌环境：Chrome 扩展注入代码、第三方 SDK 在 iframe 中运行、ResizeObserver 内部循环超限、网络波动触发超时错误。这些都不是你的应用代码产生的错误。

错误监控的本质是**信号提取**而非全量采集。全量采集不仅浪费配额和金钱，更危险的是：真正需要关注的 P0 错误被淹没在噪音中，团队因为告警疲劳而忽略所有告警。

Sentry 的过滤体系按优先级：`ignoreErrors` > `denyUrls` > `beforeSend` > `sampleRate`。`beforeSend` 可以在事件发送前做任意逻辑过滤（包括 rate limiting），是最灵活的。`sampleRate` 是最终兜底——如果前几层都没拦住，采样率确保你不会用光配额。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Sentinel 风暴的架构根因是**信号提取管道 (Signal Extraction Pipeline) 的缺失**。错误监控被当作"一个 SDK 调用"而非"一个多层过滤体系"。类比后端的日志系统——没有人会把所有 debug 日志直接推到生产环境的告警平台，而是经过 collect → filter → aggregate → alert 四个阶段。前端错误监控同样需要这样的管道。

更深层的架构问题是**错误可操作性 (Actionability) 建模缺失**。并非所有错误都值得被同等对待。一个可操作的错误必须具备三个属性：
1. **来源可归属**：来自你的代码而非第三方
2. **有修复路径**：你知道改什么可以消除它
3. **用户受影响**：它在用户的关键路径上

缺少这三个属性的错误就是噪音。架构上需要建立**错误分级模型**：
- **Actionable (P0/P1)**：你的代码导致的、影响核心功能的错误
- **Observable (P2)**：你的代码导致的、影响非核心功能的错误
- **Noise**：第三方代码、浏览器内部错误、用户环境问题

这个分级应该在 `beforeSend` 中被自动化判断，而非事后人工清理。

### 系统级预防策略

**静态检查与代码生成**：
```bash
# 自动生成 ignoreErrors 配置（从生产噪音中学习）
# scripts/generate-sentry-config.ts
import { Sentry } from '@sentry/node';

async function generateIgnoreErrors() {
  // 拉取过去 30 天 Top 100 错误
  const issues = await Sentry.getIssues({
    query: 'is:unresolved',
    statsPeriod: '30d',
    sort: 'freq',
    limit: 100,
  });
  
  // 自动识别不可操作错误模式
  const noisePatterns = issues
    .filter(i => 
      i.count > 1000 && // 高频
      !i.culprit?.includes('src/') && // 非自有代码
      i.status === 'ignored' // 已被手动忽略的
    )
    .map(i => i.title.replace(/[.*+?^${}()|[\]\\]/g, '\\$&'));
  
  // 生成配置代码
  console.log(`// Auto-generated from production noise analysis
export const IGNORE_ERRORS = ${JSON.stringify(noisePatterns, null, 2)};`);
}
```

**架构约束**：
- Sentry 配置作为独立模块 `src/monitoring/sentry.config.ts`，不允许在组件中单独调用 `Sentry.captureException` 绕过配置
- 新增第三方 SDK 必须在 architecture review 中评估其对 Sentry 噪音的贡献，并更新 `denyUrls` 和 `ignoreErrors`
- Staging 环境模拟噪音注入：定时触发 ResizeObserver / Script error / Network error，验证过滤规则有效性
- Error Budget：定义每月 Sentry 事件的"预算"——X% 用于 Actionable 错误，Y% 用于 Observable，Z% 用于 Noise（目标 Z% → 0）

### 跨层影响链

```
上游影响 (开发体验):
  - 开发者被调试噪音淹没 → 真 bug 被忽略 → 线上事故
  - Sentry 配额耗尽 → 临时关监控 → 部署无监控窗口期 → 高风险

下游影响 (运维/财务):
  - 按量付费 → 月度账单暴涨 → 财务审批收紧 → 后续监控预算被砍
  - 大量噪音事件 → Sentry 后端存储/索引压力 → 查询变慢 → 排查效率降低
  - 配额耗尽后拒绝新事件 → 真正的 P0 错误无法被记录 → 用户投诉才知问题

部署影响:
  - 新版本上线 → 突发大量新错误 → 被迫回滚 → 但其实很多是已知噪音
  - 噪音导致 CI 中 Sentry release tracking 数据异常 → 无法判断新版本质量

监控影响:
  - 告警疲劳是整个可观测性体系的癌症: 一次误报 → 团队忽视该告警 → 更多误报 → 所有告警被静音
  - 与 pit-044 互补: 噪音过多 = 信号被淹没 = 另一种形式的监控盲区
  - 与 pit-046 构成监控三角: 盲区(044) + 失真(046) + 噪音(047) = 完全失去可观测性
```

### 相关架构模式
- **Signal Extraction Pipeline** (`patterns.md`) — 采集层过滤 → SDK 层限流 → 服务层去重 → 告警层阈值，四阶段信号提取管道。
- **Error Budget Pattern** (`decisions.md`) — 借鉴 SRE 的 Error Budget 理念，为 Sentry 事件设定"噪音预算"，超过预算必须进行噪音治理而非简单扩容配额。
- **Client-Side Circuit Breaker** (`decisions.md`) — 当检测到同一错误在短时间内爆发（> 100/min）时，自动暂停该 fingerprint 的上报，防止后端被"错误 DDoS"。
- 与 **pit-044 (监控盲区)** 互补——噪音治理过激可能导致过滤掉真正的错误，需平衡过滤粒度与覆盖面。

## 预防
1. **Sentry 配置模板**：每个新项目必须从模板创建 Sentry 配置，模板包含完整的 `ignoreErrors` + `denyUrls` + `beforeSend` + `sampleRate`
2. **噪音审计**：每月分析 Sentry Issues Top 100，将被标记为 "wontfix" 或 "ignored" 的 issue 的 pattern 加入 `ignoreErrors`
3. **配额告警**：设置 Sentry 配额使用率告警（70% / 85% / 95%），在配额耗尽前预警
4. **分级采样**：production 环境 `sampleRate: 0.1`（10% 错误采样），关键错误通过 `beforeSend` 中的逻辑保证 100% 上报
5. **Fingerprint 规范化**：使用 `beforeBreadcrumb` 去除 URL/参数中的动态部分，防止同一逻辑错误碎片化为多个 issue
