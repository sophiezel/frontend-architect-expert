# pit-048: 埋点数据污染 — 脏数据/类型错误/丢失重复未治理

## 环境
- 自建埋点系统或使用 GrowingIO / 神策 / Mixpanel / Amplitude 等第三方埋点平台
- 前端埋点分为手动埋点（`track('event_name', props)`）和自动埋点（点击/曝光/页面浏览）
- 数据侧有 ETL 管线处理埋点数据并写入数据仓库

## 症状
数据分析师的日常工作被中断："这个漏斗转化率 120% 是怎么回事？""用户路径里怎么有 300 步？""昨天这个按钮的点击量怎么突然从 5000 掉到了 30？"。数据看板出现物理上不可能存在的数字（转化率 > 100%、同一用户在同一秒内"进入页面"10 次）。产品经理开始不信任数据看板，转而靠"拍脑袋"做决策。

从工程侧看，症状表现为：
- 同一事件 `property_name` 字段有时是 `"123"`（string），有时是 `123`（number），ETL 无法做聚合
- 必填字段 `user_id` 时常缺失，导致无法关联用户行为漏斗
- 页面曝光事件在 scroll 事件中重复触发，单个用户单次访问产生数百条曝光事件
- 事件定义混乱：`click_button` / `btn_click` / `button_tap` 指向同一个按钮但被当作三个不同事件

## 根因
埋点数据污染的根因是**缺少埋点全生命周期治理**。大多数团队的埋点流程是：PM 说"加个埋点"→ 开发在代码里加一行 `track('xxx')` → 上线 → 数据分析师自己去"猜"这个事件是什么。

具体根因分解：
1. **无注册机制**：埋点事件名和 Schema 未在平台注册，开发和数据分析师对"这个事件叫什么、有什么属性、属性的类型是什么"没有统一认知。开发可能用 `track('purchase', { amount: 99 })`，数据分析师期望的是 `order_complete`。
2. **无类型校验**：`track('event', { count: '10' })` 和 `track('event', { count: 10 })` 混用，ETL 无法确定字段类型。
3. **无必填检查**：关键字段（如 `user_id`、`session_id`）缺失时未拦截，直接上报。
4. **无重复去重**：曝光埋点绑定在 scroll 事件上，每次滚动都触发 `track`；手动埋点被误加在 render 函数中，每次重渲染都上报。
5. **无数据质量监控**：没有对"脏数据率""丢失率""重复率"的持续监控，污染积累到数据不可用才被发现。

```typescript
// ❌ 典型的"加一行"式埋点——完全没有治理
```

```bash
  max(timestamp) as last_seen

WHERE date = today()
```

```diff
+function validateEvent(name: string, props: Record<string, any>): { 
+  valid: boolean; 
+  errors: string[];
+  sanitized: Record<string, any>;
+} {
+  const schema = TRACKING_REGISTRY[name];
+  const errors: string[] = [];
+  const sanitized: Record<string, any> = {};
+  
+  if (!schema) {
+    errors.push(`事件 "${name}" 未注册，将被丢弃`);
+    return { valid: false, errors, sanitized };
+  }
+  
+  for (const [key, propSchema] of Object.entries(schema.properties)) {
+    const value = props[key];
+    
+    // 必填检查
+    if (propSchema.required && (value === undefined || value === null || value === '')) {
+      errors.push(`必填字段 "${key}" 缺失`);
+      continue;
+    }
+    
+    // 类型检查与转换
+    if (value !== undefined && value !== null) {
+      const actualType = typeof value;
+      if (propSchema.type === 'number' && actualType !== 'number') {
+        const parsed = Number(value);
+        if (isNaN(parsed)) {
+          errors.push(`字段 "${key}" 期望 number，实际 ${actualType}: ${value}`);
+        } else {
+          sanitized[key] = parsed;
+          continue;
+        }
+      }
+      if (propSchema.type === 'enum' && !propSchema.enumValues?.includes(String(value))) {
+        errors.push(`字段 "${key}" 不在枚举值 [${propSchema.enumValues}] 中: ${value}`);
+      }
+    }
+    
+    sanitized[key] = value;
+  }
+  
+  return { valid: errors.length === 0, errors, sanitized };
+}
```

## 原理
埋点数据污染的根源是**数据生产者和数据消费者之间的契约缺失**。在后端，API 有 OpenAPI/Swagger 定义契约；在数据库，有 DDL 定义 Schema。但前端埋点通常没有任何契约——开发者在代码中直接写 `track('xxx')`，数据分析师在另一边猜这个事件的含义。

字节跳动埋点治理的三个核心原则：
1. **SDK 管控**：所有埋点必须通过 SDK 的统一入口上报。SDK 负责终端计算（自动填充公共字段）、合规丢弃（验证失败的拒绝上报）、离线缓存（网络异常时暂存）。
2. **注册制**：埋点事件名和 Schema 必须在埋点管理平台注册后才能生效。未注册的事件 SDK 不上报，ETL 不处理。
3. **全生命周期管理**：设计（PM 提交需求）→ 注册（元数据录入）→ 开发（SDK 调用）→ 验证（Staging 环境测试）→ 上报（生产环境）→ ETL（数据清洗）→ 使用（数据分析）→ 下线（废弃事件清理）。每个阶段都有质量检查点。

数据类型不一致的后果在 ETL 阶段尤为严重。例如 `amount: 100` 和 `amount: "100"` 混在同一个字段中，聚合函数（`sum`、`avg`）对 string 返回 0 或报错，导致聚合结果错误。

## 架构视角 (Architecture Perspective)

### 架构层面根因
埋点数据污染的架构根因是**数据契约 (Data Contract) 在前端生态中的缺失**。后端 API 有成熟的契约机制（gRPC Protobuf、OpenAPI、GraphQL Schema），但前端埋点——同样是跨团队的数据接口——却没有契约保障。这是前端工程化中"数据生产严肃性"不足的体现。

更深层的架构问题是**埋点被当作"业务代码的副作用"而非"独立的数据产品"**。开发者视角中，埋点是"顺手加一行 track"；但架构视角中，每个埋点事件都是一个数据产品——它被数据分析师、产品经理、运营团队消费，驱动着公司的数据决策。数据产品的质量直接影响决策质量。

前端架构中的埋点治理属于**数据架构与前端架构的交叉领域**，需要建立一个从"加一行代码"到"可观测数据产品"的完整链路。

**架构约束**：
- 埋点注册 = 上线前置条件：未在平台注册的事件，SDK 和 ETL 两层拒绝
- SDK 是唯一出口：所有埋点必须通过统一 `track()` 函数，封禁直接调用第三方 SDK
- 公共字段由 SDK 自动填充：`user_id`、`session_id`、`device_type`、`os`、`network_type`、`page_url`、`app_version`——业务代码不传也不应该传
- 埋点质量 SLA：脏数据率 < 0.1%、必填缺失率 = 0%、重复率 < 0.5%

### 跨层影响链

```
上游影响 (数据决策):
  - 脏数据 → 错误的数据分析 → 错误的业务决策 → 产品方向偏离
  - 漏斗转化率 > 100% → 数据看板信任崩塌 → PM 不用数据做决策 → 产品靠直觉 → 迭代效率降低
  - 事件缺失 → 用户漏斗断裂 → 不知道用户在哪个环节流失 → 优化方向错误

下游影响 (后端/数据工程):
  - 类型不一致 → ETL 需要额外的清洗逻辑 → 数据管道复杂度上升
  - 字段缺失 → 下游数仓出现大量 NULL → 聚合时被排除 → 分析结果偏差
  - 重复数据 → 存储成本上升（ClickHouse 不便宜）→ 查询变慢

部署影响:
  - 未注册埋点上线 → ETL 无法处理 → 数据丢失
  - Schema 变更未同步 → 上下游断裂 → 新埋点数据在管道中被丢弃数天

监控影响:
  - 埋点质量看板的三项指标（脏数据率/丢失率/重复率）应成为前端可观测性的一部分
  - 与 pit-046 和 pit-047 构成"数据质量三角": 
    pit-046 管性能数据质量，pit-047 管错误数据质量，pit-048 管埋点数据质量
```

### 相关架构模式
- **Data Contract Pattern** (`patterns.md`) — 数据生产者（前端）和数据消费者（分析平台）之间通过契约（Schema）通信，契约变更是 breaking change 需要版本管理。
- **Registry-Driven Development** (`decisions.md`) — 所有埋点事件先在注册表定义，代码中的 `track()` 调用由注册表驱动类型推导，PR Review 时自动检查注册状态。
- **SDK as Gatekeeper** (`decisions.md`) — SDK 不是简单的上报通道，而是埋点质量的守门人：验证、去重、填充、限流、离线缓存。
- 与 **pit-046 (CLS 数据失真)** 同属数据质量范畴——一个是语义错误导致值不对，一个是治理缺失导致数据不可用。

## 预防
1. **埋点注册平台**：所有埋点事件在平台注册（名称/Schema/描述/负责人），未注册不上报不处理
2. **SDK 统一入口**：禁止直接调用第三方埋点 SDK，所有埋点必须通过 `@/tracking` 模块
3. **TypeScript 类型推导**：`track()` 函数根据注册表自动推导第二个参数的类型，编译时检查必填和类型
4. **CI 集成检查**：埋点注册完整性检查 + Schema 一致性检查 + Staging 环境数据质量检查
5. **埋点质量看板**：三项指标实时监控（脏数据率、丢失率、重复率），异常自动告警

```bash
function getUserId(): string {
  try {
    return authStore.userId || localStorage.getItem('uid') || cookies.get('uid') || '';
  } catch {
    monitor.warn('tracking: user_id 获取失败');
    return 'unknown'; // fallback
  }
}
```
