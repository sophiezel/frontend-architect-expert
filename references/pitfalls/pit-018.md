# pit-018: Date 解析时区陷阱

## 环境
- 所有 JavaScript 环境
- 国际化应用、跨时区系统

## 症状
`new Date('2024-01-01')` 在不同环境（浏览器 vs Node.js、UTC vs 本地时区）得到不同的时间戳。用户在中国看到 2024-01-01，服务器在 UTC 时区看到 2023-12-31。

## 根因
ECMAScript 规范对日期字符串解析存在歧义：
- `'2024-01-01'`（仅日期）：ES5.1 及之前默认 UTC，ES6（ES2015）规范改为 local timezone，但实现不统一。大多数现代浏览器将其作为 UTC 解析，Node.js 也是 UTC。
- `'2024-01-01T00:00:00'`（不含时区）：视为 local time。
- `'2024-01-01T00:00:00Z'`（含 Z）：明确 UTC。

同一字符串在不同解析路径下产生不同结果。

## 修复

```diff
// ❌ 问题：日期字符串不含时区信息
function getStartOfDay(dateStr) {
-  return new Date(dateStr); // '2024-01-01' → 不确定是 UTC 还是 local
}

// ✅ 修复方案一：明确指定 UTC
function getStartOfDay(dateStr) {
+  return new Date(dateStr + 'T00:00:00Z'); // 明确 UTC 午夜
}

// ✅ 修复方案二：使用时间戳
function getStartOfDay(timestamp) {
+  const date = new Date(timestamp);
+  date.setHours(0, 0, 0, 0);
+  return date;
}

// ✅ 修复方案三：使用 Day.js / date-fns
+ import { parseISO, startOfDay } from 'date-fns';
+ const date = startOfDay(parseISO(dateStr));

// ✅ 修复方案四：手动解析避免歧义
function parseDateSafe(dateStr) {
+  const [year, month, day] = dateStr.split('-').map(Number);
+  return new Date(Date.UTC(year, month - 1, day)); // 明确 UTC
}
```

```diff
// ❌ 问题：展示日期时未考虑用户时区
function formatDate(utcTimestamp) {
-  return new Date(utcTimestamp).toLocaleDateString();
}

// ✅ 修复：传递时区参数
function formatDate(utcTimestamp, locale = 'zh-CN') {
+  return new Intl.DateTimeFormat(locale, {
+    dateStyle: 'long',
+    timeZone: 'Asia/Shanghai'
```

## 原理
ECMAScript 规范（ECMA-262）对日期字符串的解析经历了多次修订。ES5.1 指定仅日期字符串 `YYYY-MM-DD` 按 UTC 解析，但 ES6 (ES2015) 将其改为按本地时区解析。然而，为了向后兼容，大多数实现（V8、SpiderMonkey）仍然将仅日期字符串按 UTC 解析。这导致了规范与现实的不一致。

明确时区的格式：
- `YYYY-MM-DDTHH:mm:ssZ` → UTC
- `YYYY-MM-DDTHH:mm:ss+08:00` → 指定偏移
- `YYYY-MM-DDTHH:mm:ss` → 本地时区（不推荐，有歧义）

## 架构视角 (Architecture Perspective)

### 架构层面根因
日期时区陷阱的架构根因是**时区策略未全局统一**：
- **存储时区、传输时区、展示时区三者未显式分离**：架构中缺少对"日期三态"的明确定义——存储态（UTC 时间戳）、传输态（ISO 8601 + 时区）、展示态（用户本地时区 + 格式）。这三态之间的转换规则没有统一封装，导致每个开发者自己做转换，产生不一致。
- **时区信息在序列化边界丢失**：API 返回 `"createdAt": "2024-01-01T00:00:00"` 没有时区标识（缺少 Z 或 +08:00），时区信息在 JSON 序列化时丢失。前端接收后无法确知这个字符串的时区语义。架构应该在 API Schema 层面强制所有日期字段带时区标识。
- **缺少全局时间策略声明**：应用没有声明"时间策略"——默认时区是什么（UTC？服务端时区？用户时区？）、如何获取用户时区（浏览器 `Intl`？用户设置？IP 推断？）、跨时区业务逻辑如何处理（如"北京时间今天"的订单截止时间）。

### 系统级预防策略

**静态防护**：
```ts
// TypeScript: 品牌类型强制带时区
type UTCDateString = string & { __brand: 'utc-date' };
// 如 "2024-01-01T00:00:00Z"

type LocalDateString = string & { __brand: 'local-date' };
// 如 "2024-01-01"

// 禁止裸用 new Date(string) — 强制通过工厂函数
function parseUTCDate(isoString: string): Date {
  if (!isoString.endsWith('Z') && !isoString.includes('+')) {
    throw new Error('日期字符串必须包含时区标识 (Z 或 +08:00)');
  }
  return new Date(isoString);
}
```

```js
// ESLint: 禁止模糊的 Date 构造
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'NewExpression[callee.name="Date"]:not([arguments.length=0])',
    message: '禁止裸用 new Date(string)，请使用 parseUTCDate() 或 parseLocalDate()'
  }]
}
```

**架构防护**：
- **日期三态模型**：
```typescript
// @app/time/README.md
// 存储态: Unix timestamp (number) 或 UTC Date，存储在数据库/状态管理
// 传输态: ISO 8601 + 时区 "2024-01-01T00:00:00Z"，经过 API/JSON
// 展示态: 根据用户时区格式化的字符串，仅在渲染层使用

// 转换规则:
export const time = {
  // API → 存储: 解析为 UTC timestamp
  fromAPI: (iso: string) => new Date(iso).getTime(), // 要求 iso 带时区
  // 存储 → API: 格式化为 ISO 8601
  toAPI: (ts: number) => new Date(ts).toISOString(),
  // 存储 → 展示: 格式化为用户时区
  toDisplay: (ts: number, timeZone: string) => 
    new Intl.DateTimeFormat('zh-CN', { dateStyle: 'long', timeZone }).format(new Date(ts)),
};
```
- **API Schema 规范**：OpenAPI schema 中所有日期字段使用 `format: date-time`（RFC 3339，强制带时区），或 GraphQL 使用自定义 `DateTime` scalar。
- **用户时区来源策略**：在 `decisions.md` 中声明用户时区的获取优先级。

**流程防护**：
- Code Review 检查项："日期解析是否使用了专门的工具函数？是否带有明确的时区信息？"
- API 设计评审：所有日期字段是否包含时区标识？前后端是否对时区语义达成一致？

### 跨层影响链

```
上游影响 (消费者):
  - API 返回不含时区的日期 → 前端所有 format 都基于错误的时间戳
  - 用户在不同设备上看到不同的日期（本地时区差异）

下游影响 (依赖):
  - 报表/统计中按日期分组的数据错位（订单统计少一天或多一天）
  - 定时任务/提醒/到期时间计算偏差（如"今天到期"的判断错误）
  - 缓存的日期数据在跨时区用户间共享时混乱

部署影响:
  - 服务器部署在不同区域（us-east vs ap-southeast）时，SSR 渲染的日期不同
  - CDN 边缘节点的时区与用户时区不一致 → SSR 水合不匹配 (hydration mismatch)
  - Docker 容器的默认 TZ 可能不是 UTC

监控影响:
  - 日期解析异常的错误率（按地区分组）
  - SSR hydration mismatch 警告（React/Vue 水合不一致的主要来源）
  - E2E 测试: CI 中强制使用 UTC 运行所有日期相关测试
```

### 相关架构模式
- **渲染模式 — SSR vs SSG vs ISR** (`decisions.md` 第一节) — SSR 中日期格式化依赖服务器时区，CSR 水合后可能日期不一致。应在 SSR 中只传递 UTC 时间戳，格式化完全在客户端完成。
- **API Schema 契约** — 日期字段必须使用 RFC 3339 格式（`2024-01-01T00:00:00Z` 或 `2024-01-01T08:00:00+08:00`），schema 层面由 Spectral/OpenAPI 工具校验。
- **CSS 方案选型** (`decisions.md` 第三节) — 这看起来很遥远，但如果日期/时间选择器（DatePicker/TimePicker）的样式方案影响了交互行为，也会间接影响日期输入的时区处理。

## 预防

1. **日期字符串始终包含时区**：`'2024-01-01T00:00:00Z'` 而非 `'2024-01-01'`。
2. **API 中使用时间戳或 ISO 8601 + 时区**。
3. **统一使用 UTC 存储**，展示时转换为用户时区。
4. **使用 date-fns / Day.js / Luxon** 代替原生 Date，API 更明确。
