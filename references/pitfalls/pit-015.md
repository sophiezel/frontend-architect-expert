# pit-015: 浮点数 0.1 + 0.2 ≠ 0.3

## 环境
- 所有 JavaScript 环境（遵循 IEEE 754 双精度浮点标准）

## 症状
`0.1 + 0.2 === 0.3` 返回 `false`，实际值为 `0.30000000000000004`。涉及金额计算时出现分差，累计后产生财务差异。

## 根因
JavaScript 使用 IEEE 754 双精度浮点数表示所有数字。0.1 和 0.2 在二进制中是无限循环小数，存储时被截断为 52 位尾数。两个近似值相加后再与 0.3 的近似值比较，细微的舍入误差导致不等。

## 诊断方法

```bash
# 1. 控制台直接验证
node -e "console.log(0.1 + 0.2)"           
# 输出: 0.30000000000000004

node -e "console.log(0.1 + 0.2 === 0.3)"  
# 输出: false

# 2. 查看完整精度
node -e "console.log((0.1).toPrecision(55))"
# 输出: 0.1000000000000000055511151231257827021181583404541015625

node -e "console.log((0.2).toPrecision(55))"
# 输出: 0.2000000000000000111022302462515654042363166809082031250

# 3. Number.EPSILON 检查
node -e "console.log(Math.abs(0.1 + 0.2 - 0.3) < Number.EPSILON)"
# 输出: true（容差比较通过）

# 4. 实际金额精度测试
node -e "
let sum = 0;
for (let i = 0; i < 100; i++) sum += 0.01;
console.log(sum); // 期望 1.00 但不是
"
```

## 修复

// ✅ 修复方案一：使用 Number.EPSILON 容差比较
function isApproximatelyEqual(a, b, epsilon = Number.EPSILON) {
+  return Math.abs(a - b) < epsilon;

// ✅ 修复方案二：金额计算转为整数（分为单位）

+  const totalCents = items.reduce(
+    (sum, item) => sum + Math.round(item.price * 100), 0
+  );
+  return totalCents / 100;

function formatPrice(price) {

// ✅ 修复方案四：使用成熟库
+ import Decimal from 'decimal.js';

## 原理
IEEE 754 双精度浮点数使用 64 位表示一个数字：1 位符号位、11 位指数、52 位尾数（有效数字）。十进制小数（如 0.1）转换为二进制时是无限循环小数，只能在 52 位处截断，产生舍入误差。这并非 JavaScript 独有，Python、Java、C 等语言的浮点数都有此问题。

十进制 0.1 → 二进制：0.0001100110011001100110011001100110011001100110011001101...（无限循环 0011）

## 架构视角 (Architecture Perspective)

### 架构层面根因
浮点精度问题的架构根因是**数值类型策略缺失**：
- **"数值"被当作单一类型处理**：应用中有多种不同语义的数值——金额（必须精确）、比例（容差可接受）、显示值（四舍五入）、科学计算（精度优先）。但它们都被当作 `number` 类型处理。架构应该定义不同的数值类型（Money、Percentage、DisplayNumber），每种类型有不同的运算和比较规则。
- **精度边界在模块间传递时丢失**：API 返回价格时，JSON 的 `{"price": 9.99}` 在 `JSON.parse` 后已经是 `9.99` 的近似值。从 API → 状态管理 → 计算 → UI 展示的链路上，每一层都在传递不精确的值。架构应该让精度信息"穿越"序列化边界——使用字符串传输金额（`"9.99"`），在业务层解析为精确类型。
- **缺少全局数值策略声明**：应用没有声明"本应用的数值策略"——金额精度（小数点后几位）、比较容差（用于什么场景）、舍入模式（四舍五入 vs 银行家舍入）。这些应该在架构文档和代码规范中明确定义。

### 系统级预防策略

**静态防护**：
```ts
// TypeScript: 品牌类型（Branded Types）区分数值语义
type Money = number & { __brand: 'money' };
type Percentage = number & { __brand: 'percentage' };
type RawNumber = number;

// 金额运算必须使用专门的函数
function addMoney(a: Money, b: Money): Money {
  return (Math.round(a * 100 + b * 100) / 100) as Money;
}

// 禁止直接用 === 比较 Money 类型
// ESLint 规则
```

```js
// ESLint: 禁止浮点数直接等值比较
rules: {
  'no-floating-decimal': 'error',
  // 自定义规则: 禁止 number === number（涉及小数的场景）
}
```

**CI 防护**：
```bash
# CI 精度测试套件
test('金额累加 100 次 0.01 结果应为 1.00', () => {
  const result = Array(100).fill(0.01).reduce((a, b) => a + b, 0);
  expect(Number(result.toFixed(2))).toBe(1.00);
});

test('订单总价与明细之和一致', () => {
  const lineItems = [{ price: 9.99, quantity: 3 }, { price: 4.50, quantity: 2 }];
  const total = calculateTotal(lineItems);
  const expected = Math.round(9.99 * 3 * 100 + 4.50 * 2 * 100) / 100;
  expect(total).toBe(expected);
});

# CI 检测: 禁止裸用 + - * / 做金额运算
grep -rn "price.*\+" src/ --include="*.ts" && echo "WARNING: 金额运算请使用专门的 Money 工具函数"
```

**架构防护**：
- **数值类型体系**：
```typescript
// @app/types/numeric.ts
export class Money {
  private cents: number; // 内部以分为单位存储
  constructor(amount: string | number) { /* 解析并转为分 */ }
  add(other: Money): Money { /* 精确加法 */ }
  equals(other: Money): boolean { /* 精确比较 */ }
  toJSON(): string { return (this.cents / 100).toFixed(2); } // API 传输用字符串
}
```
- **API Schema 策略**：金额字段在 JSON 中使用字符串传输（`"price": "9.99"`），前后端统一约定。OpenAPI/GraphQL schema 中明确标注金额字段类型为 `DecimalString`。
- **财务精度声明**：在架构决策文档中明确声明：
  - 金额精度：小数点后 2 位（分）
  - 汇率/税率精度：小数点后 6 位
  - 舍入模式：四舍五入（或银行家舍入）
  - 比较容差：不适用（金额必须精确比较）

**流程防护**：
- Code Review 检查项："是否涉及金额/财务计算？如果是，是否使用了 Money 类型或 decimal 库？"
- 架构评审：新增涉及数值计算的模块时，确认是否在架构的数值策略范围内。

### 跨层影响链

```
上游影响 (消费者):
  - 价格从 API 的字符串 → JSON.parse → number 时精度损失
  - 多个前端模块各自转换导致精度不一致累积

下游影响 (依赖):
  - 购物车/订单的总价计算误差
  - 发票/对账单的 tax 计算偏差 → 财务合规风险
  - 价格比较（如"满 9.99 包邮"）永远不触发 → 用户体验问题

部署影响:
  - 不同 Node.js 版本/V8 版本的 Math.round 行为一致（IEEE 754 保证）
  - 但 decimal 库版本更新可能导致舍入行为变化 → 需要锁定版本

监控影响:
  - 财务差异告警：订单总价与明细之和偏差 > 0.01
  - 后端对账：前端提交的金额与后端计算的金额不一致
  - E2E 测试：金额字段精确断言（不仅检查显示值）
```

### 相关架构模式
- **数据流模式 — CQRS 前端化** (`patterns.md` 第三节) — 金额计算作为 Command 层（计算订单总价），查询展示作为 Query 层（格式化显示），分离更清晰。
- **类型安全策略** (`decisions.md` 第三节 — CSS 中的 type-safe) — 品牌类型的概念同样适用于数值类型，TypeScript 的品牌类型可以防止裸 number 进入金额计算。
- **API Schema 契约** — `decisions.md` 未直接覆盖，但架构应扩展：API Schema（OpenAPI/GraphQL）中应使用 `format: decimal` 标注金额字段。

## 预防

1. **金额计算永远用整数**（以分为单位）或专门的 decimal 库（decimal.js / big.js）。
2. **避免浮点等值比较**：使用 `Number.EPSILON` 或直接 `toFixed`。
3. **ESLint 规则**：`no-floating-decimal` 甚至可以考虑自定义规则检测浮点比较。
4. **后端交换**：金融数据用字符串传输，避免 JSON 解析二次精度损失。
