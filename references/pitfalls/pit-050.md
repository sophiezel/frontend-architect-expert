# pit-050: 性能预算形同虚设 — CI 门禁配置了但从未阻止回归

## 环境
- CI/CD 中集成了 Lighthouse CI / Webpack Bundle Analyzer / size-limit
- 使用了 `@lhci/cli` 或 `bundlesize` 或 `size-limit` 作为性能检查
- 项目有明确的性能目标文档（如"首屏 JS < 200KB""LCP < 2.5s"）

## 症状
团队代码库的 bundle 在一年内从 180KB 膨胀到 890KB，LCP 从 2.1s 退化到 5.8s。但令人费解的是——CI 中的 Lighthouse 和 Bundle Size 检查一直在通过，从来都是绿色 ✓。性能持续的温水煮青蛙式退化，每次持续集成都没有阻止。

更讽刺的是，性能预算文档写得很漂亮：
- "JS bundle 上限 200KB"
- "Total Blocking Time (TBT) < 300ms"
- "LCP < 2.5s"

实际上 CI 中的配置却是另一回事：
- `maxSize: '5MB'`——比实际推荐值宽松 25 倍
- `assertions: { 'categories.performance': ['warn', { minScore: 0.6 }] }`——warning 而非 error
- Bundle 检查只看了总大小没按路由拆分，首屏 bundle 包含了所有懒加载的 chunk

当团队终于发现这个问题时，已经积重难返——需要大刀阔斧的"性能优化专项"才能回到基线，而不是每次 PR 的渐进式控制。

## 根因
性能预算形同虚设的根因不是"没配置"，而是**配置与目标脱节**，以及**缺少回退与强制的闭环机制**。具体分解：

1. **阈值设置过于宽松**：预算阈值以"不可能触及"的方式设定。例如首屏 JS bundle 目标 200KB，但 CI 阈值为 5MB——开发时无论怎么加依赖都不会触发。等发现 bundle 退化到 890KB 时已经晚了。

2. **只设 warning 不设 error**：Lighthouse CI 的 `assertions` 可以设两种级别：`warn`（黄色警告但不阻止合入）和 `error`（红色阻止合入）。大多数团队只设了 `warn`，因为"不想因为性能问题 block 业务需求"。

3. **Lab 数据无法反映真实用户场景**：Lighthouse CI 在 CI 容器中运行——CPU/内存固定、网络模拟固定、缓存为空。而真实用户的设备千差万别（低端 Android 机、3G 网络、后台进程占用）。Lab 中 LCP 可能是 1.8s，真实用户 p75 是 4.2s。

4. **预算未按路由拆分**：所有路由共享一个 bundle 总大小阈值。开发者在首屏路由上引入了一个 500KB 的图表库，但总 bundle 仍在阈值内（因为其他路由 bundle 很小）——首屏性能已经烂了但 CI 不知道。

5. **缺少性能回归的自动回退机制**：即使预算被触发了（warning），也没有自动化机制回退或标记。只是 CI 输出中多了一行黄色文本，在 PR 的 2000 行 diff 和 50 条 comment 中被完全忽略。

```javascript
// ❌ 典型的"形同虚设"配置
// lighthouse.config.js
module.exports = {
  ci: {
    collect: { url: ['http://localhost:3000/'] },
    assert: {
      assertions: {
        'categories:performance': ['warn', { minScore: 0.6 }],  // warning + 极低门槛
        'first-contentful-paint': ['warn', { maxNumericValue: 5000 }], // 5 秒也 OK
        'largest-contentful-paint': ['warn', { maxNumericValue: 8000 }], // 8 秒——离谱
        'total-byte-weight': ['warn', { maxNumericValue: 5000000 }], // 5MB
      },
    },
  },
};

// ❌ bundlesize 配置——形同虚设
// package.json
{
  "bundlesize": [
    { "path": "./dist/**/*.js", "maxSize": "5 MB" }  // 所有 JS 加总不超过 5MB
  ]
}
```

  grep -E "warn|error|minScore|maxNumericValue"

# 2. 检查 bundlesize/size-limit 配置
grep -rn "bundlesize\|maxSize\|size-limit\|sizeLimit" package.json .size-limit* -A 10

# 如果项目的 CI artifacts 中有历史 bundle 报告:

for commit in $(git log --oneline --since="6 months ago" --grep="build" | cut -d' ' -f1); do
  git show $commit:dist/report.json 2>/dev/null | jq '.totalSize // empty' && echo " $commit"
done | sort -rn | head -20
# 观察 bundle 大小的单调递增趋势

# 4. 对比 Lab 数据 vs RUM 数据
# Lab (Lighthouse CI):

  jq '.categories.performance.score, .audits["largest-contentful-paint"].numericValue'

# RUM (从 Datadog / Grafana 拉取):
curl -s "https://api.datadoghq.com/api/v1/query?query=avg:rum.largest_contentful_paint{p75}" \

## 原理
性能预算的核心是**不可逆约束**。Google 的实践是 "performance budget as a contract"：新功能不能降低 p75 用户体验成绩，否则要么优化，要么砍功能。这在组织中需要文化转变——性能不再是一个"Nice to have"的优化项，而是一个与功能正确性同等级别的"必须满足的契约"。

三级预算体系（Good / Warning / Error）的设计原理：
- **Good**：当前生产环境 p75 值，是新代码的"基线"。任何不劣于 Good 的 PR 自动通过。
- **Warning**：Good + 20%，是"需要讨论"的阈值。触发 Warning 时 CI 通过但记录 warning，要求 PR 作者在 PR 描述中解释为什么这次性能退化是可接受的，并在 JIRA 中创建性能优化 Task。
- **Error**：Good + 50%，是"不可接受"的阈值。触发 Error 时 CI 直接 block merge，PR 不能被合入。

Lab 数据 vs RUM 数据的本质区别：
- **Lab**（Lighthouse CI）：可控环境，稳定可复现，适合做"有没有回归"的判断
- **RUM**（Real User Monitoring）：真实用户环境，反映 p75/p95 真实体验，适合做"预算设置"的基准

最佳实践是：用 RUM p75 决定 Good 预算值，用 Lab 做每个 PR 的回归检测。两者互补——Lab 防止已知回归，RUM 验证整体趋势和 Lab-RUM gap。

## 架构视角 (Architecture Perspective)

### 架构层面根因
性能预算形同虚设的架构根因是**性能被当作"质量属性"而非"功能需求"**。在大部分团队的架构中，功能需求有 acceptance criteria（"用户能下单"→ 测试用例验证），而性能只有"best effort"文档（"LCP 应该 < 2.5s"→ 没人验证）。

这反映了前端架构中**非功能性需求 (NFR) 的可测试性缺失**。安全性有 SAST/DAST，可靠性有 SLO，但性能只有 Lighthouse 的 warning——一个没有人会看的黄色文本。真正有效的 NFR 约束必须具备三个要素：
1. **自动化验证**（CI 集成，无需人工判断）
2. **回退机制**（不满足时自动回退或阻止推进）
3. **反馈闭环**（触发后开发者收到可操作的反馈）

更深层的架构问题是**性能所有权的模糊**。当性能退化时谁负责？前端团队说"后端 API 慢了"；后端说"前端 bundle 太大了"；DevOps 说"CDN 配置没问题"。缺少端到端的性能契约——"首屏 LCP < 2.5s"中的 2.5s 怎么分配到前端和后端？前端自有 1.2s (bundle + render)，后端 1.0s (API)，网络 0.3s (CDN + TLS)。

Google 的"RAIL 模型"给出了答案：前端自有部分（资源加载 + 渲染）应控制在 1s 以内。这个分配在组织层面需要成为契约。

### 系统级预防策略

**静态检查**：
```bash
# ESLint 规则: 禁止导入超过阈值的大型依赖
rules: {
  'import/no-extraneous-dependencies': 'error',
  'no-restricted-imports': ['error', {

# import cost 检查 (VS Code 插件也可 CI 化)
npx import-cost --max-size=50KB --directory=src/
# 任何 import 超过 50KB (gzip) → CI warning
```

**CI 门禁**：
```bash
# 1. Bundle 预算门禁 (error 级别，block merge)
- name: Size Limit
  run: npx size-limit --json > size-report.json
- name: Check Bundle Budget
  run: |
    VIOLATIONS=$(jq '[.[] | select(.passed == false)] | length' size-report.json)
    if [ "$VIOLATIONS" -gt 0 ]; then
      echo "::error::Bundle budget exceeded for $VIOLATIONS entries"
      jq '.[] | select(.passed == false)' size-report.json
      exit 1
    fi

# 2. Lighthouse 门禁 — performance score < 85 → block merge
- name: Lighthouse CI
  run: |
    npx lhci autorun --collect.url=http://localhost:3000/ \
                     --collect.numberOfRuns=3 \
                     --assert.assertions.categories:performance='["error",{"minScore":0.85}]' \
                     --assert.assertions.largest-contentful-paint='["error",{"maxNumericValue":2500}]' \
                     --assert.includePassedAssertions=false
  env:
    LHCI_GITHUB_APP_TOKEN: ${{ secrets.LHCI_GITHUB_APP_TOKEN }}

# 3. 性能回归自动检测（对比上次 merge 的基线）
- name: Perf Regression Detection
  run: |
    BASELINE_LCP=$(curl -s "$LHCI_SERVER/latest/lcp" | jq '.value')
    CURRENT_LCP=$(jq '.audits["largest-contentful-paint"].numericValue' lhci-report.json)
    REGRESSION=$(echo "scale=2; $CURRENT_LCP / $BASELINE_LCP" | bc)
    if (( $(echo "$REGRESSION > 1.2" | bc -l) )); then
      echo "::error::LCP regressed by $(( (REGRESSION - 1) * 100 ))% — from ${BASELINE_LCP}ms to ${CURRENT_LCP}ms"
      exit 1
    fi

# 4. Bundle diff comment (PR 中自动评论 bundle 变化)
- name: Bundle Diff
  uses: actions-cool/bundle-diff@v1
  with:
    base: ${{ github.base_ref }}
    report: size-report.json
    # 自动在 PR 中评论: "This PR adds +12KB to home-js (now 162KB / 180KB warning)"
```

**架构约束**：
- 性能预算必须作为代码管理在 repo 中（`perf-budget.config.ts`），与业务代码接受同等 Review
- 预算变更 PR 必须经过性能 Owner 审批（CODEOWNERS 中指定）
- 预算违规（Error level）不允许通过配置宽松来"修复"——只能优化代码或 Code Review 中达成一致后提升预算基线
- 所有路由必须有独立的性能预算，不允许"所有路由共用一个总预算"
- 每个季度根据 RUM 数据同步一次 Good 预算基线（预算不是一次性设定永不改变的）

### 跨层影响链

```
上游影响 (产品/业务):
  - 性能持续退化 → 用户流失（Google: LCP 从 1s → 3s, bounce rate +32%）
  - 性能退化未被阻止 → 优化成本指数增长 → 需要专项优化（停业务需求 2~4 周）
  - 移动端的性能退化 → Google 搜索排名下降 → SEO 流量损失

下游影响 (后端/基础设施):
  - Lab 数据与 RUM 严重脱节 → 后端团队不认可前端性能指标
  - Bundle 膨胀 → CDN 带宽成本上升（890KB × 日活 100 万 = 每天多传输 700GB）
  - 大 bundle 导致低端设备白屏 → 客服 tickets 增多 → 客服团队压力增大

部署影响:
  - CI 性能检查只有 warning → 性能退化与 PR 合并脱钩 → 无法回溯"谁引入的退化"
  - 大 bundle 导致部署包变大 → 部署时间延长 → CI/CD 效率降低
  - CDN 缓存命中率下降（bundle 越大越容易变，cache invalid）

监控影响:
  - 性能退化未被 CI 发现 → 依赖 RUM 告警发现 → 告警滞后数小时到数天
  - 与 pit-044 形成"监控与执行"脱节——能监控到退化（044 解决盲区），但没有阻止退化的机制（050 解决执行）
  - 与 pit-046 形成"预算-采集-执行"三角: 046 管数据准确性，050 管基于数据的执行
```

### 相关架构模式
- **Performance Budget as Code** (`patterns.md`) — 性能预算与代码同仓管理，通过 PR Review 变更，在 CI 中强制执行，有明确的回退机制。
- **RAIL 性能分配模型** (`decisions.md`) — Response (100ms) / Animation (16ms) / Idle (50ms) / Load (1000ms)，每个 RAIL 维度对应不同的预算策略。
- **Progressive Performance Governance** (`patterns.md`) — 三级预算体系（Good/Warning/Error）实现渐进式的性能治理，既不过于严格阻碍开发，也不过于宽松形同虚设。
- **Lab-RUM Bridge** (`decisions.md`) — Lab 检测回归、RUM 设定基线、两者的 gap 作为环境配置质量指标。
- 与 **pit-044 (监控盲区)** 互补——044 解决"能不能看到性能数据"，050 解决"看到数据后能不能阻止退化"。

## 预防
1. **三级预算体系**：Good（当前值）/ Warning（+20%）/ Error（+50%），Error 级别 block merge
2. **按路由拆分预算**：首屏路由独立预算（JS < 150KB, total < 300KB），内页路由独立预算
3. **Lab-RUM 交叉验证**：每月对比 Lab 数据与 RUM p75，gap > 1.5x 时告警并调整 Lab 环境
4. **预算审查闭环**：每个 Warning 触发需要 PR 中附带解释 + 创建性能优化 task；每个 Error 触发必须优化代码才能合入
5. **自动化回归检测**：每次 PR 与上次 merge 的性能基线做 diff，超过 20% 退化自动 block
6. **性能契约文档化**：将"LCP < 2.5s"分解为各团队的责任分配（前端 render: 1.2s, API: 1.0s, CDN: 0.3s）
