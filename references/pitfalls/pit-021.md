# pit-021: iOS 软键盘弹出遮挡输入框 / 页面不恢复

## 环境
- iOS Safari 12+
- WKWebView (iOS 混合应用)
- 移动端 H5 页面

## 症状
用户在 iOS Safari 或 WebView 中聚焦输入框，软键盘弹出后将输入框遮挡。输入完成后键盘收起，但页面没有恢复到原始位置，上方出现空白区域。

在 React Native 的 WebView 中尤其常见：键盘弹出时将整个 WebView 向上推，收起后无法自动回弹。

## 根因
iOS Safari 的 `visualViewport` 行为与标准不同。键盘弹出时：
1. Safari 会调整 `visualViewport` 位置来确保焦点元素可见
2. 但不会同步调整 `layoutViewport`，导致页面布局与可视区域不一致
3. 键盘收起后，`visualViewport` 恢复但某些布局信息未更新

在 WKWebView 中，原生键盘事件不会自动调整 WebView 内容区域。

## 诊断方法

```bash
# 1. 在 Safari Web Inspector 中连接设备
# 打开 Safari → 开发 → [设备名] → [页面]

# 2. 在控制台监听 visualViewport 变化
window.visualViewport.addEventListener('resize', () => {
  console.log('visualViewport height:', window.visualViewport.height);
  console.log('offsetTop:', window.visualViewport.offsetTop);
});

# 3. 检查页面是否有 fixed 定位元素使用了 bottom: 0
grep -rn "position:\s*fixed" src/ | grep "bottom"
```

## 修复

```diff
// ❌ 未处理键盘事件
<input type="text" />

// ✅ 方案1: 使用 scrollIntoView + onFocus
+ const handleFocus = (e) => {
+   // 延迟等键盘动画完成
+   setTimeout(() => {
+     e.target.scrollIntoView({ 
+       behavior: 'smooth', 
+       block: 'center' 
+     });
+   }, 300);
+ };
+ <input type="text" onFocus={handleFocus} />

// ✅ 方案2: 监听 visualViewport 变化
+ useEffect(() => {
+   const handleResize = () => {
+     const currentHeight = window.visualViewport?.height || window.innerHeight;
+     document.documentElement.style.setProperty('--viewport-height', `${currentHeight}px`);
+   };
+   window.visualViewport?.addEventListener('resize', handleResize);
+   return () => window.visualViewport?.removeEventListener('resize', handleResize);
+ }, []);
```

```css
/* ✅ CSS 适配 */
.page-container {
  /* 使用动态 viewport height，不用 100vh */
  min-height: var(--viewport-height, 100vh);
  /* 避免键盘弹出时重新计算 */
  min-height: 100dvh; /* iOS 15.4+ 支持 dynamic viewport height */
}

/* fixed 底部元素需要适配 */
.fixed-bottom {
  position: fixed;
  bottom: 0;
  /* 键盘弹出时跟着上移，但确保不超出安全区 */
  padding-bottom: env(safe-area-inset-bottom);
}
```

## 原理
`visualViewport` API 是专门为解决移动端键盘问题设计的。通过监听 visualViewport 的 resize 事件，可以精确感知键盘弹出/收起，并动态调整页面布局。

`100dvh` 是 CSS 的新单位（dynamic viewport height），iOS 15.4+ 支持，它会根据当前可视区域（含键盘）自动调整，优于 `100vh`。

## 架构视角 (Architecture Perspective)

### 架构层面根因
iOS 键盘问题的架构根因是**移动端平台适配策略未系统化**：
- **键盘是移动端的基础交互设施，但被当作"输入框的问题"**：键盘弹出/收起影响的是整个页面的视口（所有 fixed 定位、滚动区域、全屏布局），但代码中键盘处理逻辑散落在各个输入框组件中，没有统一在"移动端视口管理层"处理。这类似于 pit-006（100vh）的问题——键盘动态改变了视口高度，但应用没有统一的视口管理。
- **WKWebView 和 Safari 的差异未在架构中建模**：同一个 Web 应用可能运行在 Safari、WKWebView、Android WebView、微信内置浏览器中。但代码中将它们当作"同一个浏览器"处理，没有根据容器类型应用不同的键盘策略。
- **fixed 定位与键盘的交互未在布局系统中定义**：`position: fixed; bottom: 0` 的元素在键盘弹出时的行为取决于浏览器实现（iOS: 不跟随键盘；Android: 跟随但可能溢出）。架构需要在设计系统中明确定义"底部固定元素在键盘场景下的行为"。

### 系统级预防策略

**静态防护**：
```css
/* 设计系统: 键盘安全的 CSS 变量 */
:root {
  --keyboard-safe-bottom: 0px;
  --app-viewport-height: 100vh;
  --app-viewport-height: 100dvh;
}

@supports not (height: 100dvh) {
  :root {
    --app-viewport-height: -webkit-fill-available;
  }
}

/* 底部固定元素使用变量 */
.bottom-bar {
  position: fixed;
  bottom: calc(var(--keyboard-safe-bottom) + env(safe-area-inset-bottom, 0px));
}
```

```js
// 运行时: 监听 visualViewport 更新 CSS 变量
// src/boot/keyboard-manager.ts
function initKeyboardManager() {
  const updateViewport = () => {
    const vh = window.visualViewport?.height || window.innerHeight;
    const offsetTop = window.visualViewport?.offsetTop || 0;
    document.documentElement.style.setProperty('--viewport-height', `${vh}px`);
    document.documentElement.style.setProperty('--keyboard-offset', `${offsetTop}px`);
  };
  
  window.visualViewport?.addEventListener('resize', updateViewport);
  window.visualViewport?.addEventListener('scroll', updateViewport);
  updateViewport();
}
```

```bash
test('键盘弹出不应遮挡输入框', async ({ page }) => {
  await page.setViewportSize({ width: 390, height: 844 }); // iPhone 14
  await page.goto('/form');
  const input = page.locator('input[type="text"]').last();
  await input.focus();
  // 等待键盘动画
  await page.waitForTimeout(500);
  // 输入框应在可视区域内
  const isVisible = await input.evaluate(el => {
    const rect = el.getBoundingClientRect();
    const vh = window.visualViewport?.height || window.innerHeight;
    return rect.bottom <= vh && rect.top >= 0;
  });
  expect(isVisible).toBe(true);
});
```

**架构防护**：
- **移动端视口管理层**：与 pit-006 共享视口管理策略，额外增加键盘状态管理：
```typescript
// @app/mobile/viewport.ts
class MobileViewportManager {
  private keyboardHeight = 0;
  private isKeyboardVisible = false;
  
  init() {
    window.visualViewport?.addEventListener('resize', this.handleViewportChange);
  }
  
  private handleViewportChange = () => {
    const currentHeight = window.visualViewport?.height || window.innerHeight;
    const fullHeight = document.documentElement.clientHeight;
    this.isKeyboardVisible = currentHeight < fullHeight - 100;
    this.keyboardHeight = fullHeight - currentHeight;
    
    // 更新全局状态
    document.documentElement.classList.toggle('keyboard-visible', this.isKeyboardVisible);
    document.documentElement.style.setProperty('--keyboard-height', `${this.keyboardHeight}px`);
  };
  
  // 确保元素在可视区域内
  ensureVisible(element: HTMLElement): void {
    // 根据键盘状态和元素位置计算需要的滚动调整
  }
}
```
- **键盘行为声明**：在设计系统中为布局组件（`<BottomBar>`、`<FixedFooter>`、`<FullScreenModal>`）声明键盘行为——"键盘弹出时上移并保持可见"或"键盘弹出时保持原位"。

### 跨层影响链

```
上游影响 (消费者):
  - 用户无法完成关键表单 → 转化率下降
  - 混合应用（RN WebView）的键盘体验差 → App Store 差评

下游影响 (依赖):
  - 第三方组件（DatePicker、Select、RichTextEditor）的键盘行为不一致
  - 聊天应用的输入框被键盘遮挡 → 核心功能不可用
  - 支付页面的密码输入框在键盘弹出后不可见 → 支付失败

部署影响:
  - iOS 系统版本升级（如 iOS 15.4 引入 dvh）改变了键盘行为 → 回归测试
  - WKWebView 的键盘策略受 App 原生配置影响 → 需要与移动端团队协调
  - SSR 中键盘相关代码不应在服务端执行 → 需要 `typeof window !== 'undefined'` 守卫

监控影响:
  - RUM: 按平台/版本统计键盘相关体验指标（填写表单放弃率）
  - 自定义事件: keyboardVisible 和 keyboardHeight 的变化
  - E2E: iOS 真机测试（不能用模拟器，模拟器键盘行为不同）
```

### 相关架构模式
- **pit-006: iOS 100vh 问题** — 同源的视口管理问题。键盘动态改变视口高度，与 Safari 工具栏影响 100vh 的问题同构，解决方案（visualViewport API + dvh + CSS 变量）可以共享。
- **CSS 方案选型** (`decisions.md` 第三节) — `env(safe-area-inset-bottom)` 和 `dvh` 等新 CSS 特性需要 CSS 方案支持特性检测和 fallback，Tailwind 和 CSS Modules 都支持通过 `@supports` 实现。
- **React Native WebView 架构** — WKWebView 的键盘行为需要在原生层配置 `keyboardDisplayRequiresUserAction` 和 `allowsInlineMediaPlayback`，前端架构需要与移动端架构协调。

## 预防
1. 避免在移动端页面使用 `height: 100vh`，改用 `min-height: 100dvh` 或 JS 计算的 `--viewport-height`
2. 所有 `fixed` 定位的元素必须测试键盘场景
3. React Native WebView 设置 `keyboardDisplayRequiresUserAction={false}` 并配合原生键盘监听
4. 在测试矩阵中覆盖 iOS Safari + 软键盘 场景
