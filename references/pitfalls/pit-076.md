# pit-076: GraphQL N+1 查询 — resolver 未使用 DataLoader

## 环境
- GraphQL 服务端 (Apollo Server / Yoga / GraphQL-js / Hot Chocolate / graphene)
- schema 中存在嵌套类型关系（如 `User → Posts → Comments`）
- 每个字段的 resolver 独立查询数据库

## 症状
一个简单的 GraphQL 查询发出数百甚至数千条 SQL 语句。数据库 CPU 飙升到 100%，GraphQL 接口响应时间从 50ms 暴涨到 5-10 秒。数据库连接池耗尽，其他服务也被拖垮。

```
查询:
query {
  posts(first: 20) {
    title
    author {
      name          ← 每个 post 发起一次 author 查询
    }
    comments(first: 10) {
      body
      author {
        name        ← 每个 comment 又发起一次 author 查询
      }
    }
  }
}

实际 SQL 执行次数:
  posts: 1 次
  author (for posts): 20 次
  comments: 20 次 (每 post 一次)
  author (for comments): 200 次 (每 comment 一次)
  总计: 241 次 SQL 查询  ← 而不是 4 次
```

症状在小型数据集上不明显（开发环境数据少），生产环境数据量增大后突然暴露。监控显示：数据库慢查询数量正常（每条查询很快），但 QPS 极高（频繁的小查询）。

## 根因
**GraphQL resolver 的本质是"每个字段独立解析"，默认实现会导致 N+1 查询**。核心机制：

1. **Resolver 链式执行模型**：每个字段的 resolver 独立调用，父级 resolver 返回数据后，子字段的 resolver 逐一执行。如果子字段需要查数据库，每次调用都会发起一次新查询。

2. **缺少批处理（Batching）**：没有 DataLoader 这样的批处理层，相同的数据库查询（如"查 user by id"）被重复执行 N 次而不是合并为一次 `WHERE id IN (1,2,3...N)`。

3. **缺少缓存（Caching）**：每个 resolver 独立查询，同一个 user ID 在不同层级被多次查询（post.author 和 comment.author 可能查询同一个 user），但没有请求级别的缓存复用。

4. **嵌套深度放大**：N+1 在最内层被指数级放大（posts → comments → authors → ...）。

+ const DataLoader = require('dataloader');

+ function createLoaders() {
+   return {
+     userLoader: new DataLoader(async (ids) => {
+       // N 个 id 合并为 1 条查询
+       const users = await db.query(
+         'SELECT * FROM users WHERE id = ANY($1)', [ids]

## 原理
GraphQL 的类型系统天然支持嵌套查询，但 resolver 执行模型是**深度优先 + 分层执行**。每个字段的 resolver 被独立调用，如果不使用批处理层，每个 resolver 都会独立访问数据源。DataLoader 的核心是两阶段：`load()` 将 key 收集到队列中，在同一个事件循环 tick 结束时，将所有 key 合并为一个批量 `loadMany()` 调用。它同时提供**请求级缓存**（同一 key load 多次只查一次）。

关键限制：DataLoader 实例**必须为每个请求创建**（否则请求间的缓存污染）。这通常通过 Apollo Server 的 `context` 工厂函数实现。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **数据访问层缺失**：GraphQL resolver 直接操作数据库，没有中间的 Data Access Layer 统一管理查询批处理和缓存。
- **Schema 设计未考虑查询效率**：深层嵌套的类型（User → Posts → Comments → Author）天然容易 N+1，但 Schema 设计时没有评估每个字段的"批处理友好度"。
- **GraphQL 的优势被滥用**：客户端可以自由组合字段的能力非常好，但服务端没有提前评估"哪些组合会导致 N+1 爆炸"。

### 系统级预防策略
- **强制 DataLoader 覆盖**：所有返回列表的 resolver 必须使用 DataLoader 做子字段查询。CI 中检查 resolver 是否直接调用 db.query。
- **Persisted Queries**：只允许经过审批的查询，提前分析查询代价（query cost analysis）。
- **查询复杂度限制**：`graphql-query-complexity` 设置最大查询深度和调用次数上限。

### 跨层影响链
- 数据库：连接池耗尽 → 其他服务延迟 → 级联故障
- 用户体验：5 秒+ 响应 → 用户放弃 → 收入损失
- 基础设施：需要更多数据库只读副本 → 成本增加

## 预防

1. **每个新 resolver 必须使用 DataLoader**，禁止裸 SQL 查询。
2. **启用 query complexity analysis**：设置最大深度（如 8 层）、最大节点数。
3. **Persisted Queries**：仅允许预先注册的查询，生产环境禁止随意 GraphQL 查询。
4. **Code Review 清单**：任何 Post/User 等嵌套类型的 resolver 是否有 loader？
