# pit-074: Edge Function CPU 超时 — 超过 50ms 限制被 kill

## 环境
- Cloudflare Workers / Vercel Edge Functions / Deno Deploy
- 在 Edge Runtime 中执行计算密集型操作
- 使用 V8 Isolate 运行时的产品

## 症状
API 随机返回 502/520/500 错误，响应 body 不完整或为空。错误日志显示 "CPU time exceeded" 或 "Script exceeded time limit"。问题在高峰期严重（多并发时 Isolate 复用少，每个请求被分配到新 Isolate 时触发冷启动计算）。同一个 API 端点，有时正常返回（轻量请求），有时超时（处理大量数据），表现为"间歇性 500"。

更隐蔽的症状：
- 线上监控显示 p95 响应时间正常（30ms），但 p99 高达 10 秒（被 terminate 的请求计入超时）
- `ctx.waitUntil()` 中的异步任务被静默中断，数据未写入 KV/D1
- Streaming Response 的 writer 在 CPU 超时后停止，客户端收到截断的 JSON

典型日志：
```
Error: Script exceeded CPU time limit of 50ms
  at myFunction (worker.js:42:14)
```

## 根因
**Edge Runtime 的 CPU 时间模型与开发者心智模型不匹配**。核心机制：

1. **CPU time ≠ Wall time**：Cloudflare Workers 免费版限制 10ms CPU time / request（付费 30s）。CPU time 是**实际执行 JS 代码的时间**，不包括等待 I/O（`fetch()` 网络等待、`crypto.subtle` 异步操作等待）。开发者容易误以为"30 秒 wall time = 30 秒可以做任何事"，但实际上在等待网络响应的 200ms 中，CPU time 只走了开始和结束的几微秒。

2. **同步大计算瞬间爆表**：JSON.parse(大文件)、字符串操作、循环遍历大数组、加密解密——这些是纯同步 CPU 操作，在 V8 中直接消耗 CPU time。

3. **外部 API 的响应体过大**：`await resp.json()` 对一个巨大的响应体（10MB+ JSON）会消耗大量 CPU 做解析。即便 `fetch()` 本身不消耗 CPU time，解析响应体会。

4. **V8 Isolate 的 GC 也计入 CPU time**：频繁的对象创建（如在循环中 `new Date()` 或大量模板字符串拼接）会触发 GC，GC 消耗的 CPU time 同样计入限制。

5. **Vercel Edge 的特殊限制**：虽然 wall time 较长，但如果在一次执行中做大量计算（如 SSR 大型 React 树），V8 Isolate 的编译和执行成本可能超出预算。

## 原理
Cloudflare Workers 运行在 V8 Isolates 中。每个 Isolate 分配一个 CPU 时间预算，V8 在执行每个 JS 指令时累加计数器。当计数器超过预算，V8 引擎会**立即终止** Isolate 的执行——这不是超时异常可以 catch 的，而是引擎级的 kill。关键理解：`await` 暂停 JS 执行，CPU 计数器也暂停。所以最优策略是：**将计算分散到多个 `await` 之间**，或**移到 Origin 处理**。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **计算边界未标注**：路由/端点没有声明"CPU 密集"或"轻量"标签，导致重运算无意中部署到 Edge。
- **Edge 与 Origin 职责模糊**：Edge 本应是"智能路由 + 轻量转换"，但被当作全能 API Server。
- **负载建模缺失**：上线前没有对最大请求体、最大数据集做 CPU 时间预估。

### 系统级预防策略
- **计算分类部署**：定义 `CPU_BUDGET` 级别（≤10ms Edge / ≤50ms Edge Pro / >50ms Origin / >1s Worker Thread）。
- **请求体大小限制**：Edge 中间件中拦截 body > 1MB 的请求，返回 413 或直接转发。
- **OOM/CPU 监控告警**：Cloudflare GraphQL Analytics API 监控 CPU time p99。

### 跨层影响链
- 用户：间歇性 502 → 不可靠体验 → 流失
- 服务：被 kill 的请求中 `ctx.waitUntil` 未完成 → 数据不一致
- 运维：需要逐个排查"哪些端点不该在 Edge"

## 预防

1. **不在 Edge 中做大计算**：JSON 解析 > 1MB 或数组遍历 > 1000 项 → 转 Origin。
2. **使用流式处理**：对大数据使用 `ReadableStream`，不一次性加载到内存。
3. **设置守护 timeout**：`ctx.waitUntil()` 中的任务设置自己的超时和重试。
4. **CPU 预算检查**：在开发阶段用 `wrangler dev` 模拟限制，或用 `performance.now()` 自测。
