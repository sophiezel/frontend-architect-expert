# pit-052: 主包体积爆炸 — 超 2MB 限制无法上传/启动慢

## 环境
- 微信小程序主包体积接近或超过 2MB
- 使用了大量第三方库（lodash/moment/echarts）
- 未做分包或分包策略不合理

## 症状
微信开发者工具上传时提示"代码包大小超过限制，主包上限为 2MB"，无法上传发布。即使勉强控制在 2MB 以内，启动时间超过 3 秒，用户看到白屏等待超过 2 秒后大量流失。Android 低端机上启动耗时可达 5~8 秒，小程序性能评分极低。

## 根因
小程序主包包含所有页面、组件、工具函数、第三方库，在启动时需要完整下载并执行。微信的限制是：
- **主包 ≤ 2MB**（硬性限制，超过无法上传）
- **所有分包总计 ≤ 20MB**
- 主包太大 → 下载慢（弱网用户首屏加载 > 5秒）
- 主包太大 → 代码注入和执行慢（JsCore 需要解析和执行整个 bundle）

更深层根因是**模块依赖未做可视化管理**：开发者往往不知道每个 import 带来的 package 大小，缺乏 bundle 体积分析工具链。

## 诊断方法

```bash

# 所有列在 pages 数组中的都是主包页面

find ./dist -name "*.js" -exec du -h {} \; | sort -rh | head -20

# 5. 使用 webpack-bundle-analyzer (如果使用 Taro/uni-app)
npm run build:weapp -- --analyze
```

## 原理
微信小程序的启动流程：下载主包 → 注入主包代码（JsCore 解析执行）→ 渲染首页。所有主包代码都在启动阶段处理，体积越大越慢。分包机制允许将非启动路径的代码拆分为独立文件，在用户首次访问分包页面时才下载和注入。独立分包更进一步——完全不下载主包，可以独立启动，适合分享/扫码直接进入的页面（如活动落地页）。

`lazyCodeLoading: "requiredComponents"` 的作用：阻止全局自定义组件在启动阶段全部注入，改为页面渲染时按需注入。对于有大量全局组件的项目，这个配置可以减少 20~40% 的启动代码注入时间。

## 架构视角 (Architecture Perspective)

### 架构层面根因
主包体积爆炸的架构根因是**缺少 Bundle 治理体系**。在 Web 项目中，较大的 bundle 可以靠 CDN + HTTP/2 + 浏览器缓存来缓解，用户感知不强。但小程序的"一次下载、全量注入"模型让 bundle 大小与启动性能直接挂钩。架构缺陷包括：

1. **依赖引入无成本意识**：import 一个库在小程序中不仅增加 bundle 大小，还增加 JsCore 的解析成本。每个第三方库在主包中都被完整解析执行，即使首页不需要
2. **缺少体积预算**：没有为主包、分包设置 KB 预算线，体积随时间单调递增
3. **分包与业务耦合**：分包策略被当作优化手段而非架构手段，应该在项目初期就按业务域规划分包

### 系统级预防策略

**CI 门禁**：
```bash
# CI 中检查主包大小
- name: Check Package Size
  run: |
    MAIN_PKG_SIZE=$(du -sk dist | awk '{print $1}')
    MAIN_LIMIT=2048  # 2MB = 2048KB
    if [ $MAIN_PKG_SIZE -gt $((MAIN_LIMIT * 85 / 100)) ]; then
      echo "::warning::Main package is ${MAIN_PKG_SIZE}KB (warning at 85%)"
    fi
    if [ $MAIN_PKG_SIZE -gt $MAIN_LIMIT ]; then
      echo "::error::Main package exceeds ${MAIN_LIMIT}KB limit!"
      exit 1
    fi

# 检查大型依赖
- name: Check Heavy Dependencies
  run: |
    npx miniprogram-slim check --max-file-size 200KB
```

**架构防护**：
```js
// 建立依赖引入规范 — 白名单机制
// .depcheckrc.js
module.exports = {
  allowedPackages: {
    'main': ['dayjs'],           // 主包只允许 dayjs
    'subpackage': ['echarts'],   // 分包允许 echarts
  },
  // 禁止在主包中引用的库
  bannedInMain: ['lodash', 'moment', 'echarts', 'three.js'],
};
```

### 跨层影响链

```
上游影响 (业务):
  - 超 2MB 无法上传 → 发版阻塞 → 紧急减包 → 可能误删必要功能
  - 启动慢 → 用户跳出率 +30%~50% → DAU 下降

下游影响:
  - 分包异步化组件加载延迟 → 页面出现占位闪烁
  - 预下载消耗用户流量 → 用户投诉

监控影响:
  - 监控指标: "小程序启动耗时" / "首屏渲染时间"
  - 告警阈值: 启动耗时 > 3000ms (p75)
```

### 相关架构模式
- **Bundle 预算模式** (`patterns.md`) — 与 pit-050 同源，设主包/分包 KB 上限和 CI 门禁
- **分包设计模式** — 按业务域垂直切分分包，而非按类型水平切分
- **按需注入模式** — lazyCodeLoading + 组件占位机制
- **反模式: 巨型主包** — 所有代码堆在主包，是移动端和小程序的经典架构反模式

## 预防
1. **项目初始化即规划分包**：首页核心路径放主包，其余按业务域分包
2. **体积预算**：主包 ≥ 1.6MB warning，≥ 1.8MB error（留 10% 安全余量）
3. **禁止重型库进主包**：echarts/moment/lodash 等必须放分包或按需加载
4. **CI 门禁检测**：每次 PR 检查主包大小，超过阈值 block merge
5. **启用 lazyCodeLoading**：减少启动代码注入量
6. **packOptions.ignore** 过滤非代码文件（DS_Store/.gitignore/docs）
