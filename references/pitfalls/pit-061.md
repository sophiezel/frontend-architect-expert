# pit-061: Flutter Web 首屏过大

## 环境
- Flutter Web (stable channel, 3.x+)
- 默认渲染器: CanvasKit (桌面端) / HTML (移动端)
- 部署到 CDN / 静态托管 (Firebase Hosting, Vercel, Cloudflare Pages)

## 症状
- 首屏白屏时间过长（3-10 秒），用户看到空白页面后离开
- 首次加载时浏览器下载了 2-10MB 的资源（canvaskit.wasm + main.dart.js）
- Lighthouse 性能评分 < 50，LCP > 5 秒
- 低速网络（3G/4G）下页面完全不可用，加载超时
- `canvaskit.wasm` 文件从 unpkg.com CDN 加载缓慢（国内用户更甚）
- 浏览器显示"页面无响应"警告
- SEO 完全失效（搜索引擎爬虫无法索引 Flutter Web 应用）

## 根因
Flutter Web 默认的 CanvasKit 渲染器需要加载 **canvaskit.wasm**（约 1.5MB gzip，2.8MB 未压缩）和 **main.dart.js**（业务代码，通常 200KB-2MB+）。这两个文件是**渲染阻塞**的，在它们完全下载、解析、编译完成之前，页面显示为白屏。

核心瓶颈分解：

1. **CanvasKit 体积**：1.5MB (gzip) — Skia 图形库编译为 WebAssembly，包含了完整的 2D 渲染管线、字体引擎、路径光栅化等。这是必须的体积，但加载策略可以优化。

2. **Dart2JS 编译产物体积**：即使最简单的 `Hello World`，dart2js 编译产物也接近 200KB (gzip)。因为 Flutter 框架本身（Widget 系统、渲染引擎绑定、手势系统、路由）都被编译进去。Tree shaking 只能移除未被引用的代码，但框架核心代码无法移除。

3. **默认 CDN 问题**：`canvaskit.wasm` 默认从 `https://unpkg.com/canvaskit-wasm@latest/bin/` 加载。unpkg.com 在国内访问极慢（或无响应），且 `@latest` 标签意味着缓存策略不稳定。

4. **渲染阻塞链**：`index.html` → `<script src="flutter.js">` → `flutter.js` 初始化 → 下载 `canvaskit.wasm` → 编译 WASM → 下载 `main.dart.js` → 执行 Dart → 首次渲染。这是一个**长串行链**，每一步都等待上一步完成。

## 原理
Flutter Web 的 CanvasKit 渲染器本质上是将 Flutter 的整个渲染管线编译为 WebAssembly（Wasm），在浏览器的 Canvas 2D 上下文中运行。这个过程的物理限制：

1. **Wasm 下载 + 编译时间**：1.5MB Wasm 在 4G 网络下下载约需 2-4 秒，编译（浏览器将 Wasm 字节码转为机器码）约需 0.5-1.5 秒（取决于设备 CPU）。二者是串行的（必须先下载完成才能编译），总计 2.5-5.5 秒。

2. **Dart2JS 体积构成**：dart2js 的 Tree Shaking 依赖于静态分析。但 Flutter 框架使用了很多运行时特性（如 `runtimeType`、`noSuchMethod`、动态调用），这些特性无法被完全 tree-shake。此外，泛型的类型参数在 dart2js 中被保留为运行时元数据，增加了"残留体积"。

3. **字体体积**：Material Icons 字体包含 ~2000+ 图标字形，总量约 1.1MB。如果应用只使用了其中 10 个图标，dart2js 可以 tree-shake 掉未引用的图标类引用，但字体文件本身无法被 tree-shake。需要在 pubspec 中明确限制字体加载。

4. **HTML renderer 的取舍**：HTML renderer 通过 DOM + CSS + Canvas 组合模拟 Flutter 渲染，不需要 Canvaskit。它的初始负载约 500KB (gzip)，远小于 CanvasKit。但 HTML renderer 的性能上限较低——复杂动画、大量组件时会出现明显的重排/重绘开销。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Flutter Web 首屏过大的表面原因是"CanvasKit 体积大"，但架构层面根因是**渲染方案与 Web 平台设计哲学的根本错配**：
- Web 平台的设计哲学是"渐进式加载"——先加载 HTML/CSS 显示内容，再加载 JS 增强交互。但 Flutter Web 采用"全有或全无"策略——在引擎完全就绪之前，没有任何内容可渲染。这与 Web 的"流式渲染"理念背道而驰。
- **编译策略过度保守**：dart2js 将"整个 Flutter 框架"视为一个整体进行编译，而没有利用 Web 的天然代码分割能力（动态 import、route-based chunking）。虽然 Flutter Web 支持 `deferred as`（代码分割），但框架核心的 Widget 库、渲染引擎、手势系统仍然是一个整体 chunk，无法按需加载。
- **CDN 策略缺少区域性优化**：默认使用 unpkg.com 作为 CanvasKit CDN，这是一个全球统一的 CDN，但在亚太地区（尤其是中国大陆）的延迟和可用性不可接受。Flutter 没有提供区域性 CDN 切换的配置指导，导致开发者需要手动解决 CDN 问题。

### 系统级预防策略

**静态防护**：
```yaml
# pubspec.yaml 中的静态体积守卫
flutter:
  # 限制字体加载
  fonts: []  # 不引入 Material Icons (请使用自定义图标)
  # 明确声明需要的字体子集

# analysis_options.yaml
analyzer:
  errors:
    # 禁止引入重型包 (如 google_fonts 全量引入)
    deprecated_member_use: error
```

**CI 防护**：
```bash
# CI 中体积和性能预算
# 1. Bundle 体积检查
BUNDLE_SIZE=$(du -sb build/web/ | cut -f1)
if [ "$BUNDLE_SIZE" -gt 5000000 ]; then  # 5MB 阈值
  echo "ERROR: Bundle size ${BUNDLE_SIZE}bytes exceeds 5MB limit"
  exit 1
fi

# 2. Lighthouse CI 性能门禁
lhci autorun --config=lighthouserc.js
# lighthouserc.js:
# assertions: {
#   'categories.performance.score': ['error', { minScore: 0.7 }],
#   'largest-contentful-paint': ['error', { maxNumericValue: 3000 }],
#   'total-byte-weight': ['error', { maxNumericValue: 3000000 }],
# }

# 3. 引入新依赖时的体积影响分析
npx bundle-wizard --entry build/web/main.dart.js --baseline last-build.json
```

**架构防护**：
- **渲染器降级链**：自动检测设备能力和网络速度，选择最佳渲染器。类似 Flutter 3.10+ 的 `auto` 模式，但应增加网络速度检测（通过 `navigator.connection.effectiveType`）。
- **CanvasKit 预热策略**：如果应用有登录/引导页（不需要高性能渲染），使用 HTML renderer 渲染这些页面，同时在后台预加载 CanvasKit，待主功能页面再切换。
- **代码分割架构**：将 Flutter Web 应用按路由 level 做代码分割。首屏只加载最少的 Widget（AppBar + BottomNav + 首页内容），其他页面的代码延迟加载。
- **自建 CDN + 区域加速**：将 `canvaskit.wasm` 托管到自有 CDN，配置多区域加速（全球 → Cloudflare CDN，国内 → 阿里云 CDN）。

**流程防护**：
- 引入新依赖包时，必须在 PR 中报告其对 `main.dart.js` 体积的增量。
- 每次发布前运行 Lighthouse CI 并确保分数不降。
- 监控 RUM 的 LCP 和 FCP——不仅是平均值，p75 和 p95 也要达标。

### 跨层影响链

```
上游影响 (用户层):
  - 3G/4G 用户完全无法访问 → 用户流失（尤其是新兴市场）
  - 白屏 > 3s → 53% 的移动端用户会离开 (Google 数据)
  - 国内用户因 unpkg.com 被墙/慢 → 页面完全不可用

下游影响 (后端/基础设施):
  - CDN 流量费用暴涨 (每次访问下载 2-5MB)
  - 服务器带宽压力 (高并发场景下 10MB × 10万 PV = 1TB/天)
  - 搜索引擎爬虫因白屏而无法索引 → SEO 流量为零

部署影响:
  - 大体积 Bundle 拖慢 CI 构建和上传时间
  - CDN 预热 (purge/invalidate) 更大、更慢
  - 无法使用 CodePush 式热更新 (Flutter Web 不是 JS，无动态特性)

监控影响:
  - 需要监控 CanvasKit 加载成功率和加载时间
  - LCP 监控按渲染器拆分 (CanvasKit vs HTML)
  - 需要采集 `navigator.connection.effectiveType` 和 `navigator.hardwareConcurrency` 来分析性能瓶颈是否与设备和网络有关
```

### 相关架构模式
- **渐进增强** (`patterns.md`) — 从最轻量的渲染方案开始（HTML renderer），检测设备能力后升级到 CanvasKit。
- **延迟加载 (Lazy Loading)** — Flutter 的 `deferred as` + 路由层级的代码分割。
- **感知性能优化** — 骨架屏 / 加载动画 / 进度条，减少用户感知的白屏时间。
- **CDN 架构** (`decisions.md`) — 自建 CDN vs 第三方 CDN 的选择，区域性加速策略。

## 预防

1. **项目启动阶段就设定性能预算**：`main.dart.js` < 1MB (gzip)，总首屏资源 < 2.5MB (gzip)。
2. **避免全量 Material Icons**：使用 `fluttericon.com` 或 `iconfont` 自定义图标字体，仅包含需要的图标。
3. **使用 `flutter build web --web-renderer auto --release`**：让 Flutter 根据设备能力自动选择渲染器。
4. **自托管 CanvasKit**：将 `canvaskit.wasm` 部署到自有 CDN，配置长期缓存 (`Cache-Control: immutable`)。
5. **首屏代码最小化**：将非首屏功能放在 `deferred` 库中，路由切换时按需加载。
6. **监控 RUM 指标**：在应用启动时采集 FCP/LCP/TTI，上报到分析平台。
