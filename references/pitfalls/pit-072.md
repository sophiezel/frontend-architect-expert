# pit-072: i18n 翻译键缺失 — 未翻译的 key 直接展示给用户

## 环境
- 多语言 Web 应用（react-intl / vue-i18n / next-intl）
- 翻译文件通过翻译平台 (Lokalise/Crowdin) 管理或自维护
- 部分语言的翻译覆盖率不足 100%

## 症状
用户在非英语语言页面看到原始翻译键名（如 `dashboard.recentOrders.emptyState`）而非翻译文本。更严重的是：部分页面在阿拉伯语/日语下大段显示英文 fallback，与页面整体语言不一致，看起来像半成品。用户截图发到反馈群："你们中文版是不是机翻的？为什么还有英文混在里面？"

隐蔽症状：ICU Message Format 中缺少某个复数形式导致运行时抛出 `FORMAT_ERROR`，整个组件白屏。这在测试覆盖中很难发现——因为通常只测试了 `other` 形式。

典型投诉：
- "为什么设置页有 3 个按钮全显示 `common.save`、`common.cancel`、`common.delete`？"
- "切换到日语后整个 Sidebar 都是英文"
- "阿拉伯语页面从右往左时部分文本重叠"

## 根因
**翻译覆盖的"侥幸工程"——缺失的 key 没有编译时或 CI 时的硬性阻断**，导致缺失被传播到生产环境。

1. **运行时 fallback 过于宽松**：react-intl 的 `defaultMessage` 机制允许在 `messages` 中缺失时使用组件内定义的默认值（通常是英文）。这抹平了缺失，但用户看到的是语言不一致的"半翻译"页面。

2. **翻译文件未做完整性校验**：`zh-CN.json` 有 823 个 key，`ja.json` 只有 789 个。新增 key 后被 `JSON.stringify` 静默吞掉。CI 中只校验了 JSON 格式合法性，未做跨语言 key 完整性对比。

3. **翻译平台同步延迟**：开发者在代码中添加了新 key → 源文件提取到平台 → 译员翻译 → PR 合并，这个流程中如果有任何延迟或遗漏，CI 无法检测。

4. **ICU Message Format 语法错误静默传播**：`{count, plural, one {# item} other {# items}}` 如果某个语言缺少 `other`（实际上是必需的），只有在该特定条件下才抛出错误。

5. **动态 key 拼接**：`t('status.' + order.status)` 如果 `order.status` 是新枚举值但翻译文件未同步，直接暴露原始 key。

## 原理
i18n 库的核心是一个简单的查表操作：`messages[id]`。如果 `messages` 对象中不存在该 key，库的行为取决于 fallback 策略。react-intl 会检查 `defaultMessage` prop，如果存在就使用它（通常是英文）。这就导致了一种**幸存者偏差**：测试英文时一切正常（因为 defaultMessage = 英文），测试其他语言时有 defaultMessage 兜底似乎也"正常"，但用户看到的是混杂语言。vue-i18n 的 `fallbackLocale` 配置会回退到指定语言，但同样掩盖了缺失。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **翻译完整性未成为 CI 门禁**：翻译是"数据"而非"代码"，传统 CI 只校验代码，留下了数据质量缺口。
- **动态 key 是类型系统的缺口**：`t('status.' + order.status)` 无法做静态分析，运行时可能生成任何字符串，TypeScript 完全无法约束。
- **翻译工作流未与开发流程耦合**：新增 key → 翻译 → 部署是三个独立步骤，缺少强制同步机制。

### 系统级预防策略
- **翻译完整性 CI Gate**：任何 PR 引入新的翻译 key 时，CI 必须校验所有 locale 文件都包含该 key。
- **类型安全的翻译 key**：使用 TypeScript 模板字面量类型或 `as const` 将 key 提升为类型，禁止任意字符串拼接。
- **翻译平台 webhook → PR 自动化**：译员完成翻译后自动生成 PR，由 CI 校验后自动合并。

### 跨层影响链
- 用户：看到未翻译文本 → 品牌形象受损 → 怀疑产品质量
- 客服：收到大量"翻译错误"反馈，增加成本
- 部署：紧急修复翻译缺失需要完整部署流程

## 预防

1. **TypeScript 类型化 key**：使用 `next-intl` 的类型推断或手动维护 key 的 union type。
2. **CI 强制翻译完整性**：任一新 key 必须在所有 locale 文件中存在。
3. **禁止动态 key 拼接**：使用显式映射表替代字符串拼接。
4. **翻译覆盖率仪表盘**：实时展示各语言的翻译覆盖率，低于 95% 自动告警。
