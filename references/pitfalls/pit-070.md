# pit-070: Service Worker 缓存过期 — 用户看到旧版本页面

## 环境
- PWA 应用，注册了 Service Worker
- 使用 Cache First 或 Stale-While-Revalidate 策略缓存 HTML/JS/CSS
- 部署新版本后用户仍看到旧版内容

## 症状
部署新版本后，用户打开页面看到的是旧 UI、旧逻辑、旧的错误提示。刷新页面无效（SW 拦截了请求直接返回旧缓存）。Chrome DevTools Network 面板中资源 Size 列显示 "(ServiceWorker)"。更严重的是：新部署的 API 响应与新 UI 不兼容，旧 JS 调用新 API 导致功能完全不可用。用户在群里投诉"页面挂了"，但开发者本地一切正常。

典型时间线：
1. 10:00 部署新版本前端（新的 JS/CSS/HTML）
2. 10:05 用户 A 反馈"页面白屏" / "功能异常"
3. 开发者本地测试 — 一切正常（未安装旧 SW）
4. 排查 30 分钟才发现是 SW 缓存了旧版本

## 根因
Service Worker 的缓存策略设计缺陷导致**部署与缓存的生命周期脱节**：

1. **HTML 被缓存**：如果 HTML 入口文件被 Cache First / SWR 策略缓存，新的 JS/CSS 引用路径（带新哈希）永远不会被请求，页面停留在旧版本。
2. **SW 更新不感知部署**：SW 仅在注册时检查 `sw.js` 是否有字节级变化。如果部署时未更新 `sw.js`（例如仅修改了业务 JS 但 SW 文件未变），浏览器不会触发 SW 更新。
3. **skipWaiting + clients.claim 缺失或滥用**：不恰当的更新策略导致新 SW 安装后迟迟不激活。
4. **CACHE_NAME 未版本化**：使用固定缓存名如 `'my-cache'`，不清除旧缓存，新旧资源混存。

核心机制：SW 的缓存是**不可变 URL 假设**的设计，但部署改变了 URL 背后的内容。

## 诊断方法

```bash
# 1. Chrome DevTools → Application → Service Workers
#    查看当前注册的 SW 版本/状态
#    勾选 "Update on reload" 后刷新 → 看问题是否消失（确认是缓存问题）

# 2. Network 面板
#    查看资源的 Size 列：显示 "(ServiceWorker)" → 来自 SW 缓存
#    刷新后对比资源 URL（新旧哈希是否不同）

# 3. 检查 sw.js 是否在新部署中更新
#    对比部署前后的 sw.js 文件哈希
diff <(curl -s https://example.com/sw.js | shasum) \
     <(curl -s https://staging.example.com/sw.js | shasum)

# 4. 检查 CACHE_NAME
#    Application → Cache Storage → 查看缓存名是否版本化
```

## 修复

```diff
// ❌ 问题代码 1: HTML 使用 Cache First
registerRoute(
  ({ request }) => request.mode === 'navigate',
-  new CacheFirst({ cacheName: 'pages' })
+  new NetworkFirst({
+    cacheName: 'pages',
+    networkTimeoutSeconds: 3,
+    plugins: [new ExpirationPlugin({ maxEntries: 10, maxAgeSeconds: 300 })]
+  })
);

// ❌ 问题代码 2: 固定缓存名，不版本化
- const CACHE_NAME = 'static-cache';
+ const CACHE_NAME = `static-cache-v${BUILD_VERSION}`;

// ❌ 问题代码 3: 未清理旧缓存
self.addEventListener('activate', (event) => {
+  const validCaches = [CACHE_NAME];
+  event.waitUntil(
+    caches.keys().then((keys) =>
+      Promise.all(
+        keys.filter((k) => !validCaches.includes(k)).map((k) => caches.delete(k))
+      )
+    ).then(() => self.clients.claim())
+  );
-  // 之前无清理逻辑
});
```

## 原理
Service Worker 拦截网络请求的优先级高于浏览器 HTTP 缓存。当 SW 注册后，`fetch` 事件处理器决定响应来源：缓存 或 网络。Cache First 策略的致命缺陷在于：如果 HTML 被缓存，浏览器永远不会发起对入口 HTML 的新请求，也就不会发现新的 JS/CSS 资源（带新哈希），形成**版本锁定**。只有确保 HTML 使用 NetworkFirst（带超时回退），或 SW 文件本身更新，才能打破这个锁定。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **资源生命周期未声明**：不同类型的缓存（HTML vs JS vs 数据）没有不同的生命周期声明，统一使用 CacheFirst 是懒惰设计。
- **部署原子性无保障**：前端部署是"逐步替换 CDN 文件"，与 SW 的缓存在时序上没有协调，缺少"部署版本号"作为缓存一致性锚点。
- **更新通道未设计**：没有用户可见的"新版本可用"提示通道，也没有 `window.location.reload()` 的优雅触发机制。

### 系统级预防策略
- **缓存分层治理**：将缓存分为 shell（永久预缓存）、content（短期 SWR）、data（IndexedDB + 后台同步）三层，每层独立的 TTL 和更新策略。
- **部署版本号驱动缓存失效**：每次部署注入 BUILD_ID，SW 激活时通知页面端比对，不一致则触发更新 UI。
- **可观测性**：监控 `sw.js` 文件的 `Last-Modified` / `ETag` 变化频率，告警 SW 7 天未更新的异常。

### 跨层影响链
- 上游: 用户看到旧 UI + 新 API → 功能异常 → 差评/流失
- 下游: 旧 JS bundle 中的埋点/错误监控代码过期 → 线上错误无法追踪
- 部署: 新部署的热修复/安全补丁对已有用户完全无效

## 预防

1. **HTML 入口永远使用 NetworkFirst**（带网络超时回退）。
2. **CACHE_NAME 必须版本化**：使用构建哈希或语义化版本号。
3. **activate 事件中清理旧缓存**：永远在 `activate` 中过滤无效缓存名。
4. **使用 Workbox 插件自动化**：`workbox-webpack-plugin` 或 `vite-plugin-pwa` 自动处理版本化和 precache。

## 回退

```bash
# 方案A: 紧急清除所有用户的 SW 缓存
# 部署一个"自杀式" sw.js:
self.addEventListener('install', () => {
  self.skipWaiting();
});
self.addEventListener('activate', () => {
  caches.keys().then(keys => Promise.all(keys.map(k => caches.delete(k))));
  self.registration.unregister();
});
# 这会清除所有缓存并注销 SW，页面恢复为普通 Web 应用

# 方案B: 强制页面刷新
# 在主线程代码中添加:
navigator.serviceWorker.getRegistration().then(reg => {
  if (reg) { reg.unregister(); window.location.reload(); }
});
```
