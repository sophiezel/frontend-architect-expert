# pit-011: Webpack 代码分割后 chunk 加载 404

## 环境
- Webpack 代码分割（`import()` / `React.lazy` / `SplitChunksPlugin`）
- 部署到 CDN 或子路径

## 症状
用户访问页面时报错 `Loading chunk X failed`、`ChunkLoadError`，页面白屏或功能模块加载失败。通常在部署后发生，本地开发正常。

## 根因
动态导入生成的 chunk 文件 URL 计算错误：
1. `publicPath` 配置错误——chunks 从根路径加载但部署在子目录下。
2. 部署时旧 chunk 被删除，用户浏览器缓存了旧 HTML，引用了不存在的旧 chunk 文件名（内容哈希变化）。
3. CDN 未同步或回源失败。
4. chunk 文件名包含 `[contenthash]` 但部署时新旧文件不共存。

## 诊断方法

```bash
# 1. 浏览器 Network 面板
# 筛选 .js 请求 → 查看 404 的 chunk URL
# 对比 404 URL 与实际部署路径

# 2. 检查 publicPath
# 浏览器控制台执行：
const scriptSrc = document.querySelector('script[src*="main"]')?.src || '';
console.log('实际 JS 路径:', scriptSrc);
// 对比 webpack 配置中的 publicPath

# 3. 检查 HTML 中的 chunk 引用
# 查看页面源代码 → 搜索 "chunk" 或 "async"
# 对比引用的文件名与服务器上实际存在的文件

# 4. 构建产物检查
ls -la dist/js/*.js | head -20
# 对比文件名与 HTML 中引用是否一致

# 5. 查看构建日志
# webpack --json > stats.json
# 在 stats.json 中搜索 "assets" → 确认 chunk 文件名列表
```

## 修复

```js
// webpack.config.js

// ❌ 问题：publicPath 为 '/' 但部署在 '/app/' 子路径
module.exports = {
  output: {
    publicPath: '/',
  },
};

// ✅ 修复
module.exports = {
  output: {
    publicPath: '/app/',
  },
};

// ✅ 更好的方案：运行时动态确定（Webpack 5）
module.exports = {
  output: {
    publicPath: 'auto', // 让运行时自动推断
  },
};
```

```js
// ✅ 前端错误处理：chunk 加载失败时自动重试并刷新
// src/bootstrap.js — 在 React 挂载前注入
// 也可以在入口文件顶部添加全局错误处理：
window.addEventListener('error', (event) => {
  if (event.message.includes('Loading chunk') || 
      event.message.includes('ChunkLoadError')) {
    // 重试一次，失败后刷新页面
    window.location.reload();
  }
});
```

## 验证

```bash
# 1. 本地构建 + 子路径部署测试
npm run build
# 使用 serve 在子路径下启动
npx serve -s dist -l 3000
# 访问 http://localhost:3000/app/ → 确认无 404

# 2. publicPath 验证
# 浏览器控制台：
console.log(__webpack_public_path__); // 应输出正确的路径前缀

# 3. Network 面板验证
# 清除缓存 → 刷新页面 → 检查所有 chunk 请求
# 所有 .js 文件状态码应为 200，无 404

# 4. 部署后验证
# 模拟用户浏览器缓存了旧 HTML：
# 1. 访问页面（版本 A）
# 2. 部署新版本（版本 B）
# 3. 不刷新页面，点击触发动态导入
# 修复后：应自动重试或刷新，不显示白屏
```

## 原理
Webpack 使用 `publicPath` 作为所有输出文件的基础 URL。代码分割产生的 chunk 文件路径 = `publicPath` + chunk 文件名。如果 `publicPath: '/'` 但应用部署在 `/app/`，chunk 会被请求到 `/chunk.abc.js` 而非 `/app/chunk.abc.js`。

版本更新时的 chunk 404 源于"部署原子性"问题：用户浏览器缓存了旧版 HTML（引用 chunk.abc.js），但服务器上只有新版文件（chunk.def.js），旧文件已被删除。

## 架构视角 (Architecture Perspective)

### 架构层面根因
chunk 404 的架构根因是**部署策略与资源加载策略解耦不足**：
- **部署原子性未被建模为架构需求**：代码分割将应用拆分为多个独立文件，但部署时这些文件被当作独立资源分别上传到 CDN，没有保证"所有文件同时就绪"。架构应该要求 CDN 部署策略提供"版本目录"（如 `dist/v2.3.1/`）而非覆盖式部署。
- **资源加载策略缺少版本感知**：HTML 引用的 chunk URL（如 `/js/chunk.abc123.js`）没有版本上下文。当用户持有旧版本 HTML 时，无法自动发现"自己处于旧版本"并请求刷新。架构缺少一个"版本协商协议"——HTML 中的 `data-app-version` 与 `/api/version` 端点协同，客户端检测到版本不匹配时主动全量刷新。
- **对 CDN/缓存基础设施做了过于乐观的假设**：架构假设 CDN 回源总是成功、缓存总是有效、用户永远不会在部署中途访问。但实际上，这些边界条件在生产环境中频繁出现，架构应该为"部署窗口期间的不一致性"设计容错机制。

### 系统级预防策略

**静态防护**：
```js
// webpack.config.js — 强制 publicPath 验证
const assert = require('assert');
// 构建时验证
assert(process.env.PUBLIC_PATH, 'PUBLIC_PATH 环境变量未设置！');
module.exports = {
  output: {
    publicPath: process.env.PUBLIC_PATH,
  }
};

// Vite 同理
// vite.config.ts
export default defineConfig({
  base: process.env.VITE_BASE_URL || '/',
  experimental: {
    renderBuiltUrl(filename, { hostType }) {
      // 自动检测 CDN 路径
      return hostType === 'js' 
        ? { runtime: `window.__cdn_base__ + ${JSON.stringify(filename)}` }
        : { relative: true };
    }
  }
});
```

**CI 防护**：
```bash
# CI 部署检查: 验证所有 chunk 可访问
npm run build
for f in dist/js/*.js; do
  HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "https://staging.example.com/js/$(basename $f)")
  if [ "$HTTP_CODE" != "200" ]; then
    echo "ERROR: $f 返回 $HTTP_CODE" && exit 1
  fi
done

# CI 构建后检查: HTML 中引用的 chunk 文件名是否存在
HTML_CHUNKS=$(grep -oP 'src="[^"]*chunk[^"]*\.js"' dist/index.html | sed 's/src="//;s/"//')
for chunk in $HTML_CHUNKS; do
  [ -f "dist/$chunk" ] || echo "MISSING: $chunk" && exit 1
done
```

**架构防护**：
- **非覆盖式部署**：使用版本目录部署（`/v2.3.1/js/`），保留最近 3 个版本的 chunk。旧版本 HTML 引用的 chunk 不会 404。
- **版本协商机制**：
```html
<!-- index.html -->
<meta name="app-version" content="2.3.1">
```
```js
// 应用启动时
const currentVersion = '2.3.1';
setInterval(async () => {
  const latest = await fetch('/api/version').then(r => r.text());
  if (latest !== currentVersion) {
    // 有更新，通知用户或强制刷新
    showUpdateNotification(() => window.location.reload());
  }
}, 60000);
```
- **ChunkLoadError 全局兜底**：Sentry 之外的架构级兜底——检测 chunk 加载失败后，回退到完整 bundle 或全量刷新。

**流程防护**：
- Code Review 检查项："`publicPath` 是否配置为 `auto` 或明确的环境变量？是否有 chunk 加载失败的全局处理？"
- 发布清单："发布前确认 CDN 是否保留最近 3 个版本的 chunk 文件？HTML 缓存策略是否为 `no-cache`？"

### 跨层影响链

```
上游影响 (消费者):
  - 用户看到白屏 → 跳出率 100%
  - PWA ServiceWorker 缓存的 HTML 与 CDN chunk 不匹配
  - 浏览器 HTTP 缓存了 404 响应 → 即使 CDN 恢复也需要清除缓存

下游影响 (依赖):
  - 懒加载的路由页面全部不可用
  - React.lazy 的 Error Boundary 如果未设置 → 整个 React 树卸载
  - 动态 import 的 i18n 翻译文件 404 → 页面显示 key 而非文本

部署影响:
  - 紧急回滚也必须保留所有历史 chunk 文件，增加回滚复杂度
  - 灰度发布时，不同用户拿到不同 HTML，引用的 chunk 版本不一致
  - CDN purge 操作与部署时序竞态 → chunk 短暂 404

监控影响:
  - Sentry: ChunkLoadError 聚合统计
  - RUM: 按版本/地区统计 chunk 加载失败率
  - 告警: chunk 404 率 > 0.1% 时触发
  - 自定义事件: chunk 加载耗时分布（P50/P95/P99）
```

### 相关架构模式
- **构建架构模式 — Webpack vs Vite** (`decisions.md` 第四节) — Vite 的按需编译天然使用原生 ESM `import()`，浏览器的模块加载器有内建的重试机制（比 Webpack 的 JSONP chunk 加载更可靠）。
- **Monorepo 管理** (`patterns.md` 第五节) — Turborepo 的远程缓存确保 CI 构建的 chunk 哈希与本地一致，避免"本地测试通过但 CI 构建产物不同"导致的 404 谜题。
- **部署架构** (`decisions.md` 第一节 — ISR + CSR 混合) — CDN 边缘部署策略直接影响 chunk 加载的可靠性。

## 预防

1. **`publicPath: 'auto'`**（Webpack 5）让运行时自动推断。
2. **CDN + 非覆盖式部署**：保留最近 N 个版本的 chunk 文件。
3. **入口 HTML 不缓存（或短缓存）**：`Cache-Control: no-cache` 确保用户始终获取最新 HTML。
4. **监控 ChunkLoadError**：前端接入 Sentry/自建埋点，主动发现 chunk 加载失败。
5. **使用 `__webpack_public_path__`** 在运行时动态设置。
