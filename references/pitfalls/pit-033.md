# pit-033: Vue Suspense/异步组件陷阱

## 环境
- Vue 3.3+（Suspense 稳定）
- 使用 `defineAsyncComponent` 或动态 `import()` 的项目
- SSR（Nuxt 3 / Vite SSR）场景
- `<Suspense>` 包裹异步组件

## 症状
页面白屏时间异常长——Suspense 内的组件任何一个异步依赖未 resolve，整个 Suspense 树都不渲染。SSR 场景下服务端 HTML 与客户端不匹配，触发 hydration mismatch 警告。异步组件加载失败时没有错误处理，用户看到空白区域或控制台错误。动态 import 的组件在 HMR 时行为异常。

具体表现：
- `<Suspense>` 内的 `#default` 插槽中多个异步组件，最快的也要等最慢的 → 水桶效应
- `onServerPrefetch` 在客户端 Suspense 中不执行，导致数据获取逻辑不一致
- `defineAsyncComponent` 的 loading/error 组件与 Suspense 的 `#fallback` 职责重叠

## 根因
1. **Suspense 的"全有或全无"语义**：`<Suspense>` 等待其内部所有异步依赖 resolve 后才渲染 `#default`，这包括所有嵌套 `async setup()`、`defineAsyncComponent`、以及具有 `suspensible: true` 的组件。任何一个慢请求都会阻塞整棵树。

2. **SSR 与客户端的 Suspense 行为差异**：SSR 时，服务器渲染 `#fallback` 内容而不是等待异步组件。客户端 hydrate 时发现 HTML 不匹配（服务端出的是 fallback，客户端期望的是异步组件结果）→ mismatch。

3. **defineAsyncComponent 与 Suspense 的边界模糊**：`defineAsyncComponent` 有自己的 loading/error/timeout 配置，`<Suspense>` 也有 `#fallback` 和 `onErrorCaptured`。两者同时使用时行为不可预测。

## 修复

```diff
// ❌ 问题 1: Suspense 内所有异步组件共用 fallback
<Suspense>
  <template #default>
    <AsyncUserProfile :id="1" />   <!-- 100ms -->
    <AsyncOrderList :id="1" />     <!-- 2000ms -->
    <AsyncNotifications />         <!-- 500ms -->
  </template>
  <template #fallback>
-    <FullPageSpinner />
  </template>
</Suspense>
// 问题：OrderList 的 2000ms 阻塞了 UserProfile（100ms）的展示

// ✅ 分离 Suspense 边界
+<Suspense>
+  <AsyncUserProfile :id="1" />
+  <template #fallback><ProfileSkeleton /></template>
+</Suspense>
+<Suspense>
+  <AsyncOrderList :id="1" />
+  <template #fallback><OrderSkeleton /></template>
+</Suspense>

// ❌ 问题 2: SSR hydration mismatch
// server-side: 渲染 fallback → HTML 包含 Loading...
// client-side: 重新执行 async setup → 渲染实际内容
// HTML 不匹配 → hydration 失败

// ✅ 方案 A: 客户端专用 Suspense
+<ClientOnly>
+  <Suspense>
+    <AsyncDashboard />
+    <template #fallback><DashboardSkeleton /></template>
+  </Suspense>
+</ClientOnly>

// ✅ 方案 B: Nuxt 3 使用 <NuxtClientFallback>
+<Suspense>
+  <AsyncComponent />
+  <template #fallback>
+    <NuxtClientFallback @ssr-error="handleError">
+      <SkeletonComponent />
+    </NuxtClientFallback>
+  </template>
+</Suspense>

// ❌ 问题 3: 异步组件加载失败无处理
-const AsyncComp = defineAsyncComponent(() => import('./Heavy.vue'));

+const AsyncComp = defineAsyncComponent({
+  loader: () => import('./Heavy.vue'),
+  loadingComponent: LoadingSpinner,
+  errorComponent: ErrorDisplay,
+  delay: 200,         // 200ms 后才显示 loading（避免闪烁）
+  timeout: 10000,     // 10s 超时
+  onError(error, retry, fail, attempts) {
+    if (attempts <= 2) { retry(); }
+    else { fail(); }
+  }
+});
```

## 原理
Vue Suspense 的协调机制：当组件树中遇到异步依赖（`async setup()` 返回 Promise），Suspense 会收集子树中所有 pending 的 Promise，用 `Promise.all()` 等待它们全部 resolve。这意味着 Suspense 的渲染延迟 = max(所有异步依赖的耗时)，而非 min。

SSR 场景下，Vue 的服务端渲染器不会等待异步组件（因为 Node.js 的 renderToString 是同步的），它直接渲染 fallback 内容。客户端 hydrate 时，异步组件可能在服务端 HTML 已经被 resolve（缓存）或在客户端需要重新 fetch，导致 hydration 树不一致。

## 架构视角 (Architecture Perspective)

### 架构层面根因
Suspense 陷阱的架构根因是**异步加载边界的粒度控制缺失**：
- **"一个页面一个 Suspense"是懒惰方案**：将整个页面包裹在一个 `<Suspense>` 中，看似省事，实则是将异步加载策略完全委托给框架。正确的做法是将页面拆分为多个独立的"加载岛"，每个岛有独立的 Suspense 边界和 fallback 策略。
- **SSR/CSR 的代码路径未在架构层面分离**：`onServerPrefetch` 和 `async setup()` 在 SSR 和 CSR 中有不同的执行语义，但代码中两者写在同一个组件里。架构需要提供 "SSR-only" 和 "Client-only" 的代码隔离机制（如 Nuxt 的 `server/` 和 `client/` 目录约定）。
- **错误处理不在异步边界的设计中**：同步组件的错误可以被 `onErrorCaptured` 捕获，但异步组件的加载失败（网络错误、chunk 404、超时）在 Suspense 层面没有一等公民的处理路径。

### 系统级预防策略

**静态检查**：
```bash
# 检测：Suspense 内是否包含多个异步组件
# scripts/suspense-audit.sh
find src -name "*.vue" -exec grep -l "<Suspense>" {} \; | while read f; do
  async_count=$(grep -c "defineAsyncComponent\|async setup" "$f")
  if [ "$async_count" -gt 1 ]; then
    echo "WARNING: $f 的 Suspense 内包含 $async_count 个异步依赖 — 建议拆分 Suspense 边界"
  fi
done
```

**CI 门禁**：
```bash
# Playwright E2E: 异步组件加载场景
test('异步组件超时后有降级UI', async ({ page }) => {
  // 模拟最慢的 chunk 请求失败
});

# Lighthouse CI: 首屏不应被异步组件阻塞
# lighthouserc.js
assert: {
  'categories.performance.score': ['error', { minScore: 0.8 }]
}
```

**架构约束**：
- Suspense 边界策略：每个 Suspense 最多包裹 1 个异步组件，除非多个组件共享同一数据源
- 异步组件必须定义 `onError` 重试策略和 `errorComponent`
- `async setup()` 中的数据获取必须抽离到 composable，并在 composable 中处理 loading/error 状态

**流程约束**：
- 架构评审清单："此页面的异步依赖有哪些？各自的加载耗时预估？Suspense 边界是如何划分的？"
- 每个使用 `async setup()` 的组件需提供：loading 态 design、error 态 design、超时配置

### 跨层影响链

```
上游影响 (用户感知):
  - 白屏时间被最慢的异步依赖拖长 → LCP 劣化 → SEO 排名下降
  - SSR hydration mismatch → 页面闪烁 → 用户困惑

下游影响 (CDN/部署):
  - 异步 chunk 404（部署时旧 hash 文件已删除）→ 用户看到空白
  - 异步组件加载失败触发 ErrorBoundary → 整个页面降级

部署影响:
  - 新版本部署后旧的 async chunk 被删除，仍在线的用户路由到新页面时 chunk 404
  - 需要在 nginx 中保留旧 chunk 一段时间，或使用 import() 错误重试机制

监控影响:
  - RUM: async chunk 加载时间（分位数 p50/p95/p99）
  - Error tracking: chunk load error 按 chunk 名称分组
  - 自定义事件: Suspense resolve 时间、fallback 显示时长
```

### 相关架构模式
- **Islands Architecture** (`patterns.md` 第九节) — 将页面拆分为独立的"交互岛"，每个岛拥有独立的加载状态和 Suspense 边界。与 Astro 的 Islands 思想一致。
- **SSR 渐进式水合** (`decisions.md` 第六节) — 与 Vue 的 `createSSRApp` + `hydrateOnInteraction` 结合，避免一次性 hydrate 整棵树。
- **pit-001: useEffect 无限循环** — 同为"副作用/异步边界"问题，React Suspense 与 Vue Suspense 共享等待语义的架构挑战。

## 预防
1. 每个 `<Suspense>` 只包裹 1 个异步组件（或共享同一数据源的组件组）
2. `defineAsyncComponent` 必须配置 `errorComponent` 和 `onError` 重试策略
3. SSR 场景中异步组件用 `<ClientOnly>` 包裹或使用 Nuxt 的 `<NuxtClientFallback>`
4. 异步组件不在 `async setup()` 中直接 fetch，而是通过 composable 暴露 `isLoading`/`error` 状态
