# pit-005: setState 异步批处理带来的状态不一致

## 环境
- React 16/17（事件处理中的同步批处理）
- React 18（所有场景下自动批处理）

## 症状
连续多次调用 `setState` 后，读取 state 得到的仍是旧值；依赖当前 state 计算新值时结果错误。例如连续三次 `setCount(count + 1)` 最终只加了 1 而非 3。

## 根因
React 将多个 `setState` 调用批量合并为一次更新。`setState` 不会立即修改 state 值。如果连续调用 `setState(count + 1)` 三次，每次读取的 `count` 都是同一个旧值，最终效果等同于只调用了一次。

## 诊断方法

```bash
# 1. 在多次 setState 之间打印 state
const handleClick = () => {
  console.log('[DEBUG] before:', count);
  setCount(count + 1);
  console.log('[DEBUG] after first setState:', count); // 仍是旧值
  setCount(count + 1);
  console.log('[DEBUG] after second setState:', count); // 仍是旧值
};
# 如果三次打印都是同一个值 → 确认批处理问题

# 2. React 18 检测：使用 setTimeout 打破批处理（仅 React 17-）
setTimeout(() => {
  setCount(count + 1); // React 17: 立即更新；React 18: 仍批处理
}, 0);

# 3. useEffect 监听验证
useEffect(() => {
  console.log('[DEBUG] count updated to:', count);
}, [count]);
# 如果连续三次 setState 只触发一次 effect → 确认批处理
```

## 修复

```diff
// ❌ 问题：连续 setState 基于过时的 state
function Counter() {
  const [count, setCount] = useState(0);

  const handleTriple = () => {
-    setCount(count + 1); // count = 0 → 结果 1
-    setCount(count + 1); // count 仍是 0 → 结果 1
-    setCount(count + 1); // count 仍是 0 → 结果 1（非 3）
+    setCount(c => c + 1);
+    setCount(c => c + 1);
+    setCount(c => c + 1); // 最终结果为 3
  };
}
```

```diff
// ❌ 问题：useEffect 依赖 state 可能导致竞态
function Search() {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState([]);

  useEffect(() => {
    fetchResults(query).then(setResults);
  }, [query]);

  const handleChange = (e) => {
-    setQuery(e.target.value);
-    // 如果快速输入 "abc"，三次渲染三次请求，
-    // 后到达的响应可能覆盖先到达的
+    const value = e.target.value;
+    setQuery(value);
+    // 需要配合 AbortController 或 debounce 处理竞态
  };
}
```

## 原理
React 的批处理（Batching）机制将同一个事件处理函数中的多个 setState 调用合并为一次更新，以提高性能。在 React 17 中，批处理仅发生在 React 事件处理器中；React 18 通过 `createRoot` 引入了自动批处理（Automatic Batching），将批处理扩展到 setTimeout、Promise、原生事件等所有场景。

函数式更新 `setState(prev => newValue)` 不会读取闭包中的旧值，而是 React 内部将最新的 state 作为参数传入，因此不受批处理影响。

## 架构视角 (Architecture Perspective)

### 架构层面根因
批处理导致状态不一致的架构根因是**命令式状态更新模型与 React 声明式渲染模型之间的阻抗不匹配**：
- **多步状态转换未被建模为原子操作**：连续三次 `setCount(count + 1)` 期望的是"将 count 增加 3"这个原子操作，但实现方式是三个独立的、各自读取快照的命令式操作。架构缺少一个"状态转换原子化"的机制——应该使用 `useReducer` 将多步更新封装为单个 dispatch。
- **更新策略与读取策略耦合**：`count + 1` 这种写法将"如何计算新值"和"读取哪个版本的值"绑定在一起。函数式更新 `c => c + 1` 解耦了这两者——计算逻辑独立于读取时机。架构应从设计层面就鼓励函数式更新，将其作为默认的状态更新模式。
- **时序依赖被隐藏**：当代码写成 `setCount(count + 1)` 三次时，开发者隐含地假设了三件事：(a) 同步执行、(b) 顺序执行、(c) 每次读取最新值。React 的批处理打破了这三个假设中的前两个，但代码没有表达出这个时序依赖。架构应通过 useReducer 或 XState 将时序依赖显式化。

### 系统级预防策略

**静态防护**：

```js
// TypeScript 严格模式

```

**CI 防护**：
```bash

# 使用 MSW (Mock Service Worker) 模拟慢网络 + 快速输入
test('快速输入时不应出现竞态条件', async () => {
  const { user } = render(<Search />);
  await user.type(screen.getByRole('textbox'), 'abc');
  // 等待所有请求完成
  await waitFor(() => {
    // 验证最终结果对应最新输入
    expect(screen.getByText('results for: abc')).toBeInTheDocument();
  });
});

# lint-staged 检查
{ '*.{ts,tsx}': ['eslint --rule "react-hooks/exhaustive-deps: error"'] }
```

**架构防护**：
- **优先使用 useReducer**：当一次交互触发多个 state 更新时，用 `dispatch({ type: 'INCREMENT_BY', amount: 3 })` 替代多个 `setState`。Reducer 的更新是原子的，不存在"部分应用"的问题。
- **引入状态机**：对于复杂的状态转换序列（多步骤表单、支付流程），使用 XState (`decisions.md` 第二节) 确保状态转换的合法性。
- **API 请求管理分层**：将竞态处理（AbortController）封装到数据获取层，组件不直接管理请求生命周期。

**流程防护**：
- Code Review 检查项："是否有连续的 setState 调用？如果是，它们是否使用了函数式更新或应合并为 useReducer？"
- 架构评审：检查所有 useEffect + fetch 的竞态处理是否完备，是否有统一的请求取消机制。

### 跨层影响链

```
上游影响 (消费者):
  - 依赖该 state 的派生计算（useMemo）基于错误的中间值
  - 父组件传入的 setState callback 可能在批处理中被覆盖

下游影响 (依赖):
  - useEffect 在批处理中只触发一次，跳过了预期的中间状态
  - 日志/埋点记录的状态值不连续（跳过了中间值）
  - WebSocket 同步发送的是中间值而非最终值

部署影响:
  - React 17 → 18 升级行为变化：原本 setTimeout 中不批处理的 setState 在 React 18 中会被批处理
  - 需要团队级别的"从 React 17 迁移到 18"清单，包含批处理行为变更检查
  - A/B 测试中不同 React 版本的表现可能不一致

监控影响:
  - Sentry 中状态快照与预期不符 → 排查困难
  - 性能监控：useReducer vs 多个 useState 的渲染次数差异
  - E2E 测试：验证连续操作后的最终状态而非中间状态
```

### 相关架构模式
- **单向数据流** (`patterns.md` 第三节) — `state → view → action → state` 循环要求每个 action 产生确定性的状态变化，批处理保证了多 action 合并为单次 view 更新，但要求 action 本身不依赖中间读取。
- **XState 状态机** (`patterns.md` 第一节) — 多步骤状态转换天然需要状态机来管理，一次事件只触发一次状态转换，避免了批处理的不确定性。
- **状态管理选型 — useReducer** (`decisions.md` 第二节) — 当一次交互产生多个 state 变化时，应使用 useReducer 将更新建模为单个 dispatch，消除 batch 不确定性。

## 预防

1. **函数式更新**：当新 state 依赖旧 state 时，始终使用 `setState(prev => ...)`。
2. **React 18 自动批处理**：了解升级后 setTimeout/Promise 中的 setState 也会被批处理。
3. **竞态条件**：快速变化的 state 触发异步操作时，务必使用 AbortController 或 ignore flag。
4. **代码搜索**：`grep -rn "set\w+(\w+ \+" src/` 查找直接使用 state 变量的 setState 模式。
