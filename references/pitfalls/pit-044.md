# pit-044: 前端监控盲区

## 环境
- 任何生产环境的前端应用
- 使用 Sentry / LogRocket / Datadog RUM / 自建监控

## 症状
生产环境出了问题，但监控大盘一片绿色。用户群里在投诉"页面加载不出来""按钮点了没反应"，但 Dashboard 上的 JS 错误率是 0%、API 成功率是 99.8%。更糟的是：某个页面白屏三天，直到老板自己遇到才被发现。

根本原因是监控"看起来"有，实则全是盲区：
- 只捕获了 `window.onerror`，但 `Promise.reject` 未被处理
- 请求监控只看了 HTTP 200，没检查 response body 中的业务错误码 `{ code: -1 }`
- 没有性能监控，不知道 First Contentful Paint 从 1.2s 退化到 4.5s
- 告警无降级策略——一次偶发错误触发了 5000 条告警，团队直接把告警渠道静音了

## 根因
监控盲区的根因不是"没装 SDK"，而是**监控策略没有覆盖前端可观测性的四个维度**：

1. **错误监控**：`window.onerror` 只能捕获同步错误。`Promise.reject('error')`、`nextTick` 中的错误、`requestAnimationFrame` 中的错误、跨域脚本的 `Script error`（CORS 未配置）——都是盲区。

2. **性能监控**：只看了首屏加载（LCP），没看交互延迟（INP）、累积布局偏移（CLS）、路由切换性能、长任务（Long Task API）。

3. **行为监控**：只有错误抓取，没有用户行为回放——拿到一个 JS Error 但不知道用户做了什么操作导致的。

4. **告警降级**：所有错误都发告警 = 没有告警。需要按严重性、频率、影响面分级。

## 诊断方法

```bash
# 1. 检查全局错误捕获覆盖率
grep -rn "window\.onerror\|window\.addEventListener.*error\|unhandledrejection" src/

# 2. 检查是否有 Script error 吞没
# 在 Sentry 中搜索:  message:"Script error" 
# 这通常意味着跨域脚本错误被浏览器屏蔽

# 3. 检查是否有 Promise rejection 未处理
# Node.js 环境
node --unhandled-rejections=strict app.js

# 在代码中添加:
process.on('unhandledRejection', (reason) => {
  console.error('UNHANDLED:', reason);
});

# 4. 审计监控 SDK 配置
grep -rn "Sentry\.init\|datadogRum\.init\|monitor\.init" src/

# 5. 检查跨域脚本的 crossorigin 属性
grep -rn "<script" public/ --include="*.html" | grep -v "crossorigin"

# 6. Web Vitals 集成检查
grep -rn "web-vitals\|onLCP\|onINP\|onCLS\|onFCP" src/
```

## 修复

// ❌ 问题 1: 只捕获同步错误
-window.onerror = (msg, url, line, col, error) => {
-  Sentry.captureException(error);
-};

+// ✅ 完整的错误捕获体系
+// 1. 同步错误
+window.addEventListener('error', (event) => {
+  Sentry.captureException(event.error || event.message, {
+    tags: { type: 'window.onerror' }
+  });
+});
+
+// 2. Promise rejection
+window.addEventListener('unhandledrejection', (event) => {
+  Sentry.captureException(event.reason, {
+    tags: { type: 'unhandledRejection' },
+    extra: { promise: event.promise?.toString?.() }
+  });
+});

+// React: ErrorBoundary 组件
+// Vue: app.config.errorHandler
+// Vue 3
+app.config.errorHandler = (err, instance, info) => {
+  Sentry.captureException(err, {
+    tags: { framework: 'vue' },
+    extra: { componentInfo: info }
+  });
+};

// ❌ 问题 2: 无性能监控
+// ✅ Web Vitals 监控
+import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';
+function sendToAnalytics({ name, delta, id, rating }) {
+  // 发送到 Datadog / Grafana / 自定义后端
+  if (rating === 'poor') {
+    // P0 告警
+  }
+}
+onCLS(sendToAnalytics);
+onINP(sendToAnalytics);
+onLCP(sendToAnalytics);
+onFCP(sendToAnalytics);
+onTTFB(sendToAnalytics);
+
+// ❌ 问题 3: 无 API 响应内容校验
-// 只检查 HTTP status
-if (res.ok) return res.json();

+// ✅ 检查业务错误码
+const data = await res.json();
+if (data.code !== 0 && data.code !== 200) {
+  Sentry.captureMessage(`API business error: ${data.message}`, {
+    level: 'warning',
+    tags: { api: url, errorCode: data.code },
+    extra: { response: data }
+  });
+  throw new ApiError(data.code, data.message);
+}

+// ❌ 问题 4: 告警无降级 → 告警疲劳
+// ✅ 告警分级策略
+// P0: 核心功能不可用（白屏、登录失败率 > 5%）→ 电话/短信
+// P1: 非核心功能错误率 > 2% → 即时通讯
+// P2: 性能退化（LCP > 4s）→ 创建工单
+// P3: 偶发错误 < 0.1% → 日报汇总
+
+// Sentry 中配置 rate limiting
+Sentry.init({
+  sampleRate: 1.0,          // 100% 错误捕获
+  tracesSampleRate: 0.1,    // 10% 性能追踪
+  replaysSessionSampleRate: 0.1,  // 10% session 回放
+  replaysOnErrorSampleRate: 1.0,  // 100% 错误回放
+  beforeSend(event) {
+    // 过滤已知的无价值错误
+    const ignoredErrors = [
+      'ResizeObserver loop limit exceeded',
+      'Non-Error promise rejection captured with value: undefined',
+    ];
+    if (ignoredErrors.some(msg => event.message?.includes(msg))) {
+      return null;
+    }
+    return event;
+  }
+});

## 原理
`window.onerror` 的局限：
- 只能捕获到 "error" 事件（同步 throw）
- `Promise.reject()` 触发的是 "unhandledrejection" 事件
- 跨域脚本抛出的错误会被浏览器 sanitize 为 "Script error"（安全策略），除非 script 标签添加 `crossorigin="anonymous"` 且 CDN 返回 CORS 头

Web Vitals 是 Google 定义的前端性能核心指标：
- **LCP** (Largest Contentful Paint): 最大内容绘制时间，衡量加载性能
- **INP** (Interaction to Next Paint): 交互延迟，衡量响应性
- **CLS** (Cumulative Layout Shift): 累积布局偏移，衡量视觉稳定性

## 架构视角 (Architecture Perspective)

### 架构层面根因
监控盲区的架构根因是**可观测性（Observability）没有被作为架构质量属性来设计**：
- **监控被视为"运维的事"而非"架构的事"**：架构评审中讨论了组件拆分、数据流、缓存策略，但没有讨论"当这个组件出错时如何被观测到"。可观测性应该像安全性一样，在设计阶段就纳入每个模块的契约中。
- **前端缺少"I/O 模型"的错误分类法**：后端的错误天然分层（网络/数据库/业务逻辑/内存），而前端的错误通常被笼统地分为"JS Error"和"Network Error"。实际上前端 I/O 包括：用户输入（不可预测）、API 响应（网络波动）、浏览器 API（兼容性）、第三方脚本（不可控）——每种 I/O 类型都需要各自的观测策略。
- **告警策略是"追加"的而非"设计"的**：告警规则随着时间累积——产品说"这个接口很重要加个告警"→ DevOps 加上——几个月后告警数量让所有人都麻木了。告警需要基于影响面（blast radius）和频率（frequency）的二维矩阵进行系统设计。

### 系统级预防策略

**静态检查**：
```bash
# 检查 crossorigin 属性
grep -rn "<script" public/ --include="*.html" | grep -v "crossorigin" && echo "WARNING: 跨域脚本缺少 crossorigin 属性"

# 检查 web-vitals 集成
grep -rn "web-vitals\|onCLS\|onINP\|onLCP\|onFCP" src/ || echo "WARNING: 未集成 Web Vitals 监控"

# 检查 Promise rejection 处理
grep -rn "unhandledrejection" src/ || echo "WARNING: 未处理 unhandledrejection"
```

**CI 门禁**：
```bash
# E2E 中验证监控数据上报
test('API 业务错误被上报', async ({ page }) => {
  const sentryRequests: any[] = [];
  page.on('request', req => {
    if (req.url().includes('sentry.io')) sentryRequests.push(req);
  });
  await page.goto('/page-with-business-error');
  // 应该上报了包含业务错误码的 Sentry 事件
  expect(sentryRequests.length).toBeGreaterThan(0);
});

# Web Vitals 不应劣化
test('LCP 在阈值内', async ({ page }) => {
  const lcp = await page.evaluate(() => {
    return new Promise(resolve => {
      new PerformanceObserver(list => {
        resolve(list.getEntries()[0]?.startTime || 0);
      }).observe({ type: 'largest-contentful-paint', buffered: true });
    });
  });
  expect(lcp).toBeLessThan(2500); // < 2.5s
});
```

**架构约束**：
- 每个模块必须定义其"可观测性接口"：暴露的错误类型、关键性能指标、关键用户行为事件
- 错误必须有分级 (severity)：P0(blocking) / P1(degraded) / P2(cosmetic) / P3(background)
- 所有 API 调用必须统一经过带业务错误码检查的 wrapper（而非裸 fetch）
- 告警规则必须定义"静默期"和"聚合窗口"

**流程约束**：
- 架构评审模板包含"可观测性"章节：该模块的错误分级、关键指标、告警策略
- 告警规则变更必须通过 PR Review（像代码一样管理告警规则）
- 每季度进行"告警有效回放"——统计告警中被实际处理的 P0/P1 比例

### 跨层影响链

```
上游影响 (用户体验):
  - 错误被吞 → 用户遇到白屏 → 无法反馈（连错误提示都没有）
  - 性能退化未被发现 → 用户流失 → 收入下降

下游影响 (后端):
  - 前端不报业务错误码 → 后端不知道前端遇到了哪些问题
  - API 响应校验失败不告警 → 后端改了接口格式导致前端白屏而无人知晓

部署影响:
  - Sentry release 与部署版本关联 → 快速定位"新版本引入的错误"
  - 监控 SDK 本身崩溃导致雪崩 → 需要降级方案

监控影响:
  - 监控系统的"自监控"：Sentry 自身是否正常接收数据
  - 采样率调整（错误量突然暴增 100 倍 → Sentry 配额耗尽）
```

### 相关架构模式
- **Telemetry as Code** (`patterns.md` 第十八节) — 将监控配置（错误分类、告警规则、Dashboard）作为代码管理在 repo 中。
- **Circuit Breaker 在前端** (`decisions.md` 第十七节) — 当监控检测到 API 5xx 率超过阈值时，前端自动降级（展示缓存数据、禁用非核心功能）。

## 预防
1. 错误捕获三件套：`window.onerror` + `unhandledrejection` + 框架 `errorHandler`
2. 集成 `web-vitals` 库并上报 LCP/INP/CLS/FCP/TTFB
3. 所有 API 响应统一检查业务错误码，不只看 HTTP status
4. 告警分级（P0~P4）并配置降噪策略（rate limit + dedup）
5. 跨域 `<script>` 标签必须添加 `crossorigin="anonymous"` 属性
