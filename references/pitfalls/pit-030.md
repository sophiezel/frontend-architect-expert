# pit-030: Vue 响应式丢失

## 环境
- Vue 3.0+（Composition API）
- 使用 `reactive()` / `ref()` / `shallowRef()` 的项目
- TypeScript 或 JavaScript

## 症状
数据已更新但视图不刷新，DevTools 中数据确实变了但模板无反应。使用 `reactive()` 定义的对象，解构后修改属性不触发渲染。使用 `ref()` 时忘记 `.value`，导致 `.value` 被当作普通属性传递。使用 `shallowRef()` 深层属性修改无响应。

表现为：表单提交了但列表没刷新、页面加载后数据是旧的、watch/watchEffect 不触发。

## 根因
Vue 3 响应式系统基于 Proxy（`reactive`）和 getter/setter（`ref`）。核心陷阱：

1. **reactive() 解构**：`const { name } = reactive({ name: 'foo' })` 将响应式 Proxy 对象的属性提取为普通值，切断与响应式系统的连接。Proxy 拦截只能捕获 `obj.name` 的访问，无法追踪解构后变量 `name` 的读写。

2. **ref .value 遗漏**：`<script setup>` 的编译期语法糖在模板中自动解包 ref，但在 JS/TS 代码中 ref 始终是 `{ value: T }` 包装对象，忘记 `.value` 相当于只读/写了包装对象本身。

3. **shallowRef 误用**：`shallowRef` 只追踪 `.value` 的替换，不追踪深层属性，适合大对象性能优化但容易误用。

```bash

# 观察组件重渲染次数和触发源
```

## 修复

```diff
// ❌ 问题 1: reactive() 解构破坏响应性
function useUser() {
  const state = reactive({ name: 'Alice', age: 30 });
-  return { ...state }; // 展开后全部丢失响应性
+  return toRefs(state); // 每个属性变成独立 ref，保持响应性
}

// 或者直接返回原对象，在模板中用 state.name
function useUser() {
  const state = reactive({ name: 'Alice', age: 30 });
  return { state }; // 模板中用 state.name
}

// ❌ 问题 2: ref 忘记 .value
const count = ref(0);
-function increment() { count++; } // 错误：自增的是 RefImpl 对象
+function increment() { count.value++; }

// 在 watch 中也常见
-watch(count, (v) => console.log(v)); // v 是 RefImpl 对象
+watch(count, (v) => console.log(v)); // 正确：count 作为 ref 传入，v 自动解包
+watch(() => count.value, (v) => console.log(v)); // 或显式 getter

// ❌ 问题 3: shallowRef 深层更新无效
const list = shallowRef([{ id: 1, name: 'A' }]);
-list.value[0].name = 'B'; // 不触发更新
+list.value = list.value.map(item => 
+  item.id === 1 ? { ...item, name: 'B' } : item
+); // 替换整个 .value

// 或者使用 triggerRef 手动触发
import { triggerRef } from 'vue';
list.value[0].name = 'B';
triggerRef(list);
```

## 验证

```bash
# 1. 单元测试：验证响应性是否保持
# __tests__/reactive.test.ts
import { reactive, toRefs, isReactive, isRef } from 'vue';

test('toRefs 保持响应性', () => {
  const state = reactive({ count: 0 });
  const refs = toRefs(state);
  expect(isRef(refs.count)).toBe(true);
  refs.count.value = 5;
  expect(state.count).toBe(5); // 同步回原始对象
});

test('解构破坏响应性', () => {
  const state = reactive({ count: 0 });
  const { count } = state;
  expect(isReactive(count)).toBe(false);
});

# 2. E2E: Cypress/Playwright 中操作后截图对比
# 确认 UI 状态与数据一致

# 3. 手动验证清单
# - 修改 reactive 对象属性 → UI 刷新 √
# - 修改 ref.value → UI 刷新 √
# - toRefs 后的属性修改 → 原始对象同步 √
# - shallowRef 深层修改 + triggerRef → UI 刷新 √
```

## 原理
Vue 3 响应式系统将 `reactive()` 返回的对象包装为 Proxy，拦截 `get/set/deleteProperty` 等操作。当模板或计算属性访问 `obj.prop` 时，Proxy 的 get 拦截器记录依赖（track）；修改 `obj.prop` 时，set 拦截器触发更新（trigger）。

解构赋值 `const { name } = obj` 等价于 `const name = obj.name`，只触发一次 get 拦截，之后 `name` 是原始值的副本，与响应式系统断开连接。

`ref()` 内部创建 `{ value: T }` 对象，`.value` 的 getter 触发 track，setter 触发 trigger。模板编译器自动在模板中展开 `.value`，但 JS 代码中必须显式访问。

## 架构视角 (Architecture Perspective)

### 架构层面根因
响应式丢失的架构根因是**Composition API 的边界契约未在设计系统中明确定义**：
- **函数返回值契约缺失**：`useXxx()` composable 返回什么类型——是 `Reactive<T>` 还是 `ToRefs<T>` 还是裸值？没有统一的 type-level 约束，每个 composable 自行决定，调用方不知道是否可以直接解构。
- **响应式"传染性"未被建模**：一旦某个中间环节丢失了响应式引用（例如 API 层返回普通对象），下游所有依赖该数据的 watcher/computed/组件全部失效。响应式的"源头"与"消费端"之间的链路没有类型安全保障。
- **模板编译器的"魔法"造成认知断层**：`<script setup>` 中模板自动解包 ref，让开发者误以为 ref 在任何上下文中都是"透明"的。这层编译期魔法跨越了 JS 运行时的真实语义，形成心智模型断层。

### 系统级预防策略

**静态防护**：
```bash
# ESLint 规则：强制 ref 使用 .value
# .eslintrc.js
rules: {
  'no-restricted-syntax': ['error', {
    selector: 'AssignmentExpression[left.type="Identifier"][operator="="] > Identifier[name=/ref$/i]',
    message: 'ref 变量赋值请使用 .value'
  }]
}

# TypeScript：响应式安全的类型工具
type DeepReadonly<T> = { readonly [K in keyof T]: DeepReadonly<T[K]> };
type RequireReactive<T> = T extends Ref<infer V> ? Ref<V> : never;

// composable 返回值类型检查
function useUser(): { name: Ref<string>; age: Ref<number> } { ... }
```

**CI 门禁**：
```bash
# 在 CI 中运行响应性审计脚本
# scripts/audit-reactivity.sh
echo "=== 检查 reactive 解构 ==="
grep -rn "const\s*{.*}\s*=\s*reactive(" src/ --include="*.ts" --include="*.vue"
echo "=== 检查潜在 ref .value 缺失 ==="
grep -rn "const\s+\w+\s*=\s*ref(" src/ --include="*.ts" | while read l; do
  var_name=$(echo "$l" | sed 's/.*const \(.*\) = ref(.*/\1/')
  # 检查同一文件中是否有 .value 访问
  ...
done
```

**架构约束**：
```typescript
// src/composables/types.ts — 统一 composable 返回值规范
export type ComposableReturn<T extends Record<string, unknown>> = {
  [K in keyof T]: T[K] extends object ? ToRef<T[K]> : Ref<T[K]>;
};

// 使用约定
function useUser(): ComposableReturn<{ name: string; age: number }> {
  const state = reactive({ name: 'Alice', age: 30 });
  return toRefs(state) as any;
}
```

**流程约束**：
- Code Review 清单："composable 返回值是否全部通过 toRefs 导出？"
- 如果 composable 返回 reactive 对象，必须在 JSDoc 中标注"不可解构，请使用 .value 或 toRefs"
- 新增 composable 时要求同时提交响应性单元测试

### 跨层影响链

```
上游影响 (组件层):
  - 模板渲染冻结 → 用户看到过期数据 → 业务操作基于旧状态（危险）
  - watch/watchEffect 失效 → 表单联动、搜索建议等交互链断裂

下游影响 (数据层):
  - API 响应通过解构赋值后失去响应性 → 所有消费组件无法感知数据更新
  - Pinia store 导出普通对象 → 全局状态改变后视图不刷新

部署影响:
  - 生产环境 SSR hydration 后客户端接管时数据不一致
  - 此类 bug 在 dev 环境可能偶现（HMR 刷新掩盖），生产环境持续触发

监控影响:
  - 难以通过 JS Error 捕获（无异常抛出，仅是"数据与 UI 不同步"）
  - 需要自定义 "响应性断链" 检测：对比 Vue DevTools 中的 state 快照与 DOM 内容
  - 用户行为回放录像中表现为"操作了但界面无反应"
```

### 相关架构模式
- **Reactive Boundary 模式** (`patterns.md` 第四节) — 在应用层定义明确的"响应式边界"，边界之外（API 响应、第三方库数据、WebSocket 消息）的普通对象在进入边界时统一用 `reactive()` 或 `ref()` 包装。
- **单向数据流在 Vue 中的实现** (`decisions.md` 第五节) — Vue 3 的事件系统和 provide/inject 可以与响应式系统结合实现安全的单向数据流，但前提是每一层都保持响应性引用。

## 预防
1. composable 统一使用 `toRefs()` 导出，禁止返回裸 reactive 对象
2. 用 `unplugin-auto-import` 自动导入 `ref/reactive/toRefs` 避免遗漏
3. 在 TS 中给 composable 定义严格的返回类型，利用类型系统保证响应性
4. 团队约定：`<script setup>` 之外的 JS/TS 文件中，ref 必须以 `.value` 访问——ESLint 规则强制执行
