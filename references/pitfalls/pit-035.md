# pit-035: Vite 开发环境与生产构建不一致

## 环境
- Vite 4.x / 5.x
- 使用 `process.env` 或 Node.js 内置模块的前端代码
- 依赖 esbuild/Rollup 差异的场景

## 症状
开发环境 (vite dev) 全部功能正常，发布到生产 (vite build) 后出现白屏、API 调用失败、第三方库报错。或者反过来——本地 dev 报错但 build 正常。最隐蔽的情况：功能正常但一些边界行为不一致（如条件分支走错、动态 import 路径解析不同）。

典型场景：
- 代码中使用了 `process.env.MY_VAR`，dev 模式下通过 Vite 的 `define` 替换正常工作，但生产构建中 `process` 未定义导致 `ReferenceError`
- `import.meta.url` 在 dev 返回源码路径，在 build 后返回打包后的路径
- CSS 的 `@import url()` 在 dev 中内联，在 build 中被提取为独立文件，加载顺序不同

## 根因
Vite 使用了双引擎架构：
- **Dev**: esbuild（Go 编写，超快）处理 TS/CSS/JSX 转换，只做预构建不做打包
- **Build**: Rollup（JavaScript 编写）做完整的 tree-shaking、chunk splitting、产物优化

esbuild 和 Rollup 在以下方面存在差异：
1. `define` 替换：esbuild 做字符串替换，Rollup 做 AST 级别的替换，对 `typeof process` 等特殊模式处理不同
2. Tree-shaking：Rollup 更激进，可能把 dev 中有效的副作用代码摇掉
3. CommonJS 处理：esbuild 宽松兼容，Rollup 严格转换
4. 动态 import 的路径解析算法不同

## 诊断方法

```bash
# 1. 对比 dev 和 build 产物的差异
vite build --mode development  # 以 dev 模式构建（关闭压缩）
diff -r dist/ node_modules/.vite/deps/  # 对比依赖预构建

# 2. 检查是否直接使用了 process.env
grep -rn "process\.env" src/ --include="*.ts" --include="*.vue" --include="*.js"

# 3. 检查 Node.js 内置模块引用
grep -rn "require\s*\(\s*['\"]\(fs\|path\|os\|crypto\|http\)" src/ --include="*.ts" --include="*.js"

# 4. 对比 dev/build 的 define 配置
# 在 vite.config.ts 中临时添加
console.log('define:', JSON.stringify(config.define));

# 5. 检查动态 import 路径
grep -rn "import\s*(" src/ --include="*.ts" --include="*.vue"
```

## 修复

```diff
// ❌ 问题 1: 使用 process.env
-const apiUrl = process.env.VITE_API_URL;
-const isDev = process.env.NODE_ENV === 'development';

+const apiUrl = import.meta.env.VITE_API_URL;
+const isDev = import.meta.env.DEV; // Vite 内置

// ❌ 问题 2: 期望 Node.js 模块可用（浏览器端）
-import path from 'path';
-const fullPath = path.join(baseUrl, segment);

+// 浏览器端使用 URL 或手写路径拼接
+const fullPath = `${baseUrl.replace(/\/$/, '')}/${segment.replace(/^\//, '')}`;

+// 或仅在 Node.js 端使用（需要 Vite 配置 external）
+// vite.config.ts
+define: { 'process.env': {} }, // 为浏览器提供空的 process.env

// ❌ 问题 3: 依赖开发/生产行为差异的代码
-if (process.env.NODE_ENV === 'development') {
-  // dev 专用的代码路径
-  return mockData;
-}

+// 使用 import.meta.env.DEV（Vite 明确支持，两种模式一致）
+if (import.meta.env.DEV) {
+  return mockData;
+}

+// 或者更好：将 mock 数据分离为独立模块，用 Vite 的 conditional config
+// vite.config.ts: 仅在 dev 中注入 mock 插件

// ❌ 问题 4: 动态 import 路径使用变量（dev/build 解析不同）
-const moduleName = 'views/Home';
-import(`./${moduleName}.vue`); // esbuild 乐观解析，Rollup 可能报 warning

+// 使用明确的相对路径 + @rollup/plugin-dynamic-import-vars 限制
+const modules = import.meta.glob('./views/*.vue');
+const component = modules['./views/Home.vue'];
```

## 验证

```bash
# 1. 本地 build + preview 验证（非 dev 模式）
vite build && vite preview
# 在 preview 模式下完整走一遍业务链路

# 2. 产物分析：检查是否有 process 残留
grep -r "process\." dist/assets/ 2>/dev/null && echo "WARNING: process 残留" || echo "PASS"

# 3. 自动化 E2E 覆盖
# Playwright 同时在 dev 和 production 模式下运行
npx playwright test --project=chromium --reporter=html
# 截图对比（dev vs build）

# 4. 检查 bundle 中是否包含意外的 Node.js polyfill

# 如果某个 chunk 异常大（>1MB），可能包含了 polyfill
```

## 原理
Vite 的双引擎设计出于以下考虑：
- esbuild 的速度（Go 语言，比 JS 快 10-100 倍）使其成为 dev server 的最佳选择
- Rollup 的成熟生态和 tree-shaking 能力使其成为生产构建的标准

但双引擎意味着两套不同的解析/转换/优化流程。esbuild 在 dev 模式中慷慨地处理各种语法（宽松模式），而 Rollup 在生产构建中严格遵守标准。例如：esbuild 会将 `typeof window` 保留原样，但 Rollup 的 tree-shaking 可能将其视为 dead code 移除（如果代码中其他地方没有 window 引用）。

`import.meta.env` 是 Vite 专门为解决双引擎一致性问题设计的 API——Vite 确保在 esbuild 和 Rollup 中对 `import.meta.env.*` 的处理行为一致（通过静态替换为字面量值）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
开发/生产环境不一致的架构根因是**构建工具抽象层缺失**：
- **环境变量被当作运行时配置而非编译时常量**：`process.env` 来自 Node.js 的运行时模型，但在前端构建中它是在编译期被替换为字符串字面量（`define` 替换）。开发者混淆了"运行时环境变量"和"编译时常量定义"两种语义。架构需要明确区分两者并提供统一的访问接口。
- **双引擎带来了"构建不确定性"**：同一份源码经过两个不同的编译器产生不同结果，这是"编译不可重复性"（non-reproducible build）。架构应尽可能缩小 dev/build 的差异——要么统一引擎（如用 esbuild 做生产构建），要么在 CI 中使用与生产相同的 Rollup 配置运行 dev。
- **浏览器/Node.js 环境边界模糊**：前端代码和构建配置混在同一个仓库中，构建配置中可以引用 Node.js 模块（path/fs），但如果在源码中也引用了这些模块且未被 externalize，就会在浏览器端爆炸。

### 系统级预防策略

**静态检查**：
```bash
# ESLint 规则：禁止 process.env
rules: {
  'no-restricted-globals': ['error', {
    name: 'process',
    message: '请使用 import.meta.env 代替 process.env（Vite 专用）'
  }],
  'no-restricted-imports': ['error', {
    paths: [
      { name: 'fs', message: 'Node.js 模块不可在浏览器代码中使用' },
      { name: 'path', message: 'Node.js 模块不可在浏览器代码中使用' },
      { name: 'crypto', message: '请使用 Web Crypto API (crypto.subtle)' },
    ]
  }]
}
```

**CI 门禁**：
```bash
# CI pipeline 中对比 dev 和 production 构建
# .github/workflows/check.yml
- name: Build (production mode)
  run: vite build
- name: Build (dev mode)  
  run: vite build --mode development
- name: Check process.env residues
  run: |
    if grep -rq "process\." dist/assets/; then
      echo "ERROR: process.* found in production build"
      exit 1
    fi
- name: E2E on production build
  run: vite preview & npx playwright test

# Bundle diff 检测
- name: Bundle size diff (dev vs prod)
  run: npx vite-bundle-visualizer --compare
```

**架构约束**：
- 环境变量统一通过 `src/env.ts` 访问，该文件是项目中唯一引用 `import.meta.env` 的地方
- 浏览器端代码禁止 import Node.js 内置模块（通过 ESLint `no-restricted-imports`）
- 需要在 Node.js 端运行的代码放在 `scripts/` 或 `server/` 目录，前端代码放在 `src/`，两者不允许互引

**流程约束**：
- 发布前必须在本地执行 `vite build && vite preview` 验证
- 每个 PR 的 CI 必须跑 production build + E2E

### 跨层影响链

```
上游影响 (开发体验):
  - dev 正常但 build 失败 → 开发者对构建工具失去信任
  - dev 和 build 行为不一致 → 调试方向错误（在 dev 中找问题）

下游影响 (用户):
  - 生产环境白屏 → 直接用户流失
  - 边界行为不一致 → 难以复现的幽灵 bug（部分用户触发特定条件）

部署影响:
  - CDN 部署后才发现问题 → 紧急回滚 → 发布流程被破坏
  - Hash 文件名变更但 HTML 未刷新 → 缓存问题

监控影响:
  - 需要同时监控 dev 和 production 两个维度的错误
  - 生产特有的 JS Error 往往与构建配置相关，需要记录构建 commit hash
```

### 相关架构模式
- **十二要素应用** (`patterns.md` 第十一节) — 配置与代码分离原则：编译时常量、运行时常量、功能开关分别管理。
- **Build Reproducibility** (`decisions.md` 第八节) — 使用 Nix/docker 锁定构建环境，确保 CI 构建与本地构建一致。

## 预防
1. 严格使用 `import.meta.env.*` 替代所有 `process.env`
2. CI 中强制执行 production build + preview 验证
3. 在 `vite.config.ts` 中显式定义所有 `define` 常量（而非依赖 implicit 行为）
4. 使用 `vite-plugin-checker` 在 dev 时运行 TypeScript 检查，尽早发现类型错误

## 回退
```bash
# 若 build 失败，回退到上一次成功的 commit
git log --oneline -5
git revert <broken-commit>

# 紧急方案：如果某个依赖在 build 时报错，尝试 externalize

build: {
  rollupOptions: {
    external: ['problematic-package'],
  }
}
```
