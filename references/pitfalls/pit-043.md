# pit-043: 状态管理架构选错

## 环境
- React / Vue 中大型单页应用
- 使用 Redux / Vuex / Pinia / Zustand / MobX / React Context 等
- 同时处理服务端数据（API 响应）和客户端状态（UI 状态）

## 症状
Redux store 中一半的数据是表单临时输入——每次按键都 dispatch action 更新 store → 性能极差。用 React Context 做全局状态管理——Provider 值变了整个组件树重渲染，一个用户头像更新导致整个 Dashboard 页面卡顿。

更隐蔽的问题：API 请求的 loading/error/data 状态在 store 中手写管理——每个 API 都要写 `FETCH_USERS_REQUEST / FETCH_USERS_SUCCESS / FETCH_USERS_FAILURE` 三个 action 类型的样板代码。服务端状态和客户端状态混在同一个 store 中，缓存失效策略互相干扰。

## 根因
核心问题是**状态被当作单一类别处理，没有按生命周期的维度分类**。前端状态实际上是四大类：

| 类型 | 生命周期 | 示例 | 合适方案 |
|------|---------|------|---------|
| 服务端状态 | 与服务端同步 | 用户列表、订单详情 | React Query / SWR / Pinia Colada |
| 客户端状态 | 会话/页面级 | 表单输入、UI 开关 | useState / Pinia / Zustand |
| 全局状态 | 应用级 | 当前用户、权限配置 | Context + reducer / Pinia |
| URL 状态 | 浏览器 URL | 搜索词、分页、筛选 | useSearchParams / router |

当用单一方案（如 Redux）管理所有类型时，就会出现：
- 服务端状态的"缓存失效/重试/后台刷新"逻辑手写（Redux 不提供）
- 表单临时状态经过 Redux middleware（logger/persist）造成不必要的序列化和副作用

## 原理
状态管理的本质是**"数据的生命周期管理"**：

- **服务端状态**的真相源在服务端，前端只是缓存。它的生命周期由"何时过期(staleTime)、何时重新获取(refetchInterval)、失败重试次数"定义。React Query/SWR 天然支持这些生命周期管理。
- **客户端状态**的真相源在前端，生命周期由"组件挂载到卸载"或"用户会话"定义。useState/useReducer/Pinia 适合管理这类状态。
- **URL 状态**的真相源在浏览器地址栏——搜索词、分页、模态框状态。应该通过 URL 管理（Vue Router/React Router），而非 store。

Redux 的原始设计灵感来自 Flux 架构和 Elm 语言——适合管理应用级的客户端状态（如购物车、多步骤表单向导）。用于服务端数据管理则是反模式，因为它需要手动编写缓存失效、去重、后台刷新等逻辑。

## 架构视角 (Architecture Perspective)

### 架构层面根因
状态管理选错的架构根因是**状态建模的维度缺失**——生命周期、所有权、一致性的三维分析框架未建立：
- **生命周期维度被忽略**：状态被看作"变量"而非"具有生命周期的实体"。服务端状态的 TTL、表单状态的"未提交/已提交"阶段、URL 状态的"前进/后退"行为——这些都是生命周期属性，但统一放在 Redux store 中后，这些差异被抹平为"一个 action 就是一个状态转换"。
- **所有权维度模糊**：数据是被服务端拥有的（server owns the data），还是被用户拥有的（client owns the data），还是被 URL 拥有的？所有权决定了缓存策略和冲突解决策略。当多个所有者混在同一 store 中时，更新冲突无解。
- **一致性维度不可见**：服务端状态需要"最终一致性"（允许短暂过期，后台刷新），客户端状态需要"强一致性"（立即反映），表单状态需要"临时一致性"（提交前允许不一致）。这些不同的 consistency model 被 Redux 的同步 dispatch 全部统一为强一致性——牺牲了性能和用户体验。

**静态检查**：

**架构约束**：
```typescript
// 状态分类决策矩阵（写入 ARCHITECTURE.md）
// | 问题 | 服务端状态 | 客户端状态 | URL 状态 |
// |------|----------|----------|---------|
// | 真相源在哪？ | 服务端 | 浏览器内存 | 地址栏 |
// | 用什么管理？ | React Query/SWR | Pinia/Zustand | Router |
// | 缓存策略？ | staleTime + refetch | 不需要 | share via URL |
// | 示例 | 用户列表、订单 | 主题、表单草稿 | 搜索词、分页 |

// 禁止模式：
// - store 中定义 FETCH_* action
// - Context 中放频繁变化的值
// - 表单状态放入 store（提交前的草稿除外）
```

**流程约束**：
- 每个新增状态变量必须在 PR 中标注"生命周期分类"（server/client/url）
- 新增 API 调用时默认使用 React Query/SWR，手动写入 store 需要额外审批
- 季度审计：统计 store 中服务端状态的占比，超过 30% 触发迁移计划

### 跨层影响链

```
上游影响 (开发效率):
  - 手写 loading/error 样板代码 → 每个 API 多 30-50 行
  - Context 全局重渲染 → 性能问题 → 大量 useMemo/useCallback 补丁

下游影响 (用户体验):
  - 服务端数据缓存失效不当 → 看到过期数据 → 操作基于错误信息
  - 表单状态在全局 store → 组件卸载后状态残留 → 重新打开表单看到上次的输入

部署影响:
  - Redux 中间件（如 redux-persist）将服务端数据持久化到 localStorage
  → 用户看到几天前的旧数据 → 以为功能挂了

监控影响:
  - Redux DevTools 中 action 数量爆炸（表单按键也触发 action）
  - 按 action 类型统计频率，区分 server/UI/URL 三类
```

### 相关架构模式
- **CQRS 前端化** (`patterns.md` 第三节) — 将"读服务端状态"(React Query)和"写服务端状态"(form + mutation)分离。
- **Lifting State Up 的反模式** (`decisions.md` 第十六节) — Context 的 Provider 应该在组件树中尽可能低的位置，而非包裹整个应用。

## 预防
1. 状态四分类法（server/client/global/URL）写入团队编码规范
2. 新增 API 调用默认使用 React Query / SWR / Pinia Colada
3. Context 读写分离原则：值 Context 和 Dispatch Context 独立
4. ESLint 规则禁止在 store 中手写 `FETCH_*_REQUEST/SUCCESS/FAILURE`
