# pit-041: 微前端架构陷阱

## 环境
- qiankun 2.x / wujie / micro-app / Module Federation
- 多团队协作的大型前端项目
- 主应用 + 多个子应用的架构

## 症状
子应用之间样式互相污染——子应用 A 的按钮样式影响了子应用 B。公共依赖（如 `vue`、`element-plus`）被多个子应用各自打包，导致页面加载 3 份相同的 Vue 运行时。子应用间通信变成"先找主应用中转，主应用再广播"的混乱链路。某个子应用 JS 报错后，整个主应用白屏（沙箱未隔离）。

典型场景：
- 子应用 A 使用 `ant-design-vue` 4.x，子应用 B 使用 3.x → CSS 变量和 class 命名冲突
- qiankun 的 `strictStyleIsolation` 使用 Shadow DOM，但第三方组件（如弹窗、下拉菜单）挂载到 `document.body` 上，样式全部丢失
- 主应用路由 `/app1/*` 到子应用 A，但子应用内部又定义了 `/app1/settings`，和主应用的路由规则冲突

## 根因
1. **CSS 全局作用域的根本矛盾**：浏览器的 CSS 是全局的。无论 qiankun 的 `experimentalStyleIsolation`（给样式加前缀）还是 Shadow DOM（`strictStyleIsolation`），都无法完美解决第三方组件库挂载到 `document.body` 的问题。

2. **沙箱隔离的"漏桶"**：qiankun 的 JS 沙箱（Proxy 模式）拦截 `window` 对象的读写，但无法拦截 `document.body.appendChild()`、`history.pushState()`、`setInterval` 等全局副作用。

3. **共享依赖的版本对齐灾难**：Module Federation 的 `shared` 配置看似解决了版本共享，但当版本不兼容时（Vue 3.2 vs 3.4），`singleton: true` 会导致其中一个子应用拿到不兼容的版本而崩溃。

4. **通信的隐式耦合**：`initGlobalState` 本质是全局事件总线加一个状态快照，没有类型约束、没有版本管理——子应用 A 期望 `user` 有 `email` 字段，但子应用 B 只设置了 `{ name }`。

## 诊断方法

```bash
# 搜索是否有全局的 body/div/a 样式重置（微前端中这是定时炸弹）

# 2. 检测公共依赖重复加载
ls -la dist/ | grep -E "vue|react|antd|element"
# 或在浏览器 Network 面板中搜索 "vue" / "react" → 看加载了几份

# 3. 检测 document.body 操作
grep -rn "document\.body\." src/ --include="*.ts" --include="*.vue" --include="*.js"

# 4. 检测全局副作用
grep -rn "window\.\|global\.\|history\.\|setInterval\|setTimeout" src/ --include="*.ts" --include="*.js"

# 5. 检查微前端框架的沙箱配置
grep -rn "sandbox\|strictStyleIsolation\|experimentalStyleIsolation" main-app/
```

## 修复

```diff
// ❌ 问题 1: CSS 污染 — 全局样式影响子应用
// main-app/src/styles/global.css
-* { margin: 0; padding: 0; box-sizing: border-box; }
+/* ✅ 全局样式限定在主应用的容器内 */
+#main-app-root, #main-app-root * { margin: 0; padding: 0; box-sizing: border-box; }

// ❌ 问题 2: qiankun 配置 — 沙箱未正确配置
// main-app/src/micro.ts
registerMicroApps([
  {
    name: 'app1',
    entry: '//localhost:7101',
    container: '#subapp-container',
    activeRule: '/app1',
-    // 默认 sandbox: true（Proxy 沙箱，不够严格）
+    sandbox: {
+      experimentalStyleIsolation: true,  // 给样式加前缀隔离
+      // 注意：如果子应用使用 antd/element 等弹窗组件，
+      // 考虑 strictStyleIsolation: true (Shadow DOM) 但要处理 body append 问题
+    }
  }
]);

// ❌ 问题 3: 公共依赖未共享
// Module Federation 配置 (rspack/webpack)
new ModuleFederationPlugin({
  name: 'main',
+  shared: {
+    vue: {
+      singleton: true,
+      requiredVersion: '^3.4.0',
+      eager: true,  // 主应用立即加载，避免异步竞争
+    },
+    'vue-router': { singleton: true, requiredVersion: '^4.0.0' },
+    pinia: { singleton: true, requiredVersion: '^2.0.0' },
+  }
});

// ❌ 问题 4: 子应用间通信混乱
// main-app.store.ts
-import { initGlobalState } from 'qiankun';
-const actions = initGlobalState({ user: null });
-actions.onGlobalStateChange((state, prev) => { /* ... */ });

+// ✅ 用类型安全的共享事件系统
+// packages/shared-events/src/index.ts
+import { createEventBus, TypedEventBus } from './bus';
+interface AppEvents {
+  'user:login': { userId: string; name: string };
+  'user:logout': void;
+  'theme:change': { theme: 'light' | 'dark' };
+}
+export const appBus: TypedEventBus<AppEvents> = createEventBus();
+// 所有子应用通过共享库使用同一个事件总线实例

// ❌ 问题 5: 子应用挂载到 document.body 的组件
// 子应用中的 antd Modal/Drawer 默认 getContainer={() => document.body}
// 在 Shadow DOM 模式下这些组件不可见
+// ✅ 让弹窗挂载到子应用容器内
+// antd 4.x+
+import { ConfigProvider } from 'ant-design-vue';
+// 在子应用的根组件中
+<ConfigProvider getPopupContainer={() => document.getElementById('subapp-root')!}>
+  <router-view />
+</ConfigProvider>
```

## 原理
qiankun 的 JS 沙箱实现：
- **SnapshotSandbox**：激活时把 `window` 快照，失活时恢复。简单但对 `document.body` 等操作无法恢复。
- **ProxySandbox**：为每个子应用创建 `window` 的 Proxy 对象，拦截 set/get。但无法拦截 DOM API 的副作用（如 `appendChild`）。

CSS 隔离方案对比：
- **experimentalStyleIsolation**：遍历 CSS 规则，给每个选择器加前缀（`.app1 .btn`）。问题是无法处理动态创建的样式、CSS-in-JS 生成的类名。
- **strictStyleIsolation**：把子应用放入 Shadow DOM。隔离最彻底，但导致弹窗组件（挂载到 document.body）丢失样式。需要配置 `getContainer`。

## 架构视角 (Architecture Perspective)

### 架构层面根因
微前端陷阱的架构根因是**微前端被当作"团队解耦"的工具而非"技术解耦"的工程**：
- **CSS 是"全局第一公民"与微前端"隔离为第一原则"的根本冲突**：浏览器平台将 CSS 设计为全局的、级联的；微前端追求物理级别的隔离。这两种哲学在 Web 平台上没有干净的调和方案，只能通过工程约束（如 BEM 命名、CSS Modules、Shadow DOM）缩小冲突面。
- **共享依赖被当作"打包优化"而非"架构契约"**：Module Federation 的 shared 配置看似是 webpack 的构建选项，实际是团队间的运行时契约——主应用和 N 个子应用共同依赖同一个 vue 实例。当任一团队升级 vue 版本时，这个契约被打破。微前端项目缺少一个"共享依赖锁定机制"（类似 monorepo 的固定版本策略）。
- **子应用通信没有"接口定义语言 (IDL)"**：微服务之间有 protobuf/GraphQL/OpenAPI 定义通信格式，但微前端的 `initGlobalState` 只是一个无类型的键值对，相当于微服务直接用 JSON over TCP。缺少 Schema-first 的通信层。

### 系统级预防策略

**静态检查**：
```bash
# 子应用构建产物审计脚本
# scripts/audit-subapps.sh
echo "=== 子应用独立依赖检查 ==="
for app in subapp-*/; do
  echo "--- $app ---"
  # 检查是否打包了应共享的依赖
  grep -E "vue|react|antd|element" "$app/dist/assets/*.js" && echo "WARNING: $app 独立打包了共享依赖"
done

echo "=== 全局样式检查 ==="
grep -rn "^\*\s*{" */src/styles/ --include="*.css" --include="*.scss" && echo "WARNING: 全局通配符样式"

# 检查挂载到 body 的第三方组件
grep -rn "document\.body" */src/ --include="*.ts" --include="*.vue"
```

**CI 门禁**：
```bash
# E2E: 子应用间路由切换测试
test('所有子应用可独立加载和切换', async ({ page }) => {
  const apps = ['app1', 'app2', 'app3'];
  for (const app of apps) {
    await page.goto(`/${app}`);
    await expect(page.locator(`#${app}-root`)).toBeVisible();
  }
});

# E2E: 样式隔离
test('子应用样式不互相污染', async ({ page }) => {
  const app1BtnColor = await page.$eval('#app1 .primary-btn', el => getComputedStyle(el).color);
  await page.goto('/app2');
  const app2BtnColor = await page.$eval('#app2 .primary-btn', el => getComputedStyle(el).color);
  // 验证两个按钮样式独立
});

# 共享依赖版本检查
- name: Check shared dep versions
  run: |
    for app in subapp-*/; do
      vue_ver=$(node -e "console.log(require('./$app/package.json').dependencies.vue)")
      if [ "$vue_ver" != "$MAIN_VUE_VER" ]; then
        echo "VERSION MISMATCH: $app has vue@$vue_ver"
      fi
    done
```

**架构约束**：
- CSS 隔离策略必须在架构文档中明确，并以此选型微前端框架（qiankun/Module Federation 的选择受 CSS 隔离需求影响）
- 共享依赖版本由主应用的 `package.json` 锁定，所有子应用使用 `file:` 或 `workspace:` 协议引用
- 子应用通信通过类型安全的共享事件库（而非 `initGlobalState`），共享库发布到内部 npm registry
- 所有 UI 组件库（antd/element）必须全局配置 `getPopupContainer` 指向子应用根节点

**流程约束**：
- 新增子应用前必须通过架构评审："是否需要共享 UI 组件库？是否需要与已有子应用通信？样式隔离策略？"
- 任何一方升级共享依赖时，必须同步更新所有子应用的 `package.json` 并在 CI 中验证

### 跨层影响链

```
上游影响 (用户):
  - 样式错乱 → 用户对品牌信任度下降
  - 子应用加载失败 → 整个页面不可用（微前端的"雪崩效应"）
  - 版本冲突 → 某些功能神秘失效

下游影响 (团队):
  - 子应用自主要升级依赖 → 破坏主应用构建 → 团队间信任崩塌
  - 样式隔离需要子应用配合改造 → 迁移成本被推给子应用团队

部署影响:
  - 子应用独立部署需要处理版本兼容 → 主应用路由 404 处理旧 chunk
  - CDN 缓存策略：子应用的 index.html 和 static assets 缓存时长不同

监控影响:
  - Sentry 按子应用分组错误
  - 监控: 子应用加载时间、加载失败率
  - 自定义事件: 子应用间通信消息的延迟和大小分布
```

### 相关架构模式
- **BFF 与微前端的协调** (`patterns.md` 第十六节) — 微前端子应用各自对接自己的 BFF（或共享 BFF），需要定义 BFF 的边界和聚合策略。
- **Module Federation vs qiankun 选型** (`decisions.md` 第十四节) — 构建时集成 vs 运行时集成的决策矩阵。

## 预防
1. 所有子应用强制使用 CSS Modules 或 scoped styles，禁止全局样式
2. UI 组件库统一版本（由主应用锁定），通过 Module Federation shared 或 external 共享
3. 子应用间通信一律通过类型安全的共享事件库，禁用 `initGlobalState`
4. 新增子应用必须通过架构评审清单（样式隔离/依赖共享/通信方式/错误隔离）
