# pit-071: Workbox 缓存策略选错 — Cache First 用于 API 导致数据过期

## 环境
- PWA/SPA 应用使用 Workbox 管理 Service Worker 缓存
- API 响应被 Service Worker 缓存
- 用户看到过时的数据（库存、价格、消息列表）

## 症状
用户打开页面后看到的价格是昨天的，库存数量与实际不符，消息列表停留在上次打开时的状态。用户刷新页面（Ctrl+R）后数据更新了，但下次打开又回到旧数据。更隐蔽的是：部分 API 看似正常，但后台已经修改了数据（其他用户的操作），当前用户完全无感知。用户投诉"数据不准"但开发者本地禁用 SW 后一切正常。

典型场景：
- 电商商品详情页：显示"库存 0"（实际已补货），用户放弃购买
- 社交应用：消息通知显示旧数量（实际有新消息未读）
- 控制台 Dashboard：KPI 数据显示昨日数据（今天的数据已更新）

## 根因
**缓存策略与数据新鲜度要求不匹配**。核心机制：

1. **Cache First 用于动态 API**：Workbox 的 `CacheFirst` 策略对 API 请求返回缓存命中后**永不发起网络请求**，除非缓存中无此请求。如果 API 响应在 SW 安装时或首次访问时被缓存，后续所有请求都返回该陈旧数据。

2. **Stale-While-Revalidate 的"谎言"**：SWR 策略虽然会做后台更新，但**当前请求返回的是缓存**（陈旧数据），更新后的数据要在**下一次请求**才生效。对于需要即时展示正确数据的 API（如支付状态、验证码校验），SWR 的"一次陈旧"也是不可接受的。

3. **query param / header 变化导致的缓存分离**：带 `?timestamp=` 或 `Authorization` header 的请求可能生成不同的缓存条目，部分被缓存的响应可能来自已过期的 token。

4. **ExpirationPlugin 配置不当**：`maxAgeSeconds` 设置过长（如 7 天），远超过 API 数据的实际有效窗口。

## 修复

```diff
// ❌ 问题代码 1: API 使用 CacheFirst
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
-  new CacheFirst({
-    cacheName: 'api-cache',
-    plugins: [new ExpirationPlugin({ maxAgeSeconds: 86400 })]
-  })
);

// ✅ 修复 1: API 使用 NetworkFirst（最安全）
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
+  new NetworkFirst({
+    cacheName: 'api-cache',
+    networkTimeoutSeconds: 3,    // 3 秒后回退到缓存
+    plugins: [new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 300 })]
+  })
);

// ✅ 修复 2: 按 API 类型分层缓存
registerRoute(
  ({ url }) => url.pathname === '/api/product/inventory',
+  new NetworkOnly()  // 库存永远从网络获取
);
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/product/'),
+  new StaleWhileRevalidate({
+    cacheName: 'product-cache',
+    plugins: [new ExpirationPlugin({ maxAgeSeconds: 1800 })] // 30分钟
+  })
);
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/user/preferences'),
+  new NetworkFirst({
+    cacheName: 'user-preferences',
+    networkTimeoutSeconds: 5,
+    plugins: [new ExpirationPlugin({ maxAgeSeconds: 3600 })]
+  })
);
```

```diff
// ❌ 问题代码 2: 所有 GET 请求统一缓存
- registerRoute(
-   ({ request }) => request.method === 'GET',
-   new StaleWhileRevalidate({ cacheName: 'all-get' })
- );

    !url.pathname.startsWith('/api/verification/') &&

    cacheName: 'api-safe',
    plugins: [new ExpirationPlugin({ maxAgeSeconds: 600 })]
  })
);
```

## 原理
Workbox 的 `CacheFirst` 策略实现：`caches.match(request)` → 命中则返回 → 未命中则 `fetch(request)` 并 `cache.put()`。对于带哈希的静态资源（资源内容不可变），此策略完美。但对于动态 API（资源内容随时可能变化），**不发起网络请求就无法感知数据变化**。`NetworkFirst` 恰好相反：先 `fetch`，成功则返回并更新缓存，失败则回退到缓存。`NetworkOnly` 是最严格的——不做任何缓存。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **API 新鲜度未建模**：系统中存在不同新鲜度要求的 API（强实时/弱实时/可缓存），但没有在架构层面区分它们，统一使用泛化的缓存策略。
- **离线策略缺失**：没有声明"哪些 API 可以离线工作"和"哪些 API 离线时应该优雅降级"。
- **缓存治理的"默认开放"反模式**：默认所有 GET 请求可缓存是危险的。正确做法是"默认不可缓存，逐个声明可缓存"。

### 系统级预防策略
- **API 新鲜度分级**：定义 `FRESH_ALWAYS`（支付/验证）/ `FRESH_WITHIN_30S`（库存）/ `FRESH_WITHIN_1H`（用户配置）/ `STALE_OK`（历史数据）四类，映射到 NetworkOnly / NetworkFirst(3s) / SWR(1800s) / CacheFirst。
- **缓存声明契约**：每个 API Route 显式声明 `cacheStrategy`（类似 GraphQL 的 `@cacheControl` 指令），拒绝未声明的请求进入 SW 缓存。
- **离线降级设计**：对标记为 `STALE_OK` 的 API 在加载失败时回退到 IndexedDB 中的语义化缓存，而不是 SW 的 URL-keyed 缓存。

### 跨层影响链
- 用户：看到假数据 → 做出错误决策（购买/出售）→ 投诉/赔偿
- 业务：库存超卖 / 价格错误 / 支付状态不一致
- 后端：因用户重复操作（以为操作失败）导致重复提交

## 预防

1. **默认使用 NetworkFirst**：对动态 API 不假设缓存优先。
2. **API 路由分层缓存策略**：按数据新鲜度要求划分缓存策略。
3. **Workbox ExpirationPlugin 严格配 TTL**：API 缓存的 `maxAgeSeconds` 不超过 10 分钟。
4. **Code Review 清单**：每个 `registerRoute` 必须注释说明选择该策略的理由和数据新鲜度要求。

## 回退

```bash
# 紧急：回退到 NetworkOnly 模式（彻底禁用 API 缓存）
# 部署新的 sw.js，将 API 路由改为 NetworkOnly
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
  new NetworkOnly()
);

# 并在 activate 中清除所有 API 缓存
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.delete('api-cache')
  );
});
```
