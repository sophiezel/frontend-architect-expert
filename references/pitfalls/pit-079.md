# pit-079: 低代码 Schema 迁移 — 平台升级导致历史页面渲染异常

## 环境
- 低代码平台版本升级 (Retool 3.x → 4.x / 阿里低代码引擎 1.x → 2.x)
- 大量历史页面 (100+) 由旧版本平台生成
- Schema/DSL 格式变更或组件 API 发生 breaking change
- 平台承诺"向后兼容"但实践中有边界情况

## 症状
平台升级后部分历史页面表现异常：
- 表单页面正常渲染但提交按钮无响应（事件处理器的 DSL 路径在升级中被重命名）
- 数据表格显示的列数与旧版不同（Schema 中 `columns` 数组格式从 `[{key, title}]` 变为 `[{field, headerName}]`）
- 图表组件的 `type: 'bar'` 在新版组件中被废弃，需要使用 `type: 'bar-chart'`，旧页面静默渲染为空白
- 页面布局整体偏移：旧版 Schema 使用 `margin: 12` 的隐式单位（px），新版要求显式 `margin: '12px'`，解析失败后回退为 0

更隐蔽的症状：
- 数据查询的 SQL / API endpoint 在新版平台中被序列化方式不同（旧版不带 quotes，新版带），但平台未提供自动迁移，需要手动逐个页面修复
- 条件渲染 (visibility condition) 的表达式引擎从 `{{ state.active }}` 迁移到 `{{ $context.active }}`，旧语法不被解析，所有条件渲染失效（元素永远不可见或永远可见）
- 自定义组件 (custom widget) 的 `props` 传递方式变了：旧版 `props={{ user }}` → 新版 `context={{ user }}`

当这种问题影响 100+ 页面时，人工逐个修复的时间成本可能是数周。

## 根因
**低代码平台的抽象程度越高，跨版本迁移的代价越大**——因为 Schema/DSL 不是标准化的，而是平台专属的"代码"。核心机制：

1. **Schema 格式不是公开标准**：React 有 JSX（标准化），低代码的 Schema 是平台私有的 JSON/YAML DSL，格式由平台维护者单方面定义。Breaking change 发生时，没有标准的迁移工具或社区支持。

2. **隐式默认值依赖**：旧版 Schema 中省略了 `componentVersion` 字段（默认为 v1），新版默认 v2。平台升级后所有省略该字段的页面自动使用 v2 组件渲染 → v2 的 API 与 v1 不兼容 → 静默功能异常。

3. **数据流绑定的表达式语言变更**：`{{ }}`、`${}`、`@{}` 等不同的绑定语法在版本间发生变化，且语言没有版本声明。

4. **"向后兼容"的承诺不完整**：平台通常只保证核心组件（Button/Input/Table）的兼容，但生态中的第三方组件和自定义组件不受保障。

5. **Schema 没有版本号**：最关键的缺失。Schema 文件中没有 `schemaVersion: "2.1"` 字段，平台无法判断应该使用哪个版本的解析器。

6. **测试覆盖为零**：低代码页面通常没有自动化测试（不像代码有单元测试）。平台升级后只能靠人工 QA 或用户反馈发现 bug。

```javascript
// ✅ 修复 2: 自动迁移脚本
// migrate/v1-to-v2.js
function migrateSchema(oldSchema) {
  if (oldSchema.schemaVersion === '2.1') return oldSchema;

  return {
    schemaVersion: '2.1',
    page: {
      title: oldSchema.page.title,
      layout: migrateLayout(oldSchema.page.layout),
    },
  };
}

function migrateLayout(oldLayout) {
  return {
    layoutType: oldLayout.type || 'flex',  // type → layoutType
    children: (oldLayout.children || []).map(migrateWidget),
  };
}

function migrateWidget(widget) {
  const migrated = {
    ...widget,
    widgetVersion: widget.widgetVersion || '2',
  };

  // chart type 迁移映射
  const TYPE_MAP = { bar: 'bar-chart', line: 'line-chart', pie: 'pie-chart' };
  if (widget.widget === 'chart' && widget.type) {
    migrated.chartType = TYPE_MAP[widget.type] || widget.type;
    delete migrated.type;
  }

  // 表达式语法迁移: {{ }} → {{ $ }}
  if (widget.config?.dataSource) {
    migrated.config = {
      ...widget.config,
      dataSource: widget.config.dataSource.replace(
        /\{\{\s*queries\./g, '{{ $queries.'
      ),
      width: typeof widget.config.width === 'number'
        ? `${widget.config.width}px`
        : widget.config.width,
    };
  }

  return migrated;
}

// CI 中运行: npx migrate-schemas --from=v1 --to=v2 --pages='./pages/*.json'
```

// ✅ 修复 3: Schema 版本声明（平台层面强制）
// 平台解析 Schema 时:
function parseSchema(raw) {
  const version = raw.schemaVersion || '1.0'; // fallback 到 v1
  const parser = PARSER_REGISTRY[version];
  if (!parser) {
    throw new Error(`Unknown schema version: ${version}. ` +
      `Supported: ${Object.keys(PARSER_REGISTRY).join(', ')}`);
  }
  return parser.parse(raw);
}

// 保留多版本解析器
const PARSER_REGISTRY = {
  '1.0': new SchemaParserV1(),
  '2.0': new SchemaParserV2(),
  '2.1': new SchemaParserV2(), // 小版本共享解析器
};

## 原理
低代码平台的 Schema/DSL 本质上是平台专属的"中间表示 (IR)"。与 Web 标准 (HTML/CSS/JS) 不同，这些 IR 没有标准化委员会、没有 RFC、没有浏览器引擎级别的向后兼容承诺。平台开发者修改 IR 格式的成本极低（改一个字段名即可），但修复下游影响（重写 100 个页面的 Schema）的成本极高。

这类似于数据库 Schema Migration 但更困难——因为低代码 Schema 不仅是数据结构，还包含了 UI 布局、交互逻辑、数据流绑定、样式定义，是**四种关注点的混合序列化**。

## 架构视角 (Architecture Perspective)

### 架构层面根因
- **Schema 作为"可执行代码"但无版本管理**：低代码 Schema 是图灵完备的（包含表达式、条件、循环），但没有像代码仓库一样的 git history 和版本管理。
- **平台与页面生命周期耦合**：平台升级 = 所有页面被强制迁移，违背了"独立部署/独立演进"原则。
- **缺少契约测试**：平台新版本与历史 Schema 之间没有自动化契约测试来确保兼容性。

### 系统级预防策略
- **Schema 必须声明版本号**：每一个 Schema 第一行是 `schemaVersion: "x.y"`。
- **多版本解析器共存**：平台保留所有历史版本的 Schema 解析器，不强制迁移。
- **自动迁移工具 + CI 集成**：每次平台升级附带 Schema 迁移 CLI，CI 中自动执行并检查。
- **契约测试**：对 100 个最常见的历史页面 Schema 做渲染截图回归测试。

### 跨层影响链
- 用户：平台升级后部分功能不可用 → 用户失去信任
- 维护团队：每次升级 = 数周的迁移工作 → 平台升级恐惧症
- 业务：关键仪表盘不可用 → 业务决策延迟

## 预防

1. **从 Day 1 起 Schema 带版本号**：`schemaVersion: "1.0"`。
2. **升级脚本与平台版本绑定发布**：平台发布新版本时附带 Schema 迁移工具。
3. **灰度迁移**：先迁移非关键页面，验证无误后再迁移关键页面。
4. **迁移的自动化测试**：对迁移后的页面做截图对比和功能冒烟测试。
