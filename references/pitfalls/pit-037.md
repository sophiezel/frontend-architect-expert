# pit-037: Vite HMR 边界与失效

## 环境
- Vite 3.x / 4.x / 5.x
- 任何使用 HMR（Hot Module Replacement）的开发场景
- 包含循环导入或非组件模块的项目

## 症状
修改了源码文件，浏览器没有自动刷新——需要手动 F5 才能看到变化。或者更糟：HMR 触发了但组件状态丢失（如输入框内容清空、弹窗关闭）。某些文件无论如何修改都不触发任何 HMR 事件。

典型场景：
- 修改了一个 composable/工具函数（非 .vue 文件），依赖它的 10 个组件都不更新
- 两个模块存在循环引用（A imports B, B imports A），修改其中任一个都不触发 HMR
- 修改了 `vite.config.ts` 中 `define` 的常量值，HMR 无效

## 根因
Vite HMR 的工作原理：当一个模块变更时，Vite 沿着 import 链向上查找，找到所有接受热更新的边界（HMR boundary），然后从这个边界开始重新执行模块代码。

HMR 失效的四个核心原因：

1. **非组件模块没有 HMR 接受者**：`.vue` 文件被 Vite 插件自动注入了 `import.meta.hot.accept()`，但 `.ts`/`.js` 文件默认没有。修改 composable 后，Vite 找不到 HMR boundary，执行 full reload。

2. **循环导入打断 HMR 链**：`A → B → A` 的循环依赖导致 Vite 无法构建完整的依赖图。修改 A 时，Vite 需要确定哪些模块受 A 影响——但 B 既依赖于 A 又导出给 A，形成"鸡生蛋"死锁，Vite 退化为 full reload。

3. **import.meta.glob 动态导入的模块**：Vite 可以追踪 `import.meta.glob` 的结果，但仅限匹配的文件。新增文件不在已有 glob 范围内时不会触发 HMR。

4. **模块副作用被清除**：`import.meta.hot.accept()` 等同于"接受自身更新"，但接受后旧的模块状态（闭包变量、定时器）会被清除。

## 诊断方法

```bash
# 1. 查看 HMR 日志
# 浏览器控制台 → 筛选 "[vite]"
# 观察是 "hmr update" 还是 "full reload"
# full reload 意味着 HMR 链条断裂

# 2. 开启 Vite HMR 调试日志
# vite.config.ts
export default defineConfig({
  customLogger: { /* ... */ }
});
# 或设置环境变量
DEBUG=vite:hmr vite dev

# 3. 检测循环依赖
npx dpdm src/**/*.ts --no-warning
# 或
npx madge --circular --extensions ts,vue src/
# madge 输出所有循环依赖的模块路径

# 4. 检查文件的 HMR 边界
# 对于非 .vue 文件，检查是否定义了 accept:
grep -rn "import\.meta\.hot" src/ --include="*.ts" --include="*.js"
```

## 修复

```diff
// ❌ 问题 1: composable 修改后不触发热更新
// src/composables/useCounter.ts
export function useCounter() {
  const count = ref(0);
  function increment() { count.value++; }
  return { count, increment };
}
// 修改 useCounter 后 → full reload，所有组件状态丢失

+// ✅ 在模块末尾添加 HMR accept
+if (import.meta.hot) {
+  import.meta.hot.accept((newModule) => {
+    // composable 通常不需要特殊处理
+    // accept 本身就能让 Vite 知道这是一个 HMR boundary
+  });
+}

// ❌ 问题 2: 循环导入导致 HMR chain 断裂
// A.ts imports { b } from './B'; B.ts imports { a } from './A'
-// 这种循环依赖既破坏 HMR，也破坏 tree-shaking

+// ✅ 消除循环依赖
+// 提取共享类型/常量到第三个文件 shared.ts
+// shared.ts
+export interface SharedType { /* ... */ }
+export const SHARED_CONSTANT = 42;
+// A.ts 和 B.ts 都从 shared.ts 导入，不再互相引用

+// 如果必须保留双向引用，在其中一个方向使用动态 import
+// A.ts: const { b } = await import('./B');

// ❌ 问题 3: 修改 vite.config.ts 的 define 常量
// HMR 对配置文件不生效 → 必须重启 dev server

+// ✅ 对需要热更新的常量，放在专门的 env 模块并接受 HMR
+// src/env.ts
+export const API_URL = import.meta.env.VITE_API_URL;
+if (import.meta.hot) {
+  import.meta.hot.accept(); // 成为 HMR boundary
+}

// ❌ 问题 4: HMR 导致副作用丢失（如 store 状态）
// store/counter.ts
const count = ref(0);
export function useCounterStore() { return { count }; }

+// ✅ 保留 store 状态跨越 HMR
+if (import.meta.hot) {
+  import.meta.hot.accept((newModule) => {
+    // 什么都不做 — Pinia 已经在内部处理了
+  });
+  import.meta.hot.dispose(() => {
+    // 清理副作用（如 event listener、timer）
+  });
+}
```

## 验证

```bash
# 1. 测试各类文件的 HMR 行为
# - 修改 .vue 文件 → 组件更新，状态保持
# - 修改 .ts composable → 使用该 composable 的组件更新
# - 修改纯工具函数 → 依赖它的组件更新
# - 修改 store → 组件更新且 store 状态保持

# 2. 循环依赖必须为 0
npx madge --circular src/
# 输出 "No circular dependencies found!" → 通过

# 3. 手动验证：打开页面，在输入框中输入内容
# 修改代码后确认输入框内容未丢失（状态保持成功）

# 4. 浏览器控制台监控：
# 一小时内不应出现 "Full reload"
# 预期 100% 的变更是 "HMR update"
```

## 原理
Vite 的 HMR 基于原生 ES Modules。每个模块在浏览器中作为独立的 `<script type="module">` 加载。当文件变更时，Vite dev server 通过 WebSocket 通知浏览器，浏览器 `import()` 新的模块版本，沿着依赖图向上查找 `import.meta.hot.accept()` 边界，然后热替换。

HMR boundary 的分发算法（简化）：
1. 找到变更的模块 M
2. 向上追溯 M 的导入者（importer），直到找到定义了 `hot.accept()` 的模块 B
3. 从 B 开始重新执行
4. 如果没有找到任何 B，执行 full reload

这意味着：如果变更图的最顶层没有任何 accept，整个页面就刷新。

## 架构视角 (Architecture Perspective)

### 架构层面根因
HMR 失效的架构根因是**模块图的健康度未被视为架构质量指标**：
- **循环依赖不仅是代码风格问题，更是 HMR 的杀手**：循环依赖破坏的不仅是静态分析（tree-shaking、类型推断），还直接破坏了 Vite 的 HMR 依赖图遍历算法。但多数项目把"消除所有循环依赖"视为 nice-to-have 而非架构门禁。
- **模块角色未分类**：代码库中的模块有不同的"HMR 角色"——组件（天然 HMR boundary）、composable（需要显式 accept）、纯函数工具（不需要状态保持，full reload 也可接受）、store（需要状态保持）。项目中没有按角色为模块配置对应的 HMR 策略，而是把所有非 `.vue` 文件都交给 Vite 默认的 full reload。
- **HMR 是"免费午餐"的幻觉**：Vite 的 `.vue` 文件自动 HMR 导致开发者以为所有文件都能自动热更新。这种便利掩盖了非 `.vue` 文件的 HMR 配置需求。

### 系统级预防策略

**静态检查**：
```bash
# 循环依赖检测加入 lint-staged
# .lintstagedrc.js
module.exports = {
  'src/**/*.{ts,vue}': [
    'madge --circular --extensions ts,vue src/',
    'eslint --fix',
  ]
};

# 非 .vue 文件 HMR accept 覆盖率检测
find src -name "*.ts" ! -name "*.d.ts" ! -path "*/__tests__/*" | while read f; do
  if ! grep -q "import\.meta\.hot" "$f"; then
    echo "No HMR: $f"
  fi
done
```

**CI 门禁**：
```bash
# CI 中必须通过循环依赖检测
- name: Check circular dependencies
  run: npx madge --circular --extensions ts,vue src/ --no-spinner
  # madge 发现循环依赖会 exit 1

# HMR E2E 测试（可选，昂贵但有价值）
- name: HMR E2E
  run: |
    vite dev &
    sleep 5
    # 修改一个 composable 文件
    echo "export const TEST = 1;" >> src/composables/useTest.ts
    # 等待 WebSocket 消息确认是 hmr:update 而非 full-reload
```

**架构约束**：
- 循环依赖数量必须为 0（作为架构质量硬性指标）
- 所有 `src/composables/` 下的文件必须包含 `import.meta.hot.accept()`
- 所有 `src/stores/` 下的文件必须包含 `import.meta.hot.accept()` 和 `import.meta.hot.dispose()`
- 禁止模块间的循环引用——通过 `madge` 在 pre-commit hook 中检查

**流程约束**：
- Code Review："此模块需要 HMR 吗？是否添加了 `import.meta.hot.accept()`？"
- 新人入职培训中包含"HMR boundary 概念与如何配置"

### 跨层影响链

```
上游影响 (开发效率):
  - Full reload 每次 2-3 秒 → 一天 100 次变更 = 浪费 5 分钟
  - 状态丢失需要手动恢复 → 调试表单、复杂 UI 状态时极其痛苦

下游影响 (架构质量):
  - 循环依赖过多 → 不仅 HMR 失效，tree-shaking 也失效
  - 循环依赖 → 单元测试困难（无法独立 mock 单一模块）

部署影响:
  - 无直接部署影响（HMR 只在 dev 模式）
  - 但循环依赖导致 bundle 体积增大（tree-shaking 失效）

监控影响:
  - Vite dev server 日志中 full-reload 频率
  - 浏览器控制台 [vite] 消息分类统计
```

### 相关架构模式
- **Module Boundary 模式** (`patterns.md` 第十三节) — 为每种模块类型定义角色和边界规则
- **Dependency Inversion** (`decisions.md` 第十节) — 通过接口/抽象打破循环依赖
- **pit-042: Monorepo** — monorepo 中的跨包循环依赖更难检测和修复

## 预防
1. pre-commit hook 中使用 madge/dpdm 检测循环依赖，发现即阻止提交
2. 项目模板中预先给 composables 和 stores 添加 HMR accept 模板代码
3. 在 ESLint 中自定义规则：非 `.vue` 但被多个文件导入的模块必须有 `import.meta.hot`
4. Monorepo 场景：每个包的 `src/` 目录独立运行循环依赖检测

## 回退
```bash
# 若 HMR 配置导致每次修改后多次触发更新（抖动）
# 检查是否有太多 accept 边界，移除不必要的 accept
git checkout -- src/

# 临时禁用 HMR（使用 polling full reload）
vite dev --force
```
