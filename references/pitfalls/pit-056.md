# pit-056: 分包加载失败 — 分包预下载超时/分包异步化组件加载失败

## 环境
- 微信小程序分包架构（基础库 1.7.3+ 支持分包，2.24.3+ 支持分包异步化）
- 配置了 `preloadRule` 分包预下载
- 使用了分包异步化 `require.async` 或组件级 `componentPlaceholder`
- 弱网环境（3G/4G 信号差 / 超时）

## 症状
用户从主包页面跳转到分包页面时，页面白屏 3~8 秒后显示"页面加载失败"或直接停留在空白。更隐蔽的情况：页面框架渲染了（能看到顶部导航），但内容区域是空白的——因为分包异步化组件加载失败了，占位组件也没有正确显示错误状态。

预下载配置看似有效（配置了 preloadRule），但分包在用户实际进入前并未下载完成，导致进入分包页面时仍需等待下载。有时预下载在 WiFi 下正常，但在 4G 下超时。日志中可见 `subpackage load fail: timeout` 或 `download fail` 但未被业务代码捕获。

## 根因
分包加载涉及网络下载 + 代码注入两个阶段，任一环节失败都导致页面不可用。核心问题有：

1. **预下载超时**：`preloadRule` 触发的预下载默认超时为 60 秒，弱网环境可能超时。且预下载是"尽力而为"——`wx.loadSubPackage` 没有重试机制，一次超时就永久失败，不会自动重试。

2. **分包异步化组件加载失败静默**：`componentPlaceholder` 只在加载中有占位，但加载失败后不会自动降级到错误状态——需要开发者手动监听 `onError` 事件。

3. **分包下载失败未缓存**：微信会缓存下载成功的分包（提升二次访问速度），但下载失败或超时不会自动重试，导致用户必须手动退出重进才能触发重下载。

4. **独立分包与预下载不兼容**：独立分包（`independent: true`）不支持预下载——它需要在用户首次访问时才下载，如果该页面超过 5 秒未完成下载，用户直接流失。

// 4. 诊断代码: 监控分包加载
// utils/subpackage-monitor.js
class SubpackageMonitor {
  constructor() {
    this.loadTimes = {}; // 记录每个分包的加载耗时
    this.errors = [];    // 记录加载失败
    this.preloadStatus = {};
  }

  // 监控分包加载
  trackLoad(name) {
    const start = Date.now();
    console.log(`[subpkg] Loading ${name} at ${start}`);

    return {
      done: () => {
        const duration = Date.now() - start;
        this.loadTimes[name] = duration;
        console.log(`[subpkg] ${name} loaded in ${duration}ms`);
        
        // 上报监控
        if (duration > 3000) {
          console.warn(`[subpkg] ⚠️ ${name} load too slow: ${duration}ms`);
        }
      },
      fail: (err) => {
        this.errors.push({ name, error: err, time: Date.now() });
        console.error(`[subpkg] ❌ ${name} load failed:`, err);
      }
    };
  }

  // 检测预下载是否生效
  checkPreload(packageName) {
    // 预下载的状态可以通过 app.json 中配置推断
    // 但微信没有直接 API 查询预下载状态
    const preloadTime = this.preloadStatus[packageName];
    if (preloadTime) {
      console.log(`[subpkg] ${packageName} preloaded at ${preloadTime}`);
      return true;
    }
    return false;
  }
}

const monitor = new SubpackageMonitor();

// 在分包页面的 onLoad 中使用
Page({
  onLoad() {
    const tracker = monitor.trackLoad('subpkg-user');
    // 如果 3 秒还没完成，说明加载慢或失败
    setTimeout(() => {
      if (!this._loaded) {
        console.warn('[subpkg] page not loaded after 3s — likely network issue');
      }
    }, 3000);
    this._tracker = tracker;
  },
  onReady() {
    this._loaded = true;
    this._tracker.done();
  }
});

// ✅ 修复: 分包加载重试 + 降级
// utils/subpackage-loader.js
class SubpackageLoader {
+ static MAX_RETRIES = 2;
+ static RETRY_DELAY = 2000;

  static async load(root, options = {}) {
+   let lastError;
+   
+   for (let attempt = 0; attempt <= this.MAX_RETRIES; attempt++) {
+     try {
+       await this._doLoad(root, options);
+       return; // 加载成功
+     } catch (err) {
+       lastError = err;
+       console.warn(`[subpkg] ${root} load attempt ${attempt + 1} failed:`, err);
+       
+       if (attempt < this.MAX_RETRIES) {
+         // 等待后重试
+         await new Promise(r => setTimeout(r, this.RETRY_DELAY));
+         console.log(`[subpkg] retrying ${root}...`);
+       }
+     }
+   }
+   
+   // 所有重试都失败 → 降级处理
+   this._handleFailure(root, lastError);
  }

+ static _doLoad(root, options) {
+   return new Promise((resolve, reject) => {
+     const timeout = setTimeout(() => {
+       reject(new Error(`Subpackage ${root} load timeout`));
+     }, options.timeout || 15000); // 15s 超时
+     
+     // 使用 wx.loadSubpackage（基础库 2.30.2+）
+     // 对于不支持该 API 的版本，依赖页面跳转加载
+     if (wx.loadSubpackage) {
+       const task = wx.loadSubpackage({
+         name: root,
+         success: () => {
+           clearTimeout(timeout);
+           resolve();
+         },
+         fail: (err) => {
+           clearTimeout(timeout);
+           reject(err);
+         }
+       });
+       // 监听进度
+       task.onProgressUpdate((res) => {
+         console.log(`[subpkg] ${root} progress: ${res.progress}%`);
+       });
+     } else {
+       // 降级: 通过页面跳转触发加载
+       resolve();
+     }
+   });
+ }

+ static _handleFailure(root, error) {
+   // 统一降级处理
+   wx.showToast({
+     title: '网络不佳，请重试',
+     icon: 'none',
+     duration: 3000
+   });
+   
+   // 导航到错误页面
+   wx.redirectTo({
+     url: '/pages/error/error?reason=subpackage_load_fail&pkg=' + root
+   });
+   
+   // 上报
+   wx.reportEvent('subpackage_load_fail', {
+     package: root,
+     error: error?.message || 'unknown',
+   });
+ }
}

## 原理
分包加载分为两个步骤：
1. **下载阶段**：从微信 CDN 下载分包文件（.wxapkg 格式）。受网络速度和分包大小影响。微信客户端会缓存已下载的分包。
2. **注入阶段**：将下载的代码注入到 JsCore 和 WebView。这个阶段快（100~500ms），但依赖下载完成。

分包的"加载"实际上是这两个步骤的管道。预下载（preloadRule）在触发页面的 onLoad 之后异步开始下载，但**不保证在用户跳转前完成**。如果用户在预下载完成前就跳转，页面仍需等待下载完成。

`componentPlaceholder` 的分包异步化组件加载失败后，微信不会自动重试——开发者的错误回调只有在组件 `definitionFilter` 中通过 `wx.onError` 全局捕获才能感知。这也是为什么很多分包异步化失败表现为"白屏"——组件加载失败没有任何 UI 反馈。

## 架构视角 (Architecture Perspective)

### 架构层面根因
分包加载失败的架构根因是**分包可用性被当作"环境假设"而非"运行时状态"**。在架构设计中，分包被假设为"一定能加载成功"——代码直接跳转、直接使用组件、没有加载中和失败态处理。这在弱网或高并发场景下暴露出两个缺陷：

1. **缺少分包加载状态机**：分包有三种状态（未加载、加载中、已加载、加载失败），但业务代码没有建模这些状态，导致加载失败时 UI 无反馈
2. **缺少全局加载协调器**：多个页面可能同时触发同一分包的加载，缺少去重和队列管理，可能导致重复下载或请求风暴

### 系统级预防策略

**架构防护**：
```js
// 全局分包加载协调器
class SubpackageCoordinator {
  constructor() {
    this._loading = new Map();   // root → Promise
    this._status = new Map();    // root → 'idle'|'loading'|'loaded'|'failed'
  }
  
  // 去重加载: 同一分包多个页面同时请求，只加载一次
  async load(root) {
    if (this._status.get(root) === 'loaded') return;
    if (this._loading.has(root)) return this._loading.get(root);
    
    const promise = this._doLoad(root);
    this._loading.set(root, promise);
    this._status.set(root, 'loading');
    
    try {
      await promise;
      this._status.set(root, 'loaded');
    } catch (e) {
      this._status.set(root, 'failed');
      throw e;
    } finally {
      this._loading.delete(root);
    }
  }
  
  // 预下载 + 降级
  preload(root, { network = 'wifi', timeout = 60000 } = {}) {
    // 检查网络类型
    wx.getNetworkType({
      success: (res) => {
        if (network === 'wifi' && res.networkType !== 'wifi') {
          console.log('[subpkg] skip preload on', res.networkType);
          return;
        }
        this._doLoad(root).catch(() => {}); // 预下载失败静默
      }
    });
  }
}

// 全局单例
const subpkgCoordinator = new SubpackageCoordinator();
```

### 跨层影响链

```
上游影响 (业务):
  - 分包页面白屏 → 用户以为功能不可用 → 跳出 → 转化率下降
  - 支付页面在分包中加载失败 → 订单流失 → 直接收入损失

下游影响:
  - 分包下载失败 → CDN 带宽浪费（重复下载未缓存）
  - 错误日志爆炸 → 监控告警疲劳

监控影响:
  - 核心指标: 分包加载成功率、p95 加载耗时
  - 告警: 分包加载成功率 < 99% → P1; < 95% → P0
```

### 相关架构模式
- **加载状态机模式** (`patterns.md`) — idle → loading → loaded → error 四态建模
- **全局加载协调器模式** — 去重 + 队列 + 降级
- **反模式: Happy Path 假设** — 假设网络总是好的、分包总是能加载成功的

## 预防
1. **分包加载必须有错误处理和降级 UI**：加载中 → 加载失败 → 重试机制
2. **关键路径分包不要设为独立分包**：独立分包无预下载，首次访问必须等待下载
3. **预下载配置使用 `"network": "all"`**（非 WiFi 也会预下载，考虑流量提示）
4. **建立全局分包加载协调器**：去重 + 重试 + 状态管理
5. **监控分包加载成功率和耗时**：低于阈值立即告警
6. **弱网测试作为 CI 或发版前必需步骤**
