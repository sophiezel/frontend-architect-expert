# pit-012: Tree Shaking 误删 CSS 副作用

## 环境
- Webpack + CSS Modules / Tailwind / 第三方组件库
- Webpack 4+ production mode 默认启用 tree shaking

## 症状
引入第三方 CSS 或全局样式后，构建产物的样式丢失、组件样式错乱。在生产构建中出现，开发模式正常。

## 根因
Tree shaking 依赖 ES module 的静态分析。如果 CSS 文件被标记为 `sideEffects: false`（`package.json`）或 Webpack 误判模块无副作用，导入的 CSS 文件会被整体移除。典型场景：
1. 项目的 `package.json` 设置了 `"sideEffects": false`，但没有排除 `.css` 文件。
2. 使用了 `import 'xxx.css'` 的副作用导入，被 tree shaking 移除。
3. CSS Modules 的 `:global` 样式被认为无副作用。

## 诊断方法

```bash
# 1. 对比 dev 和 prod 构建的 CSS
npm run build:dev  # 开发构建
npm run build:prod # 生产构建
diff <(ls dist/css/) <(ls dist-dev/css/)
# 如果 prod 缺少某些 CSS 文件 → 确认 tree shaking 误删

# 2. 检查 package.json 的 sideEffects
cat package.json | grep -A 5 "sideEffects"
# 如果是 "sideEffects": false（全量标记）→ 问题根源

# 3. 使用 Webpack stats 分析
npx webpack --json > stats.json
# 搜索 "unused" 或 "sideEffects"
# 检查哪些模块被标记为 unused

# 4. 临时禁用 tree shaking
# webpack.config.js:
# optimization: { usedExports: false, sideEffects: false }
# 重新构建 → 对比产物 → 如果样式恢复 → 确认是 tree shaking 导致

# 5. 检查 CSS 导入方式
grep -rn "import.*\.css" src/ --include="*.ts" --include="*.tsx" --include="*.js"
# 查找副作用导入: import 'xxx.css'（没有 from 的导入）
```

## 修复

```json
// package.json — ❌ 问题
{
  "sideEffects": false
}

// package.json — ✅ 修复
{
  "sideEffects": [
    "*.css",
    "*.less",
    "*.scss",
    "*.global.css"
  ]
}
```

```js
// webpack.config.js — module.rules
{
  test: /\.css$/,
  use: ['style-loader', 'css-loader'],
  sideEffects: true, // 显式告知 Webpack CSS 文件有副作用
}
```

```js
// ❌ 问题：副作用的全局样式导入可能被移除
import './reset.css';
import 'antd/dist/antd.css';

// ✅ 确认 package.json sideEffects 包含 *.css
// 或在导入时标记（但 package.json 方案更好）
import './reset.css';
```

## 验证

```bash
# 1. 构建产物 CSS 对比
npm run build
# 检查 dist/ 中 CSS 文件是否包含所有预期样式
# 尤其检查 reset.css / 第三方库的全局样式

# 2. 页面视觉检查
# 部署生产构建 → 浏览器打开
# 对比 dev 模式下的样式
# 所有组件的样式应一致

# 3. CSS 选择器数量对比
# 开发构建：
npx webpack --mode development | grep -c "\.css"
# 生产构建：
npx webpack --mode production | grep -c "\.css"
# 两者选择器数量应接近

# 4. Webpack stats 验证
npx webpack --json > stats.json
# 检查 sideEffects 相关警告
# 确认没有 CSS 模块被标记为 "unused"

# 5. CI 检测脚本
# 对比 dev/prod CSS 文件大小差异不应超过 20%
```

## 原理
Webpack 4+ 在 production mode 下默认启用 tree shaking。Tree shaking 通过 ES module 的静态结构分析模块的导入/导出关系，移除"未被使用"的导出。但 `import './style.css'` 这种没有导入绑定的"副作用导入"如果被标记为无副作用，也会被移除。

`sideEffects` 字段告诉 Webpack："即使这些文件的导出没有被直接引用，它们也可能有副作用，不要移除。"CSS 文件的副作用就是向 DOM 注入样式。

## 架构视角 (Architecture Perspective)

### 架构层面根因
CSS 被 Tree Shaking 误删的架构根因是**副作用声明策略错误**：
- **`sideEffects: false` 是全局否定，而非精确声明**：设置全局 `"sideEffects": false` 是"踢皮球"式架构决策——把识别副作用的负担从工具链转移到了开发者。正确的架构应该采取白名单策略：**默认所有文件有副作用，仅对已验证无副作用的模块标记 `false`**。如果必须使用全局优化，则应用白名单排除所有可能的副作用文件类型。
- **CSS 作为副作用的本质未被架构承认**：CSS 文件的导入（`import './style.css'`）的"副作用"是将样式注入 DOM——这是一种跨模块的副作用，与纯 JS 导入的本地副作用不同。但架构中没有区分"本地副作用"（polyfill.js、日志初始化）和"全局副作用"（CSS 注入、全局变量注册），两种副作用被同等对待。
- **缺少构建产物的健康检查机制**：构建流程只有"成功/失败"两种结果，没有"产物健康度"检查。CSS 文件大小、选择器数量、第三方样式是否包含等指标应作为构建后的自动化检查项，类似数据库迁移后的数据完整性检查。

### 系统级预防策略

**静态防护**：
```json
// package.json — 推荐的白名单策略
{
  "sideEffects": [
    "*.css",
    "*.scss",
    "*.less",
    "*.global.css",
    "**/polyfills/**",
    "**/init/**"
  ]
}
// 然后在特定模块的 package.json 中逐个标记无副作用：
// packages/utils/package.json
{ "sideEffects": false }
```

```js
// webpack.config.js — 为 CSS loader 显式标记
{
  test: /\.css$/,
  sideEffects: true, // 强制! 即使 package.json sideEffects 不包含也保留
  use: [
    MiniCssExtractPlugin.loader,
    'css-loader'
  ]
}

// 构建时验证: 检测是否误删了关键的第三方样式
// webpack 插件
class CSSIntegrityPlugin {
  apply(compiler) {
    compiler.hooks.afterEmit.tap('CSSIntegrityPlugin', (compilation) => {
      const cssFiles = Object.keys(compilation.assets).filter(f => f.endsWith('.css'));
      cssFiles.forEach(f => {
        const content = compilation.assets[f].source();
        // 验证 reset.css 不应为空
        if (f.includes('reset') && content.length < 50) {
          throw new Error(`CRITICAL: ${f} 疑似被 tree shaking 误删 (仅 ${content.length} 字节)`);
        }
      });
    });
  }
}
```

**CI 防护**：
```bash
# CI 构建后验证: CSS 完整性检查
npm run build
EXPECTED_SIZE=50000  # KB
ACTUAL_SIZE=$(du -k dist/css/*.css | awk '{sum+=$1} END{print sum}')
if [ "$ACTUAL_SIZE" -lt "$(echo "$EXPECTED_SIZE * 0.8" | bc)" ]; then
  echo "ERROR: CSS 产物过小 ($ACTUAL_SIZE KB)，可能存在 tree shaking 误删"
  exit 1
fi

# 检查关键选择器是否存在
for selector in '.reset' '.ant-btn' '.global-header'; do
  if ! grep -q "$selector" dist/css/*.css; then
    echo "WARNING: 关键选择器 '$selector' 未在产物中发现"
  fi
done

# 视觉回归测试
npx chromatic test --build-script-name=build:prod
```

**架构防护**：
- **副作用分类体系**：在 Monorepo 的根 `package.json` 中定义副作用分类：
```json
{
  "sideEffects": [
    "*.css",            // 类别: 样式注入
    "**/polyfills.js",  // 类别: 全局补丁
    "**/init.js",       // 类别: 初始化
    "**/register.js"    // 类别: 全局注册
  ]
  // 注意: 不包含 *.test.*、*.stories.*、**/__mocks__/**
}
```
- **构建产物完整性测试**：类似数据库迁移的验证步骤，构建后运行"产物健康检查"套件：CSS 选择器数量、JS chunk 数量、第三方库样式是否包含、i18n 翻译文件是否完整。
- **`sideEffects: false` 审批流程**：任何包要设置 `sideEffects: false`，必须经过架构评审，确认包内无任何副作用导入。

**流程防护**：
- Code Review 检查项："是否修改了 `package.json` 的 `sideEffects` 字段？新增的副作用文件类型（如 `.svg`、`.wasm`）是否已加入白名单？"
- 发布清单："发布前执行 `npm run build:prod && npm run check:build-integrity`"

### 跨层影响链

```
上游影响 (消费者):
  - reset.css 被删除 → 所有页面的基础样式失效 → 浏览器默认样式差异
  - 第三方组件库(antd/element-plus)样式丢失 → 组件显示为裸 HTML
  - 全局 CSS Variables(Design Token)丢失 → 整个设计系统崩溃

下游影响 (依赖):
  - CSS Modules 的 :global 样式丢失 → 第三方组件内部样式缺失
  - Tailwind 的 @layer base 被移除 → 基础层的 CSS reset 失效
  - CSS-in-JS 的 GlobalStyle 注入的字体/图标样式丢失

部署影响:
  - 生产环境样式丢失 → staging 正常（dev mode 不 tree shake）
  - 回滚只能回退构建产物，无法修复 sideEffects 配置错误
  - CDN 缓存了错误版本的 CSS → purge 后才能恢复

监控影响:
  - 页面视觉异常 → 用户可能不报 bug 直接离开
  - CSS 文件大小监控: 设置波动阈值，突变 >30% 时告警
  - 视觉回归测试: Chromatic/Percy 截图对比
  - RUM: CLS 突然增加（样式缺失导致布局变化）
```

### 相关架构模式
- **构建架构 — Webpack vs Vite vs Rolldown** (`decisions.md` 第四节) — Vite/Rolldown 的 tree shaking 基于 Rollup 机制，对 `sideEffects` 的处理更保守（默认不删除有副作用的导入），但仍然需要正确的 `sideEffects` 声明。
- **Monorepo 管理** (`patterns.md` 第五节) — Monorepo 中每个 package 都有独立的 `package.json`，`sideEffects` 声明需要逐包精确配置，Turborepo 可以并行验证所有包的 sideEffects 正确性。
- **CSS 方案选型** (`decisions.md` 第三节) — CSS-in-JS (运行时) 方案不受 tree shaking 影响（因为样式是 JS 执行注入的），但零运行时方案 (CSS Modules/Tailwind) 需要正确的 sideEffects 声明。

## 预防

1. **`sideEffects` 数组化**：不要用全局 `false`，用数组精确排除：
   ```json
   "sideEffects": ["*.css", "*.scss", "*.less", "**/polyfills.js"]
   ```
2. **Webpack 配置中为 CSS loader 显式设置 `sideEffects: true`**。
3. **组件库作者**：`package.json` 必须正确声明 `sideEffects`。
4. **CI 添加 CSS 缺失检测**：对比 dev/prod 构建的 CSS 选择器数量。

## 回退

```bash
# 如果 sideEffects 数组导致某些文件被保留但不应保留
# 方案A：缩小范围，只包含必要的 CSS 文件
# "sideEffects": ["src/**/*.css"]  // 只包含源码目录

# 方案B：使用更精确的 glob 排除
# "sideEffects": ["*.css", "!*.module.css"]  // 排除 CSS Modules

# 方案C：完全禁用 tree shaking（不推荐，仅临时）

# 完全回退
git checkout -- package.json webpack.config.js
```
