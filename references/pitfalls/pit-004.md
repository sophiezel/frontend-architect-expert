# pit-004: useRef vs useState 混用导致非预期行为

## 环境
- React 16.8+

## 症状
修改 `ref.current` 后组件没有重新渲染（UI 不更新），或误用 state 存储不需要触发渲染的值导致不必要的重渲染甚至死循环。

## 根因
`useState` 更新触发重新渲染；`useRef` 修改不触发渲染。开发者混淆两者的语义：
- 把需要响应式更新的数据放进 ref → 改了值但 UI 不更新。
- 把不需要 UI 更新的可变值放进 state → 引发不必要的渲染甚至死循环。

## 诊断方法

```bash
# 1. 打印诊断
# 在组件中添加：
console.log('[DEBUG] ref.current:', someRef.current);
console.log('[DEBUG] state:', someState);
# 如果 ref 值变了但 UI 没变 → 应改用 state
# 如果 state 值变了但不需要触发 UI 更新 → 应改用 ref

# 2. React DevTools Profiler 查看渲染次数
# Components → 选中组件 → 查看 "Rendered" 计数
# 如果计数远高于预期 → 可能有不必要的 setState

# 3. 检查 JSX 中使用的是 ref 还是 state
grep -rn "ref\.current" src/ --include="*.tsx" --include="*.jsx" | grep "return ("
# 如果 ref.current 出现在 JSX 返回中 → 红旗信号，应改用 state
```

## 修复

```diff
// ❌ 问题：用 state 存 interval ID，不需要触发渲染
function Timer() {
-  const [intervalId, setIntervalId] = useState(null);
+  const intervalRef = useRef(null);

  const start = () => {
-    const id = setInterval(() => tick(), 1000);
-    setIntervalId(id); // 不必要的重渲染
+    intervalRef.current = setInterval(() => tick(), 1000);
  };

  const stop = () => {
-    clearInterval(intervalId);
+    clearInterval(intervalRef.current);
  };
}

// ❌ 问题：用 ref 存需要在 JSX 中显示的值
function Counter() {
-  const countRef = useRef(0);
+  const [count, setCount] = useState(0);

  const increment = () => {
-    countRef.current += 1;
-    console.log(countRef.current); // 10, 但 UI 显示 0
+    setCount(c => c + 1);
  };

-  return <div>{countRef.current}</div>; // 永远不会更新
+  return <div>{count}</div>;
}
```

## 验证

```bash
# 1. Timer 验证
# 点击 start → 观察 React DevTools 是否触发重渲染
# 修复前：每次 start/stop 都会渲染（不必要的）
# 修复后：start/stop 不触发渲染（定时器 tick 使用独立 state）

# 2. Counter 验证
# 点击 increment 多次
# 修复前：控制台打印的值递增，但 UI 始终显示 0
# 修复后：UI 跟随点击递增

# 3. 渲染计数验证
# 在组件内添加渲染日志:
# const renderCount = useRef(0);
# renderCount.current++;
# console.log('render count:', renderCount.current);
# 执行各种操作后确认渲染次数是否符合预期

# 4. 决策检查表
# 值在 JSX 中渲染？ → useState ✓ / useRef ✗
# 值在 useEffect 依赖中？ → useState ✓ / useRef ✗  
# 值仅在回调/定时器中用？ → useRef ✓ / useState ✗
```

## 原理
React 的渲染机制依赖 state 和 props 的变化。`useState` 的 setter 调用会触发调度更新（scheduleUpdateOnFiber），而 `useRef` 直接修改 `.current` 属性不会通知 React。`useRef` 本质上是一个 `{ current: ... }` 的普通 JavaScript 对象，React 保证它在整个组件生命周期中引用不变。

## 架构视角 (Architecture Perspective)

### 架构层面根因
useRef/useState 混用的架构根因是**状态分类体系缺失**：
- **状态未被建模为不同的"类别"**：所有组件内的可变值被当作同类处理，没有按"是否驱动渲染""是否需要历史追溯""是否参与协调"等维度分类。正确的架构应该首先对状态进行分类，然后为每个类别选择正确的存储原语。
- **缺少"可变值契约"**：每个可变值应该有一个明确的契约声明——"修改此值会触发渲染"或"修改此值仅用于副作用/回调"。当开发者在 state 和 ref 之间选择时，实际上是在定义这个契约，但架构层面没有提供框架来显式声明它。
- **渲染优化策略缺失**：不必要的 state 导致过度渲染，本质上是性能优化策略没有系统化。架构应对渲染频率设定预算和约束。

### 系统级预防策略

**静态防护**：
```js
// ESLint: 检测 ref.current 出现在 JSX 中（红旗信号）
'no-restricted-syntax': ['warn', {
  selector: 'JSXExpressionContainer MemberExpression[property.name="current"]',
  message: 'ref.current 出现在 JSX 中，请确认是否需要改用 useState'
}]

// TypeScript: 使用条件类型辅助分类
type RenderState<T> = {
  value: T;
  set: (v: T | ((prev: T) => T)) => void;
  // 标记此值会触发渲染
  __brand: 'renderState';
};
type SideEffectValue<T> = {
  current: T;
  // 标记此值不触发渲染
  __brand: 'sideEffectValue';
};
```

**CI 防护**：

**架构防护**：
- **建立状态分类编码规范**：在团队编码规范中明确定义四类可变值及其存储方案：
  ```
  类别A: UI 驱动值(chart data, input value, toggle) → useState
  类别B: 副作用引用(timer ID, WS handle, request ID) → useRef
  类别C: 派生值(从 A 计算) → useMemo / 直接计算
  类别D: 服务端状态(API data) → React Query / SWR
  ```
- **组件编写模板强制执行**：通过脚手架或代码生成工具强制组件按状态类别组织代码区域。
- **引入状态管理库解决跨组件共享的 ref 问题**：如 Zustand 允许在 React 外部访问状态，天然分离了"存储"和"渲染触发"。

**流程防护**：
- Code Review 检查项："每个 useRef 是否回答了'为什么它不需要触发渲染'？"
- 架构评审：新增状态变量时问三个问题：谁修改它？谁读取它？修改后谁需要感知？

### 跨层影响链

```
上游影响 (消费者):
  - 如果本应渲染的值在 ref 中，父组件传入的 props 变化不会反映到 UI
  - 兄弟组件通过共享状态库读取到的值可能是过时的

下游影响 (依赖):
  - useEffect 依赖了 ref.current → 永远不会重新执行（ref 变化不触发 effect）
  - useCallback 依赖了 ref.current → 回调永远缓存旧值
  - 第三方组件接收 ref 作为 prop → 值变化时组件不更新

部署影响:
  - 线上 hotfix 修改渲染逻辑时，可能因 ref/state 分类错误引入新 bug
  - Sentry 错误日志中 state 值与 UI 显示值不一致 → 增加排错时间

监控影响:
  - 需要监控: React DevTools Profiler 中渲染次数异常（过多或过少）
  - 烟雾测试: 检查关键交互是否触发预期的渲染次数
  - 自定义断言: useEffect 清理函数是否被正确触发
```

### 相关架构模式
- **状态管理模式决策框架** (`patterns.md` 第一节) — "单个组件内 → useState / ref / reactive" 决策分支，架构师需要进一步定义何时选 ref 何时选 state。
- **决策树: 状态更新频率** (`decisions.md` 第二节) — "高频 (>1次/秒: 实时输入/动画) → Jotai / ref + forceUpdate" 明确指出了高频更新场景下 ref 的正确用法。
- **反模式: 巨型组件** (`patterns.md` 第六节) — 组件内同时存在过多 state 和 ref 是巨型组件的信号，应拆分为更小的单元，每个单元只管理自己职责范围内的状态类别。

## 预防

1. **决策树**：
   - 值需要在 JSX 中渲染？ → `useState`
   - 值仅用于事件/Effect/定时器且不需要触发渲染？ → `useRef`
   - 值需要驱动 useEffect？ → `useState`
2. **Code Review**：遇到 `ref.current` 出现在 JSX 中即为红旗信号。
3. **理解 React 渲染机制**：只有 state 和 props 变化才触发渲染。

