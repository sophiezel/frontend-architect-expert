# pit-029: 流式 SSR 中断 / 部分渲染

## 环境
- Next.js 13+ App Router (Streaming SSR)
- React 18+ (Suspense + Server Components)
- `renderToPipeableStream` (自定义 Node.js 服务器)
- Vercel / AWS Lambda / 自建 Node.js 服务器

## 症状
1. **Streaming SSR 中途断开**：页面加载到一半停止，用户看到不完整的 UI（Header 渲染了，但内容区域是空白或 Suspense fallback 一直转圈）。
2. **React 水合报错 "did not match"**：服务端渲染的 HTML 与客户端水合不一致，片段化渲染导致组件嵌套层级错位。
3. **Suspense boundary 缺失**：某组件的数据源超时（如数据库慢查询），拖慢整个页面的首个字节时间（TTFB），而非仅为该组件显示 fallback。
4. **流式响应中断后浏览器放弃重试**：浏览器收到不完整的 HTML 流，未触发客户端重渲染降级，页面永远不完整。
5. **`onError` 未被处理**：`renderToPipeableStream` 中数据源异常时，错误未被捕获 → 流被 shell abort → 浏览器收到空响应或半截 HTML。

## 根因
`renderToPipeableStream` 未处理 `onError`、关键路径数据未包裹 Suspense boundary、数据源超时无降级逻辑。

核心机制：React 18 的 Streaming SSR 依赖 HTTP 持久连接传输 HTML chunk。如果：
1. 某个 Server Component 的 `fetch` 抛出异常且未在 `onError` 中处理 → 流被销毁
2. 关键路径（如 Layout）包裹了 Suspense 边界但 data source 超时 → 整个 shell 被阻塞
3. `renderToPipeableStream` 的 `bootstrapScripts` 缺失 → 流结束后客户端未启动水合

```ts
// ✅ renderToPipeableStream: 正确的 onError 处理
// server/render.ts
import { renderToPipeableStream } from 'react-dom/server';

function renderApp(req, res) {
  const { pipe } = renderToPipeableStream(<App />, {
+    onShellReady() {
+      res.setHeader('Content-Type', 'text/html');
+      pipe(res);
+    },
+    onShellError(err) {
+      res.statusCode = 500;
+      res.send('<!DOCTYPE html><html><body>' +
+        '<h1>Server Error</h1><p>Please try again later.</p>' +
+        '</body></html>');
+    },
+    onError(err, errorInfo) {
+      console.error('Streaming error:', err);
+      // 记录到 Sentry/日志系统
+      // 不 abort 流！React 会自动降级该 Suspense boundary 到客户端渲染
+    },
+    bootstrapScripts: ['/assets/client.js'], // 确保客户端水合
  });
}
```

// ✅ 数据抓取组件: 超时 + 降级到客户端渲染

## 原理
React 18 的 `renderToPipeableStream` 将渲染分为两个阶段：
1. **Shell 阶段**：立即渲染所有非 Suspense 组件（Layout、Header、加载指示器），流式发送 HTML。
2. **Data 阶段**：异步 resolve 的 Suspense boundary 依次完成，每个完成后将对应 HTML chunk + inline `<script>` 替换 fallback `<!--$?-->` 占位符。

`onError` 决定错误处理策略：若在 `onError` 中调用 `res.end()` → 流终止 → 浏览器收到半截 HTML（无 `</html>` ）→ 永不完整。正确做法是在 `onError` 中记录日志但不中断流，React 将该 Suspense boundary 降级为客户端渲染。

## 架构视角 (Architecture Perspective)

### 架构层面根因
流式 SSR 中断的架构根因是**流式渲染容错设计缺失**：

- **Streaming 被当作"免费的性能优化"而非"弹性渲染模式"**：团队启用 Streaming 后假设所有组件都能快速 resolve，没有为"某个组件 slow/fail"设计降级路径。Streaming 需要部分降级（Partial Degradation）、渐进增强（Progressive Enhancement）、客户端补渲染（Client Fallback）三层容错。
- **Suspense boundary 没有按组件可用性分层**：理想的分层是：Layout（必须有，shell 的一部分）→ Above-fold content（优先级高，可容忍短暂 fallback）→ Below-fold content（可接受长时间 fallback）。但实际代码中 Suspense 随机放置，关键组件未包裹，非关键组件反而阻塞了 shell。
- **数据源超时/错误与渲染流的交互未协议化**：Server Component 的 `fetch` 超时后，React 默认会永久等待（无超时机制）。`AbortSignal.timeout` 需要显式传入，但多数项目未配置。

### 系统级预防策略

**静态防护**：

```

**CI 防护**：

test('slow API does not block page shell', async ({ page }) => {

# Lighthouse CI: TTFB < 800ms assertion
npx lhci autorun --collect.url=https://staging.example.com/slow-page
```

**架构防护**：
- **定义 Streaming Resilience 三层模式**：
```
Layer 1: 部分降级 (Partial Degradation)
  - 数据源超时 → 降级到 default value → Suspense resolve
Layer 2: 渐进增强 (Progressive Enhancement)  
  - 数据源超时 → 服务端留空 → 客户端 hydrate 后 fetch
Layer 3: 客户端补渲染 (Client Fallback)
  - 服务端渲染失败 → <Suspense> fallback → 客户端重试
```

- **按组件可用性分层放置 Suspense**：
```tsx
// Suspense 分层策略
<Shell>  {/* 无 async — 始终立即渲染 */}
  <Suspense fallback={<HeaderSkeleton />}>
    <AsyncHeader />  {/* Above-fold — 中等 fallback */}
  </Suspense>
  <Suspense fallback={<ContentSkeleton />}>
    <AsyncProductList />  {/* Below-fold — 可接受长时间 fallback */}
  </Suspense>
</Shell>
```

- **数据源超时协议**：所有 Server Component 的 `fetch` 必须设置 `AbortSignal.timeout()`，超时值按优先级分级：P0（2s）/ P1（5s）/ P2（15s）。

**流程防护**：
- Code Review: "此 Server Component 的 fetch 是否包裹在 `<Suspense>` 中？超时后降级路径是什么？"
- 发布清单: "是否在 staging 环境模拟了慢查询/数据库不可用场景验证 Streaming 降级行为？"

### 跨层影响链

```
上游影响 (消费者):
  - 用户看到"半成品"页面 → 以为网站挂了 → 刷新 → 更多请求 → 雪崩
  - 搜索引擎爬虫收到不完整 HTML → SEO 排名下降
  - 辅助技术（读屏器）遇到片段化内容 → 导航困难

下游影响 (依赖):
  - 数据库慢查询阻塞 Server Component → 连锁阻塞其他 React render
  - 第三方 API 超时 → Suspense fallback 无限旋转 → 该位置永远不填充
  - CDN/Edge Function 的超时（Vercel 10s / AWS Lambda 30s）早于数据源超时 → 用户收到 504

部署影响:
  - Node.js 服务端内存：被阻塞的 render 持有 request 引用直到超时
  - HTTP 连接池：长时间 streaming 占用连接
  - Vercel Edge Function 有 30s 硬限制

监控影响:
  - APM: Streaming 中断次数 / Suspense boundary resolve 时间
  - 自定义事件: onError 调用次数和错误类型
  - RUM: 页面完整渲染时间（最后一个 Suspense resolve）
  - 日志: Suspense fallback 展示时长分布
```

## 预防
1. 所有 Server Component 的 `fetch` 必须设置 `AbortSignal.timeout()` 超时。
2. Layout 中的 async 数据获取必须包裹 `<Suspense>`，避免阻塞 shell。
3. `renderToPipeableStream` 的 `onError` 记录日志但不中断流，React 自动降级。
4. 按内容优先级放置 Suspense boundary：Shell（必须）→ Above-fold（优先）→ Below-fold（可延迟）。
5. 在 staging 环境运行慢查询/断网混沌测试验证降级完整性。
