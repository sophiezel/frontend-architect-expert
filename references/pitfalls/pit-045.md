# pit-045: BFF 层设计反模式

## 环境
- Node.js BFF (Backend For Frontend) 层
- 微服务架构的下游聚合场景
- Express / Koa / Fastify / Hono 等 HTTP 框架

## 症状
BFF 层的代码就是"把 A 服务的响应 + B 服务的响应拼接起来返回给前端"——没有任何业务逻辑，纯粹是 HTTP 请求转发代理。更糟糕的是：一个页面一个 BFF 接口——`/bff/home-page`、`/bff/product-detail`，BFF 接口数量与前端页面一一对应，代码量膨胀到后端微服务之外的第二个后端。

最隐蔽的问题是：BFF 调用下游服务 C 时没有超时和重试设置，C 服务慢响应（5s）导致 BFF 层堆积大量等待中的请求，最终 Node.js 事件循环被拖垮。下游服务任何一个出错，BFF 就直接把错误透传给前端——无容错、无降级、无兜底。

## 根因
BFF 层的设计初衷是"为特定前端裁剪的后端"，解决的是"一个通用 API 无法适配多个前端（Web/iOS/Android）"的问题。但实践中 BFF 常常退化为四种反模式：

1. **转发代理（Passthrough Proxy）**：BFF 不做任何数据转换，只是 `fetch(backendUrl) → res.json()`。这层除了增加网络跳转，没有任何价值。

2. **过度聚合（Over-aggregation）**：一个页面一个 BFF 接口，把所有数据聚合在一个端点。看起来减少了前端请求数，但实际上：单个下游服务慢响应拖慢整个页面（最慢的决定所有）、缓存粒度太粗（一个字段变了整个接口缓存失效）。

3. **无容错/无降级**：下游服务出错 → BFF 透传错误 → 前端白屏。没有"部分数据降级返回"、"过期缓存兜底"、"重试策略"。

4. **BFF 变成业务逻辑层**：业务逻辑开始往 BFF 中渗透（判断用户权限、计算折扣、校验库存），BFF 从"数据聚合层"变成了"第二个业务服务层"。

## 诊断方法

```bash
# 1. 检查 BFF 接口是否有数据转换逻辑
grep -rn "fetch\|axios\.\(get\|post\)" src/ --include="*.ts" -A 3 | 
  grep "res\.json\|ctx\.body" | head -20
# 如果是 fetch 后直接透传 → Passthrough Proxy 反模式

# 2. 检查 BFF 接口数量与页面数量的关系
find src/routes -name "*.ts" | wc -l  # BFF 路由数
find ../web/src/pages -type d | wc -l # 前端页面数
# 如果 BFF 路由数 ≈ 前端页面数 → Over-aggregation

# 3. 检查是否有超时/重试/降级策略
grep -rn "timeout\|retry\|circuit.*breaker\|fallback\|degrad" src/ --include="*.ts"
# 如果搜索结果很少 → 无容错

# 4. 检查是否有业务逻辑
grep -rn "if.*user\.\|if.*permission\|if.*role\|price\s*[*/+\\-]\|discount\|inventory" src/ --include="*.ts"
```

```diff
+// ✅ 多层容错策略
+async function fetchWithFallback<T>(
+  primaryUrl: string,
+  fallbackData: T,
+  options: { timeout: number; retries?: number }
+): Promise<T> {
+  for (let attempt = 0; attempt <= (options.retries || 1); attempt++) {
+    try {
+      const controller = new AbortController();
+      const timer = setTimeout(() => controller.abort(), options.timeout);
+      const res = await fetch(primaryUrl, { signal: controller.signal });
+      clearTimeout(timer);
+      if (!res.ok) throw new Error(`HTTP ${res.status}`);
+      return await res.json();
+    } catch (err) {
+      if (attempt === options.retries) {
+        console.warn(`Service ${primaryUrl} failed after retries, using fallback`);
+        return fallbackData; // 降级到兜底数据
+      }
+    }
+  }
+  return fallbackData;
+}

+// 使用
+const recommendations = await fetchWithFallback(
+  'rec-service/personalized',
+  { items: DEFAULT_RECOMMENDATIONS }, // 静态降级数据
+  { timeout: 1500, retries: 1 }
+);

// ❌ 反模式 4: 业务逻辑渗透到 BFF
-// BFF 中判断 user.vipLevel 决定是否显示"VIP 专享价"
-if (user.vipLevel === 'gold') {
-  product.price = product.vipPrice;
-}

+// ✅ 业务逻辑留在业务服务中
+// BFF 只透传 user 的上下文
+const res = await fetch('product-service/pricing', {
+  headers: { 'X-User-Vip-Level': user.vipLevel },
+  body: JSON.stringify({ productId, rawPrice: product.price }),
+});
+// 定价逻辑由 product-service 负责，BFF 不做判断
```

## 原理
BFF 的正确姿势是"为前端体验定制的 API Gateway"，而非"请求转发代理"。核心职责：
1. **字段裁剪**：微服务返回 50 个字段，但移动端只需要 5 个 → BFF 裁剪
2. **数据聚合**：把一个页面的数据从 N 个服务聚合为 1 个响应 → 减少前端 HTTP 请求数
3. **协议转换**：gRPC → REST/GraphQL；SOAP → JSON
4. **容错降级**：下游服务故障时返回兜底数据或部分结果

过度聚合的危险在于"水桶效应"——一个请求聚合了 N 个下游服务，只要 N 越来越大，可用性会收敛为零。如果每个下游服务可用性 = 99.9%，聚合 10 个服务的可用性 = 99.9%^10 = 99.0%（即每分钟有 1% 的概率返回错误）。

## 架构视角 (Architecture Perspective)

### 架构层面根因
BFF 反模式的架构根因是**BFF 层的职责边界未在架构中定义**：
- **BFF 被当作"前端的后厨"而非"API 体验的编排器"**：当团队抱怨"微服务接口太乱"时，BFF 被当作解决方案引入——"再包一层就统一了"。但这一层如果没有明确的职责边界，就会"吸附"越来越多的逻辑，最终变成无法维护的"大泥球"。
- **BFF 的粒度被前端路由驱动而非业务域驱动**：BFF 接口按页面组织（`/bff/home`、`/bff/detail`、`/bff/profile`）看起来直观，但页面变化频繁（改版、AB test）导致 BFF 随之变动。正确的粒度应按业务聚合根组织（`/bff/user-session`、`/bff/order-context`），接口组合由前端 GraphQL 或 BFF 的组合层完成。
- **容错策略是事后追加的**：BFF 在初期"先跑通"时没有容错设计（因为下游服务很稳定），当系统规模增长、下游服务变多变慢时，容错变成紧急补丁——但此时已经几十个接口需要逐一改造。

### 系统级预防策略

**静态检查**：
```bash
# BFF 透传检测脚本
# scripts/audit-bff.sh
echo "=== 直传检测 ==="
grep -rn "res\.json\|ctx\.body" src/routes/ --include="*.ts" | while read line; do
  file=$(echo "$line" | cut -d: -f1)
  # 检查同一路由中是否有数据转换逻辑
  if ! grep -q "map\|filter\|select\|pick\|transform\|omit" "$file"; then
    echo "WARNING: $file 可能为透传代理"
  fi
done

echo "=== 容错策略检测 ==="
grep -rn "timeout\|catch\|fallback\|circuit\|AbortController" src/ --include="*.ts" -l | wc -l
```

**架构约束**：
- BFF 接口按业务聚合根组织（而非前端页面）：
  ```
  推荐: /bff/user-context, /bff/order-summary, /bff/product-detail
  反模式: /bff/home-page, /bff/checkout-page, /bff/profile-page
  ```
- 每个 BFF 接口必须定义：
  - 下游服务超时时间（per-service）
  - 降级策略（fallback data / partial response / stale cache）
  - 重试策略（是否幂等 → 重试次数）
- BFF 不做业务逻辑判断——所有条件分支（if/permission/role/pricing）必须留在业务服务中
- BFF 代码必须纯粹（pure aggregation + transformation），复杂度用 ESLint 圈复杂度限制（max: 10）

**流程约束**：
- 新增 BFF 接口必须在 PR 中说明"聚合了哪几个下游服务、各自的 SLA 和降级策略"
- 季度审计：统计 BFF 接口中透传（零转换）的比例，超过 10% 触发重构

### 跨层影响链

```
上游影响 (前端):
  - BFF 过度聚合 → 前端看到的数据量膨胀（未裁剪的 50 个字段）
  - BFF 无容错 → 前端错误处理变复杂（需要处理 BFF 的各种下游错误）
  - BFF 变成转发代理 → 前端请求延迟多一次网络往返

下游影响 (微服务):
  - BFF 超时设置太短 → 频繁超时重试 → 下游服务压力增加
  - BFF 不做数据聚合 → 每个前端请求打 N 个微服务 → 微服务 QPS 膨胀

部署影响:
  - BFF 与前端版本耦合 → 必须同步部署（或需要版本兼容策略）
  - BFF 的降级数据需要随 BFF 一起部署（静态 JSON 文件）

监控影响:
  - BFF 层引入的延迟（从 BFF 到下游服务的 P50/P95）
  - BFF 降级触发频率（按下游服务和降级类型分组）
  - BFF 自身的错误率（与下游服务错误率对比，监控 BFF 本身引入的错误）
```

### 相关架构模式
- **API Gateway 模式** (`patterns.md` 第十九节) — BFF 是 API Gateway 的一个特化实例，共享网关的通用职责（限流、认证、路由、聚合）。
- **GraphQL 作为 BFF** (`decisions.md` 第十八节) — 当 BFF 的接口组合复杂度超过阈值时，考虑用 GraphQL 的 resolve chain 替代硬编码的 Promise.all 聚合。

## 预防
1. BFF 接口按聚合根组织（`/bff/user-context`），而非按页面组织（`/bff/home-page`）
2. 每个 BFF 接口必须硬编码：超时时间 + 降级策略 + 重试次数（写入代码注释）
3. BFF 只做数据聚合和字段裁剪，禁止包含业务逻辑
4. 使用 `Promise.allSettled` 而非 `Promise.all`（部分失败不影响整体）
5. 为 BFF 添加响应缓存层（如 Redis），避免重复聚合
